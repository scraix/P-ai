# 消息仓库状态机拆分计划

日期：2026-05-27

## 背景

当前消息仓库 manifest 使用 `messageStoreKind + migrationState` 表达会话消息存储状态，其中 `migrationState=building` 同时承担了“正在迁移”“正在写入快照”“当前快照不可读”“可能是崩溃残留”等多种语义。

这个颗粒度过粗，导致在远程联系人会话流式回复期间打开独立聊天窗口时，迁移预检可能把正在写入中的 `building` manifest 误判为异常会话，并弹出“抛弃异常会话并继续迁移”确认。

## 目标

- 区分“迁移状态”“写入事务状态”“快照可读状态”，避免把正常活跃写入误判为异常迁移。
- 保证写入期间旧的 `ready` 快照仍然可读，避免正式 manifest 长时间暴露 `building`。
- 让迁移版本记录只表达 schema/布局迁移是否完成，不被活跃写入状态阻塞。
- 为崩溃残留提供明确恢复路径，而不是直接要求用户抛弃会话。

## 非目标

- 不改变现有 JSONL block 文件格式。
- 不引入数据库。
- 不重写会话流式持久化链路。
- 不改变用户可见的会话列表和归档功能入口。

## 建议状态模型

### 数据格式状态

- `conversationJson`
- `jsonlSnapshot`
- `jsonlEventLog`

### 快照状态

- `ready`：当前 active snapshot 可读。
- `unavailable`：没有可用快照，只能回退 legacy JSON 或进入恢复。

### 写入事务状态

- `idle`
- `writing`
- `committing`
- `failed`

建议写入事务放入独立字段，例如：

```json
{
  "activeStoreKind": "jsonlSnapshot",
  "activeSnapshotState": "ready",
  "writeTransaction": {
    "state": "writing",
    "transactionId": "...",
    "startedAt": "...",
    "updatedAt": "..."
  },
  "migration": {
    "version": 1,
    "state": "done"
  }
}
```

## 实施步骤

1. 梳理现有 manifest 读写点
   - `message_store/manifest.rs`
   - `message_store/persist.rs`
   - `message_store/migration.rs`
   - `message_store/store.rs`
   - 启动迁移 gate：`message_store_migration.rs`

2. 引入兼容结构
   - 保留旧字段读取能力。
   - 新增可选 `writeTransaction` 与 `migration` 字段。
   - 旧 manifest 读入后规范化为新语义。

3. 改造写入为 staging/atomic commit
   - 写入期间不把 active manifest 改成不可读。
   - 新快照写入临时位置或事务标记。
   - 校验通过后一次性切换 active snapshot。

4. 调整读取逻辑
   - 优先读取 active ready snapshot。
   - 遇到活跃 writing 事务时继续读取旧 ready。
   - 无 ready 且 legacy JSON 存在时回退 legacy。

5. 调整迁移预检
   - 迁移预检只判断“是否有旧 schema/旧布局待迁移”。
   - 活跃写入事务不计入 blocked。
   - 超时事务进入 recoverable，而不是直接要求用户抛弃。

6. 增加恢复逻辑
   - `writing` 超过阈值且无进展：尝试校验 staging。
   - staging 可用：提交为 ready。
   - staging 不可用但旧 ready 存在：清理事务并继续使用旧 ready。
   - 两者都不可用：进入 blocked，并给出明确原因。

## 验证计划

- 单测覆盖：
  - ready snapshot 写入期间旧 ready 仍可读。
  - 活跃 writing 不触发迁移 blocked。
  - 超时 writing 可恢复到旧 ready。
  - 旧 manifest 兼容读取。
  - 崩溃残留事务恢复。

- 手工验证：
  - 远程联系人会话流式中打开独立窗口，不出现迁移弹窗。
  - 主窗口启动不反复跑迁移预检。
  - 完成/打断流式后消息可在主窗口、独立窗口、归档窗口读取。

## 风险

- manifest 兼容处理不完整可能影响旧用户数据读取。
- staging 提交流程若未覆盖所有写入路径，可能出现新旧状态混用。
- 恢复策略过激会误清理仍在写入的事务，需要依赖更新时间和进程内锁配合。

## 当前临时修复

本计划执行前，先保留当前止血策略：

- 独立聊天窗口不运行全局消息仓库迁移 gate。
- 最近更新过的 `building` manifest 视为 busy，不作为 blocked 异常。
- busy 不阻止记录迁移版本，避免每次窗口启动反复预检。
