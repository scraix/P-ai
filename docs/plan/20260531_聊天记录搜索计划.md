# 聊天记录搜索计划

## 目标

在记忆页面新增一个 BM25-only 的聊天记录搜索入口，用于观察和手动检索已压缩/归档的对话内容。首版只做实验性搜索，不接入主动 RAG，不使用向量检索，也不影响 AI 当前回答。

## 范围

- 只索引已压缩或已归档的对话材料。
- 本地会话、归档会话、联系人会话纳入范围。
- 委托会话不纳入范围。
- 按人格控制可见性：只有某个人格作为 LLM 发言人出现在压缩段内，该人格才能搜到该段产生的切片。
- 搜索暂时只使用关键字 BM25，分词沿用记忆检索的 CJK 1/2-gram 方案。

## 分段模型

消息仓库 JSONL 已经按压缩边界分 block。首版按以下规则生成可索引段：

- 已归档会话：所有 ready JSONL block 都可作为索引段。
- 未归档会话：只索引非 latest 的 ready JSONL block，因为 latest block 可能仍是实时会话内容。
- 没有 ready JSONL block 的会话：回退使用会话快照里的压缩消息作为段来源。

## 切片文本

消息渲染格式为：

```text
[昵称]: 正文
```

切片规则：

- 短消息按完整消息边界拼接，目标长度约 256 字符。
- 普通短消息之间不做重叠。
- 单条渲染后消息超过 256 字符时，才在该消息内部切分，并保留 32 字符重叠。
- 发言人既写入文本，也保留在 metadata 中。

## 切片元数据

每个切片保留：

- `sourceKind`：`localConversation`、`archive` 或 `contact`
- `sourceId`
- `sourceTitle`
- `segmentId`
- `sliceIndex`
- `content`
- `speakers`
- `visibleAgentIds`
- `timeStart`
- `timeEnd`
- `messageStartId`
- `messageEndId`

`visibleAgentIds` 来自该段内所有非空的 `speakerAgentId`。如果段内没有 LLM 人格发言，则跳过该段。

## 搜索流程

记忆页面调用后端命令，传入：

- `agentId`
- `query`
- `limit`

后端流程：

1. 检查 app 数据目录下的持久化 Tantivy 索引。
2. 只有 metadata signature 过期或索引缺失时，才重新从聊天存储生成切片并重建索引。
3. Tantivy 索引落盘，切片 metadata 写在索引目录旁的 `metadata.json` 中。
4. 在 Tantivy 查询层加入 `visibleAgentIds` 人格过滤。
5. 使用 BM25 返回排序结果和统计信息。

首次搜索或聊天存储变化后的第一次搜索可能触发重建；正常重复搜索复用磁盘索引和进程内 reader 缓存。

## 观测信息

搜索响应返回：

- 总切片数
- 当前人格可见切片数
- 已索引会话数
- 跳过的委托会话数
- 跳过的实时 latest block 数
- 跳过的无人格段数
- 本地/归档/联系人来源切片数
- Tantivy 索引目录占用体积
- 当前进程内切片缓存估算体积

## 后续工作

- 如果要接入主动 RAG，再考虑持久化 `chat_recall_slice` 关系表。
- 将重建时机从搜索时兜底迁移到压缩/归档事件触发。
- 观察搜索质量后，再决定是否接入主动 RAG。
- 只有当人格可见性和切片粒度稳定后，再评估是否增加向量检索。
