# 会话 metadata 字段级服务计划

## 需求

- 会话服务应持有会话 metadata 的权威内存缓存。
- 会话 metadata 的每个可变字段都应通过会话服务提供的字段级方法修改。
- 字段级方法必须表达明确业务意图，例如：
  - 设置会话指定模型。
  - 设置计划模式。
  - 设置标题。
  - 设置未读数。
  - 设置工作区信息。
  - 设置会话运行相关 metadata。
- 修改 metadata 时，应先更新内存缓存，再由会话服务负责安排落盘。
- metadata 落盘应从当前内存缓存读取最新值。
- 多次连续修改同一会话 metadata 时，应以最新内存状态为准。
- 会话消息 block 已经按原子化方式管理，metadata 也应具备同等级别的字段级原子修改能力。
- 普通业务调用方不应直接读出整个 `Conversation`，修改一个 metadata 字段后再写回整个对象。
- 会话服务应统一管理 metadata 变更产生的会话概览刷新、内存派生索引更新和持久化调度。
- 会话级指定模型应成为字段级 metadata 服务的首个明确使用场景。

## 界限

- metadata 字段级服务只管理会话头信息，不管理消息 block 内容。
- 消息追加、截断、替换仍由现有消息仓库/block 机制负责。
- 字段级 metadata 修改不得修改部门配置、全局模型配置或全局选中模型。
- 字段级 metadata 修改不得依赖前端传入完整会话对象。
- 字段级 metadata 修改不得通过旧会话快照覆盖其他 metadata 字段。
- 整会话持久化路径如果仍然存在，写入 metadata 时必须合并会话服务中的最新 metadata 内存状态。
- 任何后台持久化 worker 不得持有过期 metadata 快照并直接覆盖 `meta.json`。
- 会话服务以外的模块不得直接写 `meta.json`。
- 前端只表达用户意图，不参与 metadata 并发一致性处理。
- `SessionSelector.apiConfigId` 只表示内部单次强制模型调用，不属于会话 metadata 字段。
- `preferredApiConfigId` 是会话 metadata 字段，绑定单个会话。
- `conversationCallPrimaryApiConfigId` 是派生展示值，不是持久化字段。
