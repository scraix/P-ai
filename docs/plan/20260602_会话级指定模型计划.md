# 会话级指定模型计划

## 背景

聊天窗口的模型下拉框此前混用了多个概念：部门模型队列、前端显示模型、发送会话里的 `apiConfigId`、独立窗口临时模型，以及 VS Code sidebar 的模型选择。用户实际需求是给每个会话增加一个独立的“会话指定模型”，用于决定该会话下一次调度的第一候选模型，而不是修改部门或全局配置。

## 需求

- 每个会话可以保存一个可选的指定模型：`preferredApiConfigId`。
- 未指定模型时，会话调度队列为部门模型队列，例如 `A -> B -> C`，下拉框显示部门第一模型 `A`。
- 指定模型 `D` 后，会话调度队列为 `D -> A -> B -> C`，自动去重。
- 指定模型 `C` 后，会话调度队列为 `C -> A -> B`，自动去重。
- 指定模型绑定会话，不修改部门模型队列，不修改全局选中模型。
- 流式中允许切换模型，但只影响下一次调度，并提示“模型已切换，将在下一次调度开始时生效”。
- 主聊天窗口、独立聊天窗口、VS Code sidebar 都应遵循同一语义。

## 命名边界

- `preferredApiConfigId`：会话显式指定模型，是持久化字段。
- `conversationCallPrimaryApiConfigId`：会话 call 第一模型，是展示/派生值，来源为 `preferredApiConfigId ?? 部门模型队列第一项`。
- `SessionSelector.apiConfigId`：只保留给 delegate/internal retry 等内部单次强制模型调用；普通聊天发送不依赖它表达用户下拉框选择。

## 当前实现

- 后端 `Conversation` 增加 `preferred_api_config_id`，并写入会话 `meta.json`。
- 切换会话模型时，后端更新当前会话内存缓存、会话概览派生缓存与 `meta.json`。
- 后端调度候选模型时：
  - 内部单次强制模型优先，用于 delegate retry 等场景。
  - 普通会话使用 `preferred_api_config_id + 部门模型队列`。
  - 如果已保存的会话指定模型已被删除、禁用或不再是聊天文本模型，则跳过该指定模型并回退部门队列。
- 前端模型下拉框只表达“当前会话指定模型”，不会修改部门队列。
- 前端发送前会等待当前会话正在进行的模型持久化完成，避免用户切换后立即发送时使用旧模型。
- 快速连续切换模型时，旧请求失败不会回滚较新的选择。

## 后续收敛方向

当前实现仍依赖 `Conversation` 大对象承载 metadata。更理想的长期方向是拆出字段级会话 metadata 内存缓存：

- 每个 metadata 字段有独立 mutation 方法，例如 `set_conversation_preferred_api_config_id`。
- 所有调用方只修改 metadata 内存缓存。
- metadata writer 从当前内存缓存读取最新值写入 `meta.json`。
- 整会话消息持久化写入 `meta.json` 时必须合并最新 metadata 内存值，避免旧快照覆盖字段级更新。
