# 20260311 私有部门与私有人格加载方案计划

## 1. 目标
- 不新增高权限“部门/人格管理工具”。
- 改为让 LLM 在自己的私有工作区内，通过 JSON 文件声明私有部门与私有人格。
- 由现有 `reload` 流程统一扫描、校验、加载这些私有对象。
- 保证用户仍然拥有最高控制权，而私有目录中的对象统一归属于“助理私域”，不绑定具体人格。

## 2. 核心结论
- 这不是“运行时配置工具”方案。
- 这是“Skill 指导 + 私有目录 + JSON 声明 + reload 生效”方案。
- Skill 负责教 LLM 如何组织自己的团队。
- JSON 负责承载结构化定义。
- `reload` 负责发现、校验、加载、报错。
- 私有组织资产属于“助理侧”，而不是某个具体人格。

## 3. 为什么不做工具
- 工具 schema 会很重，容易污染上下文。
- 人格、部门、委任关系天然带权限边界，做成工具后运行时校验复杂度很高。
- 一旦工具允许直接修改主配置，调试和回滚都很困难。
- 文件声明更透明，用户也可以直接查看、修改、备份。

## 4. 总体方案

### 4.1 交互方式
- 提供一个预设 Skill，指导 LLM：
  - 如何设计私有部门
  - 如何设计私有人格
  - 如何写 JSON
  - 写完后调用 `reload`
- LLM 不直接改主配置。
- LLM 只在自己的私有目录中新增或修改 JSON 文件。

### 4.2 生效方式
- `reload` 扫描私有目录。
- 对 JSON 做校验、归一化、去重与错误收集。
- 合法对象作为“外部来源对象”注入运行时。
- 非法对象不生效，并输出明确错误。

## 5. 目录设计

### 5.1 推荐目录
- `llm-workspace/private-organization/personas/`
- `llm-workspace/private-organization/departments/`

### 5.2 设计原因
- 人格与部门统一属于“私有组织”概念。
- 后续若增加私有团队规则、模板、说明文档，可以继续放在同一层级。
- 比单独拆成 `private-personas` / `private-departments` 更容易扩展。

### 5.3 文件约定
- 每个 JSON 文件只声明一个对象。
- 文件名不参与业务身份判定，只作为存储载体。
- 真正的身份以 JSON 内部 `id` 为准。

## 6. 私有人格 JSON 方案

### 6.1 基础字段
- `id`
- `name`
- `prompt`
- `avatar_path` 或后续等价字段

### 6.2 约束
- 不允许声明内建人格 ID。
- 不允许覆盖用户主配置中的同 ID 人格。
- 私有人格不提供私有记忆开关，工具人默认不使用私有记忆。

### 6.3 语义
- 私有人格不是系统默认人格。
- 私有人格是助理私域组织中的附属资产。
- 用户可以看见并最终决定是否保留，运行时来源由系统自动标记。

## 7. 私有部门 JSON 方案

### 7.1 基础字段
- `id`
- `name`
- `summary`
- `guide`
- `agent_ids`
- `api_config_id`（可选）

### 7.2 约束
- 不允许声明内建助理部门 ID。
- 不允许覆盖用户主配置中的同 ID 部门。
- `agent_ids` 中若引用私有人格，必须能在本轮加载结果中解析到。
- 若未声明 `api_config_id`，默认继承主助理部门当前模型。
- 若声明了 `api_config_id`，它必须指向可用于文本对话的模型配置。

### 7.3 语义
- 私有部门属于助理私域组织结构。
- 它不等于主系统部门。
- 它只在通过校验后被临时挂载到运行时。

## 8. 所有权与权限边界

### 8.1 用户权限
- 用户始终可以查看和修改全部内容。
- 用户可以直接删除私有目录中的 JSON。
- 用户可以决定是否保留某个私有对象。

### 8.2 LLM 权限
- LLM 只能维护助理私域目录中的对象。
- LLM 不能直接修改主配置中的内建人格、内建部门、用户人格、用户部门。
- LLM 不能通过 JSON 声明接管已有对象。

### 8.3 必须禁止
- 覆盖内建助理部门
- 覆盖系统人格
- 覆盖用户人格
- 覆盖用户创建的部门
- 通过 JSON 伪造来源或归属

## 9. reload 的加载流程

### 9.1 扫描
- 扫描私有人格目录
- 扫描私有部门目录
- 忽略非 JSON 文件

### 9.2 解析
- 逐个 JSON 解析
- 收集解析错误
- 不因单个文件失败而中断全部加载

### 9.3 校验
- 校验字段完整性
- 校验 ID 合法性与保留 ID 冲突
- 校验部门引用的人格是否存在
- 校验模型配置是否可用

### 9.4 注入
- 合法对象进入运行时临时集合
- 运行时将这些对象视为助理私域附加资产
- 运行时展示时可明确标记来源
- 主配置文件本体不被这些对象反向改写

### 9.5 错误输出
- `reload` 返回：
  - 成功加载数量
  - 跳过数量
  - 错误数量
  - 每个错误文件的简要原因

## 10. 运行时合并策略

### 10.1 主配置优先
- 主配置中的对象优先级最高。
- 私有对象不能覆盖主配置同 ID 对象。

### 10.2 私有对象增量注入
- 私有人格、私有部门只作为附加来源出现。
- 运行时需要能区分：
  - 主配置对象
  - 私有工作区对象

### 10.3 来源可视化
- 前端后续可考虑展示：
  - 来源：主配置 / 私有工作区
  - 作用域：助理私域
- 但这不是第一阶段必须实现项。

## 11. Skill 设计

### 11.1 推荐名称
- `private-organization-guide`

### 11.2 Skill 职责
- 教 LLM 如何设计私有团队
- 教 LLM 如何编写私有人格 JSON
- 教 LLM 如何编写私有部门 JSON
- 教 LLM 写完后调用 `reload`

### 11.3 Skill 不负责
- 不直接改主配置
- 不直接获取超级权限
- 不直接绕过 reload 校验

## 12. 第一阶段最小落地范围
- 落目录约定
- 落 JSON schema
- 在 `reload` 中增加扫描与加载
- 加载结果注入运行时
- 增加预设 Skill：`private-organization-guide`

## 13. 暂不做
- 不做专用高权限工具
- 不做自动审批
- 不做复杂 RBAC
- 不做私有对象的前端全量管理面板
- 不做跨 owner 的自动接管

## 14. 风险点

### 14.1 JSON 漂移
- Skill 文案、JSON schema、加载器三者必须一致。

### 14.2 owner 伪造
- 不能让 JSON 自行声明归属。
- 私有对象的归属必须由目录位置与加载流程自动决定。

### 14.3 引用悬挂
- 私有部门引用了不存在的人格时，必须整条报错而不是半生效。

### 14.4 reload 可见性
- 若错误信息太弱，LLM 会反复写错 JSON 却不知道原因。

## 15. 待评审问题
- 私有目录是否需要再按子类做更细分的命名约定。
- 私有对象是否允许在前端直接编辑。
- 私有对象是否允许被用户一键“转正”为主配置对象。
- reload 返回的错误格式应该多详细。

## 16. 归档条件
- 私有目录扫描已落地。
- 私有人格 / 私有部门 JSON 已能被 `reload` 加载。
- 运行时能区分主配置对象与私有对象。
- `private-organization-guide` 已加入预设 Skill。
