# 远程IM联系人管理计划

## 解决什么问题

接入多个 IM 渠道后，消息来源变得杂乱。QQ 群里几十个人、飞书群里一堆同事、钉钉私聊若干——这些消息全部涌进同一个聊天窗口，用户根本分不清谁是谁，也没办法控制助理该回谁、不回谁。

现在的状态是：渠道配好了，消息能进来，助理能回复，但用户对"谁在跟我说话"没有掌控感。

## 核心思路

**不要让用户提前配置联系人，让消息自己带人进来。**

每个 IM 平台的每条消息都携带发送者信息（用户ID、昵称、头像、群ID、群名）。第一次收到某个人/某个群的消息时，自动在系统里注册一条联系人记录。用户打开联系人列表，看到的就是"所有跟我说过话的人和群"，然后逐个决定怎么对待他们。

## 联系人的生命周期

### 1. 发现

消息从 IM 进入系统时，检查这条消息的来源（渠道 + 平台侧 ID）是否已有对应联系人：

- **已有**：更新昵称（IM 平台上昵称会变）、更新最后消息时间、标记有新消息
- **没有**：创建新联系人，回复策略默认为**不回复**——新来的人默认不理睬，用户主动开启才会回复

用户不需要做任何操作，联系人自然积累。

### 2. 管理

用户在设置页看到联系人列表，可以做的事情：

- **设置回复策略**：不回复 / 始终回复 / 有消息才回复
- **查看基本信息**：平台、昵称、类型（私聊还是群聊）、最后消息时间
- **删除联系人**：不想要的联系人可以删掉，下次对方再发消息会重新创建

### 3. 参与出站判断

助理回复完成后，遍历所有联系人，根据每个人的回复策略决定要不要转发。这部分已经实现了。

## 联系人需要记录什么

每个联系人本质上是"某个渠道下的某个对话方"，需要记录：

**身份信息（来自 IM 平台，自动填充）：**
- 所属渠道
- 平台类型（飞书 / 钉钉 / NapCat）
- 对方的平台 ID（QQ 号、飞书 open_id、钉钉 staff_id）
- 对方的昵称（群名或用户昵称）
- 联系人类型（私聊 / 群聊）

**用户设定（手动管理）：**
- 回复策略
- 备注名（可选，用户自己起的名字，方便辨认）

**运行时状态（系统维护）：**
- 是否有未回复的新消息
- 最后收到消息的时间
- 最后转发回复的时间

## 联系人列表的展示

### 列表应该怎么组织

按渠道分组展示，每个渠道下列出该渠道的所有联系人：

```
飞书 · 工作机器人
  ├── 产品讨论群          群聊 · 始终回复 · 最近: 5分钟前
  ├── 张三                私聊 · 有消息才回复 · 最近: 1小时前
  └── 李四                私聊 · 不回复 · 最近: 昨天

NapCat · QQ机器人
  ├── 技术交流群          群聊 · 有消息才回复 · 最近: 10分钟前
  ├── 王五                私聊 · 始终回复 · 最近: 3小时前
  └── 智能家居上报        群聊 · 不回复 · 最近: 刚刚
```

### 列表需要传达的信息

每个联系人条目一眼能看到：
- 这是谁（昵称 + 备注名）
- 来自哪个平台
- 是群还是私聊
- 当前的回复策略
- 最后活跃时间
- 是否有未回复的新消息（醒目标记）

### 快捷操作

- 单击回复策略标签可以快速切换（不回复 → 有消息才回复 → 始终回复 → 不回复，循环切换）
- 右键或展开菜单可以删除、设备注名

## 与聊天窗口的关系

聊天窗口里远程消息的展示现在已经有来源信息（通过 provider_meta.origin）。联系人管理不改变聊天窗口的展示方式，它只影响：

- 出站判断（回不回复这个人）
- 设置页的管理界面

## 群聊中的联系人粒度

一个 QQ 群是一个联系人，不是群里每个人各算一个联系人。

原因：回复策略是"要不要往这个群里转发"，不是"要不要回复群里的某个人"。群里张三说了话和李四说了话，助理的回复都是发到同一个群里，粒度到群就够了。

群内具体是谁在说话，靠消息本身的发送者信息区分，不需要联系人层面去管。

## 期望的完整场景

### 场景一：首次接入

1. 用户配好了 QQ 机器人渠道，连接成功
2. 联系人列表是空的
3. QQ 群里有人发了一条消息
4. 联系人列表自动出现了这个群，回复策略默认是"不回复"
5. 消息进入了系统历史，助理不会回复，也不会往群里转发任何东西
6. 用户看到这个联系人后，觉得需要互动，手动切换成"有消息才回复"或"始终回复"

### 场景二：管理联系人

1. 用了几天，联系人列表积累了 8 个群和 3 个私聊
2. 用户发现"摸鱼闲聊群"不需要助理回复，点一下切换成"不回复"
3. 用户发现"老板"的私聊需要始终转发，切换成"始终回复"
4. 其他人保持默认的"有消息才回复"

### 场景三：智能家居设备

1. 一个 QQ 群专门用来接收智能家居上报的温度、湿度数据
2. 消息进来后联系人自动创建
3. 用户把它设成"不回复"——数据进入系统给助理当上下文用，但不需要往群里发回复
4. 用户可以给它加个备注名叫"智能家居上报"，方便辨认

### 场景四：联系人昵称变更

1. QQ 群改了群名
2. 下次收到该群消息时，联系人记录的昵称自动更新
3. 用户不需要手动改任何东西
