# 远程IM联系人页面 UI 方案

## 当前问题

打开"联系人"设置页，看到的第一屏几乎全是渠道配置：名称、平台下拉框、默认策略、6 个能力复选框、一大块 JSON 凭证文本框。联系人列表被挤到最下面，只有一行"暂无远程联系人记录"加一个刷新按钮。

这和实际使用场景完全相反——用户日常操作的是联系人（谁该回、谁不该回），渠道配置只在首次接入时设一次。

## 设计原则

1. **联系人是主角，渠道是配角**——页面打开后，第一眼看到的应该是联系人列表
2. **渠道配置收起来**——参考 MCP 页面的 `collapse` 折叠面板，凭证和能力开关默认折叠
3. **一眼看清状态**——每个联系人一行，核心信息（谁、什么类型、什么策略、最后活跃）一眼扫到
4. **操作要快**——切换回复策略一次点击完成，不需要弹窗

## 页面整体布局

页面分两个区域，从上到下：

### 区域一：渠道配置（紧凑，可折叠）

```
┌──────────────────────────────────────────────────────┐
│  联系人渠道                          [新增渠道] [保存] │
│  在这里维护 NapCat / 飞书 / 钉钉 渠道配置             │
├──────────────────────────────────────────────────────┤
│  ▶ #1 QQ机器人  ·  NapCat  ·  已启用                  │
│  ▶ #2 工作飞书  ·  飞书    ·  已启用                   │
└──────────────────────────────────────────────────────┘
```

每个渠道默认显示为**一行摘要**（序号 + 名称 + 平台 + 启用状态），点击展开才看到完整配置。

展开后的渠道配置内部结构：

```
▼ #1 QQ机器人  ·  NapCat  ·  已启用                [删除]
┌─────────────────────────────────────────────────────┐
│  渠道名称: [QQ机器人    ]  平台: [NapCat ▼]          │
│  启用: [✓]                                          │
│                                                      │
│  ▶ 能力与行为（折叠）                                 │
│  ▶ 凭证 JSON（折叠）                                  │
└─────────────────────────────────────────────────────┘
```

**"能力与行为"** 折叠面板包含：
- 激活助理、接收文件、允许发送文件、允许主动发送、流式发送、发送工具调用

**"凭证 JSON"** 折叠面板包含：
- JSON 文本框

这样，没展开时一个渠道只占一行。两三个渠道加在一起也就三四行高度，不会抢联系人的空间。

## 联系人条目的信息结构

每个联系人条目分为**一行主信息**和**一行附属信息**：

**主信息行（左 → 右）：**
- 新消息指示器：有未回复新消息时显示红色圆点（DaisyUI badge-error），无则不显示
- 昵称/备注名：如果有备注名显示备注名，否则显示平台昵称。字体 `font-medium`
- 类型标签：`badge badge-sm badge-ghost` 显示"群聊"或"私聊"
- 回复策略按钮：`btn btn-xs`，显示当前策略文字，**点击循环切换**
- 最后活跃时间：`opacity-70 text-xs`，显示相对时间（刚刚 / 5分钟前 / 1小时前 / 昨天）

**附属信息行（可选展示）：**
- 备注名输入框：`input input-xs input-bordered`，紧凑的行内编辑
- 删除按钮：`btn btn-xs btn-ghost text-error`，放在最右侧

## 回复策略按钮的视觉区分

三种策略用不同的 DaisyUI 样式区分：

| 策略 | 按钮样式 | 含义 |
|------|---------|------|
| 不回复 | `btn btn-xs btn-ghost opacity-50` | 灰色，表示"什么都不做" |
| 有消息才回复 | `btn btn-xs btn-outline btn-info` | 蓝色轮廓，表示"有条件地回复" |
| 始终回复 | `btn btn-xs btn-info` | 蓝色实心，表示"积极回复" |

点击按钮后循环切换：不回复 → 有消息才回复 → 始终回复 → 不回复。每次点击直接调用后端更新，不需要额外的保存按钮。

## 分组标题的样式

每个渠道分组的标题行：

```
NapCat · QQ机器人
```

用 `text-xs font-medium opacity-70` 显示，格式为 `{平台名} · {渠道名称}`。如果渠道已禁用，追加一个 `badge badge-xs badge-warning` 显示"已禁用"。

分组之间用 `mt-4` 间隔。

## 空状态

联系人列表为空时：

```
┌──────────────────────────────────────────────┐
│  远程联系人                          [刷新]   │
│                                              │
│  暂无联系人记录。                              │
│  配置并启用渠道后，收到的消息会自动创建联系人。  │
│                                              │
└──────────────────────────────────────────────┘
```

提示文字用 `text-sm opacity-60`，告诉用户联系人是怎么来的（自动创建），避免用户困惑"为什么列表是空的"。

## 最后活跃时间的显示

不显示原始的 ISO 时间戳，转为人类可读的相对时间：

- 1 分钟内 → "刚刚"
- 1-59 分钟 → "X分钟前"
- 1-23 小时 → "X小时前"
- 1-6 天 → "X天前"
- 7 天以上 → 显示日期（如 "3/8"）

## 渠道配置的默认值修正

新建渠道时 `defaultReplyMode` 应默认为 `"none"`（不回复），不是当前代码里的 `"reply_once"`。

## 不需要做的事

- 不改聊天窗口的消息展示
- 不加联系人搜索功能（联系人数量不会太多，最多几十个）
- 不做联系人排序（按渠道分组已经够了）
- 不做联系人批量操作（逐个管理足够）
- 备注名不需要弹窗编辑，行内输入框就够了

## 与现有 UI 模式的对齐

| 模式 | 参考来源 | 在本页面的应用 |
|------|---------|--------------|
| 折叠面板 | MCP 页面的 `collapse collapse-arrow` | 渠道配置的"能力与行为"和"凭证 JSON" |
| 分组列表 | MCP 工具列表的 `divide-y` | 联系人按渠道分组，组内用分割线 |
| 行内操作 | 工具页面的 toggle 开关 | 回复策略的点击循环切换 |
| 状态徽章 | MCP 服务的状态 badge | 联系人类型标签、新消息指示器 |
| 紧凑表单 | 全局的 `input-sm`、`select-sm` | 备注名的行内编辑输入框 |

## 预期效果

页面打开后：
1. 顶部是渠道区，每个渠道只占一行摘要，不会抢空间
2. 下面是联系人列表，按渠道分组，每个联系人一眼看到名字、类型、策略、最后活跃时间
3. 有新消息的联系人有红点标记
4. 点策略按钮一下就能切换，不需要下拉框或弹窗
5. 整体视觉和 MCP、工具页面保持一致
