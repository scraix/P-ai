# 20260329 operate 工具接入计划

## 1. 背景

当前项目里已经有桌面操作相关的样本代码，但还没有真正把 `operate` 作为可用内置工具接入到聊天运行时。

这次目标不是把所有桌面自动化能力一次性做满，而是先把一个可稳定调用、参数清晰、和现有运行时体系打通的 `operate` 工具落地，让模型可以直接发起基础鼠标、键盘、文本输入。

结合当前讨论，本次设计原则为：

- 工具协议尽量贴近底层输入依赖能力，不堆过多人类语义别名。
- 优先让 LLM 使用稳定、可读、直觉的键名，而不是平台按键码。
- 先做最小可用版，后续再扩展更复杂的桌面交互。

## 2. 目标

### 2.1 功能目标

1. 新增并正式接入内置工具 `operate`。
2. 首版支持三类输入：
   - `mouse`
   - `keyboard`
   - `text`
3. 首版支持基础操作能力：
   - 鼠标按钮输入
   - 鼠标滚动
   - 键盘按键输入
   - 文本输入
4. 支持基础控制参数：
   - `pre_delay`
   - `delay`
   - `repeat`
   - `press`
5. 工具进入默认工具目录、运行时工具装配和状态检查链路。
6. 返回结构化结果，便于模型判断是否执行成功。

### 2.2 非目标

1. 本次不做 OCR、图像识别或“自动找按钮”。
2. 本次不做复杂桌面工作流编排，只做单步输入操作。
3. 本次不承诺管理员窗口、UAC 界面、受保护窗口的可操作性。
4. 本次不做跨平台细节打磨，优先保证 Windows 主路径可用。

## 3. 工具协议

### 3.1 输入

建议首版协议收敛为统一结构：

- `type: "mouse" | "keyboard" | "text"`
- `target?: { x: number, y: number }`
- `button?: string`
- `keys?: string[]`
- `text?: string`
- `repeat?: number`
- `pre_delay?: number`
- `delay?: number`
- `press?: number`

字段默认值与校验规则建议明确为：

- `type`
  - 类型：枚举字符串
  - 必填
  - 取值：`mouse | keyboard | text`
- `target`
  - 类型：对象 `{ x, y }`
  - 默认：无
  - 约束：`x`、`y` 必须是有限数字；不允许负值
  - 越界处理：若坐标超出当前屏幕可点击区域，首版建议直接返回参数错误，不做自动 clamp
- `button`
  - 类型：字符串
  - 默认：无
  - 约束：仅允许协议声明内的鼠标按钮值
- `keys`
  - 类型：字符串数组
  - 默认：无
  - 约束：数组不能为空；每个元素必须为非空键名
- `text`
  - 类型：字符串
  - 默认：无
  - 约束：`text` 模式下不能为空字符串
- `repeat`
  - 类型：正整数
  - 默认：`1`
  - 约束：`>= 1`，建议首版上限 `100`
- `pre_delay`
  - 类型：数字，单位秒
  - 默认：`0`
  - 约束：`>= 0`，建议首版上限 `300`
- `delay`
  - 类型：数字，单位秒
  - 默认：`0`
  - 约束：`>= 0`，建议首版上限 `300`
- `press`
  - 类型：数字，单位秒
  - 默认：`0`
  - 约束：`>= 0`，建议首版上限 `300`
  - 语义：`0` 表示正常点击/点按；`> 0` 表示长按指定时长

### 3.2 约束

- `mouse`:
  - `button` 必填
  - 若涉及点击目标，`target` 必填
- `keyboard`:
  - `keys` 必填
- `text`:
  - `text` 必填

### 3.3 键名策略

为降低提示负担，键盘输入不直接暴露平台按键码，首版采用 LLM 更容易直觉理解的可读键名，例如：

- `Enter`
- `Tab`
- `Escape`
- `Backspace`
- `Delete`
- `ArrowUp`
- `ArrowDown`
- `ArrowLeft`
- `ArrowRight`
- `Home`
- `End`
- `PageUp`
- `PageDown`
- `Control`
- `Shift`
- `Alt`
- `Meta`
- `F1` ~ `F12`

普通字符按单字符处理，例如：

- `a`
- `1`
- `/`

说明：

- 协议层允许大小写宽容处理。
- 内部映射到底层依赖需要的键类型。
- 仅当未来遇到必须布局无关的场景时，再单独补充 `raw_code` 能力。

### 3.4 鼠标按钮策略

首版支持：

- `left`
- `right`
- `middle`
- `scroll_up`
- `scroll_down`

如底层依赖在当前平台可稳定支持，再补充：

- `back`
- `forward`

补充说明：

- `scroll_up` / `scroll_down` 首版采用固定步长，不额外开放 `amount` 参数
- 默认步长建议定义为 `1` 个滚轮档位/逻辑行
- 当前通过 `repeat` 控制连续滚动次数，例如：`type=mouse` + `button=scroll_down` + `repeat=3`
- 若未来需要更细粒度控制，再扩展 `amount` 字段并单独约定单位（逻辑行数或像素）

### 3.5 组合规则

不单独定义 `double_click` 等高阶动作，统一由基础输入与控制参数组合得到，例如：

- 双击左键：`type=mouse` + `button=left` + `repeat=2` + `delay=0.1`
- 连按两次回车：`type=keyboard` + `keys=["Enter"]` + `repeat=2`
- 长按鼠标：`press > 0`

## 4. Rust 侧实现设计

### 4.1 数据模型

补充或调整与 `operate` 相关的请求/响应结构，至少包含：

- `OperateRequest`
- `OperateResponse`
- `OperateInputType`
- `OperateTarget`

建议把 `OperateResponse` 细化为结构化响应，而不是只返回一个布尔值：

- `status`
  - 枚举：`success | partial_success | failure`
- `duration`
  - 执行总耗时
- `requested_repeats`
  - 请求的重复次数
- `actual_repeats`
  - 实际尝试次数
- `result` / `outputs`
  - 成功时的执行摘要
- `error`
  - 结构化错误对象：`{ code, message, location, step }`
- `attempts`
  - 每次尝试的结果数组，例如第 `n` 次的状态、错误信息、时间戳

典型成功 JSON：

```json
{
  "status": "success",
  "duration": 120,
  "requested_repeats": 2,
  "actual_repeats": 2,
  "result": {
    "type": "mouse",
    "button": "left",
    "target": { "x": 300, "y": 200 }
  },
  "attempts": [
    { "index": 1, "status": "success" },
    { "index": 2, "status": "success" }
  ]
}
```

典型失败 JSON：

```json
{
  "status": "failure",
  "duration": 15,
  "requested_repeats": 1,
  "actual_repeats": 0,
  "error": {
    "code": "INVALID_PARAMS",
    "message": "target x 超出屏幕范围",
    "location": "target.x",
    "step": 1
  },
  "attempts": []
}
```

典型部分成功 JSON：

```json
{
  "status": "partial_success",
  "duration": 260,
  "requested_repeats": 5,
  "actual_repeats": 3,
  "error": {
    "code": "TARGET_LOST",
    "message": "第 3 次点击后目标窗口失焦",
    "location": "repeat[3]",
    "step": 3
  },
  "attempts": [
    { "index": 1, "status": "success" },
    { "index": 2, "status": "success" },
    { "index": 3, "status": "failure" }
  ]
}
```

说明：

- `OperateInputType` 对应 `result.type`
- `OperateTarget` 对应 `result.target`
- 若后续坚持 fail-fast，也应保留结构化错误对象，避免只返回字符串

### 4.2 执行分层

建议把实现拆成三层：

1. 参数校验层
2. 输入映射层
3. 具体执行层

其中：

- 参数校验负责检查 `type / button / keys / text / target`
- 输入映射负责把协议层键名转换为底层依赖可执行参数
- 具体执行层只负责鼠标、键盘、文本发送与等待控制

### 4.3 延迟与重复

统一执行顺序建议为：

1. 执行 `pre_delay`
2. 按 `repeat` 循环执行输入
3. 每次循环：
   - 若 `press > 0`，先按下，等待，再释放
   - 否则执行正常点击/按键
4. 循环之间执行 `delay`

补充边界行为：

- `pre_delay` 只在整个重复循环开始前执行一次
- `delay` 只在两次循环之间执行，不在最后一次执行后追加
- 首版默认失败策略建议为 `fail-fast`
  - 任一轮执行失败，立即停止后续循环
  - 若未来要支持“继续执行并累积错误”，应通过单独策略开关引入
- `press` 对键盘同样有效
  - `press > 0` 时表示长按指定时长后再释放
- `keys` 为多个键时，首版建议按“组合键 chord”处理
  - 先同时按下所有键
  - 等待 `press`
  - 再按相反顺序释放
  - 不建议首版同时支持“顺序按键”和“组合键”两种语义，以免模型混淆

## 7. 验收点

1. `operate` 能出现在工具目录与运行时工具清单中。
2. 模型可通过 `operate` 发起鼠标点击。
3. 模型可通过 `operate` 发起键盘输入与组合键。
4. 模型可通过 `operate` 发起文本输入。
5. `repeat / delay / pre_delay / press` 能按预期生效。
6. 参数错误时返回结构化可读错误，不会让主流程崩溃。
7. `cargo check` 通过。

## 8. 风险与注意事项

1. Windows 权限边界需要明确：普通权限应用未必能控制管理员窗口。
2. 文本输入在不同输入法和目标应用中可能表现不一致，首版优先保证“可输入”，不追求所有应用行为一致。
3. 鼠标侧键、原始按键码等能力是否暴露，取决于底层依赖在当前平台的稳定性。
4. 如果协议层一次性做太宽，后续 prompt 和状态检查都会变复杂，因此首版要刻意收敛。
5. 现有样本代码如果和正式接入方案不一致，应以正式工具协议为准做收敛，而不是保留多套并行语义。
6. 以下安全约束应视为首版必备：
   - 防诱导：防止模型在被提示注入时执行恶意桌面操作
   - 速率限制：对高频点击、大量键入、连续滚动设置频率控制
   - 用户确认：对敏感操作（如大规模删除、批量输入、可能提交表单）要求二次确认
   - 操作审计：记录并保留所有桌面操作日志，便于事后审查
   - 用户体验冲突处理：明确人工输入与模型输入冲突时的仲裁和回退策略
