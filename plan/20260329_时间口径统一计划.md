# 20260329 时间口径统一计划

## 1. 背景

当前项目里已经同时存在两种时间语义：

1. `真实时间 / 绝对时间`
   - 适合数据落盘、调度判断、计时器等待、到期比较。
   - 典型实现是 `now_utc()` 与 `OffsetDateTime` 比较。
2. `当地时间 / 用户可见时间`
   - 适合 LLM 阅读、任务工具入参、任务反馈、前端展示。
   - 当前主要依赖 `format_local_datetime_to_rfc3339()`、前端 `new Date(...).toLocale...` 临时格式化。

现状问题不是“完全没有本地时间支持”，而是“边界不够明确”：

- `task` 工具说明要求 LLM 使用本地 RFC3339。
- `task` 存储层会接受任意带 offset 的 RFC3339，但没有显式统一成数据层唯一语义。
- 调度层用绝对时间比较是对的，但它依赖“上游输入刚好可被正确解析”。
- `wait` 工具反馈是当地时间文本，`task` 数据与运行日志则仍是原始 RFC3339 字符串，显示依赖各处自行格式化。
- 前端不同组件各自实现时间展示格式，缺少统一 helper。

这会导致后续继续扩展计时器、任务反馈、更多 UI 时间字段时，口径容易再次漂移。

## 2. 目标

### 2.1 统一口径

本次统一为以下规则：

1. 任务工具参数使用当地时间。
2. 任务工具反馈使用当地时间。
3. UI 展示使用当地时间。
4. LLM 可见时间使用当地时间。
5. 数据层存储、调度判断、计时器计算统一使用真实时间。
6. 变量名、字段名、注释、文档说明统一使用同一套时间词汇，不再混用旧叫法。

### 2.2 真实时间定义

- 真实时间统一定义为：`UTC RFC3339（秒级精度，优先 Z）` 或等价的 `OffsetDateTime` 绝对时刻。
- 任何进入数据层的“可调度时间”都必须先转换为真实时间，再参与存储与比较。

### 2.3 当地时间定义

- 当地时间统一定义为：`当前系统本地时区下的 RFC3339（保留 offset，秒级精度）`。
- 用户、LLM、前端、任务回执看到的时间，都以这个格式或其本地文本变体呈现。

### 2.4 命名统一目标

本次不是只统一“行为”，还要统一“语言”：

1. 代码变量名统一。
2. 结构体字段名统一。
3. JSON/Tauri/tool schema 字段名统一。
4. 注释用词统一。
5. 计划文档、Skill 文档、工具描述统一。

要求是后续开发者看到名字时，不需要再猜这个时间到底是“当地显示时间”还是“真实调度时间”。

## 3. 非目标

1. 本次不设计“随系统时区变化自动保持每天早上 9 点”这类 floating wall-clock 规则。
2. 本次不改动性能耗时字段，例如 `elapsed_ms`、`durationMs`。
3. 本次不强行统一所有日志里的时间文本格式；优先处理任务、计时器、LLM 可见时间与 UI。

## 4. 统一命名规范

### 4.1 词汇表

本次固定只允许两套核心时间词汇：

1. `local`
   - 含义：当地时间、用户可见时间、带本地时区偏移的 wall-clock 时间。
2. `utc`
   - 含义：真实时间、绝对时间、数据层存储时间、调度比较时间。

说明：

- 业务说明里可以继续写“真实时间”，但代码命名统一使用 `utc`。
- 业务说明里写“当地时间”，代码命名统一使用 `local`。
- 注释中若出现“真实时间”，必须直接写成“真实时间（UTC）”，禁止只写“真实时间”不解释。

### 4.2 禁止继续使用的模糊命名

以下命名在时间字段上禁止继续新增：

- `time`
- `timestamp`
- `iso`
- `formatted`
- `display_time`
- `real_time`
- `actual_time`
- `run_at` / `end_at` / `next_run_at` 这种不带语义后缀的裸字段

除非上下文已经被强类型完全限定，否则一律不允许继续使用这些模糊名字承载时间语义。

### 4.3 字段命名规则

统一规则如下：

1. 对外输入字段
   - 一律显式带 `_local`
   - 例如：
     - `run_at_local`
     - `end_at_local`
2. 对外输出字段
   - 一律显式带 `_local`
   - 例如：
     - `next_run_at_local`
     - `triggered_at_local`
     - `started_at_local`
     - `finished_at_local`
3. 数据层 / 存储层 / 调度层字段
   - 一律显式带 `_utc`
   - 例如：
     - `run_at_utc`
     - `end_at_utc`
     - `created_at_utc`
     - `updated_at_utc`
     - `last_triggered_at_utc`
4. 中间转换变量
   - 输入解析阶段使用 `*_local`
   - 入库存储阶段使用 `*_utc`
   - 显示格式化阶段使用 `*_local_text` 或 `*_local_rfc3339`

### 4.4 函数命名规则

时间函数统一按语义命名：

1. 当前真实时间
   - `now_utc()`
2. 当前当地时间 RFC3339
   - `now_local_rfc3339()`
3. 当地时间输入转真实时间
   - `parse_local_rfc3339_to_utc(...)`
4. 真实时间转当地 RFC3339
   - `format_utc_to_local_rfc3339(...)`
5. 真实时间转当地文本
   - `format_utc_to_local_text(...)`

禁止继续使用缺少语义的 helper 名称，例如：

- `now_iso()`
- `parse_iso()`
- `format_message_time_*`

若短期内为了兼容保留旧 helper，也必须改成薄包装，并在注释里标记“待移除旧名”。

### 4.5 注释规范

注释统一只允许以下表述：

1. `当地时间（local）`
2. `真实时间（UTC）`
3. `入参使用当地时间，入库统一转 UTC`
4. `对外返回时再从 UTC 转回当地时间`

禁止再出现：

- “这里是时间字符串”
- “转成真实时间”
- “格式化一下时间”

这种没有说明语义边界的模糊注释。

## 5. 现状梳理

### 5.1 已经基本正确的部分

- `src-tauri/src/features/core/domain.rs`
  - `now_utc()`、`parse_local_rfc3339_to_utc()`、`format_utc_to_local_rfc3339()` 已具备基础能力。
- 大多数通用 `created_at / updated_at` 通过 `now_utc()` 生成，当前已是 UTC `Z` 语义。
- `task` 调度判断使用 `OffsetDateTime` 比较，本质上已经按绝对时间运行。

### 5.2 当前不够统一的部分

- `src-tauri/src/features/task/store.rs`
  - `run_at / end_at` 会被标准化为 RFC3339，但不强制落成 UTC `Z`。
  - `next_run_at`、`last_triggered_at`、运行日志时间需要明确区分“存储值”和“展示值”。
- `src-tauri/src/features/task/scheduler.rs`
  - 任务提示板会显示本地时间，但 provider meta 里的任务时间仍直接塞原始存储值。
- `src-tauri/src/features/chat/model_runtime/tools_and_builtin/tool_impls.rs`
  - `task` 工具描述已要求本地时间，但底层没有一个命名清晰的“当地 -> 真实”统一入口。
- `src-tauri/src/features/system/tools/wait.rs`
  - 等待结果返回的是当地时间文本，但没有与任务时间口径形成统一约定。
- 前端
  - `src/features/config/views/config-tabs/TaskTab.vue`
  - `src/features/chat/components/ChatMessageItem.vue`
  - 两处都在各自手写本地时间格式化逻辑。

- 命名层
  - 目前存在 `run_at`、`triggered_at` 这类未显式标注 local/utc 的名字。
  - 这些命名会让“时间处于哪一层语义”只能靠上下文猜测，不满足本次目标。

## 6. 方案设计

### 6.1 新增统一时间语义模块

建议新增 Rust 时间模块，例如：

- `src-tauri/src/features/core/time_semantics.rs`

职责固定为：

1. 定义“真实时间”和“当地时间”的转换规则。
2. 统一处理用户/LLM 输入的本地 RFC3339。
3. 统一生成数据层可落盘的 UTC RFC3339。
4. 统一把数据层 UTC 时间格式化为本地 RFC3339 或本地文本。
5. 统一提供命名明确的 helper，替换旧的模糊 helper。

建议提供的核心能力：

1. `now_utc()`
2. `now_local_rfc3339()`
3. `parse_local_rfc3339_to_utc(field_name, value)`
4. `format_utc_to_local_rfc3339(value)`
5. `format_utc_to_local_text(value)`

要求：

- 所有可调度时间入库前必须走 `parse_local_rfc3339_to_utc(...)`
- 所有用户可见时间输出前必须走 `format_utc_to_local_*`
- 不再允许任务模块自行直接决定“输入时间如何落盘”

### 6.2 模型拆分策略

为了彻底统一字段名，不再使用同一个结构同时承载“输入 local”和“存储 utc”两种语义。

建议拆成三层模型：

1. 输入模型
   - 例如 `TaskTriggerInputLocal`
   - 字段使用：
     - `run_at_local`
     - `end_at_local`
2. 存储模型
   - 例如 `TaskTriggerStored`
   - 字段使用：
     - `run_at_utc`
     - `end_at_utc`
3. 输出模型
   - 例如 `TaskTriggerView`
   - 字段使用：
     - `run_at_local`
     - `end_at_local`
     - `next_run_at_local`

这样可以从类型层就避免 local/utc 混装。

### 6.3 数据层规则

任务相关存储字段统一落真实时间：

- `task_record.run_at_utc`
- `task_record.end_at_utc`
- `task_record.created_at_utc`
- `task_record.updated_at_utc`
- `task_record.last_triggered_at_utc`
- `task_record.completed_at_utc`
- `task_run_log.triggered_at_utc`
- `task_progress_note.at_utc`

规则：

1. `run_at / end_at` 输入时允许本地 RFC3339 with offset。
2. 入库前统一转成 UTC RFC3339。
3. 读取后参与调度时统一按绝对时间解析。
4. 对外返回给前端或 LLM 时再转回当地时间。

说明：

- 若数据库层短期内因为 SQLite 物理列名不便一次性重命名，可以先在代码模型与 SQL alias 中统一成 `_utc`。
- 但代码字段名、变量名、返回字段名必须先统一，不允许继续把旧裸字段名扩散到更多调用点。

### 6.4 任务工具规则

`task` 工具统一遵守：

1. 入参 `trigger.run_at_local / trigger.end_at_local` 只接受当地 RFC3339。
2. 工具文案明确说明：
   - 这是用户当前当地时间
   - 直接复制提示板格式
   - 数据层会自动转为真实时间
3. 工具返回值中的任务时间字段统一输出当地时间，并显式使用 `_local` 字段名。

### 6.5 调度与提示板规则

调度器内部继续使用真实时间比较，但对 LLM 的暴露统一本地化：

- `currentLocalTime`
- `runAtLocal`
- `endAtLocal`
- `nextRunAtLocal`
- 任务触发卡片中的时间字段

其中 provider meta 也应输出当地时间，避免前端与 LLM 看到不同语义。

### 6.6 计时器 / wait 规则

`wait` 与其他计时相关工具遵守：

1. 等待计算继续使用 `Instant` 或绝对时间。
2. 回执中的 `started_at_local / finished_at_local` 统一定义为当地时间。
3. 若后续新增 timer/task 相关工具，默认沿用：
   - 参数给当地时间
   - 内部转换成真实时间
   - 反馈再转回当地时间

### 6.7 前端规则

建议新增前端时间展示 helper，例如：

- `src/utils/time.ts`

职责：

1. 统一格式化 RFC3339 字符串到本地展示文本。
2. 替换 `TaskTab.vue`、`ChatMessageItem.vue` 内重复的时间格式化逻辑。
3. 前端只做“展示格式化”，不参与调度时间换算。
4. 前端类型字段也统一显式区分 `*_local`。

## 7. 兼容与迁移

### 7.1 兼容策略

历史任务数据里可能同时存在：

- `2026-03-29T10:00:00Z`
- `2026-03-29T18:00:00+08:00`

两者都是合法绝对时间。

本次改造要求：

1. 读取时兼容两类旧值。
2. 写回或更新时统一改写为 UTC RFC3339。
3. 不要求额外做一次性全库强制迁移脚本；可采用“读兼容、写收敛”。
4. 对外 tool schema 若改字段名，需要同步更新 Skill、提示板、前端类型与相关测试。

### 7.2 风险点

1. 若某些旧前端/旧测试直接断言 `+08:00` 原样存储，会在改造后失效。
2. 若 provider meta 仍保留 UTC 原值，会出现“聊天卡片”和“任务面板”显示不一致。
3. 若前端继续局部自定义格式化，统一模块会被再次绕开。
4. 若只改逻辑不改命名，会继续留下“run_at 到底是 local 还是 utc”的理解成本。

## 9. 实施顺序

### 第一阶段：抽统一时间模块

1. 定义真实时间/当地时间 helper。
2. 把现有零散 helper 收口到统一命名下。
3. 保留旧 helper 的薄包装或直接替换调用点。
4. 先完成命名词汇表，禁止新旧名字并存。

### 第二阶段：统一任务数据入口

1. `task.trigger.run_at / end_at` 入参改为“先转真实时间再入库”。
2. `task_record` 与 `task_run_log` 相关写入全部走统一 helper。
3. `next_run_at` 计算结果统一按真实时间存储、按当地时间输出。
4. 把 task 相关结构体字段名显式改成 `*_local / *_utc`。

### 第三阶段：统一 LLM 可见时间

1. 更新 `task` 工具文案。
2. 更新任务提示板。
3. 更新任务触发消息 provider meta。
4. 更新 `task-guide` 说明。
5. 更新注释与日志文案中的时间语义表述。

### 第四阶段：统一 UI 展示时间

1. 抽前端时间格式化 helper。
2. 替换任务页与聊天卡片的重复格式化实现。
3. 核对任务详情、运行日志、任务触发卡片显示一致。
4. 统一前端类型字段与组件局部变量命名。

### 第五阶段：统一计时器反馈

1. `wait` 工具回执接入统一时间 helper。
2. 明确文档与类型定义里的时间字段语义。
3. 清理剩余旧字段名、旧 helper 名、旧注释。

## 10. 验证

### 10.1 Rust

1. 补充单测：
   - 本地 RFC3339 输入可正确转 UTC
   - 旧 `Z / +08:00` 存量值都能正确解析
   - `task` 创建/更新后存储值为 UTC
   - 调度比较不受本地显示格式影响
   - 输出给 LLM/前端的时间已转回当地时间
   - 字段名与 schema 中不再残留裸 `run_at / end_at / triggered_at`
2. 运行 `cargo check`
3. 运行 `cargo test`

### 10.2 Frontend

1. 运行 `pnpm typecheck`
2. 手动验证：
   - 任务详情页
   - 任务运行日志
   - 聊天任务卡片
   - `wait` 工具反馈

## 11. 验收标准

1. `task` 工具参数统一使用当地时间。
2. `task` 工具反馈统一使用当地时间。
3. UI 中任务相关时间统一显示当地时间。
4. 调度与计时内部统一基于真实时间判断。
5. 新增统一时间转换模块，后续不再各处分散自行换算。
6. 历史任务数据兼容读取，新写入任务统一收敛为真实时间存储。
7. 变量名、字段名、注释、Skill 文案、工具描述统一使用 `local / utc` 词汇体系。
8. 不再保留时间语义不明确的裸字段名与 helper 名。

## 12. 完成后的归档约定

实现完成后，先等待用户在生产环境验证并明确反馈“通过”，再根据最终实现修正本计划并归档到 `plan/done/`。
