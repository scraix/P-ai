# 20260329 统一桌面脚本工具计划

## 1. 背景

当前项目中桌面相关能力分散在多个工具与层级中：

- `operate`：桌面输入能力，现已改为 MCP 形态
- `desktop_screenshot`：给模型使用的桌面截图 MCP
- `wait`：等待能力
- `xcap`：更底层的窗口/区域/显示器截图接口

实测下来，LLM 在桌面交互场景中最常见的行为是：

1. 操作鼠标/键盘/文本输入
2. 等待界面变化
3. 再截图确认

如果这些动作拆成多个工具，调用成本高、上下文切换多，且不利于模型稳定生成一整套操作序列。

因此本次目标是把桌面交互收敛为一个统一的 MCP 脚本工具，由模型一次提交一段脚本，按顺序执行鼠标、键盘、文本、等待、截图动作。

## 2. 目标

### 2.1 功能目标

1. 提供一个统一的桌面脚本 MCP 工具。
2. 脚本支持以下动作：
   - `mouse`
   - `key`
   - `text`
   - `wait`
   - `screenshot`
3. 鼠标坐标统一只支持归一化百分比坐标。
4. 截图对模型的可见语义统一为“只保留最新一张”。
5. 截图允许显式保存到磁盘，但 `save` 必须是绝对路径。
6. 移除模型侧对桌面多工具频繁切换的依赖。

### 2.2 非目标

1. 本次不做 OCR、图像识别、自动找元素。
2. 本次不做流程控制语法，例如 `if/else`、块级循环、变量。
3. 本次不支持相对路径保存截图。
4. 本次不支持像素坐标。

## 3. 统一脚本协议

### 3.1 总体规则

- 一行一个动作
- 空行允许，执行时跳过
- 首版不支持注释行
- 参数采用稳定、有限的脚本格式，不做自由自然语言解析
- 通用参数默认值：
  - `repeat` 默认 `1`
  - `delay` 默认 `0`
  - `pre_delay` 默认 `0`
  - `press` 默认 `0`
- 通用参数范围：
  - `repeat` 建议限制在 `1~100`
  - `delay / pre_delay / press` 建议限制在 `0~300` 秒
- 执行顺序固定为：
  - 先执行一次 `pre_delay`
  - 再进入 `repeat` 循环
  - `delay` 只在两次循环之间执行，最后一次后不再追加
- `key` 中 `press > 0` 表示长按；多键组合默认按 chord 处理：先全部按下，再按相反顺序释放
- 首版执行模型固定为 `fail-fast / stop-on-failure`
  - 任一行解析或执行失败，立即停止后续动作
  - 后续行不再执行
  - 首版不采用“继续执行并累积错误”的策略

### 3.2 鼠标动作

格式：

- `mouse <button> click @x,y [repeat=n] [delay=s] [pre_delay=s] [press=s]`

示例：

- `mouse left click @0.50,0.10`
- `mouse left click @0.90,0.05 repeat=2 delay=0.1`

滚动：

- `mouse scroll_up [repeat=n] [delay=s] [pre_delay=s]`
- `mouse scroll_down [repeat=n] [delay=s] [pre_delay=s]`

示例：

- `mouse scroll_down repeat=3 delay=0.05`

约束：

- `x`、`y` 范围固定 `0.0 ~ 1.0`
- 坐标语义固定为当前整桌面比例坐标
- 不再支持像素坐标

### 3.3 键盘动作

格式：

- `key <combo> [repeat=n] [delay=s] [press=s]`

示例：

- `key Control+L`
- `key Enter`
- `key Control+Shift+P delay=0.1`

约束：

- 组合键使用 `+` 连接
- 键名使用可读名称，例如 `Enter`、`Tab`、`Escape`、`ArrowDown`、`Control`
- 单字符直接写字符本身
- 多键默认视为组合键 chord，而不是顺序按键脚本
- `press > 0` 时，表示全部按下后保持 `press` 秒，再反向释放

### 3.4 文本动作

格式：

- `text "..." [repeat=n] [delay=s]`

示例：

- `text "B站热榜"`

### 3.5 等待动作

格式：

- `wait <seconds>`

示例：

- `wait 0.5`
- `wait 1.2`

### 3.6 截图动作

格式：

- `screenshot`
- `screenshot focused_window`
- `screenshot region=@x,y,w,h`
- `screenshot save="ABSOLUTE_PATH"`
- `screenshot focused_window save="ABSOLUTE_PATH"`
- `screenshot region=@x,y,w,h save="ABSOLUTE_PATH"`

示例：

- `screenshot`
- `screenshot focused_window`
- `screenshot region=@0.10,0.10,0.80,0.60`
- `screenshot save="E:/github/easy_call_ai/tmp/shot.webp"`

约束：

- `region` 同样使用归一化百分比
- `region=@x,y,w,h` 中的 `w`、`h` 明确表示宽度和高度，不是右下角终点坐标
- 例如 `region=@0.10,0.10,0.80,0.60` 表示：
  - 左上角在屏幕宽高的 `10%,10%`
  - 截图宽度为屏幕宽度的 `80%`
  - 截图高度为屏幕高度的 `60%`
- `save` 必须为绝对路径
- 不允许相对路径
- `save` 仅表示保存文件，不改变模型可见视觉语义

## 4. 截图的视觉语义

这是本次设计的核心约束：

1. 模型对桌面画面的感知永远只有“最新一张截图”。
2. 同一段脚本中若执行多次 `screenshot`，新的截图会覆盖旧的截图。
3. 被覆盖的旧截图对模型来说视为“画面已经离去”，不可继续回看。
4. 脚本执行结束后，最多只将最后一张截图作为最新视觉输入进入后续上下文。
5. 即使旧截图已保存到磁盘，它也不再是模型当前可见视觉的一部分。

## 5. MCP 输入输出

### 5.1 输入

统一工具首版只接收：

- `script: string`

### 5.2 输出

首版采用 `stop-on-failure`，因此输出分为两类：

1. 成功路径：返回统一 JSON
2. 失败路径：通过 MCP error 直接返回失败，不再返回 `ok=false` 的成功 payload

成功 JSON 至少包含：

- `ok`
- `executedCount`
- `elapsedMs`
- `steps`
- `latestScreenshot`

其中：

- `steps` 为逐步执行结果
- `latestScreenshot` 仅在本次脚本内确实执行过截图时存在
- 若本次脚本执行了多次截图，`latestScreenshot` 只对应最后一次

每个 `step` 至少包含：

- `line`
- `kind`
- `summary`
- `ok`
- `savedPath`（按需）

成功示例：

```json
{
  "ok": true,
  "executedCount": 3,
  "elapsedMs": 820,
  "steps": [
    { "line": 1, "kind": "mouse", "summary": "mouse click completed, repeat=1", "ok": true },
    { "line": 2, "kind": "wait", "summary": "wait completed, seconds=0.500", "ok": true },
    { "line": 3, "kind": "screenshot", "summary": "screenshot completed, mode=desktop", "ok": true }
  ],
  "latestScreenshot": {
    "mode": "desktop",
    "width": 2560,
    "height": 1440
  }
}
```

失败示例（当前首版实际采用）：

```json
{
  "mcpError": {
    "code": "INTERNAL_ERROR",
    "message": "第 3 行 screenshot save 非法：必须是绝对路径"
  }
}
```

未采用但需明确区分的备选策略示例：

- `continue-with-errors / partial_success`
  - 若未来采用，则会继续执行后续行，并在成功 payload 中显式返回 `status=partial_success`
  - 首版不采用该策略，测试也不应按该策略断言

## 6. Rust 实现设计

### 6.1 分层

建议拆成三层：

1. 脚本分行与词法切分
2. 脚本动作解析
3. 动作执行与结果汇总

### 6.2 内部结构

建议新增内部结构，例如：

- `DesktopScriptRequest`
- `DesktopScriptAction`
- `DesktopScriptActionKind`
- `DesktopScriptStepResult`
- `LatestScreenshotState`

### 6.3 截图接入策略

统一脚本工具中的 `screenshot` 动作不能只调用底层截图函数，还要明确接入运行时视觉语义：

- 执行截图
- 更新“最新视觉态”
- 若有 `save` 则落盘
- 若后续再次截图，则覆盖前一张视觉态

本次要避免直接复用“保留多张截图 artifact 给模型回看”的旧语义。

### 第一阶段：协议与命名收敛

1. 确定统一工具名称

### 第二阶段：脚本解析

1. 实现 `mouse`
2. 实现 `key`
3. 实现 `text`
4. 实现 `wait`
5. 实现 `screenshot`

### 第三阶段：截图语义接入

1. 建立“最新视觉态”覆盖语义
2. 接入保存路径校验
3. 调整历史与结果整理逻辑

### 第四阶段：验证

1. `cargo check`
2. 补脚本解析单测
3. 补截图覆盖语义单测
4. 补绝对路径校验单测
5. 补错误提示快照与边界值单测

## 9. 验收点

1. 模型只需调用一个桌面脚本工具即可完成操作、等待、截图。
2. 鼠标坐标统一使用百分比。
3. `wait` 能在脚本中执行。
4. `screenshot save=...` 仅接受绝对路径。
5. 多次截图后，模型后续只保留最后一张视觉态。
6. `cargo check` 通过。

## 10. 风险与注意事项

1. 截图“最新一张覆盖旧图”的语义会改变当前多图 artifact 的使用方式，需要谨慎修改。
2. 若统一工具上线后仍保留旧桌面工具并同时暴露给模型，模型可能继续乱用旧工具，因此需要控制挂载策略。
3. 脚本语法必须保持克制，否则解析复杂度会快速上升。
4. 保存路径必须严守绝对路径校验，避免相对路径与工作目录漂移问题。
5. 失败策略必须与文档、MCP 返回和测试保持一致；首版既然采用 `fail-fast`，就不能在样本和文档中暗示“自动跳过失败行继续执行”。

## 11. 脚本测试矩阵

上线前需要把以下场景在测试里覆盖到位：

### 11.1 成功解析用例

1. `mouse left click @0.50,0.10`
2. `mouse left click @0.90,0.05 repeat=2 delay=0.1`
3. `mouse scroll_down repeat=3 delay=0.05`
4. `key Control+L`
5. `key Enter`
6. `text "B站热榜"`
7. `wait 0.5`
8. `screenshot`
9. `screenshot focused_window`
10. `screenshot region=@0.10,0.10,0.80,0.60`
11. `screenshot save="E:/tmp/shot.webp"`
12. 多行混合脚本顺序执行

### 11.2 失败解析用例

1. 未知动作，例如 `mose left click @0.1,0.2`
2. `mouse` 缺按钮
3. `mouse click` 缺坐标
4. 坐标格式非法，例如 `@0.1`
5. 坐标越界，例如 `@1.2,0.5`
6. `scroll` 动作误带坐标或非法参数
7. `key` 缺按键
8. `text` 缺引号或空文本
9. `wait` 缺秒数
10. `wait` 秒数非法
11. `screenshot region` 参数不完整
12. `save` 不是绝对路径
13. 参数键重复
14. 不支持的参数键

### 11.3 边界值用例

1. 归一化坐标边界 `0.0`
2. 归一化坐标边界 `1.0`
3. `repeat=1`
4. `repeat` 上限
5. `delay=0`
6. `press=0`
7. `region=@0,0,1,1`

### 11.4 语义执行用例

1. 多次 `screenshot` 只保留最后一张视觉态
2. 带 `save` 的截图会落盘，但不改变“旧画面已离去”语义
3. 无截图脚本不会生成最新视觉态
4. 失败行会立即阻断后续执行，并通过 MCP error 返回失败；首版不继续执行后续行

### 11.5 工具目录一致性用例

1. 前台 `list_tool_catalog` 中的 `operate` 定义必须与 `operate` MCP `list_all_tools()` 返回值完全一致
2. 前台 `list_tool_catalog` 中的 `read_file` 定义必须与 `read_file` MCP `list_all_tools()` 返回值完全一致
3. 一致性校验必须覆盖名称、描述、参数 schema，禁止前台额外维护第二套说明文案

## 12. 报错提示规范

脚本解析与执行错误必须遵守统一格式，至少包含：

1. 行号
2. 动作名
3. 出错字段或位置
4. 人类可读原因
5. 可选的期望格式提示

目标错误文案示例：

- `第 3 行 mouse 坐标非法：x 必须在 0.0~1.0 之间，当前为 1.2`
- `第 2 行 screenshot save 非法：必须是绝对路径`
- `第 4 行未知动作：mose。可用动作：mouse、key、text、wait、screenshot`
- `第 1 行 text 非法：必须使用双引号包裹文本内容`

测试中应至少验证：

1. 行号准确
2. 错误文案包含动作名
3. 错误文案包含关键原因
4. 典型错误文案可做快照测试，避免后续退化为模糊报错
