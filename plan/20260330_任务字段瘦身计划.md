# 20260330 任务字段瘦身计划

## 1. 背景

当前任务字段对用户和 LLM 都偏重：

1. 主表单里同时存在 `title`、`cause`、`goal`、`flow`、`todos`、`statusSummary`、`stageKey`、`appendNote`。
2. 这些字段之间语义有明显重叠：
   - `title` 与 `goal`
   - `flow` 与 `todos`
   - `statusSummary` 与 `todos`
   - `cause` 与其他说明性字段
3. 当上下文被打断后，只看 `todo` 很容易机械执行而走偏；只看 `goal` 又不知道为什么现在要做这件事。

所以本轮改造目标不是“再设计一套更复杂的任务模型”，而是把**用户和 AI 真正要写的字段**收缩到最小集合。

## 2. 目标

本轮统一口径如下：

1. 用户和 AI 日常填写的正文字段只保留：
   - `goal`
   - `why`
   - `todo`
2. 调度字段继续保留：
   - `runAtLocal`
   - `everyMinutes`
   - `endAtLocal`
3. 完成态继续保留：
   - `completionState`
   - `completionConclusion`
4. 任务列表、任务卡片、提示词、工具 schema、前端表单、后端 view model 统一改成这一套词汇。

说明：

1. 你刚才消息里的 `gol` 这里按 `goal` 处理。
2. `goal` 将成为唯一主标题字段，旧 `title` 不再作为用户/AI 直接填写字段出现。

## 3. 非目标

1. 本轮不改任务调度语义。
2. 本轮不改任务运行日志结构。
3. 本轮不改任务完成/失败的判定规则。
4. 本轮不立即删除所有旧存储列，而是先做兼容迁移与写入收敛。

## 4. 新字段口径

### 4.1 用户和 AI 可填写字段

主字段：

1. `goal`
   - 最终要达成什么。
   - 同时作为任务列表标题、编辑卡标题、任务板标题。
2. `why`
   - 为什么要做。
   - 负责保留任务方向，防止只剩 `todo` 时语义漂移。
3. `todo`
   - 当前下一步做什么。
   - 允许写成“现在先做什么，接下来怎么继续”的短行动说明。

调度字段：

1. `runAtLocal`
2. `everyMinutes`
3. `endAtLocal`

完成字段：

1. `completionState`
2. `completionConclusion`

### 4.2 不再作为主填写字段出现

以下字段从用户和 AI 的主输入中移除：

1. `title`
2. `cause`
3. `flow`
4. `todos`
5. `statusSummary`
6. `stageKey`
7. `appendNote`

其中：

1. `appendNote` 这轮先不作为主字段继续暴露。
2. 若后续仍需要保留“逐轮追加过程记录”能力，可以单独再设计为次级动作，不占主表单。

## 5. 迁移方案

### 5.1 旧字段到新字段的映射

读取旧任务时，统一转换成新 view model：

1. `goal`
   - 优先取旧 `goal`
   - 若旧 `goal` 为空，则回退到旧 `title`
2. `why`
   - 以旧 `cause` 为主体
   - 把旧 `title`、`flow`、`statusSummary`、`stageKey` 这些不再单独使用的说明性内容按顺序并入 `why`
   - 只有非空字段才并入，避免制造大段空壳文本
3. `todo`
   - 优先取旧 `todos`
   - 若旧 `todos` 为空，则回退到旧 `statusSummary`
   - 若两者都存在，则先保留旧 `todos`，必要时把旧 `statusSummary` 追加到尾部，避免信息丢失

### 5.2 写入收敛策略

新字段写回时按以下策略执行：

1. 主写字段改为：
   - `goal`
   - `why`
   - `todo`
2. 旧字段不再接受前端主表单直接编辑。
3. 为了兼容旧逻辑分阶段迁移，写入时暂时镜像：
   - `title = goal`
   - `goal = goal`
   - `cause = why`
   - `todos / statusSummary` 从 `todo` 派生
4. `flow`、`stageKey` 等废弃字段不再新增真实业务意义，只保留必要兼容值或逐步清空。

## 6. 数据模型调整

### 6.1 前端

前端编辑表单改为：

1. `goal`
2. `why`
3. `todo`
4. `runAtLocal`
5. `everyMinutes`
6. `endAtLocal`
7. `completionState`
8. `completionConclusion`

移除：

1. `title`
2. `cause`
3. `flow`
4. `todosText`
5. `statusSummary`
6. `stageKey`
7. `appendNote`

### 6.2 后端

后端 `TaskEntry` / `TaskCreateInput` / `TaskUpdateInput` / 调度提示板输出逐步收口到：

1. `goal`
2. `why`
3. `todo`
4. `trigger`
5. `completionState`
6. `completionConclusion`

兼容层负责把旧字段转换进新结构，不让前台再感知旧词汇。

## 7. UI 改造

### 7.1 列表区

列表里的任务标题直接显示 `goal`。

### 7.2 编辑卡

编辑卡主表单缩成以下结构：

1. `goal`
2. `why`
3. `todo`
4. 调度
5. 完成区

`任务信息` 区也统一改成新字段名，避免继续出现旧 `title / stageKey / statusSummary` 词汇。

## 8. Prompt / Skill / 工具说明改造

### 8.1 task-guide

`task-guide` 里的字段说明、推荐工作流、质量要求全部改成：

1. `goal`
2. `why`
3. `todo`

删除旧字段说明和旧示例。

### 8.2 调度提示板

隐藏任务板、任务提醒提示、任务 JSON 摘要全部改成新字段输出：

1. `goal`
2. `why`
3. `todo`

不再对 LLM 继续暴露 `title/cause/flow/todos/statusSummary/stageKey` 这一套。

## 9. 风险与兼容

### 9.1 风险点

1. 旧数据迁移后，`why` 可能变得很长。
2. 旧 `statusSummary` 与 `todos` 的合并规则如果处理不好，可能让 `todo` 读起来不自然。
3. 现有调度日志和测试里大量使用 `title` 词汇，迁移时容易漏改。

### 9.2 应对策略

1. 先在 view model 层完成统一，再逐步收缩存储层。
2. 对旧数据转换新增专门测试，覆盖：
   - 只有旧 `title`
   - 只有旧 `goal`
   - 只有旧 `statusSummary`
   - 同时存在旧 `todos + statusSummary`
   - 同时存在旧 `cause + flow + stageKey`
3. 对列表标题、编辑卡标题、任务板输出、skill 文档各补一条回归检查。

## 10. 验证

### 10.1 Rust

1. `cargo check`
2. 任务旧字段到 `goal/why/todo` 的映射测试
3. create/update/complete 输入输出结构测试

### 10.2 Frontend

1. `pnpm typecheck`
2. 任务编辑卡只出现 `goal/why/todo`
3. 列表标题、编辑卡标题、完成区文本全部不再出现旧字段名

## 11. 实施顺序

1. 先改前后端 view model 与表单类型。
2. 再加旧字段到新字段的兼容映射。
3. 再改任务编辑卡、任务列表和本地文案。
4. 再改 task-guide、提示板与任务摘要输出。
5. 最后补测试并清理残留旧词汇。
