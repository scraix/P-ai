# 20260330 任务追踪机制简化计划

## 1. 背景

当前任务追踪与调度机制偏复杂：

1. 系统会从全局 `active` 任务中计算一个 `current_tracked_task_id`。
2. 排序规则混合了“是否到点”“立即任务优先”“时间优先”“order_index”等多套逻辑。
3. 调度器每轮还要处理“当前追踪任务”“到点任务”“队列恢复”三条心智路径。
4. 这套机制对“为什么这轮会提醒这个任务、为什么另一个任务没提醒”不够直观。

这次目标不是继续修补全局 tracked 规则，而是直接改成更简单的“按会话防重入提醒”。

## 2. 目标

把任务提醒机制简化为：

1. 先扫描所有“已到时间且未完成”的任务。
2. 按目标会话分组。
3. 每个会话只取“最旧的一条未完成任务”。
4. 如果该会话当前正在对话中，则本轮不提醒。
5. 如果该会话最后一条消息是系统人格消息，则本轮不提醒，避免重复提醒。
6. 否则向该会话发出一条任务待办提醒。

## 3. 新规则

### 3.1 会话分类

任务只按两类目标会话处理：

1. 联系人会话
2. 桌面会话

这里的用途不是做不同排序，而是决定原会话丢失后的处理方式。

### 3.2 候选任务筛选

候选任务必须同时满足：

1. `completionState == active`
2. 任务时间已到

“时间已到”的判定继续沿用当前已有 trigger 规则：

1. 无 `runAtLocal` 的立即任务
2. 一次性 `runAtLocal`
3. 周期 `runAtLocal + everyMinutes + endAtLocal`

本轮不改 trigger 语义，只改提醒分发方式。

### 3.3 按会话分组

对候选任务按“最终要发送到哪个会话”分组。

每组只保留一条任务：

1. 取该组中最旧的未完成任务
2. “最旧”优先按创建顺序/`order_index`
3. 不再引入全局 `current_tracked_task_id` 竞争

### 3.4 防重复提醒

每个目标会话在本轮真正发提醒前，检查两个条件：

1. 如果该会话当前正在对话中，跳过
2. 如果该会话最后一条消息是系统人格消息，跳过

说明：

1. 这里的“系统消息”不是看 `role=system`
2. 而是看消息的 `speaker_agent_id == SYSTEM_PERSONA_ID`
3. 当前任务提醒消息本身就是系统人格发出的提醒消息，因此这条规则可以天然阻止重复连续提醒

### 3.5 会话丢失时的处理

联系人会话：

1. 如果任务原绑定联系人会话丢失
2. 直接跳过本轮
3. 不自动回退到别的会话

桌面会话：

1. 如果任务原绑定的是桌面副会话
2. 原副会话丢失时，自动回退到桌面主会话
3. 如果连桌面主会话也不可用，则跳过

## 4. 对现有机制的替换范围

### 4.1 要移除或降级的旧机制

以下旧机制将不再作为调度核心：

1. 全局 `current_tracked_task_id`
2. 全局“谁最优先”竞争式排序
3. 基于全局 tracked 的提醒决策

### 4.2 仍保留的部分

以下部分继续保留：

1. trigger 时间语义
2. 本地时间输入、UTC 存储
3. 任务运行日志
4. 聊天繁忙时的基本保护

## 5. 数据与运行态调整

### 5.1 后端

后端调度层将改为：

1. 先枚举到点任务
2. 解析每个任务的目标会话
3. 对目标会话做分组
4. 每组取一条任务尝试提醒

新增或明确的运行态判断：

1. 会话是否正在对话中
2. 会话最后一条消息是否为系统人格消息

### 5.2 前端

前端“当前追踪任务”展示需要一起调整：

1. 如果继续保留“当前追踪任务”文案，则只能作为展示，不再代表全局调度核心
2. 更推荐改成“当前待办任务”或直接取消该概念

这块会在实现时一并收口，避免 UI 继续暗示存在单一全局 tracked 任务。

## 6. 具体实现点

### 6.1 调度入口

改造 `task_scheduler_tick`：

1. 不再先刷新全局 tracked 再取第一条 due task
2. 改成构造“按会话分组后的提醒计划”

### 6.2 会话解析

新增或整理“任务目标会话解析”逻辑：

1. 识别联系人会话
2. 识别桌面会话
3. 联系人会话丢失直接跳过
4. 桌面副会话丢失回退桌面主会话

### 6.3 防重复判断

新增或整理两个判断函数：

1. `会话当前是否正在对话中`
2. `会话最后一条消息是否是系统人格消息`

### 6.4 UI 收口

任务面板里的“追踪中”状态需要重新定义：

1. 若仍保留，只表示“这条任务最近被系统选中展示”
2. 不再把它描述为调度核心状态

## 7. 风险点

### 7.1 旧 tracked 语义残留

风险：

1. 后端已经不再按全局 tracked 调度
2. 但前端仍显示“当前追踪任务”
3. 会造成用户理解偏差

处理：

1. 实现时同步收口任务面板相关文案

### 7.2 系统人格消息识别不统一

风险：

1. 如果不同入口对“系统消息”定义不一致
2. 会导致防重复判断失效

处理：

1. 统一以 `speaker_agent_id == SYSTEM_PERSONA_ID` 为准
2. `task_trigger` 相关 meta 仅作为辅助，不作为唯一判据

### 7.3 联系人/桌面会话分类来源不稳

风险：

1. 如果当前会话类型字段不够清晰
2. 回退策略容易实现错

处理：

1. 先核对现有 conversation kind / source 标记
2. 只在现有字段能稳定区分时落地
3. 若确实存在边界模糊，再补最小必要字段

## 8. 验证

### 8.1 Rust

1. `cargo check`
2. 联系人会话丢失时跳过提醒
3. 桌面副会话丢失时回退桌面主会话
4. 会话最后一条消息为系统人格消息时不重复提醒
5. 同一会话多条到点任务时只提醒最旧一条

### 8.2 Frontend

1. `pnpm typecheck`
2. 任务面板不再错误暗示存在单一全局调度 tracked
3. 聊天里连续任务提醒不会重复刷屏

## 9. 实施顺序

1. 先核对并收口会话分类与系统人格消息判断依据
2. 再改后端调度主链路，移除全局 tracked 依赖
3. 再改前端相关展示文案
4. 最后补测试与验证
