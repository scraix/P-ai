# 20260330 任务面板编辑卡重构计划

## 1. 背景

当前任务面板还是旧形态：

1. 上方用 `join button` 做四个分类过滤。
2. 中间展示任务列表。
3. 下方是只读的任务详情卡。
4. 新建任务与编辑任务还没有形成统一的前端入口。

这不利于验证任务系统，因为：

1. 过滤区交互不够直观，不符合本次想要的 DaisyUI `filter` 形态。
2. 列表只能查看，无法直接在面板内新增任务。
3. 点击任务后看到的是只读详情，不利于快速改标题、目标、流程、触发器与 todo。
4. 详情卡嵌在页面里，空间不足，不适合承载完整编辑表单。

## 2. 目标

本次改造目标如下：

1. 任务面板四个分类过滤改成 DaisyUI `form.filter` + radio button 形式。
2. 任务面板保留任务列表，但新增“新建任务”入口。
3. 点击任务列表项时，不再展示只读详情卡，而是打开任务编辑卡。
4. 任务编辑卡作为独立卡片/弹层出现，占当前窗口约 `90%` 尺寸。
5. 新建任务与编辑任务复用同一套表单，不走两套 UI。

## 3. 非目标

1. 本次不改任务存储协议与字段定义。
2. 本次不新增新的任务分类语义，仍沿用当前四类：
   - `active`
   - `tracked`
   - `completed`
   - `all`
3. 本次不改聊天卡片里的任务展示结构。
4. 本次不改任务运行日志的数据结构，只调整面板呈现位置与联动方式。

## 4. 交互方案

### 4.1 任务过滤区

把当前顶部这组：

- `active`
- `tracked`
- `completed`
- `all`

从 `join` 按钮切换成如下语义：

```html
<form class="filter">
  <input class="btn btn-square" type="reset" value="×" />
  <input class="btn" type="radio" name="task-filter" aria-label="进行中" />
  <input class="btn" type="radio" name="task-filter" aria-label="追踪中" />
  <input class="btn" type="radio" name="task-filter" aria-label="已完成" />
  <input class="btn" type="radio" name="task-filter" aria-label="全部" />
</form>
```

实现约束：

1. 保持四类过滤语义不变。
2. `reset` 用来回到默认筛选。
3. 默认筛选仍然优先落在 `active`。

### 4.2 列表区

列表区保留，但增加两个能力：

1. 右上角增加 `新建任务` 按钮。
2. 点击任务项后打开编辑卡，而不是仅切换下方只读详情。

列表项仍保留：

1. 标题
2. 状态摘要
3. 当前追踪标识
4. 完成态/顺序号

### 4.3 编辑卡

旧的“任务详情”区域移除，改为“任务编辑卡”。

编辑卡形态：

1. 使用独立弹层或浮层卡片。
2. 尺寸为窗口的约 `90%` 宽高。
3. 内部允许滚动。
4. 新建任务与编辑任务共用同一组件。

编辑卡建议包含字段：

1. 标题
2. 起因
3. 目标
4. 流程
5. 状态摘要
6. Todo 列表
7. 阶段键
8. 追加笔记
9. 触发器：
   - `runAtLocal`
   - `everyMinutes`
   - `endAtLocal`

编辑卡底部动作：

1. `取消`
2. `保存`
3. 若是已有任务，可保留 `完成任务` 入口

### 4.4 未保存改动处理

编辑卡需要显式处理未保存改动，避免用户误关后丢数据。

规则如下：

1. 表单打开时记录一份 `initial snapshot`。
2. 当前表单值与初始快照不一致时，视为 `dirty=true`。
3. 当用户执行以下动作时，若 `dirty=true`，弹出确认提示：
   - 点击 `取消`
   - 点击编辑卡关闭按钮
   - 点击遮罩关闭编辑卡
   - 从“当前编辑任务”切换到另一条任务
   - 从“编辑已有任务”切到“新建任务”
4. 确认提示文案明确说明“未保存更改将丢失”。
5. 本轮不做自动保存草稿，避免在本地引入一套临时任务草稿语义，先保证保存边界清晰。

## 5. 技术方案

### 5.1 前端组件结构

以 `src/features/config/views/config-tabs/TaskTab.vue` 为主改造，必要时拆出：

1. `TaskEditorCard.vue`
2. `task-editor` 相关本地表单类型/helper

目标是把当前单文件里：

1. 列表逻辑
2. 详情渲染
3. 新建/编辑表单

分成更清晰的职责。

### 5.2 数据接线

复用现有后端命令：

1. `task_list_tasks`
2. `task_get_task`
3. `task_create_task`
4. `task_update_task`
5. `task_complete_task`
6. `task_list_run_logs`

前端新增本地表单态：

1. `create` 模式
2. `edit` 模式
3. dirty 状态
4. 保存中状态
5. 加载中状态
6. 错误态

错误处理策略：

1. 创建任务失败
   - 在编辑卡顶部展示错误提示。
   - 保留当前表单输入，不自动清空。
   - 不关闭编辑卡，允许用户继续修改后重试。
2. 更新任务失败
   - 在编辑卡顶部展示错误提示。
   - 保留当前未保存内容。
   - 列表不做乐观更新，避免前端先改成功、后端失败造成不一致。
3. 加载任务详情失败
   - 编辑卡进入错误态，展示“重新加载”入口。
   - 保留任务列表，不清空当前列表数据。
   - 若是从列表点击进入失败，不覆盖列表已有数据，只阻止进入可编辑状态。
4. 列表加载失败
   - 面板顶部展示错误提示。
   - 保留上一次已加载成功的列表快照；若没有历史快照，则展示空态 + 重试按钮。
5. 调用失败重试
   - `task_list_tasks`、`task_get_task` 提供显式“重试/刷新”入口。
   - `task_create_task`、`task_update_task` 不做自动重试，避免重复提交造成重复任务或覆盖错误。
6. 网络/桥接错误
   - 统一按“调用失败”处理，文案允许展示原始错误文本。
   - 若是 Tauri invoke、后端校验失败或运行时异常，都走同一错误提示区，不做静默吞错。
7. 乐观更新策略
   - 本轮不采用 create/update 的乐观更新。
   - 仅在后端成功返回后回写列表与编辑卡。
   - 因此不需要额外回滚逻辑，失败时保持现有列表与表单态不变。

### 5.3 表单映射

需要增加显式的前端表单模型，避免直接把 `TaskEntry` 当作可编辑对象。

建议拆成：

1. `TaskEditorForm`
2. `TaskEditorMode`

规则：

1. 列表返回的 `TaskEntry` 先映射到表单。
2. 保存时再映射回 `task_create_task / task_update_task` 的入参。
3. `todos` 使用多行文本和数组互转。
4. `trigger` 字段保持 `runAtLocal / everyMinutes / endAtLocal` 统一命名。

### 5.4 编辑卡状态机

建议把编辑卡状态收敛成有限状态，避免“加载中 / 保存中 / dirty / 错误”互相打架。

核心状态：

1. `closed`
2. `loading`
3. `editing:create`
4. `editing:update`
5. `saving`
6. `error`

状态规则：

1. 从列表点开已有任务时，先进入 `loading`，成功后进入 `editing:update`。
2. 点击新建任务时，直接进入 `editing:create`。
3. 保存时进入 `saving`，成功后关闭卡片并刷新列表，失败后回到原编辑态并显示错误。
4. `dirty` 不单独作为主状态，而是作为 `editing:*` 下的派生标记。
