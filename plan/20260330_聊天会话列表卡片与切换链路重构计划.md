# 20260330 聊天会话列表卡片与切换链路重构计划

## 1. 背景

当前聊天窗口的会话入口主要存在三个问题：

1. 输入区上方仍是横向会话按钮托盘，窗口较窄时可读性差。
2. 会话切换 UI、会话概览数据、前台切换逻辑混在一起，后续继续扩展会越来越难维护。
3. 现有事件链路在收到 `round_completed / round_failed / history_flushed` 后，会再次调用未归档会话列表加载，形成“事件后再拉一遍列表”的模式。

这次改造不准备继续修补旧托盘，而是直接改成“新增按钮 + 会话列表按钮 + 卡片式会话列表”的正式版方案，并同步把会话概览更新链路改成“初始化一次拉取 + 推送增量维护”。

## 2. 目标

本次改造目标如下：

1. 取消当前聊天输入区上方的横向会话工具栏。
2. 改成两个固定入口：
   - `+` 按钮：新建会话
   - `会话列表` 按钮：打开会话列表卡片
3. 点击会话列表按钮后，显示一个卡片，卡片内列出所有未归档会话。
4. 每个会话项展示：
   - 会话标题
   - 最近活跃时间
   - 主会话/当前会话状态
   - 最后 2 条消息的发言人与摘要
5. 会话列表数据不再在每次打开卡片时重新拉取。
6. 会话概览采用：
   - 初始化时一次性拉取完整快照
   - 后续依赖推送事件更新本地缓存
7. 重写会话切换链路的组织方式，但保留当前“先展示本地缓存、再回填后端快照”的无白屏体验。

## 3. 非目标

1. 本次不改归档窗口 `archives` 的信息结构。
2. 本次不改消息正文渲染、输入框、语音录制、附件上传等聊天主流程。
3. 本次不做 LLM 式“智能总结”摘要。
4. 本次摘要只基于最后 2 条真实消息做截断预览，不新增单独摘要持久化字段。
5. 本次不改会话上限语义，仍保持现有最多 8 个未归档会话。

## 4. 核心原则

### 4.1 数据只初始化拉取一次

正式版按以下原则实现：

1. 聊天窗口初始化时，拉取一次“会话概览快照”。
2. 打开/关闭会话列表卡片时，不重新拉取。
3. 后续会话标题、时间、消息数、最后两条预览、运行态等信息，统一由推送事件驱动更新本地缓存。

### 4.2 推送优先，重拉只作为恢复手段

正常路径不再做“事件来了再重新拉列表”。

只有以下场景允许重新拉取：

1. 聊天窗口冷启动
2. 明确的手动刷新入口
3. 推送丢失或状态明显失步后的恢复
4. 开发期排障需要的兜底路径

也就是说，用户日常点击会话列表按钮时，不应该再触发列表读取。

### 4.3 切换逻辑与概览逻辑分层

本次把“会话概览列表”与“前台消息切换”视为两层状态：

1. 会话概览层：
   - 管理所有未归档会话卡片数据
   - 管理当前会话 id
   - 管理卡片开关状态
2. 前台切换层：
   - 负责当前会话切换
   - 负责本地缓存首帧展示
   - 负责切换后加载快照与补消息

避免再把列表按钮、列表项数据、前台消息切换塞在一个模板和一组散落方法里。

## 5. 交互方案

### 5.1 底部入口

聊天输入区右侧动作区保持紧凑，新增两个固定按钮：

1. `+`
   - 单击后直接新建会话
   - 创建成功后立即切换到新会话
2. `会话列表`
   - 单击后打开会话列表卡片
   - 再次点击或点击外部区域关闭

说明：

1. 本次“取消会话工具栏”默认指当前输入区上方那条横向会话切换条。
2. 消息区下方现有工作区锁定条与前台人格 presence 条不在这次移除范围内，除非实现时发现它与新卡片强冲突，再单独评估。

### 5.2 会话列表卡片

卡片展示规则：

1. 以浮层卡片形式出现，锚定在“会话列表”按钮附近。
2. 卡片宽度优先适配当前聊天窗口，不做全屏抽屉。
3. 卡片支持滚动。
4. 当前会话高亮显示。
5. 主会话增加明确标记。
6. 正在后台整理上下文的会话禁用切换。
7. 创建会话达到上限时，`+` 按钮禁用并显示现有提示文案。

### 5.3 会话项内容

每个会话项展示：

1. 标题
2. 最近活跃时间
3. 工作区名称
4. 主会话标记
5. 当前会话标记
6. 运行态标记：
   - 空闲
   - 回复中
   - 整理中
7. 最后 2 条消息预览：
   - 发言人
   - 文本摘要

摘要规则：

1. 只取 `user / assistant / tool` 三类可展示消息。
2. 文本优先来自现有 `messageText` 语义。
3. 无文本但有附件时，可展示简短占位，如“[图片]”“[PDF]”“[音频]”“[附件]”。
4. 文本做单行或双行截断，不展开全文。

## 6. 数据方案

### 6.1 会话概览结构

需要为聊天窗口新增统一的“会话概览项”结构，建议包含：

1. `conversationId`
2. `title`
3. `updatedAt`
4. `lastMessageAt`
5. `messageCount`
6. `workspaceLabel`
7. `isMainConversation`
8. `runtimeState`
9. `previewMessages`

其中 `previewMessages` 固定最多 2 条，每条包含：

1. `messageId`
2. `role`
3. `speakerAgentId`
4. `speakerLabel`
5. `previewText`
6. `createdAt`

### 6.2 初始化快照接口

新增或重组一个面向聊天窗口的初始化接口，返回：

1. 当前会话 id
2. 所有未归档会话概览
3. 当前会话的首屏消息快照
4. 当前会话是否还有更多历史

目标是让聊天窗口首次进入时只拉一次，就能同时拿到：

1. 当前前台消息
2. 会话列表卡片数据

避免初始化阶段还要分别调：

1. `switch_active_conversation_snapshot`
2. `list_unarchived_conversations`
3. 工作区标签补取

### 6.3 推送更新模型

本次不采用“事件到了前端再去 invoke 拉列表”的方式。

改为后端在以下时机直接发出“会话概览更新事件”：

1. 新建会话后
2. 会话删除后
3. 用户消息刷写后
4. 助手轮次完成后
5. 助手轮次失败后
6. 历史刷写后
7. 会话被强制归档后
8. 会话运行态发生切换后

事件可选两种实现方式：

1. 发送整份最新概览快照
2. 发送单会话 patch

本次优先推荐“发送整份最新概览快照”，原因如下：

1. 当前未归档会话上限只有 8 个，数据量很小。
2. 全量快照更稳，避免前端 patch 合并出错。
3. 后端在写入 `conversations` 后即可顺手重新组装概览。
4. 可以直接替代现在事件后 `loadUnarchivedConversationListOnly()` 的做法。

事件语义补充要求：

1. 每个“会话概览更新事件”都必须携带单调递增的 `sequence` 与快照级 `lastModified` 时间戳。
2. 前端只接受 `sequence` 更大，或在 `sequence` 相同前提下 `lastModified` 更新的快照；更旧事件必须直接丢弃。
3. 后端需要对短时间内的连续变更做合并发送，采用 debounce 或窗口化 merge，只发送该窗口内最新的一份完整快照，避免单次操作触发多次重复全量快照。
4. 同一 `sequence` 对应的 payload 必须是完整且幂等的，建议同时携带 `conversationListVersion`，方便前后端统一去重。
5. 客户端重连后若收到回放的中间事件，应忽略低于当前最高 `sequence` 的快照，只应用最高序号快照；若发现序号跳变存在 gap，则必须重新拉取完整初始化快照再恢复监听。
6. 事件投递默认视为可能乱序、重复、延迟到达；前后端都要以“快照幂等 + 序号去重 + 缺口回补”作为恢复基础。

### 6.4 与现有消息推送的关系

现有聊天窗口已经监听：

1. `history_flushed`
2. `round_completed`
3. `round_failed`
4. `assistant_delta`
5. `conversation_messages_after_synced`

本次不新增轮询。

调整方向是：

1. 保留这些消息相关事件，用于正文流和补消息。
2. 新增或扩展“会话概览更新事件”，用于列表卡片数据。
3. 取消这些事件回调中的 `loadUnarchivedConversationListOnly()`。

## 7. 会话切换链路重写方案

### 7.1 保留的行为

当前切换链路有一个正确方向，需要保留：

1. 切换时先使用本地缓存立即展示
2. 首帧完成后再请求后端快照
3. 再异步补充缺失消息

这样可以避免聊天窗口切换时白屏或明显闪烁。

### 7.2 需要重写的点

本次“重写会话切换方法”不是推翻缓存优先策略，而是重写其组织方式：

1. 从 `UnifiedWindowApp.vue` 中抽离会话概览与切换职责。
2. 把“当前会话切换”“概览快照应用”“推送事件回写”“列表卡片开关”分成独立状态块。
3. 明确区分：
   - `currentChatConversationId`
   - 后端会话运行态

### 7.3 具体原则

1. 当前会话判定统一以前端 `currentChatConversationId` 为准。
2. 不依赖 `isActive` 作为唯一当前会话依据。
3. 列表项点击只负责发起切换，不直接拼装消息。
4. 切换完成后，由前端基于 `currentChatConversationId` 自行计算当前会话高亮，不从概览 payload 中读取“当前会话”字段。

## 10. 风险点

### 10.1 推送事件不够全

风险：

1. 如果只在轮次完成时发概览事件
2. 但新建、删除、归档时没发
3. 前端列表会出现失步

处理：

1. 把“会话概览更新”当成独立职责梳理一遍
2. 所有会话结构变化入口统一补事件

### 10.2 当前会话判断混乱

风险：

1. 现有后端会把多条前台聊天都标成 `active`
2. 如果新 UI 继续直接依赖 `isActive`
3. 容易高亮错会话

处理：

1. 前端当前项高亮统一由 `currentChatConversationId` 决定
2. 后端 `isActive` 仅保留为兼容字段，不作为最终高亮依据

### 10.3 会话预览构造不一致

风险：

1. 卡片摘要与聊天正文使用不同的文本提取规则
2. 用户会觉得列表摘要和实际消息不一致

处理：

1. 摘要构造尽量复用现有 `messageText` 语义
2. 对附件占位文案做统一规则

### 10.4 `UnifiedWindowApp.vue` 继续膨胀

风险：

1. 如果只往现有大文件里塞新逻辑
2. 会让这次“重写切换链路”名义成立、实际维护性更差

处理：

1. 本次默认拆出至少一个会话概览相关 composable
2. 保持入口文件只负责组装

### 10.5 初始化快照失败

风险：

1. 6.2 中的初始化快照接口可能因窗口首次打开、配置缺失、后端异常或旧版本接口不兼容而失败。
2. 如果前端把首屏渲染完全绑定在初始化快照成功上，聊天窗口可能出现空白、列表缺失或会话状态卡死。

处理：

1. 前端初始化失败时必须给出可见反馈，至少展示可重试状态，不允许静默失败。
2. 降级路径为：保留本地缓存的当前消息与 `currentChatConversationId`，同时退回旧的 `switch_active_conversation_snapshot` / `list_unarchived_conversations` 组合读取。
3. 若连降级读取也失败，则保留输入区可用并提示“会话列表加载失败，可稍后重试”。

### 10.6 WebSocket 连接断开

风险：

1. 若后续概览推送走 WebSocket 或等价长连接，连接断开会导致前端错过中间快照。
2. 连接恢复后如果直接继续应用增量或历史事件，容易把列表状态回滚到旧版本。

处理：

1. 重连成功后必须以最高 `sequence` 为基准做状态重同步，不应用重连前缓存的中间事件。
2. 若无法确认连续性，必须重新请求完整概览快照，再恢复本地监听。
3. 重连期间前端继续允许用户查看当前会话，但要将列表视为“可能过期”，并在同步完成后一次性替换为最新快照。

### 10.7 推送事件与用户操作的竞态

风险：

1. 用户连续快速切换会话时，推送事件可能晚于本地切换结果到达。
2. 如果事件里的高亮或活跃态覆盖本地状态，`currentChatConversationId` 与列表展示会短暂错位。
3. `isActive` 仅表示后端活跃态，不应反向覆盖当前前台会话。

处理：

1. 前端始终以 `currentChatConversationId` 作为当前高亮和前台消息展示的最终依据。
2. 推送事件只更新概览列表、运行态和预览内容，不直接改写当前前台选择，除非当前会话已不存在且需要恢复候选会话。
3. 若用户本地切换与推送事件并发到达，应按“本地切换结果 > 同时刻推送高亮建议 > 后端 `isActive`”的优先级合并。

### 10.8 部署期间的兼容性

风险：

1. 部署期间可能出现前端已升级但后端尚未提供概览推送事件，或后端已升级但前端仍走旧读取逻辑。
2. 混合版本下若新前端强依赖推送，列表可能永远不更新。

处理：

1. `UnifiedWindowApp.vue` 或后续 composable 必须把概览推送视为“可选能力”，注册失败或长期无事件时自动回退到主动拉取。
2. 若后端不支持新事件，前端继续使用初始化快照 + 关键操作后重拉，保证功能可用。
3. 新后端必须保持对旧前端可兼容，不因新增概览推送改变现有消息事件含义。

## 11. 验证

### 11.1 前端

1. `pnpm typecheck`
2. 打开聊天窗口时只初始化拉取一次会话概览快照
3. 反复打开/关闭会话列表卡片，不触发重新读取会话列表
4. 新建会话后列表立即更新，无需手动刷新
5. 新消息到达后，当前会话与非当前会话的卡片预览都能自动更新
6. 切换会话时仍保持缓存首帧体验
7. 正在整理上下文的会话在卡片中不可切换
8. 初始化快照失败时，前端能展示错误或降级提示，并允许重试；失败后界面不可出现永久空白。
9. 丢失或延迟一条概览推送事件后，列表最终仍能与最新快照保持一致；过期事件不会覆盖较新状态。
10. 连续快速切换多个会话时，最终高亮必须与 `currentChatConversationId` 一致，不允许回跳到旧会话。
11. 重连或重新进入聊天窗口后，前端能基于最高 `sequence` 恢复最新概览；若检测到序号缺口，会主动重拉完整快照。

### 11.2 后端

1. `cd src-tauri && cargo check`
2. 会话概览快照能正确返回最近两条预览
3. 创建、删除、归档、轮次完成、轮次失败后都能发送概览更新事件
4. 概览项中的主会话标记、时间、消息数正确
5. 并发创建/删除会话时，概览事件 `sequence` 严格递增，较旧快照不会在较新快照之后发送。
6. 概览快照构造性能可观测，最近两条 `previewMessages` 提取在目标会话上限下满足既定耗时阈值。
7. 推送失败时会记录可排障日志，并在下次状态变化或重连时允许客户端通过完整快照恢复；不得因单次推送失败破坏状态写入。

### 11.3 集成测试

1. 新前端 + 旧后端时，聊天窗口仍能通过降级拉取显示列表与当前会话；缺少概览推送不应导致功能不可用。
2. 旧前端 + 新后端时，现有消息流与会话切换行为保持兼容，不因新增概览事件产生异常。
3. 多窗口同时打开时，任一窗口中新建、删除、归档会话后，其他窗口最终都能收敛到同一份最新概览快照。
4. 多设备或多实例场景下，晚到、重复、乱序事件不会导致当前会话高亮与预览内容长期不一致。

## 12. 实施顺序

1. 先定义会话概览与预览消息类型
2. 再实现聊天窗口初始化快照接口
3. 再补会话概览更新事件，替换现有事件后的列表重拉
4. 再拆分前端会话概览与切换逻辑
5. 最后重做 `ChatView` 的按钮与卡片 UI，并补验证

## 13. 结果预期

改造完成后，聊天窗口的会话交互应满足：

1. 用户进入窗口时只初始化一次
2. 平时打开会话列表是纯本地展示，不等待请求
3. 新消息来了，卡片数据自动更新
4. 切换会话依然快速，不因为重构损失现有首帧体验
5. 后续若要继续扩展会话搜索、置顶、删除入口，也有清晰的数据层可以承接
