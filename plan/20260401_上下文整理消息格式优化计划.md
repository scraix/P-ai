## 背景

本轮围绕“上下文整理”链路做的，不只是正文样式美化，而是把压缩消息格式、压缩提示词、提示词组装边界、媒体处理、前端刷新时机、以及自动压缩触发时序一起收口到同一套设计语义。

本文件已按当前代码实现修正，不再保留旧的“回复后再压缩”口径。

## 用户确认后的设计约束

- 归档与压缩是分离的独立能力，压缩链路里不能混入归档式 fallback 文案。
- 压缩失败时允许摘要正文为空，不做多份旧摘要拼接，不再生成 `C1/C2/C3` 一类降级内容。
- 压缩消息正文固定为四段，顺序必须是：
  1. 用户画像
  2. 摘要说明
  3. 摘要正文
  4. 保留对话
- 不要“补充信息”等额外段落。
- 摘要正文要按“交接摘要 / handoff summary”口径生成，帮助下一个模型继续工作。
- 摘要正文里的对话摘要要带人格昵称，不能退回通用 `assistant`。
- 保留对话不能按固定条数硬截，必须按 token 预算向前回收，并跳过旧压缩消息与工具噪音。
- SummaryContext 请求必须是纯文本，不携带图片/音频正文，不允许把历史媒体 base64 带进压缩请求。
- 会话中允许存在多条旧压缩消息，但每次 prompt 组装只能从“最后一条压缩消息”开始。
- 自动压缩只能发生在“下一次 LLM 请求发出之前”：
  - 用户消息发出前可以检查
  - 工具回合结束、准备续调模型前可以检查
  - 回复生成后不允许再补一次压缩
  - `tool_call -> tool_result` 中间绝不允许插入压缩
- 压缩完成后属于“消息序列重写”，前端不能只追加消息，必须按强制刷新语义重载。
- `deepseek` 旧协议链路已废弃，本次不继续为其补齐新时序。

## 实际实现范围

### 1. 压缩消息正文格式已改为四段式

已在 `src-tauri/src/features/system/commands/archive_pipeline.rs` 收口为固定四段：

1. 用户画像
2. 摘要说明
3. 摘要正文
4. 保留对话

同时移除了不需要的触发原因、补充信息等可见噪音字段，保留原有 `context_compaction` / `summary_context_seed` 识别方式不变。

### 2. 压缩提示词已切换为中文版交接摘要口径

已将压缩提示词收口为中文版 “CONTEXT CHECKPOINT COMPACTION / handoff summary” 语义，要求模型输出：

- 当前进展与关键决策
- 重要上下文、约束与用户偏好
- 剩余待办与清晰下一步
- 继续工作需要的关键数据、示例或引用

实现上优先原位替换现有提示词模板，没有新增一层独立提示词系统。

### 3. SummaryContext 请求已改为纯文本

已在 `src-tauri/src/features/system/commands/prompt_assembly.rs` 处理 SummaryContext 模式下的媒体剥离：

- 历史图片/音频不再进入压缩请求
- 最新图片/音频不再进入压缩请求

目的：

- 避免非流式压缩请求携带大体积 base64
- 避免再次出现 `base64 decode fail`
- 维持“持久层存引用，压缩请求只吃文本”的设计

### 4. Prompt 组装边界已修正为“最后一条压缩消息”

已在 `src-tauri/src/features/chat/conversation.rs` 修正上下文边界逻辑：

- 会话内可以存在多条旧压缩消息
- prompt 组装时只从最后一条压缩消息开始
- 最终语义是：`最后一条压缩消息 + 其后消息`
- 不再把更早的旧压缩消息一起拼进去

### 5. 保留对话已改为 token 预算回收

已在 `src-tauri/src/features/system/commands/archive_pipeline.rs` 改成按 token 预算向前保留最近原始对话，当前预算为 `10_000`。

当前规则：

- 向前逐条累加，直到预算上限
- 跳过旧压缩消息
- 跳过工具噪音
- 保留原始换行，不再压扁为单行

### 6. 压缩失败不再做错误的归档式 fallback

已在 `src-tauri/src/features/system/commands/archive_pipeline.rs` 去掉错误实现：

- 不再把多份旧摘要拼接成新的压缩正文
- 不再把归档逻辑混进压缩逻辑
- 压缩失败时摘要正文允许为空字符串

### 7. 前端已按“消息序列重写”处理压缩结果

已完成两部分前端收口：

- `src/features/chat/composables/use-chat-flow.ts`
  - 将 `history_flushed.compactionApplied` 视为强制重置事件
  - 清理草稿 / 工具态 / 旧序列缓存后再重载
- `src/features/chat/views/ChatView.vue`
  - 增加“正在整理上下文...”独立状态提示
  - 过滤 `organize_context` 通用工具气泡，避免与整理横幅重复

目标是解决：

- 压缩后前端仍停留在旧消息序列
- 气泡显示“已压缩成功”，但页面仍卡在整理中
- 压缩消息插入后列表未按新序列刷新

### 8. 自动压缩触发时序已改为“请求前检查”

已按最新实现把时序收口为：

- 用户消息发给模型前检查
- 工具回合结束后，下一次续调模型前检查
- 回复生成后不再立即补压缩

具体落点：

- `src-tauri/src/features/system/commands/chat_and_runtime/core_send_inner.rs`
  - 保留原有 pre-send 检查
  - 删除“回复后 82% 立刻压缩”旧路径
- `src-tauri/src/features/chat/model_runtime/provider_and_stream/tool_loop.rs`
  - 统一工具循环在下一轮模型请求前检查是否需要压缩
  - 若触发压缩，则先整理，再基于最新会话和本轮暂存 `tool_result` 重建 prompt 后续调
- `src-tauri/src/features/chat/model_runtime/provider_and_stream/router.rs`
- `src-tauri/src/features/chat/model_runtime/provider_and_stream/openai_style.rs`
- `src-tauri/src/features/chat/model_runtime/provider_and_stream/gemini.rs`
- `src-tauri/src/features/chat/model_runtime/provider_and_stream/anthropic.rs`
  - 统一透传压缩检查所需上下文

当前语义已经与设计一致：

- 只在“下一次 LLM 请求前”检查
- 不在回复后补压缩
- 不在工具执行中间压缩
