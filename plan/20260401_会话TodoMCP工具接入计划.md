# 20260401 会话 Todo MCP 工具接入计划

## 1. 背景

当前项目里已经有一个持久化的 `task` 工具和任务板体系，适合跨会话、可调度、可归档的任务推进。

但你这次要的是另一类能力：

1. 更像通用编码助手里的会话内 Todo 追踪器。
2. 生命周期只在当前聊天/当前运行期内有效。
3. 核心目标不是“定时执行任务”，而是“帮助模型在复杂任务里维持步骤感、逐项推进、及时更新状态”。

仓库当前在运行时工具装配里已经明确写了约束：

1. 新工具优先实现为 MCP 形态。
2. 前端工具目录也会单独读取内建 MCP schema。
3. 聊天提示词支持插入隐藏 preamble / extra block。

因此本次最合适的路径不是继续扩展现有持久化 `task`，而是新增一个独立的、会话内的 `todo` MCP 工具。

## 2. 目标

### 2.1 功能目标

1. 新增一个内建 MCP 工具 `todo`。
2. 该工具以“全量覆盖”方式更新当前会话 Todo 列表，而不是增量 patch。
3. Todo 数据写入当前 `Conversation` 的专门字段，随会话一起持久化。
4. Todo 列表按当前会话隔离，主会话与委托线程互不干扰。
5. 当所有 Todo 都是 `completed` 时，自动清空当前会话的 Todo 字段。
6. 在聊天提示词中注入 Todo 使用规则与当前 Todo 快照，促使模型主动、正确使用。
7. 将该工具接入：
   - 运行时工具装配
   - 前端工具目录
   - 默认工具开关配置
   - 必要测试

### 2.2 非目标

1. 本次不替代现有持久化 `task` 工具。
2. 本次不做独立 Todo 面板 UI。
4. 本次不做多人共享待办或远程同步。
5. 本次不把 Todo 与任务调度、提醒、自动触发打通。

## 3. 产品定义

### 3.1 工具定位

`todo` 是“会话内工作记事板”，不是长期任务系统。

它适合：

1. 当前轮请求包含多个子任务。
2. 需要分步骤推进的复杂改动。
3. 模型需要在执行过程中持续维护“下一步做什么”。

它不适合：

1. 单个简单动作。
2. 纯解释、纯闲聊。
3. 需要跨会话保留的任务。

### 3.2 输入协议

首版协议与参考设计保持一致，只保留最小字段：

```json
{
  "todos": [
    {
      "content": "Run tests",
      "status": "pending"
    }
  ]
}
```

字段规则：

1. `todos` 必填，表示本次提交后的完整 Todo 列表。
2. `content` 必填，非空字符串。
3. `todos` 数量建议优先保持在 `3~7` 之间，但不做强制限制。
4. `status` 必填，枚举值：
   - `pending`
   - `in_progress`
   - `completed`

### 3.3 核心机制

1. 每次调用都提交完整列表。
2. 服务端直接覆盖当前会话旧列表。
3. Todo 总步数建议优先保持在 `3~7` 步，避免过少失去追踪意义，或过多导致列表失控。
4. 同一时刻最多只允许一个 `in_progress`。
5. 若所有项均为 `completed`，则自动清空当前会话 Todo。
6. 不相关或已经失效的步骤应从列表中移除，不保留历史垃圾项。

### 3.4 会话存储策略

Todo 状态建议直接挂在 `Conversation` 上，而不是额外做一份运行时内存表：

1. 当前会话本来就是提示词组装、续聊恢复和持久化的主对象。
2. 主会话与委托线程本来就是不同 `Conversation`，天然隔离。
3. 这样可以避免 “MCP 更新了一份临时状态，但会话本体看不到” 的双写问题。
4. 应用重启后，当前会话 Todo 仍能保留，续聊时也能直接恢复。

建议新增会话字段：

1. Rust `Conversation`
   - `current_todos: Vec<ConversationTodoItem>`
2. 前端对应类型
   - `currentTodos?: ConversationTodoItem[]`

其中 `ConversationTodoItem` 建议结构：

1. `content: String`
2. `status: "pending" | "in_progress" | "completed"`

MCP `todo` 工具只负责：

1. 根据当前 `tool_session_id` 解析出目标 `conversation_id`
2. 全量覆盖该 `Conversation.current_todos`
3. 当全部完成时自动写成空数组

## 4. 提示词机制

本次不仅要接工具，还要接“什么时候该用、什么时候不要用”的引导。

### 4.1 规则目标

模型应当理解：

1. 复杂任务时主动建 Todo。
2. Todo 只在确实需要时使用，且总步数优先维持在 `3~7` 步。
3. 开始做某项前先把它标记为 `in_progress`。
4. 做完一项立刻改成 `completed`。
5. 任一时刻只能有一个 `in_progress`。
6. 简单问答不要滥用 Todo。

### 4.2 提示词内容范围

建议以隐藏 block 注入，至少覆盖以下内容：

1. 何时必须使用 Todo：
   - 至少 3 个不同步骤的复杂任务
   - 用户一次提出多个要求
   - 用户明确要求使用 Todo
   - 执行过程中已经出现明显的多阶段推进需求
2. 何时不要使用 Todo：
   - 单个简单动作
   - 纯解释或纯对话
   - 追踪收益明显低于维护成本的琐碎任务
3. 状态规则：
   - `pending`
   - `in_progress`
   - `completed`
   - 同时只能一个 `in_progress`
4. 更新规则：
   - 做前更新
   - 做完即改
   - 总步数优先保持在 3~7 步
   - 新发现的后续事项要补进列表
   - 无关事项从列表移除
5. 描述要求：
   - `content` 用祈使句
   - 保持短、明确、可执行
6. 行为倾向：
   - 拿不准时，复杂任务宁可主动使用

### 4.3 Todo 系统提示词

除了运行时快照，还需要在系统提示词中固定写入一段专门的 Todo 规则说明。

这段提示词的目标不是介绍工具 schema，而是约束模型的使用习惯。

建议内容至少包括：

1. `todo` 是当前复杂任务的步骤追踪器，不是长期任务系统。
2. 只有复杂任务、多要求任务、或用户明确要求时才使用。
3. 一旦使用，步骤数建议优先保持在 `3~7` 步。
4. 每次真正开始某步前，先把该步改为 `in_progress`。
5. 任一时刻只允许一个 `in_progress`。
6. 完成某步后立刻更新状态。
7. 如果全部完成，不要继续制造新步骤，而是直接向用户汇报。
8. 如果步骤已经无关或走偏，要及时删除或改写，而不是一直堆着。

提示词形态建议：

1. 写入系统提示词固定 preamble。
2. 与 `todo board` 分开：
   - `todo guide` 负责规则
   - `todo board` 负责当前状态
3. 即使当前还没有 Todo，也要让模型先知道这套规则。

### 4.4 当前 Todo 快照注入

每轮发送前，若当前会话已有 Todo，建议把快照一并注入提示词，例如：

```xml
<todo board>
activeTodoCount: 3
todo[1].content: Inspect MCP runtime attach path
todo[1].status: completed
todo[2].content: Add todo MCP server
todo[2].status: in_progress
todo[3].content: Run cargo check
todo[3].status: pending
</todo board>
```

这样模型在续做任务时能看到自己的当前进度，而不是只依赖对话回忆。

### 4.5 上下文压缩摘要中的 Todo 段落

除了聊天时的隐藏 `todo board` 注入，还需要同步接入上下文压缩链路。

目标是：

1. 当系统生成压缩摘要时，不要只保留记忆与历史结论。
2. 如果当前会话存在 Todo，需要在压缩摘要里显式保留当前待办。
3. 这样压缩后恢复上下文时，模型仍能直接看到“当前做到哪一步”。

建议格式：

```md
## Current Todo List

- [completed] Inspect MCP runtime attach path
- [in_progress] Add todo MCP server
- [pending] Run cargo check
```

放置位置建议：

1. 在压缩摘要里，位于记忆相关段落之后。
2. 若当前无 Todo，则不输出该段。
3. 该段内容只反映当前会话的 `current_todos`，不混入持久化 `task`。

## 5. Rust 实现设计

### 5.1 新增 MCP Server

建议新增文件：

- `src-tauri/src/features/system/tools/todo_mcp.rs`

建议内容：

1. 常量：
   - `MCP_TODO_SERVER_FLAG`
   - `MCP_TODO_TOOL_NAME`
   - `MCP_TODO_SESSION_FLAG`
2. 数据结构：
   - `TodoItem`
   - `TodoWriteRequest`
   - `TodoWriteResponse`
3. 会话字段读写 helper：
   - `conversation_todo_list(...)`
   - `conversation_todo_replace(...)`
   - `conversation_todo_clear(...)`
4. 校验逻辑：
   - `content` 非空
   - `status` 合法
   - `in_progress` 数量 <= 1
5. 工具执行逻辑：
   - 接收完整列表
   - 规范化
   - 定位当前会话
   - 覆盖会话字段
   - 自动清空全完成列表
   - 返回更新后的快照

### 5.2 主程序入口

建议同步修改：

- `src-tauri/src/main.rs`
- `src-tauri/src/features/system/tools.rs`

目标：

1. `include!` 新的 `todo_mcp.rs`
2. 在启动参数分支里新增 `--mcp-todo-server`
3. 允许主程序以子进程方式启动 Todo MCP server

### 5.3 运行时工具装配

建议同步修改：

- `src-tauri/src/features/chat/model_runtime/provider_and_stream/tool_assembly.rs`
- `src-tauri/src/features/chat/model_runtime/provider_and_stream/types.rs`

目标：

1. 像 `read_file` / `operate` 一样，把 `todo` 当作内建 MCP 挂载。
2. 为运行时保存 `TodoMcpClient`，保证 tool loop 生命周期内连接不被提前释放。
3. 在 `tool_manifest` 中记录：
   - 已启用且挂载成功
   - 已启用但挂载失败
   - 未启用

### 5.4 前端工具目录与默认开关

建议同步修改：

- `src-tauri/src/features/system/commands/chat_and_runtime/tool_catalog.rs`
- `src/features/config/utils/builtin-tools.ts`

目标：

1. 前端工具目录能读到 `todo` 的 MCP schema。
2. `todo` 进入默认工具配置。
3. 默认建议 `enabled: true`，因为它是低风险、轻量级状态工具。

### 5.5 提示词注入

建议同步修改：

- `src-tauri/src/features/system/commands/chat_and_runtime/core_send_inner.rs`
- 可能补充 helper 于：
  - `src-tauri/src/features/system/commands/prompt_assembly.rs`
  - 或 `src-tauri/src/features/system/tools/todo_mcp.rs` 附近的共享函数

建议做两类 block：

1. `todo guide`
   - 固定规则说明
2. `todo board`
   - 当前会话快照

注入策略建议：

1. `todo guide` 作为系统提示词固定段接入，而不是临时弱提示
2. 仅在当前会话存在 Todo 时注入 `todo board`
3. 对委托线程同样生效，但只展示各自会话 Todo
4. `todo guide` 中要明确写入“优先保持 3~7 步”与“全部完成后直接汇报”

### 5.6 上下文压缩与摘要链路

建议同步检查并修改上下文压缩/摘要生成相关代码，确保：

1. 压缩摘要构造时可以读取 `Conversation.current_todos`
2. 在记忆段落后追加 `## Current Todo List`
3. Todo 项按当前顺序输出，包含状态与内容
4. 若当前无 Todo，则不追加该段

这样无论会话是否发生上下文压缩，当前 Todo 都能继续留在后续 prompt 可见范围内。

## 6. 与现有 task 的关系

必须明确区分：

1. `task`
   - 持久化
   - 可调度
   - 有时间触发器
   - 面向“长期任务管理”
2. `todo`
   - 绑定当前会话
   - 不调度
   - 无时间字段
   - 面向“当前复杂工作拆解与推进”

首版不做两者自动互转，避免语义混淆。

## 7. 返回结构建议

这里不应该只停留在“返回存了什么 Todo”，而应该直接服务下一步执行决策。

更合适的原则是：

1. 如果还有未完成项，返回“下一步该做什么”。
2. 如果全部完成，返回“应开始汇报/结束当前工具链”。
3. 同时保留完整快照，便于调试和前端展示。

因此建议 `todo` 工具返回“状态快照 + 下一步信号”，而不是只有一句成功：

```json
{
  "sessionId": "api::conversation",
  "allCompleted": false,
  "shouldReport": false,
  "nextTodo": {
    "content": "Add todo MCP server",
    "status": "in_progress"
  },
  "todos": [
    { "content": "Inspect MCP runtime attach path", "status": "completed" },
    { "content": "Add todo MCP server", "status": "in_progress" },
    { "content": "Run cargo check", "status": "pending" }
  ]
}
```

当全部完成后：

```json
{
  "sessionId": "api::conversation",
  "allCompleted": true,
  "shouldReport": true,
  "nextTodo": null,
  "todos": []
}
```

`nextTodo` 的选择规则建议明确为：

1. 先取唯一的 `in_progress`
2. 若没有 `in_progress`，取第一个 `pending`
3. 若没有未完成项，则返回 `null`

`shouldReport` 的语义建议明确为：

1. `false`：说明工具调用后应继续执行 `nextTodo`
2. `true`：说明当前 Todo 已全部完成，模型应停止继续追加执行，转而向用户汇报结果

## 8. 测试计划

### 8.1 Rust 单测

建议至少覆盖：

1. 合法列表可写入并覆盖旧值。
2. 两个 `in_progress` 会报错。
3. 空 `content` 会报错。
4. 全 `completed` 时自动清空。
5. 不同 `session_id` 互不影响。

### 8.2 工具目录测试

建议补充或扩展现有前端工具目录测试，确保：

1. `todo` 前端目录定义与运行时 MCP schema 一致。
2. 不会出现目录与运行时 schema 漂移。

### 8.3 集成验证

至少手动验证：

1. 启用 `todo` 后，模型能看到工具。
2. 复杂请求下模型能主动创建 Todo。
3. 更新 Todo 时不会影响其他会话。
4. 委托线程中的 Todo 不污染主线程。

## 9. 风险与注意点

### 9.1 提示词过强导致滥用

如果“拿不准就用”写得太重，模型可能在简单任务里也频繁写 Todo。

处理策略：

1. 明确保留“不该用”的负例。
2. 把“复杂任务优先使用”写成主规则。
3. 用“优先保持 3~7 步”的提示避免无限膨胀。
4. 避免把 Todo 描述成所有任务都必须先走的流程。

### 9.2 与持久化 task 混淆

如果描述不清，模型可能把短期 Todo 当成长任务来创建，或反过来。

处理策略：

1. 在 `todo guide` 中显式对比 `todo` 与 `task`。
2. `todo` schema 不出现时间字段。
3. `task` 与 `todo` 的 description 各自强调边界。

### 9.3 会话状态边界

由于 Todo 绑定在 `Conversation` 上：

1. 同一会话内可跨重启保留。
2. 新开会话默认不会继承旧会话 Todo。
3. 委托线程只应看到自己会话的 Todo，不应读写主线程 Todo。

因此实现里要严控 `tool_session_id -> conversation_id` 的解析与写入边界。

## 11. 实施顺序

1. 先给 `Conversation` 增加 Todo 字段与读写 helper。
2. 再实现 `todo_mcp` server，以 MCP 形式更新会话字段。
3. 再接主程序启动参数与运行时挂载。
4. 然后接前端工具目录与默认工具配置。
5. 再注入提示词 guide + board。
6. 再把 Todo 接入上下文压缩摘要，在记忆段落后追加 `## Current Todo List`。
7. 最后补测试并跑：
   - `cd src-tauri && cargo check`
   - 相关 Rust tests

## 12. 验收标准

1. 模型在复杂任务中可调用 `todo` 维护当前步骤。
2. `todo` 数据绑定在当前会话字段内，并且不会串到其他会话。
3. 所有 Todo 完成后会自动清空。
4. 前端工具目录可展示 `todo` schema。
5. 上下文压缩后生成的摘要会在记忆段落后追加 `## Current Todo List`。
6. 相关编译与测试通过。
