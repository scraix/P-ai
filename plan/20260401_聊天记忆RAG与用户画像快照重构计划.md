# 20260401 聊天记忆RAG与用户画像快照重构计划

## 1. 说明

本计划基于当前暂存实现反向整理，重点描述聊天主链路这次到底发生了什么变化。

这份文档只看两件事：

1. 当前轮记忆召回现在如何进 prompt。
2. 会话为什么开始带用户画像快照与初始 SummaryContext 消息。

## 2. 背景

旧链路的主要问题是：

1. 记忆命中后会尽早被展开成完整文本块，容易在持久化层和 prompt 层耦合过深。
2. 当前轮命中记忆和历史轮命中记忆之间缺少统一去重策略。
3. 新会话或归档后新前台会话太“空”，只能靠模型自己从零恢复上下文。

当前暂存实现已经把这三点重排为：

1. 先存“命中了哪些记忆 ID”
2. 后在 prompt 组装阶段决定怎么注入
3. 会话对象自身保存用户画像快照，必要时用 SummaryContext 种子消息启动

## 3. 目标

这次重构后的实际目标可以概括为：

1. 记忆召回结果先落 ID，不在消息写入阶段展开完整记忆块。
2. prompt 注入改成按消息逐条读取 `retrieved_memory_ids`，并做请求内去重。
3. 会话对象显式保存 `user_profile_snapshot`。
4. 前台聊天会话在创建和归档切换时自动带入“可继续聊天”的最小背景。

## 4. 当前方案

### 4.1 会话模型新增画像快照

`Conversation` 结构新增：

1. `user_profile_snapshot`

它的用途不是保存完整画像数据库，而是保存“本会话当前可直接拼进上下文的画像快照文本”。

### 4.2 新前台会话不再纯空白

当前前台聊天会话已经统一改走：

1. `build_foreground_chat_conversation_record(...)`
2. `ensure_active_foreground_conversation_index_atomic(...)`

创建时会自动：

1. 读取当前 agent 可见的画像快照
2. 找到最近一次非委托归档摘要
3. 生成一条初始 `[上下文整理]` 种子消息
4. 把这条消息作为新会话第一条用户消息写进去

所以现在的“活动会话兜底创建”语义已经从“空壳会话”变成“带种子上下文的会话”。

### 4.3 当前轮记忆召回改成先写 ID

发送消息时，当前实现会：

1. 从用户文本部分提取 recall query
2. 根据当前 agent 可见记忆做检索
3. 取得命中 ID
4. 只把这些 ID 写进 `provider_meta.retrieved_memory_ids`

它不再在这里直接把完整 memory board 文本塞进 `extra_text_blocks`。

### 4.4 当前轮会话也记录 recall table

虽然 prompt 注入被延后，但当前实现仍会把命中的真实 `memory_id` 写入：

1. `conversation.memory_recall_table`

这使得后续 SummaryContext 仍能知道“本次会话实际用过哪些记忆”，便于做 useful feedback、merge 和画像提炼。

### 4.5 后台队列写消息时同样补 recall IDs

`chat/scheduler.rs` 中，后台队列批量写入用户消息时也同步做了同样的动作：

1. 识别用户消息
2. 计算 recall ids
3. 写入 `provider_meta.retrieved_memory_ids`
4. 累加到 `conversation.memory_recall_table`

这保证了：

1. 前台直接发消息
2. 后台恢复队列写消息
3. 远程入口转进来的用户消息

这些路径最终会形成同一套记忆元数据形状。

### 4.6 Prompt 注入改为按消息延迟展开

当前 prompt 组装已经变成：

1. 先读取对当前 agent 可见的所有记忆
2. 对每条用户消息读 `retrieved_memory_ids`
3. 按请求内 `seen_memory_ids` 去重
4. 再把这些 ID 转成 `<memory_context>` 文本块

重点变化有两点：

1. 历史轮与当前轮都能按同一规则注入
2. 同一条记忆在同一次请求中不会被重复注入多次

### 4.7 `<memory_context>` 取代旧 MemoryBoard 包装

记忆块外层包装已由旧的：

1. `<system-reminder>`
2. `[MemoryBoard]`

变为新的：

1. `<memory_context>`
2. `<id:短ID或真实ID> ... </id:短ID或真实ID>`

这带来三个变化：

1. 语义上更像“检索出的背景上下文”，而不是高优先级系统提醒。
2. 每条记忆可带显示 ID。
3. SummaryContext 和普通聊天都能更自然地引用短 ID。

### 4.8 Prompt 前言新增 Memory RAG 规则

系统前言新增专门规则，明确要求模型：

1. 把 `<memory_context>` 当成系统检索到的背景，不是用户当前发言。
2. 仅在有帮助时自然参考。
3. 不要暴露注入机制。
4. 若和当前用户消息冲突，以当前消息为准。

这说明当前实现不只改了数据注入形状，也补了语义防护。

### 4.9 历史消息结构新增 `extra_text_blocks`

`PreparedHistoryMessage` 新增：

1. `extra_text_blocks`

意义是：

1. 历史用户消息正文和附加上下文块被明确拆开。
2. provider 适配层、调试 JSON、token 估算都会显式处理这些附加块。
3. 记忆上下文不再必须揉进 `text` 字段里。

### 4.10 最新用户消息与历史用户消息的注入边界

当前实现对“最新用户消息”和“历史用户消息”采用一致规则，但落点不同：

1. 历史用户消息的记忆块进入 `PreparedHistoryMessage.extra_text_blocks`
2. 最新用户消息的记忆块进入 `prepared.latest_user_extra_text`

这样能保证 provider 侧看到的结构仍然清晰：

1. 历史轮保留独立消息块
2. 当前轮继续使用 latest user payload 机制

### 4.11 画像快照的补种策略

当前实现会在多个关键点补种 `user_profile_snapshot`：

1. 新建前台会话时
2. 首次发送聊天消息时若发现为空
3. 后台队列处理会话时若发现为空
4. 上下文压缩完成后重新刷新快照
5. 归档后创建新前台会话时重新注入快照

这意味着画像快照已经成为会话生命周期的一部分，而不是只在归档时偶发使用。

## 5. 非目标

当前暂存代码没有体现以下目标，因此不将其写成已完成能力：

1. 本次不做附件内容本身的记忆检索。
2. 本次 recall query 仍以用户文本为主，不把图片或音频直接纳入召回文本。
3. 本次不把完整用户画像板长期写入会话消息历史。
4. 本次不为委托会话和远程联系人会话强制注入完整画像快照。

## 7. 风险点

### 7.1 ID 写入成功但 prompt 注入失败

风险：

1. 消息里已经有 `retrieved_memory_ids`
2. 但 prompt 组装时若可见记忆读取失败，就会丢失注入

处理：

1. 当前实现已做读取失败日志。
2. SummaryContext 与普通聊天都允许在缺记忆块的情况下继续走主流程。

### 7.2 recall query 过于依赖纯文本

风险：

1. 附件或图片信息未进入 recall query 时，召回可能偏弱。

处理：

1. 当前实现优先保证规则清晰和稳定。
2. 后续若要扩展多模态召回，应独立设计，而不是把现在的文字链路继续堆复杂度。

### 7.3 种子消息过多导致用户困惑

风险：

1. 会话第一条消息就是 `[上下文整理]`，如果 UI 不清楚，用户可能误以为是自己发的。

处理：

1. 当前实现沿用“用户消息承载上下文整理”的既有约定。
2. 如需进一步降低认知成本，应在聊天渲染层单独美化 compaction block，而不是回退到隐藏这条消息。

## 8. 验证

### 8.1 代码层

1. 当前轮命中记忆已写入 `provider_meta.retrieved_memory_ids`。
2. prompt 组装阶段会把这些 ID 转成 `<memory_context>`。
3. 历史消息与最新用户消息都已支持注入记忆块。
4. 新会话和归档后新会话都会自动生成初始 SummaryContext 消息。

### 8.2 行为层

1. 连续两轮用户消息命中不同记忆时，历史轮与当前轮的记忆块应各自进入正确位置。
2. 同一次请求中重复命中的同一记忆不应重复注入。
3. 归档后切到的新会话应带最近摘要和画像快照。
4. 请求体预览中应能看到 `<memory_context>` 出现在实际请求 JSON 里。

### 8.3 建议自测

1. `cd src-tauri && cargo test`
2. 构造一个“第一轮命中 A，第二轮命中 B”的连续对话
3. 预览请求体，确认历史轮只带 A，当前轮只带 B
4. 强制归档后检查新会话首条消息是否已带摘要种子

## 9. 结果预期

如果当前设计成立，那么聊天主链路已经从“即时拼接记忆文本”升级为：

1. 写入时只存结构化召回结果
2. 组装时再注入真实上下文
3. 会话层自己携带可恢复的用户画像快照

这会让后续继续调试 prompt、预览请求体、做 SummaryContext 复用都更稳。
