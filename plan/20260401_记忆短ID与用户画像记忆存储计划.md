# 20260401 记忆短ID与用户画像记忆存储计划

## 1. 说明

本计划基于当前暂存实现反向整理，聚焦记忆存储层本身的重构设计。

它回答三个问题：

1. 为什么现在要给记忆增加短 ID。
2. 为什么要单独引入用户画像记忆链接。
3. SummaryContext 如何利用这两项能力。

## 2. 背景

旧记忆存储主要有两个不足：

1. 对模型来说只有 UUID，不利于 merge、引用、调试和人工核对。
2. “哪些记忆属于用户画像”只是逻辑约定，没有独立集合，难以做快照、完整画像板和自动维护。

当前暂存实现已经为这两个问题补上正式数据层语义。

## 3. 目标

重构后的实际目标可以整理为：

1. 每条记忆同时拥有稳定 UUID 和递增短 ID。
2. 用户画像记忆具备独立链接集合，而不是散落在普通记忆里靠推断。
3. 模型可直接使用短 ID 参与 useful/merge/profile 操作。
4. 历史老数据可平滑迁移，不要求一次性推翻现有库。

## 4. 当前方案

### 4.1 记忆短 ID 字段

`memory_record` 表新增：

1. `memory_no INTEGER UNIQUE`

对应到运行时结构：

1. `MemoryEntry.memory_no: Option<u64>`

并新增：

1. `MemoryEntry.display_id()`

规则是：

1. 有 `memory_no` 时优先展示短 ID
2. 没有时退回原始 UUID

### 4.2 迁移策略

当前迁移不是“删表重建”，而是：

1. 为老表补列 `memory_no`
2. 为老数据按创建时间顺序回填
3. 确保唯一索引存在

这代表当前实现采用的是“读兼容、写收敛”思路，风险更小。

### 4.3 新写入规则

`memory_store_upsert_drafts(...)` 现在会：

1. 查询当前最大 `memory_no`
2. 插入新记忆时分配递增短 ID
3. 更新已有记忆时，如果历史上没有 `memory_no`，也会补齐

所以短 ID 已经不只是迁移补丁，而是正式写入规则。

### 4.4 用户画像记忆单独成链

当前新增表：

1. `profile_memory_link`

字段包括：

1. `id`
2. `memory_id`
3. `source`
4. `created_at`
5. `updated_at`

这说明“画像记忆”现在不是 memory_record 的一个布尔标记，而是一张独立关系表。

### 4.5 画像记忆的允许类型

当前实现只允许以下类型进入画像集合：

1. `knowledge`
2. `skill`
3. `event`

`emotion` 被明确排除。

这和 SummaryContext prompt contract 是对齐的。

### 4.6 画像记忆的读取规则

当前通过：

1. `memory_store_list_profile_memories_visible_for_agent(...)`

来读取画像记忆。

它会同时做三层过滤：

1. 先按 agent 可见性过滤
2. 再按 `profile_memory_link` 过滤
3. 再按允许类型过滤

并按链接更新时间倒序返回。

这意味着：

1. 不是所有记忆都自动算画像
2. 也不是所有画像记忆对所有 agent 都可见

### 4.7 SummaryContext 如何使用短 ID

当前解析器已支持：

1. `usefulMemoryIds` 里直接输出数字
2. `mergeGroups.sourceIds` 里直接输出数字
3. `profileMemories.memoryId` 里直接输出数字

后端再通过 alias map 把短 ID 还原为真实 UUID。

这意味着模型面向人的 ID、数据库真实 ID、日志显示 ID 已经开始被拆开。

### 4.8 画像记忆的两种写法

SummaryContext 现在可以通过 `profileMemories` 走两条路径：

1. `memoryId`
   - 引用已有记忆进入画像集合
2. `memory`
   - 直接创建新的画像记忆，再自动链接进画像集合

当前后端会统计：

1. linked count
2. created count
3. skipped count

这让“模型识别画像”和“画像真正入库”之间有了可观测结果。

### 4.9 画像快照与完整画像板

当前实现已经区分两种读取形态：

1. `build_user_profile_snapshot_block(...)`
   - 用于聊天会话和压缩消息里的轻量快照
2. `build_user_profile_memory_board(...)`
   - 用于 SummaryContext 中的完整画像板

这表明画像存储层已经支持两种消费场景：

1. 聊天继续上下文
2. 画像治理与纠错

### 4.10 前端打通情况

前端目前已把 `memoryNo` 字段打到类型层：

1. `MemoryTab.vue`
2. `use-memory-viewer.ts`

这代表短 ID 已经具备前后端数据通路，但当前暂存没有显示出大规模 UI 呈现改版，所以更准确的状态是：

1. 类型已打通
2. 存储已落地
3. UI 展示仍处于轻接入阶段

## 5. 非目标

从当前暂存代码看，本轮没有体现以下目标：

1. 本次不把短 ID 作为外部 API 的唯一主键。
2. 本次不为画像记忆提供单独的前端编辑界面。
3. 本次不支持手动重排短 ID。
4. 本次不把所有记忆自动分桶进画像集合。

## 7. 风险点

### 7.1 短 ID 与真实 ID 混用

风险：

1. 模型输出短 ID
2. 后端某些路径仍按 UUID 假设处理
3. 可能导致 merge/useful/profile 应用错位

处理：

1. 当前已在 SummaryContext 入口引入 alias map 统一解析。
2. 后续新增涉及 memory id 的模型输出时，也必须复用同一解析规则。

### 7.2 画像集合污染

风险：

1. 如果模型频繁把临时上下文送进 `profile_memory_link`
2. 长期画像会膨胀并失真

处理：

1. 当前已在 prompt contract 和类型过滤上双重限制。
2. 画像只允许 `knowledge/skill/event`。
3. 读取画像时也会再次过滤，防止脏数据直接进入快照。

### 7.3 UI 已有代码仍默认只看 UUID

风险：

1. 数据层已有 `memoryNo`
2. 前端若继续只显示 UUID，调试价值不明显

处理：

1. 当前先打通类型，保证不会阻塞后续 UI 接入。
2. 若要进一步提升可用性，可以单独安排“记忆列表显示短 ID”改造，不必和本轮存储重构绑死。

## 8. 验证

### 8.1 数据层

1. 新写入记忆应自动得到递增 `memory_no`
2. 旧库升级后历史记录应自动补齐 `memory_no`
3. `profile_memory_link` 应能正确持久化并去重
4. `emotion` 即使被链接，也不应出现在画像可见列表中

### 8.2 SummaryContext

1. 模型返回数字型短 ID 时应能解析成功
2. `profileMemories.memoryId` 与 `profileMemories.memory` 两种写法都应生效
3. useful / merge / profile 三类动作应能共享同一套 ID 解析规则

### 8.3 建议自测

1. `cd src-tauri && cargo test`
2. 新建几条记忆，确认返回列表中已有 `memoryNo`
3. 让 SummaryContext 返回短 ID，检查是否能映射回真实记忆
4. 链接一条 `knowledge` 和记一条 `emotion` 到画像集合，确认最终只读出允许类型

## 9. 结果预期

如果当前设计保持成立，那么记忆层已经从“普通记忆单表”升级为：

1. 面向数据库的真实 ID
2. 面向模型和调试的短 ID
3. 面向长期用户背景的画像链接集合

这会让 SummaryContext、prompt 预览、人工排查和后续画像管理都更可控。
