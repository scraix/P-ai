# 20260401 请求体预览与调试链路补全计划

## 1. 说明

本计划基于当前暂存实现反向整理，主要描述“请求体预览”这一块已经被重构成什么样。

它的价值不只是一个 UI，而是验证本轮大重构是否真的按预期组装 prompt。

## 2. 背景

旧请求体预览只有一个“完整请求体”入口，放在当前这次重构里已经不够用，因为现在至少存在三种关键请求：

1. 正常聊天请求
2. SummaryContext 压缩请求
3. SummaryContext 归档请求

如果预览层仍只有一种模式，就很难回答这些问题：

1. 压缩和归档是否真的共用统一 prompt 语义。
2. `<memory_context>` 是否真的进入了实际请求体。
3. 最新用户消息和历史消息的 `extra_text_blocks` 是否被 provider 适配层正确序列化。

## 3. 目标

当前暂存实现已经表现出的目标为：

1. 前端能显式切换查看三种请求体。
2. 后端预览逻辑与真实运行逻辑尽量复用。
3. 调试 JSON 与 provider 实际请求结构保持一致。
4. 系统 prompt 预览仍保留独立入口，不影响原有调试习惯。

## 4. 当前方案

### 4.1 后端新增预览模式

`get_prompt_preview(...)` 现在新增：

1. `preview_mode`

支持三种值：

1. `chat`
2. `compaction`
3. `archive`

默认仍是 `chat`。

### 4.2 三种模式的真实含义

`chat`：

1. 预览普通聊天请求体
2. 使用当前会话、当前 agent、当前系统前言

`compaction`：

1. 预览 SummaryContext 的压缩场景请求体
2. 使用 archive host agent
3. latest user payload 会被替换成压缩任务说明、记忆整理上下文和 JSON 合约

`archive`：

1. 预览 SummaryContext 的归档场景请求体
2. 与 `compaction` 共用同一模式，只在 requirement 与 JSON 说明上区分摘要用途

### 4.3 预览逻辑尽量复用真实运行逻辑

当前实现不是单独手写一套“预览版 prompt”，而是直接复用：

1. `build_prepared_prompt_for_mode(...)`
2. `build_summary_context_requirement_block(...)`
3. `build_summary_context_memory_block(...)`
4. `build_summary_context_json_contract_block(...)`

这意味着只要运行时 prompt 变了，预览大概率也会同步变化，减少两套逻辑飘离。

### 4.4 当前会话选择逻辑

预览时会优先选：

1. 用户指定的会话
2. 否则选当前可见未归档会话
3. 否则构造一个最小默认会话

并会额外记录：

1. 最新用户消息 ID
2. 该消息挂载的 `retrieved_memory_ids`

用于调试日志。

### 4.5 前端交互变为“先开弹窗，再选模式”

当前前端 `use-prompt-preview` 已改成：

1. 点击“请求体预览”只打开弹窗
2. 弹窗初始是空状态
3. 用户选择 `对话 / 压缩 / 归档` 后才真正请求后端

这样做的好处是：

1. 用户不必重复点不同入口
2. 同一次会话上下文下可以快速切换三种预览
3. 更适合拿来对照 SummaryContext 重构前后差异

### 4.6 系统 prompt 预览保留单独入口

系统 prompt 预览仍然保留原有独立逻辑：

1. `openSystemPromptPreview(...)`
2. `get_system_prompt_preview(...)`

因此当前设计不是把所有预览都合并成一个弹窗模式，而是：

1. 请求体预览支持多模式
2. 系统前言预览继续直接打开

### 4.7 调试 JSON 已支持 `extra_text_blocks`

为了保证“预览看到什么，真实请求就是什么”，当前这些层都已补上 `extra_text_blocks`：

1. `prepared_prompt_to_messages_json(...)`
2. rig provider message 转换
3. DeepSeek message 转换
4. token 估算

这很关键，因为本轮重构后：

1. 历史消息里的 `<memory_context>`
2. 附件说明
3. 消息上下文块

都不再保证会被揉进 `text` 字段。

### 4.8 前端弹窗状态机

当前弹窗状态已改为：

1. `null`
   - 未选模式，只展示提示语
2. `chat`
3. `compaction`
4. `archive`
5. `system`

这比旧的 `full/system` 两态更贴合当前真实调试需求。

### 4.9 当前前端文案现状

从暂存代码看，模式切换按钮和空状态提示已直接接入中文文案：

1. 对话
2. 压缩
3. 归档

当前实现更偏“先把功能路径打通”，多语言抽取可在后续单独整理。

## 5. 非目标

当前暂存未体现以下目标：

1. 本次不做请求体 diff 视图。
2. 本次不把三种预览并排同时展示。
3. 本次不自动高亮 `<memory_context>`、画像块或 JSON contract 差异。
4. 本次不把请求体预览结果持久化到日志文件。

## 7. 风险点

### 7.1 预览与真实运行仍可能漂移

风险：

1. 虽然现在复用了大部分组装逻辑
2. 但如果后续运行时新增 overrides，预览层忘记同步，仍可能漂移

处理：

1. 当前已经尽量直接复用真实构建入口。
2. 后续任何 SummaryContext prompt 变更，都应同步看一眼预览命令是否还在共用同一套 helper。

### 7.2 前端空状态容易被误解为加载失败

风险：

1. 弹窗先打开、再选模式的设计，如果提示不清楚，用户会误以为预览坏了。

处理：

1. 当前已加入空状态提示文案。
2. 后续可考虑在按钮区默认高亮一个推荐模式，但本轮先保持显式选择更安全。

### 7.3 多语言尚未完全收口

风险：

1. 当前模式文案直接写在组件里，后续国际化可能不统一。

处理：

1. 当前优先保证调试路径可用。
2. 如后续要收口多语言，再单独做一次小型文案整理即可。

## 8. 验证

### 8.1 行为层

1. 打开请求体预览弹窗时，应先看到模式选择，不立即拉取请求体。
2. 选择 `对话` 后，应返回普通聊天请求体。
3. 选择 `压缩` 后，应看到 SummaryContext 压缩要求与 JSON contract。
4. 选择 `归档` 后，应看到 SummaryContext 归档要求与 JSON contract。
5. 若当前消息命中过记忆，应能在请求体 JSON 中看到 `<memory_context>`。

### 8.2 代码层

1. `prepared_prompt_to_messages_json(...)` 已能序列化 `extra_text_blocks`
2. provider 转换层已能序列化历史用户消息的 `extra_text_blocks`
3. token 估算已计入 `extra_text_blocks`
4. 前端 `promptPreviewMode` 已从 `full/system` 升级为多模式状态

### 8.3 建议自测

1. `pnpm typecheck`
2. `cd src-tauri && cargo check`
3. 打开请求体预览，依次切换三种模式
4. 对照真实会话和 SummaryContext 请求内容，确认预览与实际一致

## 9. 结果预期

如果当前设计保持成立，那么请求体预览已经从“普通调试小工具”升级为这次重构的关键观察窗：

1. 能直接看聊天 prompt
2. 能直接看 SummaryContext 压缩 prompt
3. 能直接看 SummaryContext 归档 prompt

这样你就可以不用猜链路是否真的统一，而是直接看最终请求体来核对。
