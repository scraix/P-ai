# 20260401 重构设计总览与计划索引

## 1. 说明

本组文档不是“前置立项计划”，而是基于 2026-04-01 当前暂存区实现反向整理出的设计计划，用于回答两件事：

1. 这轮大重构到底改成了什么。
2. 现在的实现是否符合原本想达成的方向。

整理范围只覆盖当前暂存区里的改动，不主动把未暂存内容混进来。

## 2. 本次重构的主轴

从暂存代码看，这次重构主要收敛成四条主线：

1. `SummaryContext` 统一归档与上下文压缩
2. 聊天主链路的记忆 RAG 延迟注入与用户画像快照
3. 记忆存储层的短 ID 与用户画像记忆链接
4. 请求体预览与调试链路补全

其中真正的中心变化不是某一个 UI，而是“聊天、压缩、归档、记忆整理”四条链路被重新组织成同一套数据模型与 prompt 语义。

## 3. 模块索引

### 3.1 SummaryContext 统一归档与压缩

文档：

- `plan/20260401_SummaryContext统一归档与压缩计划.md`

核心问题：

1. 归档 prompt、上下文压缩 prompt、不同时机的记忆整理 prompt 不再分三套维护。
2. 模型统一输出 `summary/usefulMemoryIds/newMemories/mergeGroups/profileMemories` 五字段 JSON。
3. 压缩与归档都走同一套 `SummaryContext` 结果应用流程。

### 3.2 聊天记忆 RAG 与用户画像快照

文档：

- `plan/20260401_聊天记忆RAG与用户画像快照重构计划.md`

核心问题：

1. 当前轮命中的记忆不再在消息写入时直接塞成完整 memory block。
2. 改成先把 `retrieved_memory_ids` 挂到消息元数据里，再在 prompt 组装阶段按需注入。
3. 前台聊天会话创建、归档切换、后台队列恢复时，都会带上用户画像快照与初始 SummaryContext 消息。

### 3.3 记忆短 ID 与画像记忆存储

文档：

- `plan/20260401_记忆短ID与画像记忆存储计划.md`

核心问题：

1. 记忆现在同时有稳定 UUID 与面向模型/调试的短 ID `memory_no`。
2. 用户画像不再只是“普通记忆里的一部分约定”，而是通过 `profile_memory_link` 单独建链接集合。
3. SummaryContext 可以直接引用已有画像记忆，也可以创建新画像记忆。

### 3.4 请求体预览与调试链路

文档：

- `plan/20260401_请求体预览与调试链路补全计划.md`

核心问题：

1. 旧请求体预览只够看“普通聊天”。
2. 现在前端可切换查看 `chat / compaction / archive` 三种请求体。
3. 调试 JSON 与真正下发给 provider 的历史消息结构保持一致，避免“预览看起来对，真实请求不对”。

## 4. 关键关系图

当前暂存实现可以概括成下面这条主链路：

```text
用户消息进入
  -> 当前轮根据用户文本检索可见记忆
  -> 只把 retrieved_memory_ids 写入 provider_meta
  -> prompt 组装时把命中记忆转成 <memory_context>
  -> 模型正常回复

会话变长 / 用户强制归档
  -> 进入 SummaryContext
  -> 模型同时输出 summary + usefulMemoryIds + newMemories + mergeGroups + profileMemories
  -> 一次性应用到记忆存储层
  -> 压缩场景写回新的 [上下文整理] 消息
  -> 归档场景直接产出 archive summary，并切到新的前台会话

新前台会话建立
  -> 注入最近一次归档摘要
  -> 注入用户画像快照
  -> 从第一条消息起就具备可继续聊天的上下文
```

## 6. 建议阅读顺序

如果你想快速判断“当前实现是否符合预期”，建议按这个顺序看：

1. 先看 `SummaryContext` 文档，确认归档与压缩是不是已经被统一。
2. 再看聊天记忆 RAG 文档，确认当前轮记忆注入是不是延迟到 prompt 组装阶段。
3. 再看记忆存储文档，确认短 ID 与画像记忆是否有独立落盘语义。
4. 最后看请求体预览文档，确认前端是否已经能直接观察这套新链路。

## 7. 验收口径

这组文档默认以“最新实现是否已经呈现出目标设计”为验收口径，而不是以“是否还需要继续开发更多功能”为唯一判断标准。

也就是说：

1. 如果某个设计已经在暂存代码里落地，文档会把它写成“当前方案”。
2. 如果某个方向只打通了一半，文档会明确标出“当前落点”和“剩余缺口”。
3. 如果某块暂存代码只是类型打通、尚未形成完整 UI，文档也会如实标注，而不是按完成态粉饰。
