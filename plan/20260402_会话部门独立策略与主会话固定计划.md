# 20260402 会话部门独立策略与主会话固定计划

## 背景

当前前台聊天默认围绕“主部门”运行，但未归档会话实际上已经具备独立会话列表与独立 `conversationId`。用户希望进一步明确边界：

1. 主会话继续固定绑定主部门；
2. 除主会话外，其他前台会话可以绑定任意部门；
3. 新建会话时允许用户显式选择部门，创建后该会话锁定所属部门；
4. 主会话归档后自动新建的新主会话，默认仍然是主部门；
5. 修复“主会话归档后自动把某个副会话升格成主会话”的错误行为；
6. 会话列表里主会话必须固定置顶。

## 当前代码现状

### 前端

- 新建会话入口在：
  - `src/features/chat/components/ChatComposerPanel.vue`
  - 当前弹窗只有标题输入，没有部门选择。
- 创建会话调用在：
  - `src/UnifiedWindowApp.vue`
  - `createUnarchivedConversation()` 当前只传 `apiConfigId` 和 `title`，不传 `departmentId`。
- 聊天发送调用在：
  - `src/UnifiedWindowApp.vue`
  - `send_chat_message` 当前 session 只传 `apiConfigId`、`agentId`、`conversationId`，没有 `departmentId`。
- 停止生成调用同样没有带 `departmentId`。
- 前台当前人格 / 模型依赖全局：
  - `assistantDepartmentAgentId`
  - `assistantDepartmentApiConfigId`
  这意味着当前前台主链仍然默认按“主部门”解释。
- 会话列表排序存在前后端不一致：
  - 后端 `collect_unarchived_conversation_summaries()` 已把主会话置顶；
  - 但前端 `chatUnarchivedConversationItems` 又按 `updatedAt` 二次排序，导致主会话置顶被冲掉。

### 后端

- `SessionSelector` 已支持 `department_id`：
  - `src-tauri/src/features/core/domain/types_requests.rs`
- 发送链路已支持按请求部门解析：
  - `src-tauri/src/features/system/commands/chat_and_runtime/core_commands.rs`
  - `src-tauri/src/features/system/commands/chat_and_runtime/core_send_inner.rs`
- 但 `Conversation` 结构当前没有存储“会话绑定部门”：
  - `src-tauri/src/features/core/domain/types_chat.rs`
- 新建会话逻辑在：
  - `src-tauri/src/features/system/commands/config_and_persona.rs`
  - `create_unarchived_conversation()` 当前不接收部门，也不把部门写入会话。
- 主会话归一化逻辑在：
  - `src-tauri/src/features/chat/conversation.rs`
  - `normalize_main_conversation_marker()` 当前会在主会话缺失时，直接把“最新活跃会话”升为主会话；
  - 这正是用户指出的 bug 根源。

## 目标

1. 为前台未归档会话增加“绑定部门”这个一等属性。
2. 主会话永远绑定主部门，且由系统维护，不允许被普通副会话自动顶替。
3. 新建会话时用户可以选择部门，创建成功后该会话固定使用该部门的人格/模型/工具权限。
4. 聊天发送、停止、切换快照、读取消息等链路都按“当前会话绑定部门”解释，而不是偷回全局主部门。
5. 会话列表中主会话固定第一位，前端不再覆盖这个排序。

## 实施方案

### 1. 给 Conversation 增加 department_id

处理项：

- 在 `Conversation` 结构中新增 `department_id: String`；
- 新建前台聊天会话时显式写入绑定部门；
- 历史会话兼容：
  - 若旧数据没有 `department_id`：
    - 主会话回填为主部门；
    - 非主会话先按 `agent_id -> department` 推断；
    - 推断失败时回退主部门。

原因：

- 只有把“部门”落到会话本体上，才能真正实现“会话锁定部门”，否则每次发送仍会被全局状态覆盖。

### 2. 新建会话弹窗增加部门下拉框

处理项：

- 在 `ChatComposerPanel.vue` 的“新建会话”弹窗中增加部门下拉；
- 选项规则：
  - 主部门可选；
  - 其他普通部门可选；
  - 不展示无效 / 无模型部门；
- 默认值规则：
  - 普通“新建会话”默认仍选主部门，减少行为突变；
  - 主会话被归档后系统自动补建的新会话，强制使用主部门，不走用户选择。

传参调整：

- `createConversation` 事件从 `title?: string` 升级为结构化 payload；
- `UnifiedWindowApp.vue` 将 `departmentId` 一并传给 `create_unarchived_conversation`。

### 3. 聊天运行时按“当前会话绑定部门”驱动

处理项：

- 前端切换到某个会话后，读取并缓存该会话绑定的 `departmentId`；
- 发送消息、停止消息、会话维护等 session 参数统一补上 `departmentId`；
- 前台展示的人格/模型标题要改为跟随当前会话部门，而不是永远使用主部门。

注意：

- 这是本次最关键的行为变更；
- 如果只在新建时存部门，但发送时不传 / 不按会话取，表面上有了下拉，实质仍会跑偏。

### 4. 修复“副会话自动升主会话”的 bug

现状问题：

- `normalize_main_conversation_marker()` 当前会在主会话丢失时，直接从现有未归档会话里挑一个最新的顶上；
- 这会导致副会话被自动升格为主会话。

修复方案：

- 主会话缺失时，不再把现有副会话升主；
- 而是显式新建一个新的主会话，并绑定主部门；
- 删除/归档主会话后，若需要新的主会话，也只能创建新的主会话，不能复用旧副会话。

### 5. 会话列表固定主会话置顶

现状问题：

- 后端 `list_unarchived_conversations` 已按主会话优先排序；
- 前端 `chatUnarchivedConversationItems` 又自行按 `updatedAt` 重排，破坏了这个保证。

修复方案：

- 前端不再覆盖后端排序；
- 若前端仍需本地排序，只允许“主会话优先”作为第一排序键。

### 6. 会话摘要与归档视图增加部门信息

建议一起做：

- `UnarchivedConversationSummary` 增加 `departmentId` / `departmentName`；
- 方便：
  - 新建后确认绑定是否正确；
  - 会话列表直接看出每个会话属于哪个部门；
  - 归档/切换/排障时更清晰。

这一步不是必须做得很重，但至少数据结构应补齐，避免后面又靠前端猜。

## 风险与关注点

### 1. 历史会话兼容

风险：

- 旧会话没有 `department_id`，如果处理不好，会在升级后跑到错误部门。

应对：

- 增加一次性兼容回填逻辑；
- 优先用主会话标记、再用 agent->department 推断，最后才回退主部门。

### 2. 前台标题/头像/工具权限要跟会话部门走

风险：

- 只改发送链路，不改 UI 展示，会出现“界面显示主部门，实际发送到副部门”的错觉。

应对：

- 当前会话的展示人格、工具权限、模型来源，都应改为基于当前会话部门计算。

### 3. 自动新建主会话的时机要收紧

风险：

- 如果归档、删除、切换快照等多个链路都能触发“补主会话”，容易重复创建。

应对：

- 把“确保主会话存在”收敛到统一后端 helper；
- 只有检测到主会话缺失时才创建，并且创建后立即写回 `main_conversation_id`。

## 验证计划

### 前端

- `pnpm typecheck`
- 新建会话弹窗出现部门下拉；
- 选择副部门创建后，会话列表能看出绑定部门；
- 切换到副部门会话后，发送消息走副部门；
- 主会话始终在会话列表第一位。

### 后端

- `cargo check`
- 补测试至少覆盖：
  1. 新建普通会话时，部门绑定被写入会话；
  2. 主会话归档后，新建出来的新主会话绑定主部门；
  3. 主会话归档/删除后，不会把现有副会话升格为主会话；
  4. 未归档会话列表始终主会话置顶；
  5. 历史缺失 `department_id` 的会话能被兼容回填。

## 待确认决策

1. 普通“新建会话”默认部门先保持主部门，用户可改。
2. 主会话在 UI 上建议继续显示“主会话”标签，并禁止用户把它改绑到别的部门。
3. 是否在会话列表直接展示部门名：
   - 我建议展示，信息更直观，也方便确认功能正确。

## 实施顺序

1. 补后端会话 `department_id` 存储与兼容回填。
2. 修主会话缺失时的补建逻辑，禁止副会话升主。
3. 扩展会话 summary，返回部门信息，并固定主会话排序。
4. 前端新建会话弹窗增加部门下拉，并传到创建命令。
5. 前端聊天运行时改为跟随当前会话部门。
6. 验证并补测试。
