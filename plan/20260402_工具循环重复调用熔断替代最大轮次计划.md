# 20260402 工具循环重复调用熔断替代最大轮次计划

## 背景

当前工具循环使用 `toolMaxIterations` 作为全局最大轮次上限：

- 前端在工具配置页暴露“工具最大调用轮次”输入框；
- 配置读写链路会保存并回放 `toolMaxIterations`；
- 后端在 `src-tauri/src/features/chat/model_runtime/provider_and_stream/tool_loop.rs` 中将其作为 `max_rounds` 使用。

用户希望移除这个可配置项，改为更明确的防失控策略：

- 若模型**连续使用相同工具**且**参数完全相同**达到 10 次；
- 则**强行禁用下一次该工具调用**，阻断死循环。

这属于行为规则调整，不只是 UI 删除，因此需要先统一前后端边界与兼容策略。

## 目标

1. 前端不再展示“工具最大调用轮次”配置。
2. 后端不再依赖用户可配置的 `toolMaxIterations` 来控制工具循环。
3. 改为固定规则：
   - 连续 10 次出现“同工具名 + 同参数”调用；
   - 第 11 次再出现同样调用时，直接拒绝执行该工具。
4. 拒绝时要有清晰日志和工具结果，便于模型理解“已被熔断”，并促使其换方案或结束回答。
5. 尽量保持配置文件兼容，避免因为旧字段导致加载失败。

## 当前代码现状

### 前端

- `src/features/config/views/config-tabs/ToolsTab.vue`
  - 工具页头部存在“工具最大调用轮次”输入框。
- `src/features/config/views/ConfigView.vue`
  - 保存工具配置前会记录并恢复 `toolMaxIterations`。
- `src/features/config/composables/use-config-core.ts`
  - 快照与 payload 中包含 `toolMaxIterations`。
- `src/features/config/composables/use-config-persistence.ts`
  - 加载/保存配置时会归一化 `toolMaxIterations`。
- `src/UnifiedWindowApp.vue`
  - 默认配置、事件回流与数值归一化中都包含 `toolMaxIterations`。
- `src/types/app.ts`
  - `AppConfig` 类型仍包含 `toolMaxIterations`。

### 后端

- `src-tauri/src/features/chat/model_runtime/provider_and_stream/tool_loop.rs`
  - 当前使用 `max_tool_iterations` 计算 `max_rounds`，按轮次截断。
- 各 provider/router 会把 `max_tool_iterations` 传递进工具循环。

## 实施方案

### 1. 前端移除“工具最大调用轮次”入口

处理项：

- 删除 `ToolsTab.vue` 中该输入框；
- 清理 `ConfigView.vue` 中围绕该字段的保存/回滚辅助逻辑；
- 清理相关文案：
  - `zh-CN.json`
  - `zh-TW.json`
  - `en-US.json`

预期效果：

- 用户不再看到这个配置项；
- 配置页不会再围绕该字段发生编辑与保存。

### 2. 前端配置链停止把它当作活跃配置项

处理项：

- 从 `AppConfig` 前端类型中移除 `toolMaxIterations`；
- 清理：
  - `use-config-core.ts`
  - `use-config-persistence.ts`
  - `UnifiedWindowApp.vue`
  中与该字段相关的默认值、快照、payload、事件回流、数值归一化逻辑。

兼容策略：

- 前端读取旧配置时，即使后端仍返回该字段，也不再使用；
- 若后端结构暂时仍保留该字段，不在前端继续展示和编辑。

### 3. 后端改为“重复调用熔断”规则

核心规则：

- 在单次工具循环会话内，跟踪最近一段连续工具调用；
- 若当前工具调用满足：
  - 工具名相同；
  - 参数规范化后完全相同；
  - 且已连续出现 10 次；
- 则第 11 次不真正执行工具，直接返回一个“已因重复调用被禁用”的工具结果。

建议实现方式：

- 在 `tool_loop.rs` 内维护连续计数状态，例如：
  - `last_tool_name`
  - `last_tool_args_signature`
  - `same_call_streak`
- 参数签名使用稳定序列化：
  - 若参数本身是 JSON 对象，需要使用 key 稳定排序后的 JSON 字符串；
  - 若当前拿到的是原始字符串，则先尝试解析 JSON，再做稳定序列化；
  - 无法解析时退回原字符串。

熔断后的返回建议：

- 不调用真实工具；
- 向前端发送失败态工具事件；
- 给模型写回清晰结果，例如：
  - “工具调用已被系统阻止：相同工具与相同参数已连续调用 10 次，请调整参数或停止调用。”

这样模型能在下一轮看到明确反馈，而不是无提示中断。

### 4. 后端兼容处理

兼容策略建议：

- 配置结构和磁盘字段暂时可以保留 `tool_max_iterations` / `toolMaxIterations` 的反序列化兼容；
- 但运行时不再使用该字段；
- 后续如需彻底清理存量字段，可单独做一次迁移与清扫，不和本次行为调整绑死。

这样可以降低一次性改动面，避免影响旧配置读取。

## 风险与关注点

### 1. “相同参数”判定必须稳定

风险：

- JSON 参数若仅 key 顺序不同，不应被误判为不同调用。

应对：

- 使用稳定序列化生成签名，而不是直接用原始 JSON 字符串比较。

### 2. 熔断提示要对模型足够明确

风险：

- 如果只是静默失败，模型可能继续重复同样调用。

应对：

- 返回清晰、可读、可被模型消费的错误结果；
- 同时打出一条 INFO 日志，说明触发了重复调用熔断。

### 3. 不能误伤正常多步工具链

风险：

- 有些工具本来就会连续调用多次，但参数每次不同。

应对：

- 只拦“同工具 + 同参数”的连续重复，不拦不同参数或不同工具。

## 验证计划

### 前端

- `pnpm typecheck`
- 打开工具配置页，确认“工具最大调用轮次”输入框已消失；
- 确认工具页保存/切换人格不再依赖该字段。

### 后端

- `cargo check`
- 为 `tool_loop.rs` 增加测试，至少覆盖：
  1. 连续 10 次相同工具同参数，第 11 次触发熔断；
  2. 同工具但参数不同，不触发熔断；
  3. 不同工具穿插时，连续计数被重置；
  4. JSON 参数仅 key 顺序不同，仍视为同参数。

## 待确认决策

1. 本次先按“单轮会话内连续 10 次”实现，不跨会话累积。
2. 本次优先移除前端配置入口，但后端存储字段先保留兼容，不做磁盘迁移清扫。
3. 触发熔断时，默认作为一次“失败的工具结果”回喂给模型，而不是直接整轮报错退出。

## 实施顺序

1. 移除前端工具页输入框与相关文案。
2. 清理前端配置类型、快照、保存、回放中的 `toolMaxIterations`。
3. 在后端工具循环里加入“同工具同参数连续 10 次熔断”逻辑。
4. 补充测试与日志。
5. 通过 `pnpm typecheck` 与 `cargo check` 验证。
