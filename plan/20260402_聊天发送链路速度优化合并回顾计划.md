# 20260402 聊天发送链路速度优化合并回顾计划

## 1. 文档定位

本计划用于合并回顾以下两类已落地优化：

1. `20260401_聊天后台持久化服务最小改造计划`
2. `20260402_提示词构建前移裁剪优化计划`

同时补充本轮实际实现中新增的聊天链路性能优化、锁诊断与一致性修正，统一形成一份“聊天发送速度优化回顾文档”，方便后续回看与继续推进。

## 2. 本轮优化目标

本轮优化的核心目标已经从“猜测慢点”收敛为三个明确方向：

1. 降低模型请求发出前的本地前置耗时。
2. 消除聊天发送主链路的异常会话锁竞争。
3. 在不破坏会话一致性的前提下，把高成本同步写盘移出热路径。

## 3. 已确认的瓶颈演进

### 3.1 第一阶段定位结果

最早通过聊天耗时汇总确认：

1. 主链路慢点并不在模型请求前的 prompt build 本体。
2. 早期最突出的问题是全局会话锁等待，常见 `200ms+`。
3. `state_read_app_data_cached()` 本身几乎总是 `0ms`，不是瓶颈。

### 3.2 第二阶段定位结果

在补充锁耗时监控后进一步确认：

1. 原先的主要等待来自 `conversation_lock` 竞争。
2. 一个与聊天主链路无关的桌面工具路径 `desktop_tools.rs` 在会话锁内同步读盘，曾持锁接近 `300ms`。
3. 该问题修复后，发送前的会话锁等待已从接近 `300ms` 下降到约 `0~50ms`。

### 3.3 当前阶段结果

当前发送前的主要本地耗时已收敛到：

1. `send_chat_message_inner_runtime_and_session_ready`
2. `prepare_context`
3. `prompt token estimate` 或运行时准备中的大对象 clone / 组装

也就是说，链路已经从“异常卡顿”进入“正常优化阶段”。

## 4. 本轮已落地改动

### 4.1 后台持久化服务最小改造

已落地：

1. `AppState` 新增后台持久化状态槽、通知器与序号。
2. 新增 `state_schedule_app_data_persist(...)`。
3. 聊天主链路部分同步写盘改为“更新缓存 + 调度后台持久化”。
4. 后台持久化 worker 已接入 `spawn_blocking`，避免同步磁盘 I/O 直接堵塞 async runtime。

当前取舍：

1. 热路径优先走后台调度。
2. 但与“发消息入口一致性”强相关的个别入口已按实际回归结果恢复为同步提交。
3. 即：性能优化必须服从功能稳定性。

### 4.2 Prompt 构建前移裁剪

已落地：

1. `build_prompt_with_mode()` 改为先按最近压缩边界裁剪有效消息。
2. 仅对裁剪后的消息做 clone 与 enrich。
3. 避免旧历史在多轮压缩后仍被重复 clone、重复 enrich、重复遍历。

收益方向：

1. 长历史、已压缩会话的 prompt 前置开销明显下降。
2. 与 PDF enrich 相关的无效工作被明显减少。

### 4.3 会话锁与记忆锁拆分

已落地：

1. 删除原 `state_lock`。
2. 新增：
   1. `conversation_lock`
   2. `memory_lock`
3. 记忆召回、画像构建不再与会话提交绑定为一把锁。
4. 发送主链路与 scheduler 已改成“短锁读快照 -> 锁外计算 -> 短锁提交”的结构。

当前语义：

1. 会话域与记忆域不做事务绑定。
2. 记忆缺失、延迟、失败都不应阻塞聊天主流程。

### 4.4 会话锁诊断能力补齐

已落地：

1. `conversation_lock` 增加 owner 追踪能力。
2. 锁等待日志会直接输出：
   1. 当前等待者
   2. 当前 owner
   3. owner 持有时长
3. 主要聊天热点调用点已补 task name。

这一轮诊断直接帮助定位并修复了：

1. `desktop_tools.rs` 中无关路径抢占会话锁并同步读盘的问题。

### 4.5 桌面工具会话解析链路去锁化

已落地：

1. `resolve_chat_tool_session_id(...)` 去掉了对 `conversation_lock` 的依赖。
2. 同步 `read_config(...)` / `read_app_data(...)` 改为缓存读取。

这是本轮最关键的锁竞争修复之一。

### 4.6 聊天入口一致性修正

已落地：

1. 修复显式 `conversation_id` 被错误套用前台可见性过滤的问题。
2. 委派会话（delegate conversation）在显式发送时不再被错误回退到别的前台会话。
3. 该修复用于解决“当前最后一条消息来自助理自身，无需重复激活”的误判问题。

### 4.7 日志收口

已落地：

1. 聊天耗时日志改为中文汇总。
2. 高频 `0ms` 的 `[应用数据耗时]` 缓存命中日志已移除。
3. 仅保留更有价值的磁盘读取与慢锁诊断日志。
4. 部分英文锁错误和序列化错误文案已改为中文。

## 5. 当前效果总结

根据最近一次日志观测：

1. 会话锁等待已基本不再是主要问题。
2. 发送前前置本地耗时已明显下降到约 `130ms` 量级。
3. 当前主要可优化点已从“锁竞争”转移到：
   1. 运行时准备阶段的大对象 clone / 组装
   2. prompt 相关的局部构建与估算

## 6. 当前仍保留的主要优化点

### 6.1 运行时准备阶段

当前已确认：

1. `runtime_data = data.clone()` 仍有稳定开销。
2. 后续还应继续判断：
   1. 是否必须 clone 整份 `AppData`
   2. 是否能改为只抽取运行时所需子集

### 6.2 scheduler 提交阶段

当前 `scheduler_commit` 仍有数十毫秒级持锁。

后续可继续检查：

1. 锁内 Vec push / 汇总消息拼装是否过多。
2. 是否还能进一步缩减提交窗口。

### 6.3 token 估算

在未命中上一轮 provider usage 缓存时，token 估算仍可能成为一次性大头。

当前原则已经明确：

1. 优先使用上一轮 provider 返回的 usage。
2. 仅在没有可用 usage 时才走本地估算。

后续若仍有明显波动，再继续专项处理。

## 7. 当前架构结论

本轮落地后，可以明确形成以下结论：

1. 聊天链路的最大收益来自“缩短锁范围”和“挪走热路径同步写盘”。
2. 并不是所有同步写入都能无脑改成后台调度，入口一致性相关路径仍需谨慎。
3. 调试与诊断能力本身也是性能优化的一部分；若没有锁 owner 追踪，本轮很多问题无法快速定位。

## 8. 后续建议

下一阶段建议继续围绕“正常优化”推进，而不是重新做大重构：

1. 继续拆 `send_chat_message_inner_runtime_and_session_ready`
   - 判断是否能避免整份 `AppData` clone
2. 继续收缩 `scheduler_commit`
   - 让会话提交阶段更接近纯提交
3. 继续清理残余英文错误日志
   - 保持日志口径统一
4. 在确保一致性的前提下，谨慎扩展后台持久化覆盖范围
