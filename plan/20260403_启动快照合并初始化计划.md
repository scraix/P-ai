# 20260403 启动快照合并初始化计划

## 1. 背景

当前窗口启动阶段，前端会依次请求：

1. `load_config`
2. `load_agents`
3. `load_chat_settings`

而后端这三个命令目前都会进入 `conversation_lock`，并且其中两到三个命令会继续读取 `app_data`、执行 `ensure_default_agent(...)`，甚至触发写回。结果是：

1. 启动阶段出现大量重复持锁日志。
2. 同一批运行时数据被重复读取、重复组装。
3. “纯读取配置”与“修正运行时状态”混在一起，导致读接口变重。

## 2. 本轮目标

本轮只解决“启动初始化重复读取过多”的问题，不做大规模锁模型重构。

目标：

1. 将启动阶段的三次初始化请求合并为一次后端快照读取。
2. 将配置、人格、聊天设置在同一次持锁窗口内准备完成。
3. 保持前端现有数据结构和 UI 行为不变。
4. 不改动聊天发送主链路与会话提交语义。

## 3. 实现范围

### 3.1 后端

新增一个启动快照命令，例如：

1. `load_app_bootstrap_snapshot`

返回内容：

1. `config`
2. `agents`
3. `chatSettings`

实现原则：

1. 只持有一次 `conversation_lock`
2. 只读取一次 config
3. 只读取一次 app_data
4. 只做一次必要的默认人格/聊天设置修正

### 3.2 前端

前端启动刷新链路改为优先调用一次启动快照接口，再把结果分别回填到：

1. 配置状态
2. 人格状态
3. 聊天设置状态

原有 `loadConfig / loadPersonas / loadChatSettings` 保留，作为兼容和后续局部刷新入口。

## 4. 不做的事

本轮明确不做：

1. 不拆 `conversation_lock` 为多把域锁
2. 不重写 `save_config / save_agents / save_chat_settings`
3. 不调整聊天发送链路中的会话上下文准备逻辑
4. 不处理所有读接口中的“懒修复写回”问题，只先收敛启动主路径

## 5. 风险与控制

主要风险：

1. 前端合并初始化后，局部字段回填顺序变化可能影响现有 watch
2. 启动快照若少带某些字段，可能导致配置页或聊天页局部状态异常

控制方式：

1. 后端直接复用现有三条读取链路的组装语义
2. 前端保持原有字段映射，不改变业务字段命名
3. 原有三个独立读取方法保留，不做删除

## 6. 验收标准

满足以下条件即认为本轮通过：

1. 启动阶段不再连续调用三次初始化接口
2. 后端初始化相关 `conversation_lock` 持锁日志显著减少
3. 配置页、聊天页、归档页启动后数据正常
4. 人格列表、聊天设置、主题与模型配置加载行为不回退
