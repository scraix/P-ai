# Windows 安装版与便携版自动更新计划

## 背景

当前项目虽然已经有“检查更新”入口，但严格来说它只做了“请求 GitHub/Gitee Release API -> 比对版本 -> 打开发布页”，并没有真正执行下载、校验、安装与重启。

当前现状：

- 前端 `src/features/shell/composables/use-github-update.ts` 只负责显示“发现新版本”并打开发布页。
- 后端 `src-tauri/src/features/system/commands/config_and_persona.rs` 的 `check_github_update` 只负责查询远端版本信息。
- 打包配置 `src-tauri/tauri.conf.json` 当前仍同时产出 `msi`、`nsis` 以及其他平台安装包，本期需要收敛。
- 发布流程 `.github/workflows/release.yml` 当前只上传安装产物，没有生成 updater 用的签名与更新清单。

用户已经明确本期只支持 Windows，但希望同时覆盖两种运行形态：

1. 安装版：通过安装包安装到系统中。
2. 便携版：目录即程序本体，用户可直接解压运行。

同时，用户已经给出一套可接受的便携版识别口径与更新思路：

- 通过可执行文件同级目录下是否存在 `PORTABLE` 标记文件，判断当前是否为便携版。
- 安装版与便携版共用同一套应用代码，但更新时选择不同产物：
  - 安装版下载 `setup.exe`
  - 便携版下载 `portable.zip`

本次计划的目标，就是把当前“伪自动更新”改造成“Windows 安装版 + 便携版双链路自动更新”。

## 目标

### 1. 功能目标

1. Windows 安装版支持真正自动更新，而不是仅跳转发布页。
2. Windows 便携版支持真正自动更新，采用“下载压缩包 -> 校验 -> 退出主程序 -> 外部进程覆盖 -> 重启”方案。
3. 客户端通过 `PORTABLE` 标记文件稳定识别当前运行形态。
4. GitHub Release 作为唯一更新源。
5. 更新 UI 统一展示检查、下载、校验、安装/覆盖、失败等状态。
6. 更新失败时保留足够中文诊断信息，便于排障。

### 2. 体验目标

1. 用户在配置页点“检查更新”后，若有新版本，可以直接在应用内完成后续操作。
2. 安装版与便携版共用一个更新入口，但内部自动分流。
3. 便携版更新过程中，前端能实时看到阶段与失败原因。
4. 安装版更新不再误导成“自动更新”，其行为与文案应和实际能力一致。

## 非目标

1. 本期不支持 macOS / Linux 自动更新。
2. 本期完全放弃 `msi` 产物与 `msi` 自动更新链路。
3. 本期不支持“裸 `p-ai.exe` 直接作为正式便携版发布物”；便携版应是受控压缩包形态。
4. 本期不做差分更新，仍使用整包更新。
5. 本期不做多更新源聚合下载，只以 GitHub Release 为主。

## 方案总览

本次更新能力分成两条链路：

### 1. 安装版链路

- 运行形态：无 `PORTABLE` 标记文件
- 更新产物：`NSIS setup.exe`
- 更新方式：接入 Tauri 官方 `updater` 插件
- 安装方式：使用 Windows `passive` 安装模式
- 适用范围：标准安装用户

### 2. 便携版链路

- 运行形态：可执行文件同级存在 `PORTABLE`
- 更新产物：`portable.zip`
- 更新方式：项目自定义更新器
- 覆盖方式：下载后校验，由外部 helper 进程在主程序退出后完成覆盖与重启
- 适用范围：解压即运行的便携用户

### 3. 统一入口

- 前端只保留一个“检查更新 / 立即更新”入口
- 后端或运行时先识别当前形态
- 再决定：
  - 安装版走 Tauri 官方 updater
  - 便携版走自定义便携更新器

## 运行形态识别

### 1. 标记文件规则

新增统一判断逻辑：

- 若 `exeDir()/PORTABLE` 存在，则 `is_portable = true`
- 否则 `is_portable = false`

要求：

1. 判断逻辑必须只依赖文件系统，不依赖注册表。
2. 判断逻辑必须收敛到单一函数，不允许前后端各写一套不同规则。
3. 便携包发布时，根目录必须自带空的 `PORTABLE` 文件。

### 2. 影响范围

`is_portable` 至少要影响以下行为：

1. 更新产物选择
2. 数据目录选择
3. 更新 UI 提示语
4. 后续诊断日志

说明：

- 本期以自动更新为第一目标。
- 若当前项目仍未完全按 `PORTABLE` 规则切换数据目录，也应在计划中一并纳入修正，避免“识别为便携版但数据仍写入 AppData”的半成品状态。

## 发布产物收敛

### 1. Windows 目标产物

Windows 最终只保留两类正式发布物：

1. 安装版：
   - `p-ai_<version>_x64-setup.exe` 或等价命名
2. 便携版：
   - `p-ai_<version>_x64-portable.zip` 或等价命名

不再继续面向用户主推以下产物：

1. `msi`
2. 裸 `p-ai.exe`

### 2. 命名规范

需要统一一套稳定命名，让安装版与便携版都能被客户端精确匹配，例如：

- `p-ai-windows-v0.9.1-x64-setup.exe`
- `p-ai-windows-v0.9.1-x64-portable.zip`

命名规则必须编码进：

1. 发布工作流
2. 更新元数据生成逻辑
3. 客户端产物匹配逻辑

## 安装版更新设计

### 1. 官方 updater 接入

安装版接入 Tauri 官方 updater，至少包含：

1. Rust 侧增加 `tauri-plugin-updater`
2. 前端增加 `@tauri-apps/plugin-updater`
3. capability 权限放行 updater 能力
4. `tauri.conf.json` 增加 updater 端点、公钥与 Windows 安装模式配置

### 2. 更新源

安装版更新源采用 GitHub Release 的 updater 元数据，例如：

- `https://github.com/<owner>/<repo>/releases/latest/download/latest.json`

发布流程负责生成：

1. `latest.json`
2. 对应安装包签名

### 3. 前端行为

安装版若检测到新版本：

1. 显示版本号与更新日志摘要
2. 用户确认后开始下载
3. 下载期间显示进度
4. 校验通过后进入安装阶段
5. 安装完成后提示重启或自动重启

若未检测到新版本：

1. 弹窗明确提示“当前已是最新版本”
2. 同时提供“强制更新”和“取消”
3. “强制更新”用于重新下载并安装当前 latest 对应版本，方便测试更新链路

### 4. 文案要求

必须把当前“检查更新”的误导性文案修正为真实能力：

- 未接入 updater 前叫“查看新版本”
- 接入后才允许叫“自动更新”

## 便携版更新设计

### 1. 便携版更新总流程

便携版采用以下步骤：

1. 读取 GitHub Release 更新元数据
2. 根据架构与运行形态定位便携包文件名
3. 下载便携包到临时更新目录
4. 对下载文件执行校验
5. 生成并启动外部 helper 进程
6. 主程序退出，释放文件锁
7. helper 延时执行解压覆盖
8. helper 启动新版主程序
9. helper 清理临时文件

### 2. 校验策略

便携版不能只下载不校验，至少需要：

1. 校验压缩包大小或长度有效
2. 校验 SHA-256 或等价强校验值
3. 对下载的 `portable.zip` 执行密码学签名验证
4. 校验失败时中止覆盖

校验信息建议来自 GitHub Release 元数据，而不是前端临时猜测。

签名策略要求：

1. 便携包必须和安装版共用同一套发布签名体系，优先复用 updater 公钥生成 detached signature。
2. 客户端在解压前必须先验证 `portable.zip` 的签名与哈希，任一失败都必须中止覆盖。
3. 若便携包未来改用证书链签名，再增加签名者证书有效期与撤销状态校验。

本期安全边界说明：

1. 第一阶段优先实现“固定公钥 + detached signature + SHA-256”校验。
2. 第一阶段不额外引入 Windows Authenticode 证书链、OCSP 或 CRL 检查，因为当前发布链路本身更贴近 Tauri updater 的公钥签名模型。
3. 这意味着本期便携版安全保障低于“带证书链在线撤销校验”的完整模型，但仍显著高于“只校验文件长度”或“只看 GitHub 下载成功”。
4. 若签名验证失败、签名缺失、签名文件不匹配或哈希不匹配，客户端必须直接报错并停止，不允许继续覆盖。

### 3. helper 执行策略

便携版不能由当前运行中的主进程直接覆盖自己，因此必须交给外部进程。

建议实现顺序：

1. 第一版优先实现 Rust 小型 helper
2. 若某一步必须调用系统工具，也只能作为 helper 内部的受控子步骤，不允许把完整更新逻辑外包给拼接脚本

约束：

1. helper 逻辑必须可获取错误码与 stderr
2. 路径必须正确处理空格与非 ASCII
3. 覆盖前应尽量避免半成品状态

#### 3.1 Helper 完整性

1. 若 helper 作为独立二进制随应用分发，启动前必须先校验其完整性。
2. 完整性校验可选两种等价实现：
   - 使用与便携包相同的公钥签名机制做 detached signature 校验
   - 在主程序中内置 helper 的预期 SHA-256，并在执行前重新计算比对
3. 若 helper 为临时生成文件，仍必须校验生成结果与预期模板或内置摘要一致。
4. helper 完整性校验失败时，不得执行 helper，必须记录中文错误日志并弹出前端提示。

#### 3.2 权限与 UAC

1. 便携版默认按“非提权”路径执行更新。
2. 若目标目录可写，则直接在当前权限下完成 staging、备份、覆盖与重启。
3. 若目标目录不可写或位于受保护位置，再提示用户是否重试为提权更新。
4. 若用户拒绝提权，则保留已下载包并提示手动更新，不自动继续。
5. 若提权失败或 UAC 被取消，必须明确提示失败原因，并允许用户重试或改走手动下载路径。

#### 3.3 进程与参数安全

1. helper 启动必须使用 argv 风格 API，不允许使用 `cmd /C`、字符串拼接命令或 shell 模式执行整段脚本。
2. 所有运行时参数必须做严格校验与白名单限制，只允许传递：
   - 当前版本目录
   - staging 目录
   - 备份目录
   - 目标可执行文件路径
   - 受控布尔开关或固定枚举
3. 不允许把用户可编辑文本直接拼进 helper 参数。
4. helper 与其子进程的退出码、stderr、关键阶段日志都必须被捕获并写入更新日志。
5. 遇到提权失败、helper 完整性失败、参数校验失败时，前端必须收到明确错误提示。

### 4. 覆盖策略

便携版覆盖建议采用：

1. 解压到临时目录
2. 基础校验压缩包内容结构
3. 对 staging 目录做文件数、关键文件存在性、必要文件哈希校验
4. 替换前先生成带时间戳的旧版本备份目录
5. 再整体替换或逐文件覆盖目标目录
6. 覆盖完成后再次做落地校验

失败判定至少包括：

1. staging 中文件数明显不完整
2. 关键文件缺失
3. 覆盖过程中出现 I/O 错误、异常或权限错误
4. 覆盖后目标目录校验和或文件数量与 staging 不一致
5. helper 返回非零退出码

回滚策略：

1. 一旦覆盖阶段失败，立即停止继续写入。
2. 若备份目录完整存在，则自动执行回滚，把旧版本从时间戳备份目录恢复到目标目录。
3. 回滚完成后再次校验关键文件存在性与主程序可启动性。
4. 若回滚成功，前端提示“更新失败，已恢复旧版本”。

备份保留策略：

1. 默认保留最近 2 份便携版更新备份。
2. 同时清理 7 天前的旧备份。
3. 若磁盘空间不足，可在更新开始前先做一次过期备份清理。

说明：

- 直接在原目录一边解压一边覆盖虽然简单，但回滚与诊断都更差。
- 若实现成本允许，优先“解压到 staging，再覆盖目标目录”。
- 若自动回滚本身也失败，必须进入失败兜底路径：提示用户下载完整安装包或便携压缩包手动恢复，并保留 recovery 日志供排障。

### 5. 临时目录

便携版更新需要专用临时目录，例如：

- 安装版数据目录下的 `temp/updater/`
- 便携版程序目录下的 `data/temp/updater/` 或等价目录

临时目录中至少包含：

1. 下载文件
2. 解压目录
3. 日志文件
4. helper 执行脚本或参数文件

## 更新元数据设计

本次需要补一层统一更新元数据，不再只依赖“latest release API 返回 tag_name”。

建议元数据至少包含：

1. `schemaVersion`
2. `version`
3. `notes`
4. `publishedAt`
5. `platforms`

其中 `platforms.windows-x86_64` 至少要区分：

1. `installer`
2. `portable`

每个条目至少包含：

1. `url`
2. `signature` 或 `sha256`
3. `size`
4. `kind`

说明：

- 安装版字段要满足 Tauri 官方 updater 读取要求。
- 便携版字段可由我们扩展，但必须保证客户端能稳定解析。

### 1. 元数据 JSON Schema

建议把更新元数据正式定义为一份 versioned schema，例如：

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://easycall.ai/schemas/update-manifest.schema.json",
  "title": "Easy Call AI Update Manifest",
  "type": "object",
  "required": ["schemaVersion", "version", "notes", "publishedAt", "platforms"],
  "additionalProperties": true,
  "properties": {
    "schemaVersion": {
      "type": "string",
      "const": "1.0"
    },
    "version": {
      "type": "string",
      "minLength": 1
    },
    "notes": {
      "type": "string"
    },
    "publishedAt": {
      "type": "string",
      "format": "date-time"
    },
    "platforms": {
      "type": "object",
      "required": ["windows-x86_64"],
      "additionalProperties": true,
      "properties": {
        "windows-x86_64": {
          "type": "object",
          "required": ["installer", "portable"],
          "additionalProperties": true,
          "properties": {
            "installer": {
              "$ref": "#/$defs/artifact"
            },
            "portable": {
              "$ref": "#/$defs/artifact"
            }
          }
        }
      }
    }
  },
  "$defs": {
    "artifact": {
      "type": "object",
      "required": ["url", "size", "kind"],
      "additionalProperties": true,
      "properties": {
        "url": {
          "type": "string",
          "format": "uri"
        },
        "signature": {
          "type": "string",
          "minLength": 1
        },
        "sha256": {
          "type": "string",
          "pattern": "^[A-Fa-f0-9]{64}$"
        },
        "size": {
          "type": "integer",
          "minimum": 1
        },
        "kind": {
          "type": "string",
          "enum": ["installer", "portable"]
        }
      },
      "anyOf": [
        { "required": ["signature"] },
        { "required": ["sha256"] }
      ]
    }
  }
}
```

### 2. 版本与兼容策略

1. 顶层 `schemaVersion` 首版固定为 `1.0`。
2. 客户端若遇到未知的主版本，例如 `2.x`，必须直接中止解析并提示“更新元数据版本不受支持”。
3. 客户端若遇到同主版本下的次版本升级，例如 `1.1`，可按向后兼容方式继续解析已知字段，并忽略未知字段。
4. 工作流生成元数据时，也必须校验 `schemaVersion` 与当前客户端支持范围一致。

### 3. 校验规则

发布工作流与客户端都必须做同一套最小校验：

1. 顶层必须存在 `schemaVersion`、`version`、`notes`、`publishedAt`、`platforms`。
2. `platforms.windows-x86_64.installer` 与 `platforms.windows-x86_64.portable` 必须同时存在。
3. 每个 artifact 必须包含 `url`、`size`、`kind`，并至少包含 `signature` 或 `sha256` 之一。
4. `installer.kind` 必须为 `installer`，`portable.kind` 必须为 `portable`。
5. 任一必填字段缺失或格式非法时，客户端不得继续更新。
6. 对未知附加字段，客户端在记录 debug 日志后可忽略，不得因为额外字段而报错。

## 前端交互设计

### 1. 配置页入口

现有“检查更新”区块改造成明确的更新面板，至少包含：

1. 当前版本
2. 最新版本
3. 当前运行形态：
   - 安装版
   - 便携版
4. 当前更新状态：
   - 未检查
   - 检查中
   - 可更新
   - 下载中
   - 校验中
   - 安装中 / 覆盖中
   - 完成
   - 失败

### 2. 失败提示

失败文案必须中文，且包含足够上下文，例如：

- 请求更新元数据失败
- 未找到匹配当前架构的安装包
- 更新包校验失败
- 便携版 helper 启动失败
- 覆盖失败，目标文件被占用

### 3. 进度推送

安装版与便携版都应统一走事件推送或统一状态仓，至少把这些阶段推给前端：

1. 检查开始
2. 检查完成
3. 下载进度
4. 校验开始 / 完成
5. 安装开始 / 覆盖开始
6. 完成
7. 失败

## 后端设计

### 1. 运行时接口

后端建议拆出独立更新模块，避免继续把更新逻辑塞在 `config_and_persona.rs` 中。

建议新增：

- `src-tauri/src/features/system/updater/`

至少拆分为：

1. `detect.rs`
   - 运行形态、架构、路径识别
2. `metadata.rs`
   - 获取和解析更新元数据
3. `portable.rs`
   - 便携版下载、校验、helper 启动
4. `events.rs`
   - 更新进度事件推送
5. `mod.rs`
   - 对外命令与统一入口

### 2. 现有命令收敛

现有 `check_github_update` 需要重构为更明确的职责：

1. 查询更新概览
2. 返回运行形态
3. 返回可更新状态
4. 若为安装版，可返回官方 updater 所需上下文
5. 若为便携版，可返回便携包匹配信息

同时新增：

1. `start_update`
2. `cancel_update`（如可行）
3. `get_update_runtime_state`

### 3. 日志要求

日志必须符合仓库中文日志规范，建议统一前缀：

- `[自动更新]`

至少记录：

1. 更新检查开始/完成
2. 当前运行形态
3. 匹配到的目标产物
4. 下载开始/完成/失败
5. 校验开始/完成/失败
6. 安装或覆盖开始/完成/失败

## 发布流程改造

### 1. 目标

现有 GitHub Actions 发布流要从“只上传包”升级到“生成客户端可消费的更新清单与签名”。

### 2. 需要调整的点

1. Windows 构建只保留 `nsis`
2. 额外生成便携版压缩包
3. 便携包根目录写入 `PORTABLE`
4. 为安装版生成 updater 所需签名与 `latest.json`
5. 为便携版补充签名、`sha256`、大小、文件名等元数据
6. 统一上传到 GitHub Release

### 3. 便携版打包

便携版建议由工作流显式产出，而不是直接上传裸 `p-ai.exe`。

压缩包内容至少包括：

1. 主程序
2. 必要资源文件
3. `PORTABLE`
4. 运行所需的动态依赖

## 数据目录与运行环境一致性

由于用户已经明确指出“便携版/安装版识别”不只是更新问题，也会影响数据位置，因此本期必须一并检查：

1. 配置文件目录
2. 会话数据目录
3. 日志目录
4. 临时下载目录

验收要求：

1. 安装版继续使用标准应用数据目录
2. 便携版使用程序目录下的本地数据目录
3. 更新临时目录与最终运行形态一致

## 风险与应对

### 1. 便携版覆盖中断

风险：

- 主程序退出后，helper 执行失败，导致用户停留在旧版本或半更新状态

应对：

1. 先下载到临时目录
2. 先验包
3. 优先使用 staging 解压
4. 覆盖前输出详细日志

### 2. 安装版与便携版识别错误

风险：

- 识别错误会导致下载错误产物，甚至把安装版当便携版处理

应对：

1. 单点实现 `is_portable`
2. 启动时打印识别日志
3. UI 明确显示当前运行形态

### 3. 更新元数据不一致

风险：

- 客户端匹配不到正确包

应对：

1. 工作流统一命名
2. 客户端不要硬编码多套命名猜测
3. 发版时对 `latest.json` 与便携字段做自动校验

### 4. 裸 exe 误当便携版

风险：

- 用户直接运行开发构建或 release 目录中的单个 exe，行为不稳定

应对：

1. 正式发布时不再主推裸 exe
2. 便携版只认带 `PORTABLE` 的正式压缩包
3. UI 中对“非正式便携形态”给出明确提示
