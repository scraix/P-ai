# 20260407 会话列表点击标题改名计划

## 1. 背景

当前聊天侧边栏中的会话列表，整张卡片都是“切换会话”按钮，标题区域只是只读文本。

现状问题：

1. 用户无法直接在会话列表里修改会话标题。
2. 后端已有“创建会话时指定标题”能力，但没有“更新会话标题”命令。
3. 主会话当前前端始终显示为固定文案“主会话”，即使底层标题字段变化，列表里也不会体现。

## 2. 本轮目标

本轮只实现“在会话列表中点击标题后改名”的基础能力。

目标：

1. 用户点击当前会话标题即可进入内联编辑态。
2. 按 `Enter` 保存，按 `Esc` 取消，失焦时保存。
3. 保存成功后，会话列表标题立即刷新，无需手动重开窗口。
4. 非当前会话仍保持原有整卡切换行为，不破坏现有操作习惯。

## 3. 实现范围

### 3.1 前端

聊天侧边栏会话列表新增标题编辑态：

1. 普通显示态下，标题仍按现有样式展示。
2. 仅当前会话的标题区域可点击进入输入框，并自动聚焦、选中文本。
3. 编辑期间阻止事件冒泡，避免误触发其他列表交互。
4. 非当前会话继续保持整卡按钮，不开放重命名入口。
5. 正在整理上下文的会话保持禁用，不开放重命名入口。
6. 主会话暂不开放改名，继续显示固定“主会话”。

前端还需要补一条调用后端的新接口：

1. 传入 `conversationId`
2. 传入新的 `title`
3. 保存成功后复用现有会话概览刷新链路更新 UI

### 3.2 后端

新增一个轻量命令，例如：

1. `rename_unarchived_conversation`

职责：

1. 校验会话存在且属于前台未归档会话。
2. 清洗并写回新的标题。
3. 持久化 `app_data.json`。
4. 复用现有 overview emit 机制，通知前端刷新会话列表。

标题规则：

1. 仅允许保存非空标题。
2. 纯空白输入不提交，前端直接取消本次改名，避免写入无效标题。

## 4. 不做的事

本轮明确不做：

1. 不实现归档列表改名。
2. 不实现批量重命名。
3. 不调整主会话“固定标题”显示策略。
4. 不增加新的右键菜单或更多会话管理功能。

## 5. 风险与控制

主要风险：

1. 输入框失焦与列表其他交互互相影响，可能出现误保存或误取消。
2. 后端改名后若未正确发出 overview 更新事件，列表标题可能要到下次刷新才变化。

控制方式：

1. 当前会话单独渲染为可编辑容器，避免和非当前会话的切换按钮互相干扰。
2. 保存逻辑优先收口到一个前端函数，统一处理 `Enter / Esc / blur`。
3. 后端直接复用现有 unarchived overview 的构建与 emit 链路，避免新增第二套刷新机制。

## 6. 验收标准

满足以下条件即认为本轮通过：

1. 在聊天侧边栏点击当前会话标题，可进入编辑态并成功保存新标题。
2. 非当前会话点击卡片区域，仍然是切换会话，不会误进入编辑。
3. `Enter` 保存、`Esc` 取消、失焦保存行为符合预期。
4. 改名后列表标题立即更新，重启应用后标题仍然保留。
5. `pnpm typecheck` 与 `cargo check` 通过。
