# Codex 协议与账号登录接入计划

## 当前状态

- 状态：已完成开发接入，待继续按真实使用场景验证
- 实现日期：2026-04-14
- 范围：配置层、认证层、模型选择、聊天运行时、部门模型选择

## 本次目标

在项目中新增独立 `codex` 协议，支持像 `cline` 一样使用 ChatGPT / Codex 登录态，而不是继续走普通 OpenAI API Key 语义。

同时满足以下口径：

- 支持两种认证方式：
  - 读取本机 `~/.codex/auth.json`
  - 应用内自行发起 OAuth 登录
- `codex` 作为独立协议参与文本模型配置与部门绑定
- Codex 只允许调整“思维强度”
- 不向用户暴露 `temperature`、`maxOutputTokens`、上下文手动配置
- 凭证不通过 `load_config()` 明文返回到前端

## 最终实现结果

### 1. 协议与配置模型

前后端已经新增 `codex` 协议，并补齐了 Codex 专属配置字段：

- `requestFormat = "codex"`
- `codexAuthMode = "read_local" | "managed_oauth"`
- `codexLocalAuthPath`
- `reasoningEffort`

已完成的实际约束：

- `codex` 属于文本模型协议，可参与聊天与部门绑定
- `codex` 不要求 API Key
- `codex` 的 `base_url` 固定为 `https://chatgpt.com/backend-api/codex`
- 保存配置时会强制把 Codex 的上下文窗口设为 `272000`
- `temperature`、`maxOutputTokens` 等参数对 Codex 不再开放给用户，保存时也会被归一化回默认值

### 2. 前端配置页

Codex 已拆成独立前端组件，不再和通用 OpenAI 配置面板混用。

当前界面行为：

- `ApiTab` 可以新增 `codex` 协议
- `codex` 配置显示独立的 [CodexProviderPanel.vue](/E:/github/easy_call_ai/src/features/config/views/config-tabs/CodexProviderPanel.vue)
- 支持两种认证方式切换：
  - `读取本地`
  - `自行登录`
- 本地模式可填写本地凭证路径并执行“检查本地登录”
- OAuth 模式可执行“登录 Codex”与“退出登录”
- 模型区域只显示：
  - 模型选择
  - 思维强度
  - 是否启用工具
- 不再显示：
  - API Key 输入
  - Base URL 输入
  - 温度
  - 最大输出
  - 手动上下文配置

额外已完成项：

- 本地检查按钮增加忙碌态，避免重复点击并发触发
- `reasoningEffort` 已补 `xhigh`
- 模型卡片可以独立新增/删除

### 3. 认证与凭证存储

后端已新增独立 Codex 认证命令与凭证处理文件：

- [codex_auth.rs](/E:/github/easy_call_ai/src-tauri/src/features/system/commands/codex_auth.rs)

当前已落地的认证能力：

- 读取本地 `~/.codex/auth.json`
- 兼容本地文件中的 `snake_case` 字段
- 从 JWT claims 中提取 `account_id / email / exp`
- 应用内 OAuth 登录：
  - `authorize`：`https://auth.openai.com/oauth/authorize`
  - `token`：`https://auth.openai.com/oauth/token`
  - 回调：`http://localhost:1455/auth/callback`
  - 使用 PKCE
  - `originator` 固定为 `cline`
- 托管 OAuth 凭证保存到应用私有目录，不返回到前端配置对象
- 支持退出登录
- 运行时 access token 过期时支持基于 refresh token 自动刷新

当前已提供的 Tauri 命令：

- `codex_get_auth_status`
- `codex_start_oauth_login`
- `codex_logout`

说明：

- 最终没有保留单独的 `poll` / `cancel` 命令
- OAuth 状态由 `codex_get_auth_status` 结合内存 session 查询返回

### 4. 模型列表与参数口径

Codex 当前采用静态模型列表，不走远端 `/models`。

最终预设模型固定为：

- `gpt-5.4`
- `gpt-5.4-mini`
- `gpt-5.3-codex`
- `gpt-5.3-codex-spark`
- `gpt-5.2`

思维强度当前支持：

- `low`
- `medium`
- `high`
- `xhigh`

上下文窗口口径：

- Codex 所有模型统一固定为 `272000`
- 该值在保存配置时由后端强制归一化

### 5. 聊天运行时

最终实现与最初计划有一处明确偏差：

- 最初计划：为 Codex 新写一层独立运行时适配
- 最终实现：放弃自维护适配层，回到 `genai` 的 `OpenAIResp` 运行链路

当前实际做法：

- `codex` 仍是独立协议
- 认证解析独立
- 运行时会先解析并刷新 Codex 凭证
- 然后把 `access_token` 注入到 `genai::OpenAIResp` 请求
- 额外注入 `ChatGPT-Account-Id`
- `reasoningEffort` 会继续透传给 `genai`

这样做的原因是：

- 自写 Codex 事件/工具适配层维护成本偏高
- 后续更适合跟随 `genai` 升级来获得兼容性修复

### 6. 部门与全局可选模型

Codex 已接入部门模型选择链路。

已完成：

- 供应商页可配置 Codex
- 全局文本模型候选包含 Codex
- 部门页驱动模型候选包含 Codex
- 部门保存校验允许 Codex 文本模型

## 与原计划的差异

以下内容与最初计划相比发生了调整：

### 1. 不再自写独立 Codex provider

原计划曾准备把 Codex 从 `genai OpenAIResp` 完全拆出去，后续已经撤回。

最终决定：

- 保留独立认证层
- 保留独立前端组件
- 聊天运行时继续复用 `genai`

### 2. 不再向用户展示固定 Base URL

原计划曾考虑在界面上说明固定地址。

最终实现：

- 前端不展示固定地址提示
- 后端保存时强制固定 `base_url`

### 3. 模型策略从“静态候选 + 手填兜底”收紧为固定预设

当前实现更保守：

- 使用固定的 5 个预设模型
- 刷新模型时也返回同一组固定值
