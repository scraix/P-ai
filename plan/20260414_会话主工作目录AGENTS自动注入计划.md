# 20260414_会话主工作目录AGENTS自动注入计划

## 1. 目标

在聊天系统提示词组装的最后阶段，检测当前会话是否存在“用户额外挂载并设为 main 的工作目录”。若该目录根下存在全大写 `AGENTS.md`，则读取其内容并追加到系统提示词末尾；若不存在则跳过。

本次改动只服务于“当前项目工作准则自动带入 prompt”，不改变系统工作目录语义，不扫描其他目录，不新增配置项。

---

## 2. 范围

### 2.1 包含

1. 在后端 prompt 组装链路增加“会话 main workspace 的 `AGENTS.md` 注入”能力。
2. 明确只读取当前会话中用户指定的 main workspace 根目录下的 `AGENTS.md`。
3. 为聊天正式发送链路接入该能力。
4. 为 prompt 预览链路接入同一能力，保证预览与真实发送一致。
5. 增加最小必要测试，覆盖命中、跳过和边界条件。

### 2.2 不包含

1. 不读取 `agents.md` 小写文件。
2. 不回退到 system workspace。
3. 不扫描 secondary workspace。
4. 不递归搜索子目录。
5. 不新增 UI 开关。
6. 不在本次引入裁剪、摘要或分段注入策略，首版按全文注入处理。

---

## 3. 目标语义

仅当同时满足以下条件时才注入：

1. 当前会话存在 `level == main` 的工作目录。
2. 该工作目录不是 built-in system workspace。
3. 该 main workspace 根目录下存在全大写 `AGENTS.md`。
4. 文件内容可成功读取，且去除空白后非空。

任一条件不满足，则直接跳过，不报错给用户。

---

## 4. 设计约束

### 4.1 工作目录判定约束

必须基于“当前会话显式配置的 workspace 列表”做判断，不能调用“找默认工作目录并自动回退 system workspace”的逻辑。因为本需求只关心用户主动指定的项目目录，不关心系统内建工作目录。

建议实现口径：

1. 从 `conversation.shell_workspaces` 中筛选 `level == main`。
2. 排除 `built_in == true` 的 workspace。
3. 对命中的 workspace 做规范化与 canonical 解析。
4. 只取该目录根路径，不延展到其他目录。

### 4.2 注入位置约束

不要改写 `agent.system_prompt` 原文。

应复用现有 `ChatPromptOverrides.system_preamble_blocks` 机制，把 `AGENTS.md` 包装成单独 block，追加到系统 preamble 末尾。这样可以保持：

1. 现有人格提示词结构不被破坏。
2. 调试预览和正式发送使用同一注入方式。
3. 后续如果要扩展更多运行时 block，仍能统一收口。

### 4.3 文件名约束

只认全大写 `AGENTS.md`。

理由：与外部规范保持一致，避免项目内出现“小写也能工作、外部却不识别”的双重语义。

---

## 5. 实现方案

### 5.1 新增辅助能力

在 Rust 后端新增一组小型 helper，职责建议如下：

1. 解析当前会话用户指定 main workspace 根路径。
2. 在该根路径下拼接 `AGENTS.md`。
3. 判断文件是否存在且为普通文件。
4. 读取文本内容。
5. 将内容包装为 prompt block，例如：

```xml
<workspace agents>
以下内容来自当前主工作目录根下的 AGENTS.md，请视为该项目开发准则。

...原文...
</workspace agents>
```

block 名称可在实现时再定，但应表达“这是当前项目工作准则”，避免与系统人格提示词混淆。

### 5.2 聊天发送链路接入

在正式发送前构建 `ChatPromptOverrides` 的位置追加该 block。

当前这里已经在挂：

1. hidden skill snapshot
2. todo guide
3. remote im runtime activation

本次应把 `AGENTS.md` block 也作为同级 preamble block 注入。

### 5.3 Prompt 预览链路接入

`get_prompt_preview` 的 Chat 模式目前也会通过 `ChatPromptOverrides.system_preamble_blocks` 注入额外 block。

本次必须同步接入 `AGENTS.md` block，保证：

1. 预览看到什么，真实发送就是什么。
2. 用户可直接在系统 prompt 预览中验证该功能是否生效。

### 5.4 日志策略

增加一条聚合型 INFO 日志即可，避免噪音。

建议字段：

1. 任务名：`[AGENTS注入]`
2. 状态：开始 / 完成 / 跳过 / 失败
3. main workspace 路径
4. 是否命中 `AGENTS.md`
5. 读取字符数
6. 跳过原因或异常信息

注意：

1. 文件不存在属于“跳过”，不是失败。
2. 文件读取异常可记为“失败”，但不应中断主聊天流程，应降级为跳过。

## 7. 测试计划

至少覆盖以下场景：

### 7.1 命中注入

1. 会话存在用户指定 main workspace。
2. 根目录下存在 `AGENTS.md`。
3. 构建结果的 preamble 中包含注入 block 与文件正文。

### 7.2 无 main workspace

1. 会话没有用户指定 main workspace。
2. 结果应跳过注入。

### 7.3 main workspace 为 built-in / system

1. 即使其路径存在 `AGENTS.md`，也应跳过。
2. 防止系统工作目录内容被重复读入。

### 7.4 只有 secondary workspace

1. secondary 根下即使有 `AGENTS.md` 也不注入。

### 7.5 文件不存在

1. 应跳过，不报错。

### 7.6 文件读取失败

1. 应记录失败信息。
2. 主流程继续，不中断 prompt 构建。

如测试搭建成本较高，至少保证“命中注入 + system 跳过 + 无文件跳过”三类核心回归。

---

## 8. 风险与取舍

### 8.1 Token 增长

首版按全文注入，可能增加 prompt token。

当前取舍：

1. 先保证语义正确。
2. 若后续出现明显上下文压力，再补长度裁剪或摘要策略。

### 8.2 路径语义误判

最大风险是错误地把 system workspace 当成项目 main workspace。

规避方式：

1. 只看会话 `shell_workspaces`。
2. 显式排除 built-in。
3. 不使用“默认工作目录回退”逻辑。

### 8.3 预览与真实请求不一致

如果只改发送链路、不改预览链路，调试会失真。

因此本次必须把预览一并纳入范围。

---

## 9. 验收标准

满足以下条件即可视为完成：

1. 当前会话挂载项目为 main workspace 时，若根目录存在 `AGENTS.md`，系统提示词末尾能看到对应内容。
2. 未挂载 main workspace 时，不会读取 system workspace，也不会自动回退。
3. 只有 secondary workspace 时，不注入。
4. Prompt 预览与正式发送结果一致。
5. 相关测试通过，且不影响现有 prompt 组装主链路。

---

## 10. 执行顺序

1. 先补 workspace 判定与 `AGENTS.md` 读取 helper。
2. 接入正式聊天发送链路。
3. 接入 prompt 预览链路。
4. 补测试。
5. 本地跑最小验证。

---
