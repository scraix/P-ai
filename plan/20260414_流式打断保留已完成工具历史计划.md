# 20260414 流式打断保留已完成工具历史计划

## 背景

当前聊天流式阶段已经支持展示本轮工具时间线，但用户在流式过程中点击 stop 后，整条草稿消息会被前端清理。若后端 stop 持久化结果里不包含已完成工具历史，这些工具痕迹会一起消失。

现状已经确认：

- 前端 `stopChat()` 会移除当前流式 draft；
- 后端 `stop_chat_message` 只持久化部分 assistant 文本与 reasoning；
- 工具循环内部其实已经累计了已完成的 `tool_history_events`；
- 一旦用户 stop，正在执行的 chat/tool future 会被 abort，未走到正常 `assistant_message_commit`；
- 因此 stop 后已完成工具历史不会进入最终消息，导致聊天记录失真。

用户期望很明确：

- stop 只代表“后续生成被打断”；
- stop 之前已经完成的工具调用仍然成立；
- 这些已完成工具应写入最终消息并保留可见。

## 目标

1. stop 后保留已完成工具历史，不再让其随 draft 一起消失。
2. stop 持久化出的 assistant 消息应同时包含：
   - `partial_assistant_text`
   - `partial_reasoning_standard`
   - `partial_reasoning_inline`
   - 已完成的 `tool_history_events`
3. 当前正在执行、尚未完成的工具不记入最终消息。
4. 正常完成链路与 stop 链路对“已完成工具”的语义保持一致。
5. 尽量最小改动，不重做整套流式协议。

## 当前代码现状

### 前端

- `src/features/chat/composables/use-chat-flow.ts`
  - `stopChat()` 会清理当前 draft，并依赖后端返回的 `assistantMessage` 回填最终消息；
  - 若返回消息不带工具历史，流式阶段已展示过的工具就会消失。

### 后端 stop 持久化

- `src-tauri/src/features/system/commands/chat_and_runtime/core_commands.rs`
  - `stop_chat_message` 当前只接收部分文本 / reasoning；
  - 持久化 assistant message 时未写入 `tool_history_events`。

### 后端工具循环

- `src-tauri/src/features/chat/model_runtime/provider_and_stream/tool_loop.rs`
  - 每个工具完成后，会把工具调用与工具结果追加到 `tool_history_events`；
  - 这些数据目前停留在工具循环局部上下文中。

### 后端正常提交链路

- `src-tauri/src/features/system/commands/chat_and_runtime/core_send_inner.rs`
  - 正常完成时会走最终 assistant message commit；
  - stop 时由于 future 被 abort，不会进入这条正常提交链路。

## 核心设计口径

### 1. stop 语义

stop 的语义定义为：

- 中断“未完成的生成过程”；
- 保留“已经成立的历史事实”。

这里的“历史事实”包括：

- 已输出的 assistant 部分文本；
- 已输出的 reasoning；
- 已完成并已拿到结果的工具调用历史。

不包括：

- 当前尚未完成的工具调用；
- 尚未形成稳定结果的后续模型输出。

### 2. 已完成工具的判定边界

本次计划按工具循环已有写入时机定义“已完成”：

- 只有在工具实际返回后，且对应事件已追加进 `tool_history_events`，才视为已完成；
- 若 stop 发生在工具执行中，该工具不写入 stop 后的最终消息。

### 3. 最终消息表达

stop 后落地的最终 assistant message 应表达为：

- “消息生成被打断”；
- “但以下工具已经执行完成并产生结果”。

形式上不额外新增特殊消息类型，仍沿用现有 assistant message 结构，只补齐工具历史字段。

## 实施方案

### 1. 为 inflight chat 增加“已完成工具历史”运行时缓存

目标：让 stop 命令在 chat/tool future 被 abort 之后，仍能拿到本轮已经完成的工具历史。

建议方向：

- 在当前 inflight chat 对应的运行时状态中，增加可更新的 `completed_tool_history_events` 缓存；
- 工具循环每次确认工具完成并追加本地 `tool_history_events` 后，同步写入这份运行时缓存；
- 缓存内容只保存“已完成”的工具事件，不保存进行中的工具。

要求：

- 缓存更新必须是增量且幂等，不因 stop 与正常完成竞态写出重复事件；
- 缓存作用域仅限当前 inflight 会话，不跨消息复用。

### 2. stop 命令读取缓存并写入 assistant message

处理项：

- 扩展 `stop_chat_message` 所使用的持久化数据来源；
- 在 stop 时从 inflight runtime 中取出当前已完成的 `completed_tool_history_events`；
- 创建 assistant message 时，把这部分工具历史一并写入。

结果要求：

- 若没有已完成工具，则行为与当前一致，仅保存部分文本 / reasoning；
- 若已有已完成工具，则最终消息必须包含对应工具历史。

### 3. 正常完成与 stop 完成共用一致语义

处理原则：

- stop 链路补的是“原本正常 commit 才会带上的已完成工具事实”；
- 不应让 stop 结果与正常完成结果在同一批已完成工具上出现结构差异；
- 若运行时缓存已存在，正常完成链路也要检查是否需要共用或清理，避免双写冲突。

### 4. 前端保持现有 stop 回填模型，仅依赖后端结果变完整

本次前端原则上不重构：

- 继续在 stop 后移除 draft；
- 继续使用后端返回的 `assistantMessage` 刷新正式消息；
- 由于正式消息已带 completed tool history，前端自然能保留这些工具痕迹。

若联调发现前端正式消息渲染对 stop 返回的工具历史存在遗漏，再做最小补丁修正；否则不扩大前端改动面。

## 风险与关注点

### 1. stop 与工具完成之间的竞态

风险：

- 工具刚完成、缓存尚未写入时，用户立刻 stop，可能出现边界丢失。

应对：

- 把缓存写入时机放在“工具完成并追加事件”的同一临界区；
- 避免先更新局部变量、后异步补写全局缓存。

### 2. 重复写入或顺序错乱

风险：

- 正常完成链路与 stop 链路若同时读取/提交同一批工具历史，可能重复；
- 多次 stop 或异常回收时可能写出不一致消息。

应对：

- 以单条 inflight 记录为唯一数据源；
- stop 成功提交后清理对应 inflight 状态；
- 缓存结构保持与最终消息结构一致，避免转换时重排。

### 3. 未完成工具误记入历史

风险：

- 若缓存写入点放得过早，执行中的工具会被错误保留。

应对：

- 仅在工具调用完成并拿到最终结果后写入缓存；
- `running` 态、开始事件、未完成参数都不进入最终消息。

## 验证计划

### 后端

- `cargo check --manifest-path src-tauri/Cargo.toml`
- 增加或补充测试，至少覆盖：
  1. stop 发生在第 N 个工具执行中时，前 N-1 个已完成工具被保留；
  2. stop 发生在首个工具执行中且该工具未完成时，不写入任何工具历史；
  3. 正常完成链路下工具历史与当前行为一致；
  4. stop 后 assistant message 同时带部分文本 / reasoning / tool history。

### 前端联调

- `pnpm -s exec tsc --noEmit`
- 手动验证以下场景：
  1. 连续多个工具后 stop，已完成工具仍显示在正式消息中；
  2. 仅执行到一半的当前工具不会显示为已完成；
  3. stop 后 draft 消失，但正式消息保留先前已完成工具；
  4. 正常完成时展示不回退。

## 待确认决策

1. 本次只保留已完成 `tool_history_events`，不额外持久化“进行中的流式工具展示态”。
2. 本次不新增 stop 专用消息类型，仍沿用现有 assistant message 结构。
3. 本次以“工具结果已落入 `tool_history_events`”作为唯一完成判定标准。

## 实施顺序

1. 梳理 inflight chat / tool abort 状态结构，确认缓存挂载点。
2. 在工具循环完成点把已完成工具历史同步写入运行时缓存。
3. 扩展 stop 持久化链路，把缓存中的 completed tool history 写入 assistant message。
4. 补充测试与联调验证，确认 stop/正常完成两条链语义一致。
