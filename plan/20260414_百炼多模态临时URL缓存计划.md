# 百炼多模态临时 URL 缓存计划

## 背景

当前项目的多模态消息主要以本地二进制转 `base64` 的方式进入聊天链路，再由通用 OpenAI 风格适配层发给上游。

这套方式对大多数 OpenAI 兼容接口是可用的，但对阿里云百炼存在一个专门差异：

- 百炼在调用多模态模型时，支持先把本地文件上传到临时存储空间，再通过临时 `oss://` URL 调用模型
- 该临时 URL 默认有效期 48 小时
- 临时 URL 与模型名绑定，不同模型之间不能复用
- 使用临时 URL 调用时，请求头必须带 `X-DashScope-OssResourceResolve: enable`

参考文档：

- https://help.aliyun.com/zh/model-studio/get-temporary-file-url

## 目标

在不改动前端消息协议的前提下，为百炼供应商新增一套“多模态临时 URL 缓存”能力：

- 当当前 API 的 `base_url` 命中 `dashscope.aliyuncs.com` 时，认定为阿里云百炼
- 对百炼多模态请求，优先复用消息上已缓存且未过期的临时 URL
- 若没有缓存或缓存已过期，则自动重新上传本地媒体并获取新的临时 URL
- 自动把新的 URL 与过期时间写回消息元数据，供后续复用
- 自动为百炼请求补 `X-DashScope-OssResourceResolve: enable`

## 范围

本次仅覆盖聊天主链路中的多模态输入缓存：

- 用户消息中的图片
- 用户消息中的音频
- 最新用户消息与仍参与上下文的历史用户消息

本次不做：

- 前端新增配置项
- 非百炼供应商适配
- PDF 单独上传 URL 化
- 生产级永久文件存储

## 设计决定

### 1. 缓存记录放在 `provider_meta`

不修改 `MessagePart` 基础结构，也不增加新的前端输入字段。

原因：

- 当前 `MessagePart` 已被多处链路复用，直接扩字段会放大改动面
- `provider_meta` 已经承载附件、隐藏提示、召回 ID 等附加上下文，适合承载供应商专属缓存信息
- 这样可以把百炼缓存限定为“消息级 provider 扩展信息”，不污染通用消息模型

建议结构：

```json
{
  "aliyunMultimodalCache": {
    "items": [
      {
        "kind": "image",
        "mime": "image/webp",
        "contentHash": "sha256:...",
        "model": "qwen-vl-plus",
        "url": "oss://dashscope-instant/...",
        "expiresAtMs": 1770000000000
      }
    ]
  }
}
```

### 2. 命中条件

缓存命中必须同时满足：

- 当前请求供应商为百炼
- 当前模型名与缓存项的 `model` 一致
- 当前媒体内容哈希与缓存项的 `contentHash` 一致
- `expiresAtMs` 晚于当前时间
- `url` 非空

任一条件不满足，都视为未命中并重新上传。

### 3. 供应商识别规则

百炼识别采用最小且稳定的规则：

- 解析当前 `ResolvedApiConfig.base_url`
- host 命中 `dashscope.aliyuncs.com` 时视为百炼

不新增单独配置开关。

### 4. 发送前转换

实现落点放在 Rust 后端“发送前”阶段，而不是前端：

- 前端继续上传原始本地媒体
- 消息持久化仍以原始媒体引用为主
- 真正准备发给模型时，再判断是否需要把 `base64` 转成百炼临时 URL

这样做的好处：

- 不影响其他供应商
- 不改变现有聊天输入协议
- 历史消息回放时也可以统一复用同一套逻辑

## 实现方案

### 1. 新增百炼缓存数据结构与工具函数

位置建议：

- `src-tauri/src/features/core/domain/` 下的运行时/聊天类型文件
- 或放在聊天发送链路附近，保持局部封装

需要包含：

- 缓存项结构体
- 解析/写回 `provider_meta` 的辅助函数
- 媒体内容哈希函数
- 过期判断函数

### 2. 新增百炼临时 URL 上传服务

需要封装以下两个 HTTP 步骤：

1. `GET https://dashscope.aliyuncs.com/api/v1/uploads?action=getPolicy&model=<model>`
2. 按返回的表单字段上传文件到 `upload_host`

输出：

- `oss://...` 临时 URL
- 统一按“当前时间 + 48h”记录 `expiresAtMs`

说明：

- 上传凭证与模型绑定，所以缓存键必须带 `model`
- 使用当前供应商的 API Key

### 3. 在聊天发送前为百炼做媒体 URL 化

落点优先考虑：

- `core_send_inner.rs`
- `conversation.rs`
- `core_provider_calls.rs`

原则：

- 在构造 `PreparedPrompt` 或构造 `genai::ContentPart` 之前完成转换
- 对最新用户消息和参与上下文的历史用户消息都生效
- 命中缓存就直接改为 URL 输入
- 未命中则上传并把结果写回对应消息的 `provider_meta`

### 4. 百炼请求头自动补齐

当请求命中百炼且本轮存在 `oss://` 多模态 URL 时：

- 自动在请求头中加入 `X-DashScope-OssResourceResolve: enable`

该逻辑应在 provider 请求头合成处完成，避免散落在各个调用分支。

## 风险与约束

### 1. URL 与模型绑定

同一媒体对不同百炼模型不能复用，所以缓存项必须带模型名。

### 2. 临时 URL 不适合作为长期持久化资源

虽然缓存会落在消息元数据里，但它本质上只是“可失效缓存”，不是长期资源引用。

因此：

- 历史消息中的 URL 过期后必须允许自动重传
- 原始媒体引用必须继续保留，不能只剩 URL

### 3. 只对百炼生效

这次实现必须严格限制在 `dashscope.aliyuncs.com`，避免误伤其他 OpenAI 兼容服务。

## 验证计划

开发验证至少覆盖：

- `cargo check`
- 一次图片消息首发：触发上传并写回缓存
- 同模型二次发送：直接命中缓存，不重复上传
- 过期缓存或模型切换：重新上传
- 非百炼供应商：不触发该逻辑
- 请求头包含 `X-DashScope-OssResourceResolve: enable`

## 预期结果

完成后，百炼多模态输入的行为将变为：

- 默认仍从本地媒体开始
- 首次发送时自动上传到百炼临时存储
- 48 小时内同模型复用缓存 URL
- 过期后自动重建 URL
- 用户不需要手动处理上传和 URL 替换
