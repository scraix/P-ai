# 20260414_输入面板指令系统计划

## 1. 目标

在聊天输入面板中新增“指令系统”。用户可通过底部按钮或 `Tab` 快捷方式唤起指令面板，在面板中选择一个或多个预设指令，再将这些指令作为“文本附件”随本轮消息一并发送。

本次“指令”定义如下：

1. 指令是用户预设的常用提示词片段。
2. 指令由用户主动选择并发送，不属于 skill。
3. 指令表达“本轮想要什么”，通常不包含“具体怎么做”。
4. 指令可重复复用，可在不同轮次多次选择。

---

## 2. 范围

### 2.1 包含

1. 在聊天输入面板底部新增“指令”按钮。
2. 点击按钮后，在输入框上方拉起指令面板。
3. 支持在输入框空态或光标聚焦时，按 `Tab` 唤起指令面板。
4. 指令面板支持键盘导航：
   - 上 / 下：移动当前焦点
   - 左 / 右：在存在网格布局时横向移动，若首版仍为单列列表，则左 / 右允许退化为无动作或等价移动
   - `Enter`：使用当前焦点指令
5. 指令支持多选；同一条指令可在未来轮次重复使用。
6. 已选指令在发送前以“文本附件”形式挂在本轮用户消息上。
7. 发送时将所选指令写入 `extraTextBlocks`，而不是拼进系统提示词。
8. 发送后清空当前轮次已选指令，不影响预设本身。
9. 在设置页增加“指令预设”管理，至少支持：
   - 新增
   - 编辑
   - 删除
10. 为最小核心交互补充前端与后端必要测试。

### 2.2 不包含

1. 不把指令做成 skill。
2. 不把指令注入 system prompt。
3. 不做变量模板，例如 `{{topic}}`。
4. 不做分类、搜索、收藏、最近使用。
5. 不做项目级指令作用域。
6. 不做部门 / 人格差异化指令集。
7. 不做自动发送；`Enter` 仅表示“使用该指令”，不是直接发送整条消息。
8. 不做复杂拖拽排序；首版如需排序，仅保留数据顺序写回。

---

## 3. 目标语义

### 3.1 指令与 skill 的边界

1. skill 是模型按需读取的能力说明。
2. 指令是用户显式发出的本轮附加意图。
3. skill 属于系统运行时参考资料；指令属于用户消息内容。

因此本次实现必须把指令挂到用户消息上，而不是放进系统 preamble block。

### 3.2 “文本附件”的落点

当前发送链路已支持 `extraTextBlocks`，显示链路也会把这类附加文本渲染为独立块。

首版将“文本附件”解释为：

1. 指令在消息结构中进入 `extraTextBlocks`。
2. 指令不污染输入框正文。
3. 指令在历史消息中可见，便于回看与追溯。

### 3.3 多指令规则

1. 单次发送允许携带多个指令。
2. 同一轮内是否允许重复添加同一指令，建议首版去重处理；若用户再次选择同一条，仅保持一份。
3. 不同轮次可重复使用同一条指令预设。

## 4. 交互设计

### 4.1 面板开合

1. 输入面板底部新增“指令”按钮。
2. 点击后在输入框上方展开指令面板，再次点击则收起。
3. 输入框聚焦时按 `Tab`：
   - 若面板未打开，则阻止默认焦点跳转并打开面板。
   - 若面板已打开，则将键盘焦点切换到指令列表导航态。

### 4.2 指令选择

1. 指令面板展示预设指令列表，建议显示：
   - 名称
   - 可选的简短正文预览
2. 当前键盘焦点项需有明显高亮。
3. `Enter` 对当前焦点项执行“使用该指令”：
   - 将其加入本轮已选指令列表
   - 保持面板继续可用，便于连续选多个
4. 鼠标点击某条指令时，也执行同样的“使用”操作。

### 4.3 已选指令展示

在输入框附近展示当前轮次已选指令，建议采用 badge/tag 形式，至少支持：

1. 显示名称
2. 单独移除
3. 发送前可见

### 4.4 发送行为

1. 用户正文仍来自原输入框。
2. 所选指令在发送时作为 `extraTextBlocks` 附带发送。
3. 发送成功发起后，清空当前轮次已选指令。
4. 若本轮没有正文，但选了指令：
   - 证据不足，当前是否允许“空正文 + 仅指令发送”还未明确。
   - 建议实现前再确认；若不额外确认，首版可沿用现有发送校验，不主动放宽。

## 5. 数据设计

### 5.1 预设结构

建议在聊天设置中新增指令预设字段，例如：

```ts
type PromptCommandPreset = {
  id: string;
  name: string;
  prompt: string;
};
```

聊天设置扩展为：

```ts
instructionPresets: PromptCommandPreset[];
```

### 5.2 发送结构

本轮已选指令不必进入长期聊天设置，仅保存在前端当前会话输入态。

发送时将其映射为 `extraTextBlocks`。为避免与其他附加文本块混淆，建议统一包装格式：

```xml
<user instruction>
总结成表格
</user instruction>
```

若同轮选中多个，则按选择顺序依次追加多个 block。

## 7. 键盘交互约束

### 7.1 `Tab` 冲突处理

`Tab` 目前原本用于浏览器默认焦点切换。本次若要拿来打开指令面板，必须显式 `preventDefault()`。

建议口径：

1. 仅当聊天输入框聚焦且未组合输入时，拦截 `Tab`。
2. 面板打开后，方向键由面板消费；未打开时，方向键仍保持原有输入行为。
3. `Enter` 在面板聚焦态表示“使用指令”；在普通输入态表示“发送消息”。

### 7.2 上下左右导航

证据不足：你要求支持“上下左右”，但当前面板是否最终采用网格布局尚未定。

因此建议实现口径：

1. 数据层先按一维索引维护焦点。
2. 首版 UI 若为单列列表：
   - 上 / 下生效
   - 左 / 右可等价为上 / 下，或保持原位
3. 若实现时决定做双列网格，再让左 / 右按列移动。

这样能先满足快捷操作，再避免为“左右”过度设计布局系统。

## 8. 测试计划

### 8.1 前端交互

至少覆盖：

1. 点击“指令”按钮可展开 / 收起面板。
2. 输入框按 `Tab` 可打开面板。
3. 方向键能移动焦点。
4. `Enter` 能把当前焦点指令加入已选列表。
5. 同一指令重复使用时，按产品口径验证是否去重。

### 8.2 发送结果

至少覆盖：

1. 已选多个指令时，发送 payload 中包含多个 `extraTextBlocks`。
2. `extraTextBlocks` 顺序与用户选择顺序一致。
3. 发送后当前轮次已选指令清空。

### 8.3 持久化

至少覆盖：

1. 指令预设能随聊天设置正确加载。
2. 新增 / 编辑 / 删除后能正确保存。
3. 老数据缺少该字段时能安全回退为空数组。

## 9. 风险与取舍

### 9.1 `Tab` 影响原生焦点行为

这是本次最大交互风险。

处理方式：

1. 仅在输入框聚焦时拦截。
2. 仅在聊天窗口中生效。
3. 保持逻辑简单，避免把整个页面做成复杂焦点管理系统。

### 9.2 指令与其他 `extraTextBlocks` 混合

当前附件解析、PDF 解析等也可能使用 `extraTextBlocks`。

处理方式：

1. 用统一 `<user instruction>` 包装明确来源。
2. 不新增新的消息字段，避免全链路改造。

### 9.3 设置页复杂度上升

首版若把指令管理做得太重，会拖慢主链路。

处理方式：

1. 先做最小 CRUD。
2. 暂不做排序、搜索、分组、导入导出。

## 10. 验收标准

满足以下条件即可视为完成：

1. 输入面板存在“指令”按钮，并能展开指令面板。
2. 输入框按 `Tab` 能唤起指令面板。
3. 面板支持键盘导航，并可通过 `Enter` 使用指令。
4. 单次消息可携带多个指令。
5. 所选指令会作为文本附件进入 `extraTextBlocks` 发送。
6. 发送后本轮所选指令被清空，但预设仍保留可复用。
7. 设置页可维护指令预设，重启后仍保留。

---

## 11. 执行顺序

1. 先扩展聊天设置数据结构，接通指令预设持久化。
2. 在设置页落最小指令预设管理 UI。
3. 在输入面板新增指令按钮、面板与已选区。
4. 接入 `Tab`、方向键、`Enter` 的指令导航逻辑。
5. 将所选指令接入发送链路的 `extraTextBlocks`。
6. 补最小验证与回归测试。

---

## 12. 当前待确认结论

已确认：

1. 指令允许多选。
2. 指令作为文本附件加入本轮消息。
3. `Tab` 可唤起指令框。
4. 面板内可用方向键选择，`Enter` 使用指令。
5. 指令预设可重复复用。

仍待最终拍板：

1. 是否允许“空正文 + 仅指令发送”。
2. 同一轮内重复选择同一指令时，是保留重复项还是自动去重。
3. 首版单列列表下，左 / 右方向键是等价移动还是无动作。

若这三点不额外变更，可按“沿用现有发送校验 + 同轮自动去重 + 单列时左右无动作”的默认口径推进。
