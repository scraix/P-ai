# 20260415_设置标题栏搜索跳转计划

## 1. 目标

在设置窗口标题栏新增一个搜索框。用户输入关键词后，系统基于当前语言的 i18n 文案检索相关设置分类，展示结果列表；选择结果后仅切换到对应设置 tab，不做页内精准定位、不做高亮。

本次方案明确采用：

1. 不引入完整 schema。
2. 不手写大量设置项元数据。
3. 基于“各 tab 实际使用过的 i18n key”自动生成搜索索引。
4. 只跳 tab，不滚动到具体控件。

---

## 2. 范围

### 2.1 包含

1. 在设置标题栏区域新增搜索输入框。
2. 支持输入后展示匹配结果列表。
3. 搜索结果按 tab 去重，避免同一个 tab 因多个命中文案重复出现。
4. 选择结果后切换 `configTab`。
5. 当前语言切换后，搜索匹配文本同步切换到对应语言。
6. 顺手补齐当前设置页中遗漏的硬编码文案到 i18n，避免搜索遗漏。

### 2.2 不包含

1. 不做页内锚点定位。
2. 不做高亮。
3. 不做模糊排序优化到搜索引擎级别。
4. 不做所有动态 key 的全自动完美覆盖。
5. 不做欢迎页之外的“全文检索”体验。

---

## 3. 核心思路

### 3.1 搜索对象

搜索对象不是 DOM，不是 schema，而是“每个 tab 组件里实际出现过的 i18n key 对应文本”。

目标效果：

1. 搜索“视觉 / vision”命中 `chatSettings`。
2. 搜索“主题 / language / 热键 / 日志 / 技能”等命中对应 tab。
3. 用户不需要知道精确控件位置，只需快速跳到相关设置分类。

### 3.2 索引来源

为每个 tab 维护一份“该 tab 使用的 i18n key 列表”，但这份列表尽量从组件现有 `t("...")` 使用方式自动提取，而不是再人工维护一套业务搜索文本。

实现上允许两层：

1. 第一层：自动扫描 tab 组件源码中显式写出的 `t("...")` key。
2. 第二层：对扫描不到但又很关键的文本做少量补充，例如动态 key、运行时拼接文案、暂未 i18n 化的遗留文本。

这仍然属于“自动提取为主”，不是 schema，也不是纯手写索引。

### 3.3 结果粒度

索引最终聚合到 tab 级别，而不是设置项级别。

建议结果结构：

```ts
type ConfigSearchResult = {
  tab: ConfigTab;
  title: string;
  matchedTexts: string[];
};
```

其中：

1. `tab` 用于切换目标页签。
2. `title` 用于结果展示。
3. `matchedTexts` 用于辅助显示命中了什么词，可首版只取前 1-2 条。

## 4. 架构设计

### 4.1 数据层

新增一个设置搜索索引模块，例如：

`src/features/config/search/config-search-index.ts`

职责：

1. 定义 tab 与组件文件的对应关系。
2. 基于 i18n key 和当前语言翻译表生成可搜索文本。
3. 输出 tab 级聚合结果。

### 4.2 提取层

新增一个轻量提取模块，例如：

`src/features/config/search/config-search-source.ts`

职责：

1. 声明每个 tab 对应的组件文件。
2. 存放自动提取结果或最小补充映射。
3. 隔离“搜索来源”与“搜索匹配逻辑”。

首版不强求运行时真的去读源码文件；更稳妥的做法是：

1. 先人工整理一版“从各 tab 源码提取出的 key 列表”。
2. 结构设计按未来可脚本化生成来组织。
3. 后续若需要，再把这一步替换成脚本自动生成。

这样能先把功能上线，再决定要不要把“提取”彻底自动化。

### 4.3 UI 层

建议把搜索框放在设置窗口顶部、左侧 tab 菜单上方或右侧内容区顶部，避免侵入通用聊天/归档标题栏逻辑。

原因：

1. 当前标题栏是多窗口复用，加入设置专属搜索会提升耦合。
2. `ConfigView` 本身就是设置导航中心，最适合持有搜索状态。
3. 只影响设置窗口时，逻辑更收敛。

如果最终仍坚持“标题栏里搜索”，则建议：

1. 标题栏只发出 `update:configSearchQuery`。
2. 实际搜索结果列表与切 tab 逻辑仍由设置视图持有。

## 5. 实现步骤

### 5.1 第一步：补齐 i18n 遗漏

先检查设置页各 tab 中仍然写死的中文文案，补齐到 i18n。

原因：

1. 既然搜索基于 i18n，就要尽量保证搜索源统一。
2. 这也符合当前项目“所有文本都应 i18n”的目标。

### 5.2 第二步：建立 tab 搜索源

为各 tab 建立 key 列表，例如：

```ts
{
  chatSettings: [
    "config.tabs.chatSettings",
    "config.chatSettings.visionApi",
    "config.chatSettings.responseStyle",
    "config.chatSettings.pdfReadMode",
  ]
}
```

再通过当前语言 `t(key)` 转换为可搜索文本。

### 5.3 第三步：实现搜索匹配

匹配规则首版保持简单：

1. 搜索词转小写。
2. tab 标题和 tab 内所有文本转小写。
3. `includes` 命中即可。
4. 结果按 tab 去重。

### 5.4 第四步：接入设置 UI

新增：

1. 搜索输入框。
2. 下拉结果列表。
3. 点击结果后调用 `requestTabChange(targetTab)`。

### 5.5 第五步：验证

至少验证：

1. 中文界面下能搜中文。
2. 英文界面下能搜英文。
3. 同一 tab 多个命中只出现一条结果。
4. 切换 tab 后结果面板关闭。

---

## 6. 风险与边界

### 6.1 动态 key 覆盖不全

例如 ``t(`responseStyle.${id}`)`` 这类动态 key 不一定能靠静态方式完整提取。

处理方式：

1. 首版允许少量手工补充。
2. 不为了完全自动化引入额外复杂度。

### 6.2 非设置项文案噪音

如果某个 tab 内提示文案很多，可能导致“搜某个普通词也命中”。

处理方式：

1. 首版优先收录标题、区块标题、关键按钮文案。
2. 少收录冗长提示文本。

### 6.3 标题栏接入成本

如果坚持把输入框放在通用窗口标题栏，接线会比放在 `ConfigView` 更复杂。

处理方式：

1. 首版优先实现为设置视图内顶部搜索。
2. 若你确认必须放标题栏，再单独扩展标题栏 props 和事件。

---

## 7. 建议结论

本次最稳妥的 v1 为：

1. 搜索只跳 tab。
2. 搜索源以 i18n 为主。
3. 先补齐遗留硬编码文案。
4. 采用“半自动提取 + 少量手工补充”的 tab 级索引。

这样能在不引入 schema、不做大重构的前提下，得到一个可用、可维护、且与现有结构相容的设置搜索功能。
