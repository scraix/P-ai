# 20260418_工具阶段触发流式通道重建计划

## 一、文档目的

这份计划用于解决当前聊天链路中的一个顽固问题：

- LLM 流式输出期间，前端偶尔会失去实时更新
- 这类问题不一定发生在页面刷新时
- 一旦当前实时通道失效，后续工具执行与后续 LLM 输出可能继续在后台进行，但前端会出现“黑洞”

当前计划不尝试一次性重做整条实时链路，而是基于现有架构，利用“工具执行不是流式”的天然阶段边界，引入一次可恢复的流式通道重建机制。

## 二、问题收口

当前真正的问题不是：

- 页面刷新时 Tauri callback warning

而是：

- 在正常聊天过程中
- 即使不 reload
- 某些流式阶段也会偶发失去前端实时更新

已确认的一点是：

- 工具执行开始时，本轮 LLM 到“发起工具调用”为止的内容其实已经到达稳定点
- 工具本身不是 token 流，而是离散阶段

这意味着：

- 工具阶段天然适合作为“流式通道恢复点”

## 三、核心思路

本方案的核心是：

1. LLM 流式阶段继续走 `Channel`
2. 工具执行阶段不依赖 `Channel`，只走普通事件
3. 每次工具开始执行时，由后端主动通知前端“请重建流式通道”
4. 前端收到通知后立即新建 `Channel`
5. 前端通过单独命令把新 `Channel` 绑定回当前运行中的会话/轮次
6. 工具执行结束后，如果还会继续发起下一段 LLM 流式输出，则改走新绑定的 `Channel`

也就是说：

- 通道不是一轮只建立一次
- 而是按“流式阶段”短生命周期存在
- 工具阶段则成为天然的“重绑窗口”

## 四、目标效果

计划完成后，预期效果应为：

1. 某一段 LLM 流式期间即使前端偶发丢失实时更新
2. 只要进入工具执行
3. 后端就会触发一次通道重建
4. 后续新的 LLM 流式阶段可以继续恢复前端可见性

换句话说：

- 允许某一段流式出问题
- 但不能让整轮后半程一直黑洞到底

## 五、协议设计

### 1. 新增后端事件

建议新增一类普通前端事件，例如：

- `easy-call:stream-rebind-required`

建议 payload 至少包含：

- `conversationId`
- `requestId`
- `phaseId`
- `reason`

其中：

- `conversationId`
  - 标识当前会话
- `requestId`
  - 标识当前整轮请求
- `phaseId`
  - 标识下一段流式阶段
- `reason`
  - 当前阶段为什么要求重建，第一版可固定为 `tool_start`

### 2. 新增前端重绑命令

建议新增单独 Tauri 命令，例如：

- `rebind_active_chat_stream_channel`

输入建议包含：

- `conversationId`
- `requestId`
- `phaseId`
- `onDelta`

其中：

- `onDelta` 仍然由前端 `new Channel()` 提供

### 3. 后端运行态

后端需要新增一层“当前可替换流式通道”的运行态，而不是继续把 channel 只作为 `send_chat_message` 的一次性参数。

建议后端运行态至少记录：

- `conversationId`
- `requestId`
- `currentPhaseId`
- `currentDeltaChannel`

并允许在轮次中途被 `rebind_active_chat_stream_channel` 更新。

## 六、阶段模型

本方案建议把一次聊天轮次划分为多个阶段。

### 1. 流式阶段

例如：

- `llm_stream_phase_1`
- `llm_stream_phase_2`

特征：

- 有 token/delta 持续产生
- 需要 `Channel`

### 2. 工具阶段

例如：

- `tool_execution_phase`

特征：

- 工具不是流式 token
- 只需要普通事件：
  - `tool_start`
  - `tool_done`
  - `tool_failed`

### 3. 阶段切换

推荐的阶段切换为：

1. `llm_stream_phase_start`
2. `llm_stream_phase_end`
3. `tool_start`
4. 前端重建 channel
5. `tool_done / tool_failed`
6. 下一段 `llm_stream_phase_start`

## 七、前端职责

前端需要新增以下职责：

### 1. 监听重绑事件

当前聊天窗口应监听：

- `easy-call:stream-rebind-required`

并在满足以下条件时处理：

- 当前窗口处于对应会话
- `requestId` 与当前轮次一致

### 2. 收到事件后立即新建 channel

前端收到事件后：

1. `new Channel<AssistantDeltaEvent>()`
2. 挂好当前 delta handler
3. 调用 `rebind_active_chat_stream_channel(...)`

### 3. 不再要求一条 channel 贯穿整轮

当前前端逻辑默认是：

- `sendChat()` 建一次 channel
- 后续整轮都指望它活着

本方案下需要改为：

- `sendChat()` 建初始 channel
- 后续允许被新的阶段 channel 替换

## 八、后端职责

后端需要新增以下职责：

### 1. 在工具开始时发重绑事件

每次工具开始执行时：

1. 先发 `tool_start`
2. 再发 `stream-rebind-required`

或者二者合并为：

- `tool_start` payload 中带 `requiresRebind = true`

第一版建议分开，语义更清楚。

### 2. 接受新的 channel 绑定

后端需允许在当前轮次未结束时更新：

- 当前活跃流式 channel

### 3. 后续流式继续走新 channel

工具结束后，如果下一轮 provider 还会继续流式输出：

- 统一从当前最新绑定的 channel 发送 delta

### 4. 工具阶段不依赖 channel

工具开始、完成、失败都应继续走普通事件，不把工具状态绑定到当前流式 channel 是否还活着。

## 九、与当前架构的差异

### 1. 当前架构

当前大致是：

- 前端 `sendChat()` 新建一次 channel
- channel 作为 `send_chat_message` 参数传给后端
- 后端整轮都拿着这一条 channel 发 delta

问题是：

- 只要这条 channel 某个阶段失效
- 整轮后续都容易一起黑洞

### 2. 本方案

本方案改为：

- 初始流式阶段仍由 `sendChat()` 提供 channel
- 但后续每到工具阶段，都允许重建
- 后端不再认为 channel 是整轮不可变参数
- 而是认为它是“当前流式阶段的活动通道”

## 十、第一版实现范围

第一版建议只做最小闭环：

1. 工具开始时发 `stream-rebind-required`
2. 前端收到后立即重建 channel
3. 后端接受新的 `onDelta`
4. 下一段 LLM 流式输出改走新 channel

第一版先不做：

- 自动检测通道已死
- seq/replay
- snapshot 恢复
- 多窗口跨会话竞争绑定

也就是说，第一版先只实现：

- “工具阶段强制重绑一次”

## 十一、验收口径

完成后应至少满足：

1. 一轮聊天中如果发生工具调用
2. 工具开始时后端会发出重绑请求事件
3. 前端能在当前会话里收到该事件并立即重建 channel
4. 工具执行结束后，下一段 assistant 流式输出能够继续显示
5. 工具状态本身不再依赖 channel 实时存活

## 十二、后续演进方向

如果第一版有效，后续再考虑演进到：

1. 后端统一阶段 ID 与 request ID
2. 更完整的流式阶段模型
3. 失活通道检测
4. snapshot + replay
5. 更彻底的“流式阶段通道”设计

当前这份计划只解决一个现实问题：

- 即使某一段流式偶发断掉
- 至少在工具阶段可以重新拉起下一段流式可见性
