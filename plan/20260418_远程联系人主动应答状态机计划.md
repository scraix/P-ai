# 20260418_远程联系人主动应答状态机计划

## 一、目标

为远程联系人补一套明确的“主动应答状态机”，解决当前仅靠单次激活判定、缺少持续状态的问题。

本次目标不是重做远程 IM 整体架构，而是在现有：

- 远程消息入队
- 主会话调度
- `remote_im_send` 决策

之上补齐：

- 是否在看群
- 是否正在处理
- 忙时如何补跑
- 何时离场

## 二、核心设计

状态拆两层：

1. 活跃状态
- `away`：离场
- `present`：在场

2. 工作状态
- `idle`：空闲
- `busy`：忙碌

初始值：

- 所有远程联系人在运行时初始为：`away + idle`

说明：

- 活跃状态决定“这名联系人当前是否值得继续关注和参与”
- 工作状态决定“当前是否可以立刻发起一次处理”

## 三、配置项

联系人侧新增/明确两类配置：

1. 激活策略
- `never`：不激活
- `always`：始终激活
- `keyword`：关键字激活

2. 耐心策略
- `patience_seconds`
- 默认建议值：`420` 秒，即 7 分钟

补充：

- `activation_keywords` 仅在 `keyword` 下生效
- 原有 `activation_cooldown_seconds` 第一版不继续作为主规则使用，避免和“忙碌 / 待办 / 耐心离场”重复叠加

## 四、运行时字段

以下字段只放内存，不持久化：

- `presence_state: away | present`
- `work_state: idle | busy`
- `has_pending: bool`
- `last_success_reply_at: Option<String>`

说明：

- `last_success_reply_at` 不持久化
- 应用重启后，联系人一律回到 `away + idle`
- 这是运行时行为，不需要跨重启保留

## 五、消息进入时的处理规则

### 1. 离场状态 `away`

收到新远程消息时：

1. 判断是否满足激活策略
- `never`：不激活
- `always`：激活
- `keyword`：命中关键字才激活

2. 若不满足激活策略
- 不回复
- 不进入主调度
- 只更新消息游标
- 不在离场期持续维护大窗口上下文

3. 若满足激活策略
- 切换为 `present`
- 然后立即进入“在场状态”的处理逻辑

### 2. 在场状态 `present`

收到新远程消息时：

1. 如果 `work_state == busy`
- 不立即再次调度
- 只设置：
  - `has_pending = true`

2. 如果 `work_state == idle`
- 立即触发一次调度
- 切换：
  - `idle -> busy`

## 六、调度结束后的状态迁移

无论调度成功还是失败，调度结束后都必须：

- `busy -> idle`

然后继续判断后续动作。

### 1. 本轮成功回复

判定标准：

- 必须是“真正成功发出去”
- 只有真正成功外发远程联系人后，才允许更新 `last_success_reply_at`

行为：

- 更新：
  - `last_success_reply_at = now`
- 保持：
  - `presence_state = present`

### 2. 本轮 `remote_im_send = no_reply`

行为：

1. 若 `last_success_reply_at` 不存在
- 第一版建议：保持 `present`
- 但继续看 `has_pending` 是否需要补跑

2. 若存在 `last_success_reply_at`
- 计算当前时间与上次成功回复时间差
- 如果差值超过 `patience_seconds`
  - `present -> away`
- 否则保持 `present`

说明：

- 不是“只要 no_reply 就离场”
- 而是“长时间都没有必要回复，且耐心耗尽后才离场”

### 3. 调度失败

行为：

- `busy -> idle`
- 保持当前 `presence_state`
- 若存在 `has_pending = true`，仍允许下一轮继续补跑

## 七、待办标记规则

`has_pending` 只用布尔，不用计数。

原因：

- 联系人忙碌期间可能收到多条新消息
- 当前调度器本来就允许连续回复
- 第一版不要求把每条新消息都单独拆成待办计数
- 只需要确保忙完后会继续消费到最新状态，不丢消息

规则：

1. 忙碌中收到新消息
- `has_pending = true`

2. 当前调度结束后
- 如果 `has_pending == true`
  - 先清为 `false`
  - 立刻再触发一次新调度
  - 再次进入 `busy`

## 八、消息游标与压缩消息

这部分采用更轻的方案：

1. 离场期只维护消息游标
2. 真正从 `away -> present` 时，再插入一条简要压缩消息
3. 压缩消息不经过 LLM
4. 压缩消息需要持久化，作为后续继续处理的基座

### 8.1 离场期只做什么

当联系人处于 `away` 时：

1. 继续吸收新消息
2. 更新“已看到的消息游标”
3. 不做主动回复
4. 不持续维护完整大上下文快照

这里的游标需要独立设计，不要和联系人其它配置字段混在一起。

### 8.2 `away -> present` 时做什么

当一条新消息使联系人从 `away` 切到 `present` 时，先判断处理模式。

1. 如果是 `qa` / 无上下文模式
- 不生成复杂压缩
- 插入一条固定模式提示消息：
  - `当前是问答模式，尽可能快地回答问题`
- 持久化这条消息作为边界

2. 如果是连续上下文模式
- 取过去约 `20K` 规模的可读对话正文窗口
- 插入一条简要压缩消息：
  - `上下文窗口已经滑动`
- 持久化这条消息作为边界

说明：

- 这不是 LLM 摘要
- 不调用模型
- 不做复杂总结
- 只插入一条轻量边界消息

### 8.3 `20K` 的具体含义

这里的 `20K` 不是：

- `20K` 条消息
- 也不是整段原始会话 JSON

这里的 `20K` 指的是：

- 过去约 `20K` 规模的“可读对话正文窗口”

构成规则：

1. 只取可读对话消息
2. 不取工具消息
3. 不取摘要消息
4. 不取系统控制噪音

也就是只保留这些内容：

1. 远程联系人消息
2. 本地用户消息
3. 助理对远程联系人的真实回复消息
4. 必要时保留少量人工可读的失败结果消息

明确排除：

1. 工具调用消息
- `tool_calls`
- `tool`
- MCP / tool status

2. 摘要 / 压缩类消息
- 旧摘要
- 历史压缩结果本身

3. 纯系统控制消息
- 激活判定日志
- 内部路由元数据
- append / history 事件痕迹

4. 无正文的壳消息
- 空文本
- 纯附件占位且无可读说明

建议实现方式：

- 从最新往前回溯
- 累积到约 `20K` 的可读正文规模就停止
- 然后插入一条轻量压缩消息作为边界

### 8.4 压缩消息的目标

压缩消息不追求丰富信息，而是只承担两个作用：

1. 标记“这里之前的窗口已经被压缩过”
2. 作为后续继续回溯的边界

目标是：

- 让联系人重新进场时，不需要重新吃完整历史
- 后续统计窗口时最多回溯到“上一次压缩消息”为止
- 不做复杂摘要推理，降低失真风险

### 8.5 持久化边界

需要持久化的是：

- 该联系人最近一次“进场时插入的压缩消息”
- 对应的消息游标边界

补充：

- `qa` / 无上下文模式联系人，同样插入固定模式提示消息
- 但这类联系人不需要复杂上下文压缩

不需要持久化的是：

- `last_success_reply_at`
- `presence_state`
- `work_state`
- `has_pending`

也就是说：

- 运行时状态重启后清空
- 但“联系人压缩消息边界”可以保留

### 8.6 后续消息如何继续

在联系人已经 `present` 后：

1. 后续新消息不再回头重压过去 `20K`
2. 直接基于：
- 已持久化的压缩消息
- 从压缩消息之后新增的消息
继续处理

统计窗口时的规则：

1. 只统计正文数量 / 正文规模
2. 最多回溯到上一次压缩消息
3. 不跨压缩消息继续往前扫

这样能避免：

- 每次都重新扫描太长历史
- 压缩逻辑过重
- 状态机和上下文窗口强耦合

### 8.7 游标要求

这里的消息游标需要在计划和实现里单独明确：

1. 它表示压缩消息覆盖到哪条消息
2. 它表示后续增量消息从哪条开始继续算
3. 即使本轮 `no_reply`，游标也要推进
4. 防止重复吃同一批消息

## 九、与当前架构的关系

本次不重做主会话调度器。

仍然复用现有：

- 远程 IM 入队
- `scheduler`
- `send_chat_message_inner`
- `remote_im_send`

但新增一层“远程联系人主动应答运行时状态”：

1. 入站消息先查联系人运行时状态
2. 再决定：
- 只更新游标
- 还是触发主调度

也就是说：

- 现有主调度器继续负责“真正处理一轮”
- 新状态机负责“值不值得开始这一轮”

## 十、边界与异常

### 1. 应用重启

- 所有联系人回到 `away + idle`
- `last_success_reply_at` 清空
- 不保留上次耐心状态

### 2. 联系人配置变更

- 新配置在下一条新消息进入时生效
- 不要求强制中断当前正在执行的一轮

### 3. 忙碌中再次收到大量消息

- 只保留 `has_pending = true`
- 不积累多层重复任务

### 4. `no_reply` 连续出现

- 只有在超过耐心时间后才降回 `away`
- 不做瞬时离场

### 5. 调度失败

- 不直接离场
- 防止偶发失败导致联系人状态来回抖动

### 6. 用户无感

- 用户不应该直接感知到“在场 / 离场 / 忙碌 / 空闲”这些状态
- 这些状态只作为系统内部调度依据
- 不额外暴露成前台状态提示

## 十一、实施步骤

1. 新增远程联系人运行时状态结构
- 放在运行时内存，不进持久化

2. 抽象远程联系人主动应答判定入口
- 统一处理：
  - `away`
  - `present`
  - `idle`
  - `busy`
  - `has_pending`

3. 单独设计并实现联系人消息游标与压缩消息边界结构
- 明确游标字段
- 明确压缩消息持久化位置

4. 接入远程消息入站链路
- 在现有 `remote_im_enqueue_message_internal(...)` / 调度入口之前做状态机决策

5. 实现 `away -> present` 时的压缩消息插入逻辑
- 连续上下文模式：
  - 插入 `上下文窗口已经滑动`
- `qa` / 无上下文模式：
  - 插入 `当前是问答模式，尽可能快地回答问题`
- 落持久化

6. 接入调度完成回调
- 根据：
  - 成功回复
  - `no_reply`
  - 失败
  更新运行时状态

7. 保证状态更新原子化
- 防止忙碌结束、待办补跑、新消息并发进入时产生竞态

8. 增加日志
- 用中文日志明确记录：
  - 联系人状态迁移
  - 激活原因
  - 离场原因
  - 待办补跑
  - 压缩消息生成与游标推进

## 十二、测试要求

至少覆盖：

1. 初始状态为 `away + idle`
2. `away` 下未命中激活策略，只更新游标不回复
3. `away` 下命中激活策略，切到 `present` 并触发调度
4. `present + idle` 收到消息会进入 `busy`
5. `present + busy` 收到消息只打 `has_pending`
6. 调度结束后若 `has_pending=true`，会立刻补跑
7. `remote_im_send = no_reply` 且超过耐心时间后离场
8. `remote_im_send = no_reply` 但未超过耐心时间时保持在场
9. 调度失败后回到 `idle` 但不离场
10. 重启后运行时状态重置，不读取上次 `last_success_reply_at`
11. `away -> present` 时会插入压缩消息，而不是做复杂摘要
12. 后续消息会基于“压缩消息 + 增量消息”继续，而不是每次全量回扫
13. `qa` / 无上下文模式联系人会插入固定模式提示消息作为边界
14. 只有真正成功外发后才更新 `last_success_reply_at`
15. 状态更新原子化，不会出现重复补跑或错误离场

## 十三、验收标准

满足以下条件视为完成：

1. 远程联系人存在明确的 `away/present + idle/busy` 双层状态
2. “离场看群、在场处理、忙时挂待办、耐心耗尽后离场”完整跑通
3. 重启后状态重置，不保留运行时临时状态
4. 联系人的消息游标与压缩消息边界清晰，且不会反复吃旧消息
5. `qa` / 无上下文模式联系人会插入固定模式提示消息，但不会走复杂压缩
6. “成功回复”的定义严格限定为真正成功外发后才更新 `last_success_reply_at`
7. 状态更新具备原子性，不会因竞态产生重复补跑或错误离场
8. 现有远程 IM 主链不被破坏
9. 有足够测试覆盖关键状态迁移
