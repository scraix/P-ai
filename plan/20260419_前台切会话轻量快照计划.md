# 20260419_前台切会话轻量快照计划

## 一、背景

当前前台切换会话时，首屏消息依赖后端命令：

- `switch_active_conversation_snapshot(...)`

前端链路位于：

- `src/UnifiedWindowApp.vue`
- `switchUnarchivedConversation(...)`
- `requestConversationSnapshot(...)`

后端实现位于：

- `src-tauri/src/features/system/commands/config_and_persona/unarchived_conversations.rs`

实测中已经出现：

- `[会话锁] 持有完成 ... held_ms=1072`
- `[会话切换计时] ... 开始请求后台异步补消息 totalMs=1655.6`

这说明：

1. 前台切会话主路径中的“同步快照”过重。
2. 用户真正关心的是“尽快看到目标会话最近消息”。
3. 当前快照命令在热路径里顺手做了大量非首屏必要工作。

本计划目标不是立刻实现，而是把后续改造路径深化到“可直接施工”的程度，避免真正开始动手时再次重复梳理。

---

## 二、问题定义

### 1. 当前慢的不是“异步补消息”

项目里已经存在单独的异步补消息链路：

- 前端：`requestConversationMessagesAfterAsync(...)`
- 后端命令：`request_conversation_messages_after_async(...)`
- 后端查询：`resolve_unarchived_conversation_messages_after(...)`

它的职责本来就比较单纯：

- 找目标会话
- 用 `afterMessageId` 找锚点
- 取锚点之后的消息
- 锚点缺失时回退 recent limit
- materialize 媒体引用

因此，当前用户体感中的“切会话很慢”，主因不是“没有补消息方法”，而是：

- **在补消息之前，先走了一条过重的同步快照链路**

### 2. 当前慢的也不是“同步快照这个概念”

“同步快照”本身没有问题。  
前台切会话时，先拿一份“足够把首屏画出来”的当前状态，本来就是合理设计。

真正的问题是：

- 当前实现的 `switch_active_conversation_snapshot(...)`
- 并不是“只读目标会话最近消息”
- 而是被污染成了一条“顺手做很多维护、汇总、修正和写回”的大杂烩命令

所以本计划本质上是在解决：

- **前台切会话热路径职责污染**

---

## 三、当前真实链路

### 1. 前端当前切换流程

当前前端切换一个未归档会话时，大致流程为：

1. `switchUnarchivedConversation(conversationId)`
2. 先把本地缓存消息顶到前台
3. `nextTick()`
4. `requestAnimationFrame()`
5. 记录 `前台切换首帧完成`
6. 调 `requestConversationSnapshot(...)`
7. 后端执行 `switch_active_conversation_snapshot(...)`
8. 返回最近一段消息 + todo + 整份未归档会话摘要
9. 前端应用 snapshot
10. 再调用 `requestConversationMessagesAfterAsync(...)`
11. 后端后台补消息，最后 emit `easy-call:conversation-messages-after-synced`

### 2. 当前前端时序的关键事实

当前日志里的：

- `开始请求后台异步补消息 totalMs=1655.6`

并不是“异步补消息自己耗时 1655ms”，而是：

- 从切会话开始到“终于准备发起异步补消息”为止的累计耗时

这意味着：

- **前面那条同步快照链已经先吃掉了绝大多数时间**

### 3. 当前后端 snapshot 做了什么

`switch_active_conversation_snapshot(...)` 当前会在持有 `conversation_lock` 的热路径里做：

- 读取 `app_config`
- 读取 `app_data`
- `normalize_single_active_main_conversation(...)`
- `normalize_foreground_conversation_departments(...)`
- `resolve_unarchived_conversation_index_with_fallback(...)`
- `ensure_unarchived_conversation_not_organizing(...)`
- 遍历 foreground 会话修正 `status`
- 读取目标会话全部消息并截取最近 `50` 条
- 读取当前 todo / todos
- 收集整份 `unarchived_conversations` 摘要
- 如果有变更则 `persist_app_data_conversation_runtime_delta(...)`
- 解锁后再 `materialize_chat_message_parts_from_media_refs(...)`

### 4. 当前链路的本质问题

这条命令混杂了三类职责：

1. 首屏显示职责
2. 会话列表摘要刷新职责
3. 数据归一化与持久化职责

也就是说：

- 用户只是切到一个会话
- 但系统却顺手做了“全局列表刷新 + 状态修正 + 可能写回”

这就是当前热路径被污染的核心原因。

---

## 四、目标

新增一条真正面向前台首屏切换的“轻量快照”接口，要求：

- 只读取指定会话
- 只返回当前前台首屏最小必要字段
- 不同步构建整份未归档会话摘要
- 不做不必要的归一化与持久化
- 不改变现有异步补消息链路

### 目标结果

1. 切换会话后更快看到目标会话最近消息。
2. 会话锁持有时间显著下降。
3. `前台切换首帧完成 -> 开始请求后台异步补消息` 的延迟明显下降。
4. 后续列表摘要、badge、todo、补消息仍保持可用。

### 目标的衡量方式

本计划不提前承诺一个脱离上下文的固定毫秒数，但至少要求：

- 轻量快照的锁持有时间明显低于当前观测到的 `~1072ms`
- 轻量快照链不再同步触发整份 `unarchived_conversations` 摘要生成
- 切会话总链路中，“等待 snapshot 返回”不再成为压倒性大头

---

## 五、非目标

本计划先不处理：

- `request_conversation_messages_after_async(...)` 的协议重构
- 会话列表摘要缓存化重构
- badge 与预览消息规则重构
- 会话排序规则调整
- 切会话后的 replay / seq / snapshot 总线设计
- 图片/PDF 等重媒体场景下更细粒度的 materialize 降级

换句话说，本计划只解决：

- **“前台切会话首屏快照”这条热路径**

---

## 六、目标链路应该长什么样

### 1. 目标前端时序

前端切会话改成三条解耦链：

1. 本地缓存首帧
2. `get_foreground_conversation_light_snapshot(...)`
3. 前台立即应用最近消息 + todo
4. 并行或稍后：
   - 刷新会话列表摘要
   - 发起 `requestConversationMessagesAfterAsync(...)`

也就是把：

- “前台首屏内容”
- “会话列表摘要”
- “补尾巴消息”

拆成三条职责明确的链。

### 2. 目标后端原则

轻量快照命令只负责：

- 找会话
- 校验会话可用于前台显示
- 取最近 N 条消息
- 读取当前 todo
- 返回首屏最小必要字段

它不负责：

- 全局摘要
- 全局状态修正
- 全局持久化副作用

---

## 七、建议新增的专用接口

第一版建议新增新命令，暂定名：

- `get_foreground_conversation_light_snapshot`

### 1. 返回字段建议

- `conversationId`
- `messages`
- `hasMoreHistory`
- `currentTodo`
- `currentTodos`

### 2. 明确不返回

- `unarchivedConversations`
- 非当前会话 preview / unread / runtime state
- 非当前会话 badge 信息
- 前台列表摘要数据

### 3. 为什么字段只保留这些

因为切会话首屏真正要画的只有：

- 最近消息
- 当前会话的 todo 信息
- 是否还有更多历史

这些之外的数据，都不应阻塞首屏。

---

## 八、轻量快照应该保留与剥离的职责

### 1. 必须保留

轻量快照中可以保留的逻辑：

- 目标会话存在性校验
- 会话是否允许在前台显示的必要校验
- `ensure_unarchived_conversation_not_organizing(...)`
- 最近消息截取
- `hasMoreHistory` 计算
- 当前 todo / todos 读取
- 必要的媒体 materialize

### 2. 必须剥离

以下内容不应再出现在轻量快照热路径里：

- `normalize_single_active_main_conversation(...)`
- `normalize_foreground_conversation_departments(...)`
- 整份 foreground 会话 `status` 修正
- `collect_unarchived_conversation_summaries(...)`
- `persist_app_data_conversation_runtime_delta(...)`

### 3. 原因

这些逻辑的问题不是“功能错”，而是“位置错”：

- 它们更像后台维护职责
- 或更像列表刷新职责
- 不应该跟“只想尽快看到目标会话最近消息”的首屏命令绑死

---

## 九、建议的前后端拆分方案

### 第一层：首屏快照

负责：

- 返回当前会话最近消息
- 返回 todo
- 返回 `hasMoreHistory`

特点：

- 前端 `await`
- 必须足够轻

### 第二层：列表摘要刷新

负责：

- `unarchived_conversations` 摘要
- preview / unread / badge / sorting 相关展示信息

特点：

- 不阻塞首屏
- 可并行、可稍后、可独立失败重试

### 第三层：异步补消息

负责：

- 基于 `afterMessageId` 补全首屏之后的增量消息

特点：

- 维持现有协议
- 暂不重构

## 十、建议的实施顺序

### 阶段一：补齐现状日志，不改协议

目标：

- 先把当前 snapshot 的真实耗时拆清

建议日志字段：

- `lock_wait_ms`
- `lock_held_ms`
- `messages_slice_ms`
- `todo_read_ms`
- `summaries_ms`
- `persist_ms`
- `materialize_ms`
- `total_ms`

目的：

- 让后面改造不是“凭感觉”，而是能看到到底哪一段最重

### 阶段二：新增轻量快照命令

目标：

- 实现 `get_foreground_conversation_light_snapshot`

要求：

- 与旧 `switch_active_conversation_snapshot(...)` 并存
- 先不删旧命令
- 先证明它显著更轻

### 阶段三：前端切会话改走轻量快照

改造点：

- `switchUnarchivedConversation(...)`
- `requestConversationSnapshot(...)`

目标：

- 前台切会话首屏只依赖轻量快照
- 列表摘要和补消息走后续链路

### 阶段四：清理旧快照职责

待新链路稳定后，再决定：

- 旧 `switch_active_conversation_snapshot(...)` 是否保留
- 或者下沉为非热路径管理命令

## 十一、日志与指标深化

为了避免后面继续靠猜，第一版建议统一以下日志口径。

### 1. 后端轻量快照日志

- `[前台轻量快照] 开始`
- `[前台轻量快照] 完成`
- `[前台轻量快照] 失败`

建议字段：

- `conversation_id`
- `message_count`
- `has_more_history`
- `lock_wait_ms`
- `lock_held_ms`
- `materialize_ms`
- `total_ms`

### 2. 旧重型快照日志

旧命令也应该补细分日志，避免之后无法对比：

- `[前台重型快照] 开始`
- `[前台重型快照] 完成`
- `[前台重型快照] 失败`

建议细分字段：

- `conversation_id`
- `message_count`
- `summaries_count`
- `normalize_changed`
- `persist_changed`
- `lock_wait_ms`
- `lock_held_ms`
- `summaries_ms`
- `persist_ms`
- `materialize_ms`
- `total_ms`

### 3. 前端切换链路日志

保留并对齐：

- `前台切换首帧完成`
- `开始请求后台异步补消息`

新增：

- `轻量快照返回`
- `轻量快照应用完成`
- `会话列表摘要刷新开始`
- `会话列表摘要刷新完成`

### 4. 判断优化是否成功的日志特征

如果改造成功，日志上应该能看到：

1. 轻量快照总耗时明显低于旧快照
2. 轻量快照锁持有时间显著下降
3. `开始请求后台异步补消息` 提前出现
4. 摘要刷新即便稍后完成，也不再阻塞首屏

## 十二、风险与规避

### 风险 1：首屏和列表摘要短暂不同步

拆分后可能出现：

- 右侧当前消息先变了
- 左侧列表摘要稍后才刷新

这是可接受的，只要：

- 不串会话
- 不错消息
- 不错误显示空列表

### 风险 2：旧数据依赖的归一化副作用丢失

当前重型快照里顺手做的归一化，可能已经在无意中兜底了一些旧数据问题。

所以：

- 第一版不要直接删旧命令
- 先并存
- 先观测
- 再判断哪些副作用必须保留、哪些应搬离热路径

### 风险 3：媒体 materialize 仍可能是慢点

即便去掉列表摘要和持久化，若最近 `50` 条里图片/PDF 很多，`materialize_chat_message_parts_from_media_refs(...)` 仍可能占用明显时间。

这不是本计划第一阶段的主问题，但日志必须把它单独拆出来，避免再次把“媒体解析慢”误判成“快照整体逻辑慢”。

### 风险 4：前端切换链路双源数据带来抖动

若轻量快照、列表摘要刷新、异步补消息应用顺序处理不好，可能出现：

- 前台消息短暂回跳
- 列表 badge 短暂错位
- 同一会话被多次应用不同快照

因此前端改造时要明确：

- 谁是首屏真源
- 谁只能做增强补充
- 谁不能覆盖已经生效的前台消息

## 十三、验收标准

满足以下条件可视为本计划落地完成：

1. 新增 `get_foreground_conversation_light_snapshot`
2. 前台切会话改走轻量快照
3. 切会话时不再同步构建整份 `unarchived_conversations`
4. 会话锁持有时间显著下降
5. `前台切换首帧完成 -> 开始请求后台异步补消息` 的延迟明显下降
6. 不出现错会话、缺消息、todo 丢失、列表长时间不同步
7. 旧重型快照仍可保留作回滚路径，直到新链路稳定

---

## 十四、建议的验证用例

真正实施时，至少应覆盖以下场景：

### 1. 普通文本会话

- 最近消息无图片无附件
- 验证纯文本情况下切会话是否明显提速

### 2. 重媒体会话

- 最近消息中包含图片、附件、音频
- 验证 `materialize` 不会重新成为主瓶颈

### 3. 大量未归档会话

- 未归档会话列表较长
- 验证列表摘要刷新不再阻塞首屏

### 4. 正在 organizing 的会话

- 验证必要保护仍在
- 不会因为轻量化而放出错误前台状态

### 5. 切换后立刻补消息

- 验证 `afterMessageId` 路径仍正确
- 不会丢尾巴消息

---

## 十五、当前判断

当前可以明确得出三点：

1. 项目中已经存在单独的异步补消息方法。
2. 当前没有一条“专门给前台切会话首屏用、只干最少事情的高度优化快照接口”正在被使用。
3. 当前切会话慢的根因，是 `switch_active_conversation_snapshot(...)` 这条热路径承担了过多职责。

因此后续最值得做的不是继续微调现有重型命令，而是：

- **新增一条真正轻量、职责单一的前台快照接口**
