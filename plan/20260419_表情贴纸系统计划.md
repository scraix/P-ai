# 20260419_表情贴纸系统计划

## 一、目标

实现一套统一的表情贴纸系统，同时覆盖：

1. 本地聊天
   - LLM 直接在正文里输出 `:happy:` 这样的贴纸语法
   - 前端把它渲染成贴纸图片，而不是普通文本

2. 远程联系人
   - 助理回复里如果出现 `:happy:`，外发时按顺序拆成：
     - 文本
     - 对应贴纸图片
     - 文本
   - 例如：`你好:happy:我想说`
     - 先发 `你好`
     - 再发 `happy` 对应图片
     - 再发 `我想说`

3. LLM 自主管理贴纸库存
   - 贴纸资源统一放到系统工作目录 `.meme/`
   - 只提供一个很小的 `meme` 工具用于“偷图入库”
   - 删除、重命名、整理继续交给终端/`exec`

4. 随机结果稳定
   - 不是前端渲染时随机
   - 而是在助理消息写入历史记录时，就直接随机选中某一张具体贴纸
   - 该结果写入消息记录，后续本地渲染、切会话、重启、远程外发都复用同一份结果

---

## 二、最终设计边界

### 1. 不做

- 不做 `meme.list`
- 不做 `meme.send`
- 不做前端随机缓存贴纸结果
- 不做“远程联系人发原始 `:happy:` 文本”
- 不做专用删除工具

### 2. 只保留一个 `meme` 工具

工具名：`meme`

第一版只支持：

- `action=steal`

用途：

- 把当前可见图片路径或工作区内图片文件收入 `.meme/`

不承担：

- 列库存
- 直接发送
- 删除
- 重命名

---

## 三、LLM 看到什么

### 1. 可用贴纸名不再通过工具查询

当前有哪些贴纸，应直接注入到模型可见说明里，而不是让模型再调 `meme.list`。

注入位置：

1. `meme` 工具 description
2. 聊天提示词中的一段专门规则块

规则要点：

- 当前可用贴纸名直接列给模型
- 模型想使用贴纸时，直接在正文输出 `:name:`
- 不要编造不存在的贴纸名
- 若当前没有合适贴纸、但上下文里已有图片文件路径，可先调 `meme` 工具收入库存

### 2. 示例

```text
当前可用贴纸：happy、angry、派蒙震惊
使用方式：直接在正文输出 :happy:、:angry:、:派蒙震惊:
不要调用工具发送贴纸；贴纸会在消息落库时自动解析并固化。
```

## 四、资源目录

贴纸库存放在系统工作目录：

```text
.meme/
```

建议结构：

```text
.meme/
  index.json
  items/
    happy/
      001.png
      002.gif
    angry/
      001.webp
```

`index.json` 第一版建议结构：

```json
{
  "version": 1,
  "updatedAt": "2026-04-19T12:00:00Z",
  "items": [
    {
      "name": "happy",
      "aliases": ["开心", "高兴"],
      "files": [
        ".meme/items/happy/001.png",
        ".meme/items/happy/002.gif"
      ]
    }
  ]
}
```

说明：

- `name` 是主要贴纸名
- `aliases` 用于后续兼容别名匹配
- `files` 保存相对工作目录的路径，便于远程发送直接复用

## 五、助理消息落库时的固化逻辑

### 1. 核心原则

随机必须发生在：

- 助理最终文本已经确定
- 准备写入历史消息
- 但尚未真正提交到会话记录时

### 2. 输入

例如助理回复：

```text
你好 :happy: 我想说
```

### 3. 处理

后端在落库前：

1. 解析出 `:happy:` 这类 token
2. 在 `.meme/index.json` 里找到 `happy`
3. 从 `happy` 对应文件列表中选出一张具体图片
4. 把“文本 + 已选中的具体图片”写进消息元数据

### 4. 写入位置

第一版不新增 `MessagePart::Meme`，而是写到 `provider_meta.memeSegments`：

```json
{
  "memeSegments": [
    { "type": "text", "text": "你好 " },
    {
      "type": "meme",
      "name": "happy",
      "filePath": "E:/.../.meme/items/happy/001.png",
      "mime": "image/png"
    },
    { "type": "text", "text": " 我想说" }
  ]
}
```

同时：

- `parts` 里的原始文本仍保留 `你好 :happy: 我想说`
- 这样能保留模型原始输出，便于调试和导出

### 5. 一旦落库就不再随机

后续：

- 本地重渲染
- 切会话
- 应用重启
- 远程联系人外发

都只用这条消息里已经固化好的 `memeSegments`。

---

## 六、本地聊天渲染

本地渲染优先读：

- `provider_meta.memeSegments`

若存在：

- `text` 段按文本显示
- `meme` 段按贴纸图片显示

若不存在：

- 继续走当前普通 Markdown / 文本渲染链

前端不再负责随机选图，只负责：

- 读取消息记录中的已固化结果
- 按需把 `filePath` 读成 data URL 显示

---

## 七、远程联系人外发

### 1. 统一目标

远程联系人外发时，不发送原始 `:happy:` 文本，而是按段拆开发。

### 2. 自动外发

如果当前轮次命中了远程联系人自动外发链路：

- 优先消费已落库消息里的 `provider_meta.memeSegments`
- 顺序组装 `content`：
  - 文本
  - 图片
  - 文本

### 3. 显式调用 `remote_im_send`

如果模型主动调用 `remote_im_send`，同样支持解析 `text` 里的 `:happy:`。

也就是说，远程联系人路径最终统一表现为：

- 本地消息记录里是一份已固化的结构化贴纸段
- 远程发送时按这份结构或按同样语法解析结果进行顺序发送

### 4. 远程顺序要求

这一点是第一版必须保证的验收项：

```text
你好:happy:我想说
```

必须实际发成：

1. 文本：你好
2. 图片：happy 选中的具体图片
3. 文本：我想说

## 八、`meme` 工具设计

### 1. 工具名

`meme`

### 2. 第一版参数

```json
{
  "type": "object",
  "properties": {
    "action": {
      "type": "string",
      "enum": ["steal"],
      "default": "steal"
    },
    "name": {
      "type": "string",
      "description": "收入库存后的贴纸名"
    },
    "source_path": {
      "type": "string",
      "description": "当前可见图片文件路径或工作区图片路径"
    }
  },
  "required": ["name", "source_path"]
}
```

### 3. 行为

`action=steal` 时：

1. 校验 `source_path` 是否存在
2. 校验是否为支持的图片类型
3. 复制到 `.meme/items/<name>/`
4. 更新 `.meme/index.json`
5. 返回成功结果与最终文件路径

### 4. 不负责的事

- 不提供库存列表
- 不直接发送
- 不负责删除和重命名

## 九、实现分层

### 阶段 A：库存与工具

1. 新增 `.meme/` 目录辅助函数
2. 新增 `index.json` 读写
3. 新增 `meme` 工具
4. 前后端默认工具绑定加入 `meme`

### 阶段 B：提示词与工具说明

1. 每轮组装时读取当前可用贴纸名
2. 注入到 `meme` 工具 description
3. 注入到聊天提示词规则块

### 阶段 C：助理消息落库固化

1. 解析助理最终文本中的 `:name:`
2. 选定具体图片
3. 写入 `provider_meta.memeSegments`

### 阶段 D：本地渲染

1. 前端类型补充 `memeSegments`
2. 助理气泡渲染支持贴纸段
3. 不再前端随机

### 阶段 E：远程外发

1. 自动外发链优先消费 `memeSegments`
2. `remote_im_send` 支持 `text` 中贴纸语法
3. 保证文本/图片/文本顺序

---

## 十、验收标准

1. `.meme/` 中已有 `happy`
   - 助理回复 `你好 :happy:`
   - 本地聊天显示为文字 + 贴纸图片

2. 切换会话、刷新前端、重开应用后
   - 同一条消息仍显示同一张贴纸
   - 不会重新随机

3. 远程联系人场景
   - 助理回复 `你好:happy:我想说`
   - 远端实际收到：文本、图片、文本

4. 若贴纸名不存在
   - 本地保留原文 `:unknown:`
   - 不应崩溃

5. `meme` 工具可把工作区图片收入 `.meme`

---

## 十一、风险与注意事项

1. 第一版把结果挂在 `provider_meta`，后续要确认：
   - 导出
   - 归档
   - 补消息
   - 会话投送
   - 审查
   不会把 `memeSegments` 丢掉

2. 本地渲染与远程外发必须消费同一份已固化结果，不能各自随机

3. 如果未来需要：
   - 一个名字多张图随机池管理
   - 别名维护
   - 角色/联系人专属表情包
   可以在 `.meme/index.json` 结构上继续扩展

---

## 十二、结论

第一版最终方案收敛为：

- **贴纸协议**：`:meme_name:`
- **库存目录**：系统工作目录 `.meme/`
- **唯一工具**：`meme`（只负责 `steal`）
- **可用库存暴露方式**：直接注入提示词与工具说明
- **随机时机**：助理消息落库时
- **持久化位置**：`provider_meta.memeSegments`
- **本地支持**：前端按 `memeSegments` 渲染
- **远程支持**：后端按 `memeSegments` 或等价解析结果拆段顺序发送
