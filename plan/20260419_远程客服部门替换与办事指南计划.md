# 20260419_远程客服部门替换与办事指南计划

## 一、目标

废除当前内置“前台”部门，新增并启用新的内置部门“远程客服”。

本次不是简单改名，而是把远程 IM 的承接部门、默认绑定、部门办事指南、远程回复工具说明，统一切换到新的“远程客服”语义。

同时，把你提供的行为准则与决策逻辑，整理成两部分：

- 部门办事指南
- 远程联系人回复决策工具说明

二者分工不同，不再混写。

## 二、明确决策

本次按你的最新口径执行：

- 不保留“前台”部门
- 不做“前台 -> 远程客服”的兼容迁移
- 直接废除旧内置部门
- 直接新增新的内置部门“远程客服”

这意味着：

- 新版本启动后，系统内置部门集合会发生变化
- 远程联系人默认承接部门改为“远程客服”
- 旧的 `front-desk-department` 相关默认注入和修正规则全部移除

## 三、为什么要拆成两部分

你给的原文里混了两种信息：

1. 角色长期行为倾向
- 例如只回应明确提及自己的人
- 不参与敏感争议话题
- 发言频率适中

2. 本轮消息的即时决策规则
- 是否必须回应
- 是否允许主动介入
- 不回应时该如何结束本轮

在我们项目里，这两类内容应该分别落在：

- `部门办事指南`
  - 负责长期角色气质、边界、社交风格、禁区

- `remote_im_send` 工具说明
  - 负责本轮如何做“发 / 不发 / 结束本轮”的决策动作

特别是这句：

> 你的 `should_reply` 决策必须为 `false`

不应该原样保留。

因为我们项目没有让模型输出布尔决策字段，而是要求模型在远程联系人轮次里：

- 要发，就调用 `remote_im_send(action=send, ...)`
- 不发，就调用 `remote_im_send(action=no_reply, status=done)`

所以这类描述必须改写成工具动作口径。

## 四、目标状态

完成后，系统应呈现为：

1. 内置部门中不再有“前台”
2. 内置部门中新增“远程客服”
3. 新远程联系人默认绑定到“远程客服”
4. 配置页、默认注入、修正规则都只认“远程客服”
5. 远程联系人轮次里，模型看到的部门办事指南是“远程客服”版本
6. 远程回复工具说明不再提 `should_reply=true/false`，而是改成项目内真实动作

## 五、办事指南整理方案

### 5.1 部门概述

“远程客服”部门的概述，建议收成一句短的：

- 负责承接远程 IM 联系人的消息，在明确被需要时提供简短、稳妥、友好的回应；不主动卷入无关话题，不参与高风险争议讨论。

### 5.2 部门办事指南

办事指南不再使用“步骤一 / 步骤二 / should_reply”这种偏判题器的写法，而改成“办事指南”风格。

建议收口为以下几块：

#### 1. 回应边界

- 只回应明确提及自己、追问自己、或明确希望自己参与的话题
- 与自己无关的话题不主动加入
- 没有人明确需要自己时，以观察为默认动作

#### 2. 不重复回应

- 不对同一问题重复回答
- 除非对方是在追问、要求补充、或话题出现了新的角度

#### 3. 发言频率

- 保持适中发言占比
- 若近期已经发言较多，应优先观察
- 若刚刚发言但无人接话，应暂缓再次发言

#### 4. 话题风险控制

- 不参与政治敏感、争议激烈、火药味强的话题
- 对模糊、无意义、纯情绪宣泄的话题不主动介入

#### 5. 社交姿态

- 若有人对自己表达不满，应先道歉
- 若群友明确期待自己回应，应优先回应
- 若能明显改善气氛、提供正向价值，可以适度出手

#### 6. 主动介入资格审查

- 主动介入前，先判断当前话题最需要的是哪种角色
- 只有当自己确实适合作为那个角色发言时，才允许主动加入
- 只要有犹豫，就保持观察

#### 7. 最近消息分析口径

- 优先分析最近 7 条消息
- 只围绕最新、最具体、最有价值的话题做判断

### 5.3 需要从办事指南里删掉的内容

以下内容不再放在部门办事指南里：

- `should_reply=true/false`
- “输出某个布尔字段”
- “进入步骤三 / 步骤四”
- “把 should_reply 设为 false”

因为这些是工具决策说明，不是角色长期行为说明。

## 六、工具说明整理方案

远程联系人轮次最终必须通过 `remote_im_send` 工具结束。

因此，原来的：

- “你的 should_reply 决策必须为 false”

要改成项目内的真实动作说明，建议改成：

- 若不满足明确回应条件，或你对是否应该主动介入存在犹豫，则不要发信；在完成本轮分析后，立即调用 `remote_im_send(action=no_reply, status=done)` 明确结束本轮。

同理，必须回应的情况改成：

- 若最新消息明确提及你、向你提问、请求你参与，或对你之前的回答进行追问，则你必须回应，并在完成思考后通过 `remote_im_send(action=send, ...)` 发出回复。

主动介入改成：

- 若没有人明确要求你参与，只有在你确认自己确实是当前最合适发言的角色时，才允许主动回复；否则应使用 `remote_im_send(action=no_reply, status=done)` 结束本轮。

## 七、实现范围

### 7.1 内置部门定义

需要替换默认内置部门定义：

- 删除前台部门默认注入
- 新增远程客服部门默认注入
- 调整内置部门排序

### 7.2 配置修正规则

需要同步修改：

- 配置加载时对内置部门的补齐逻辑
- 配置加载时对内置部门名称/概述/办事指南的修正逻辑
- 任何只认“前台”部门 ID 的规则

### 7.3 远程联系人默认绑定

当前新联系人默认会绑定到旧的远程 IM 承接部门。

这里要改成：

- 新远程联系人默认绑定“远程客服”部门

### 7.4 提示词与工具说明

需要同步修改两条注入链：

1. 部门信息注入
- 让远程联系人轮次看到新的“远程客服”办事指南

2. 远程回复工具说明
- 把 `should_reply=false` 这类说法改成 `remote_im_send(send / no_reply)` 口径

### 7.5 测试

至少补这几类回归：

1. 默认配置生成后，内置部门中存在“远程客服”，不存在“前台”
2. 新远程联系人创建后，默认绑定到“远程客服”
3. 远程联系人提示词里能看到新的办事指南
4. 远程回复工具说明不再出现 `should_reply`

## 八、风险点

### 8.1 旧数据引用旧部门 ID

因为这次不做兼容迁移，历史配置或旧联系人如果仍然绑定旧前台部门，可能出现：

- 找不到部门
- 绑定为空
- 提示词上下文丢失

本次实现时需要明确选择一种策略：

- 启动时直接清空旧绑定
- 启动时改绑到新部门
- 或在校验时报出明确错误

建议实现时采用：

- 发现旧前台部门绑定时，直接改绑到新“远程客服”部门

这不算保留旧部门，而是一次性数据纠偏。

### 8.2 文案重复

如果部门办事指南和工具说明都重复写同一套规则，提示词会变冗长。

实现时应保持：

- 部门办事指南写“长期风格与边界”
- 工具说明写“本轮怎么做 send / no_reply”

## 九、验收口径

完成后应满足：

1. 配置页部门列表里不再出现“前台”
2. 出现新的“远程客服”内置部门
3. 新远程联系人默认绑定到“远程客服”
4. 远程联系人轮次的提示词里，能看到新的“远程客服”办事指南
5. 远程回复工具说明中，不再出现 `should_reply`
6. “必须不回应”的表达已经改成：
   - 不发信
   - 并调用 `remote_im_send(action=no_reply, status=done)` 结束本轮

2. 替换配置修正规则
