# 20260420_主工作目录Git幽灵快照撤回计划

## 背景

当前会话撤回主要依赖现有的消息/工具回退链路：

- 回退会话消息
- 尝试撤回 `apply_patch`
- 回填用户输入

这套链路对“聊天历史撤回”是成立的，但对“主工作目录代码状态恢复”并不稳定，尤其在以下场景中容易失真：

- 本轮改动不止来自 `apply_patch`
- 文件被多工具、多阶段连续改写
- Shell / 外部工具直接修改了主工作目录
- 后续轮次又在同一批文件上继续改动

此前讨论过三种方向：

1. `git reflog`
2. 自动创建普通 `git commit`
3. 类似 Codex 的 ghost snapshot（幽灵快照）

最终结论是：

- `reflog` 不适合作为业务能力依赖
- 普通 `git commit` 会污染用户提交历史，不能接受
- 参考 Codex 的 ghost snapshot 方向更合理

但本项目不应全局无差别启用 ghost snapshot，而应只对“用户添加的主工作目录”生效，并作为现有撤回链路之上的最高一级代码撤回能力。

## 本次目标

建立一套“主工作目录 Git 幽灵快照撤回”设计方案，但本轮只形成计划，不进入实现。

目标如下：

1. 仅针对当前会话的**主工作目录**
2. 仅当主工作目录本身是 **Git 仓库** 时生效
3. 为每条用户消息建立可选的 Git 幽灵快照记录
4. 撤回时优先恢复该条用户消息对应的主工作目录快照
5. 若无快照或目录非 Git，则自动降级为当前既有撤回流程

## 核心结论

### 1. 不使用 reflog

原因：

- `reflog` 是 Git 内部历史，不是稳定业务接口
- 会过期、会被 GC 清理
- 它记录的是引用移动，不是“某条用户消息对应哪个代码状态”

### 2. 不创建普通 commit

原因：

- 会污染用户正常提交历史
- 会干扰分支图、提交审查与版本管理
- 与“会话级撤回”语义不匹配

### 3. 参考 Codex 的 ghost snapshot

目标语义：

- 使用 Git 生成一个“脱离正常分支历史的工作区快照”
- 不更新 branch / HEAD
- 不污染 `git log`
- 撤回时按这个快照恢复工作区

## 范围边界

本计划的设计边界必须严格限制如下：

### 只作用于主工作目录

仅当前会话设置的“主工作目录”参与 Git 幽灵快照。

不纳入范围：

- 附加工作目录
- 系统默认工作区
- 非用户显式添加的其他路径
- shell root 列表中的其他目录

### 只作为最高一级撤回

新能力不是替代现有撤回，而是覆盖在最上层：

1. 若该用户消息存在 Git 幽灵快照记录，则优先做主工作目录 Git 恢复
2. 随后再继续执行现有消息/工具撤回收口
3. 若不存在 Git 幽灵快照记录，则直接走现有撤回流程

### 非 Git 自动降级

若主工作目录：

- 不是 Git 仓库
- 无法创建 ghost snapshot
- 创建过程中被配置关闭

则本轮不报致命错误，直接回退到当前已有撤回逻辑。

## 生命周期设计

### 1. 创建时机

触发时机为：

- 用户消息进入正式处理前

但不是全局无脑创建，而是满足下列条件时才尝试：

- 当前会话存在主工作目录
- 主工作目录是 Git 仓库
- 当前轮次允许使用“主工作目录 Git 撤回”

### 2. 记录内容

每条用户消息最多关联一条 Git 幽灵快照记录。

建议记录：

- `conversation_id`
- `user_message_id`
- `main_workspace_path`
- `ghost_commit_id`
- `created_at`
- `status`

其中：

- `ghost_commit_id` 为 Git 层的恢复依据
- `status` 用于区分 `created / skipped / failed`

### 3. 使用时机

撤回某条用户消息时：

1. 先查该消息是否存在 `ghost_commit_id`
2. 若存在，则先尝试恢复主工作目录
3. 若不存在，则直接跳过 Git 层恢复
4. 无论 Git 层是否存在，都继续进入现有撤回收口逻辑

### 4. 失效与保留

本轮先不做主动清理设计落地，只定义语义：

- 归档会话时，不立即删除 ghost snapshot 设计本身
- 删除会话时，不立即做 Git 对象级删除设计
- 是否引入内部 ref、是否主动删 ref、何时做对象清理，留待下一轮单独讨论

这样做的原因是：

- Git 对象生命周期本身复杂
- 当前优先级是先把“创建/记录/恢复/降级”链路设计正确
- 清理策略应作为独立问题处理，避免本轮把设计复杂度扩散

## 运行规则

### 规则 1：仅主工作目录

只有当前会话主工作目录参与 ghost snapshot。

### 规则 2：每条用户消息最多一条

同一条用户消息只记录一个主工作目录快照，不扩展到多目录。

### 规则 3：失败不阻断对话

若 ghost snapshot 创建失败：

- 本轮对话继续
- 只标记该消息没有 Git 快照
- 撤回时自动降级

### 规则 4：撤回优先级固定

撤回顺序固定为：

1. 主工作目录 Git 幽灵快照恢复
2. 现有消息/工具撤回流程

### 规则 5：无快照即降级

若当前用户消息没有 Git 快照记录，不做额外提示性阻断，直接进入原流程。

## 与现有撤回系统的关系

这套设计不是新建第二套独立撤回系统，而是给现有撤回链路增加“更高优先级的代码状态恢复层”。

现有系统继续负责：

- 消息裁剪
- 工具历史回退
- `apply_patch` 撤回
- 输入框回填

新设计只新增：

- 主工作目录 Git 状态恢复

## 暂不处理的内容

本轮计划明确不处理以下事项：

- 多工作目录同时快照
- 附加工作区快照
- ghost snapshot 的对象清理策略
- 是否挂内部 ref
- ghost snapshot 的 GC / prune 策略
- 非 Git 主工作目录的替代快照方案
- UI 上单独暴露 Git 撤回开关

## 后续实现顺序建议

后续若进入实现，建议顺序如下：

1. 先补“主工作目录是否为 Git 仓库”的检测能力
2. 再补“用户消息 -> ghost snapshot 记录”写入
3. 再补撤回链路的“优先 Git 恢复，否则降级”
4. 最后单独讨论清理策略与对象生命周期

## 当前结论

本方案的最终口径是：

- 只对主工作目录生效
- 只对 Git 仓库生效
- 作为新的最高一级代码撤回能力
- 不污染用户正常提交历史
- 不取代原有撤回系统
- 无快照时自动降级

本计划当前仅用于设计确认，暂不实现。
