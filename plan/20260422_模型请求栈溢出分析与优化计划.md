# 20260422 模型请求栈溢出分析与优化计划

## 一、问题概述

当前项目在以下链路中已经出现过 `thread 'tokio-rt-worker' has overflowed its stack`：

- 工具安全审查（`Tool safety review`）
- 上下文整理 / 归档摘要（`Archive summary` / `SummaryContext`）
- 工具循环长链路中的自动整理前后切换

从现象上看，用户会感知为：

- 一次工具执行或调度中途突然闪退
- 主进程 `p-ai.exe` 直接退出
- 并不是普通模型请求错误提示，也不是可恢复的 `Result::Err`

这个问题的严重性不在“某次请求失败”，而在于：

**模型请求相关代码当前运行在主进程 Tokio worker 上，一旦触发栈溢出，后端主进程会直接退出。**

---

## 二、当前问题的正确分析口径

当前问题不应再以“调用链放大”作为主因来描述。

更准确的定义是：

**单次完整物化超大请求对象、超大日志对象，本身就已经足够危险。**

也就是说：

- 问题不是“只有在长链路里才会炸”
- 问题不是“只有因为多层 `await` 叠加才会炸”
- 问题也不是“只有工具安全审查这种小请求才会触发”

真正的问题是：

- 只要某一步开始完整物化超大的 `PreparedPrompt`
- 或完整物化超大的 `request_log`
- 或完整物化超大的 provider request / 最终 HTTP body
- 即使不讨论“链路传递放大”，单次加载和处理本身也已经危险

因此，本计划后续统一采用如下口径：

1. **主问题是“单对象过大”**
2. **副本并存会进一步恶化，但不是主问题成立的前提**
3. **工具安全审查只是小样本，不是问题主体**
4. **主聊天 / 自动整理 / 归档摘要才是未来真正要承受 1MB~2MB 级请求体的主体**

### 1. 工具安全审查的意义

工具安全审查这条链的意义，不是因为它代表了主问题，而是因为它提供了一个更小的复现样本，用来证明：

- 主进程 worker 上的模型请求边界是脆弱的
- 即使请求体本身不大，发送阶段也可能暴露问题
- 但它不能代表“大请求场景”的全部风险

也就是说：

- 审查链可以帮助定位“哪里会崩”
- 但不能用来替代对主聊天 / 自动整理 / 归档摘要这些大请求路径的分析

### 2. 主聊天 / 自动整理 / 归档摘要才是核心问题场景

真正危险的场景是这些会生成巨大请求体的路径：

- 主聊天长上下文请求
- 工具循环中累计大量工具结果后的继续请求
- 自动整理 / 上下文压缩请求
- 归档摘要请求

这些场景的共同点不是“链很长”，而是：

- 历史消息本体已经非常大
- 单次请求物化时就可能接近 1MB
- 后续如果上下文继续增大，2MB 甚至更高也是可预期的
- 一旦同一轮里再保留平行副本，内存压力会进一步恶化

所以本计划后续分析的主语，统一改成：

**“超大请求对象 / 超大日志对象的单次物化与表示方式”**

而不是：

- “调用链有几层”
- “哪一层先调哪一层”

## 三、已确认事实

### 1. 崩溃发生在主进程 Tokio worker

日志明确显示：

- `thread 'tokio-rt-worker' has overflowed its stack`

说明：

- 不是前端崩溃
- 不是子进程崩溃
- 是 Tauri 后端主进程工作线程栈溢出

### 2. 工具安全审查请求内容本身不是根因

已经确认过：

- 工具安全审查请求只有极少量消息
- 手工原始 HTTP 请求可成功
- 加应用同款 `originator` / `user-agent` 请求头后仍可成功
- 独立 `genai_probe` 进程也可成功

因此可排除：

- 请求体格式非法
- headers 值非法
- 本地兼容服务天然不支持该请求

### 3. 主应用运行态具有“特异性”

同样的模型、同样的 base_url、同样的 headers、同样的 body：

- 独立 probe 成功
- `p-ai.exe` 主应用中失败并触发栈溢出

这说明问题更接近：

- 主进程 worker 上加载和处理请求对象的方式有问题
- 主应用中的请求物化和发送过程比独立样本更重
- 而不是协议本身不可用或服务端天然不支持

### 4. 大请求日志对象已经实锤存在

实际测得：

- `request_log_bytes = 848442`
- `request_log_messages = 135`

也就是：

- 单份完整请求日志接近 0.85MB
- 这还是一轮日志，不是整个进程的全部对象总量

这足以说明：

**完整请求日志对象本身已经是重量级对象。**

### 5. 工具循环使历史不断堆积

在当前实现中：

- 读文件结果
- `git diff`
- 工具输出文本
- 多轮 assistant/tool 历史

都会不断进入后续请求上下文。

这意味着：

**历史不断堆积会让下一次单次物化更大，但“单次物化危险”这个问题本身并不依赖链路堆积才成立。**

## 四、核心问题归因

### 1. 问题主因不是“调用链放大”，而是“单对象已经过大”

未来模型上下文更大，单次请求体 1MB、2MB、甚至更高，都是合理且正常的。

大请求体本身不是 bug。

真正的问题在于：

**一旦在 worker 所在线程上开始完整物化超大请求对象、超大日志对象，本身就已经危险。**

副本并存当然会进一步恶化，但不是主因成立的前提。

### 2. “副本并存”是问题放大器，不是前提条件

从当前代码结构看，较大概率同时存在这些对象：

1. 原始会话历史结构
2. `PreparedPrompt`
3. `request_log`（完整 JSON）
4. provider request（如 `ChatRequest`）
5. 最终发送给 HTTP 层的序列化请求体

如果每一层都完整保留一份，那么问题会进一步恶化。

但即便不考虑这些平行副本，单次完整物化一个 800KB ~ 2MB 级对象，本身也已经需要被当成一级问题处理。

### 3. 问题不是“要不要完整日志”，而是“完整日志与完整请求如何表示”

用户要求明确：

- 日志要完整
- 请求体要完整
- 不接受为了降压而写硬盘

因此问题不应该被理解为：

- “要不要删日志”
- “要不要截断请求”

而应该被理解为：

**在必须保留完整请求与完整日志的前提下，如何避免单次物化过重，以及避免同一份内容被重复表示。**

### 4. 栈溢出暴露的是“对象过重 / 表示过重”，不是“网络错误”

当前崩溃点最终表现为：

- `reqwest_builder.send().await`

但这不等于问题在网络发送本身。

更合理的理解是：

- `.await` 只是最后触发点
- 真正的问题在它之前，超大对象已经被完整构造出来
- 如果这些对象本身过大，或在发送前后再次复制，worker 上的处理就会变得危险

因此：

**网络发送是触发点，不是问题本体。**

---

## 五、优化目标

本次优化的真正目标不是“让请求体变小”，而是：

1. 保持完整请求体
2. 保持完整日志
3. 不写硬盘
4. 避免 worker 上单次完整物化超大对象时再额外制造副本

换句话说，目标应该是：

**完整数据继续保留，但单次物化更轻、重复表示更少。**

---

## 六、优化计划

### 阶段 A：先把“单次物化对象”彻底找清

目标：

- 明确每一类请求中，哪些对象会被完整物化
- 明确每一类请求中，哪一个对象体积最大
- 不再把主要精力放在“链路层数”上

具体工作：

1. 明确主聊天请求中完整物化的对象集合
2. 明确自动整理 / 归档摘要请求中完整物化的对象集合
3. 明确工具安全审查中完整物化的对象集合
4. 记录每一步是“主表示”还是“再次物化”

产出：

- 一份“单次物化对象清单”
- 一份“对象体积与表示方式清单”

### 阶段 B：先解决“单对象过大”，再解决“副本并存”

目标：

- 先压低单次物化对象的重量
- 再减少同一份内容的平行副本

重点方向：

1. 审查 `PreparedPrompt` 是否已经过重
2. 审查 `request_log` 是否在一开始就被完整展开成巨大 JSON
3. 审查 provider request 是否再次完整复制了同一批消息内容
4. 审查最终 HTTP body 是否与前述对象再次平行存在

判定标准：

- 先做到“单次物化不再过于臃肿”
- 再做到“同一轮不再长期共存多份完整大对象”

### 阶段 C：把日志和请求的表示方式重排，但不丢完整性

目标：

- 完整日志继续保留
- 完整请求继续保留
- 但两者不再各自完整展开为独立重量级树

重点方向：

1. 日志字段与请求字段之间尽量复用同一份主数据
2. 降低完整日志 JSON 的独立物化次数
3. 降低完整请求 JSON / request body 的重复构造次数

判定标准：

- 保留完整日志能力
- 保留完整请求能力
- 但请求发送期间不再需要同时维持多份完整重量级表示

### 阶段 D：缩短超大对象的活跃窗口

目标：

- 即使请求体仍然很大，也不要让多个超大对象在同一时间窗口里长期共存

重点方向：

1. 发送前提前释放不再需要的中间对象
2. 缩小 `invoke_model_with_policy(...)` 到 `reqwest send` 之间的活跃重量级对象集合
3. 明确哪些状态只应在：
   - 请求前使用
   - 请求后使用
   - 而不应横跨整个请求生命周期

判定标准：

- 进入 `send().await` 之前，活跃对象数量和总重量明显下降

### 阶段 E：工具循环增长治理

目标：

- 避免工具循环把下一轮“单次物化对象”继续抬高

重点方向：

1. 区分“原始结果留存”和“进入模型上下文的表示”
2. 对高膨胀内容（如长 diff、长代码）做输入层面的控制
3. 降低工具循环对后续单次请求体体积的抬升速度

判定标准：

- 在多轮工具调用中，上下文增长速度受控
- 不再每轮都无差别抬高下一轮完整物化对象的体积

### 阶段 F：结构性兜底

目标：

- 即使主进程仍然保留部分模型请求，也要降低“单次超大物化直接拖死 worker”的概率

重点方向：

1. 评估主进程内超大请求物化边界是否需要更严格隔离
2. 评估是否需要把高风险大请求和其他请求做更明确分离
3. 评估 worker 栈大小是否只作为临时止血手段，而不是长期方案

判定标准：

- 主进程不再因为单次大请求链路而直接退出

---

## 七、优先级建议

### P0

1. 临时增加栈大小以快速止血（短期缓解）
2. 梳理对象生命周期与副本清单
3. 收紧 `request_log`、`PreparedPrompt`、provider request 的单次物化体积
4. 收紧 `request_log`、`PreparedPrompt`、provider request 的并存关系

### P1

1. 工具循环高膨胀内容治理
2. 整理 / 归档链路的请求体单份化与轻量表示

### P2

1. 结构性兜底与隔离策略评估
2. 运行态容错策略评估（长期）

---

## 八、结论

当前问题不是“上下文大所以无解”，而是：

**单次完整物化超大请求对象、超大日志对象，本身就已经危险；而当前实现又让同一份内容进一步出现了平行表示。**

因此，本次优化的核心不是：

- 削弱功能
- 删除日志
- 缩短请求体

而是：

**让完整请求体和完整日志继续存在，同时优先控制单次物化重量，再减少平行副本与重复表示。**
