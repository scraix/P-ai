# 20260423_会话服务收口与异步持久化计划

## 一、背景

本轮聊天性能与会话切换排查里，已经暴露出一个更底层的共性问题：

- 会话读取入口分散在多个 command / helper 中
- 外部代码直接接触 `AppState`、`cached_app_data`、`cached_conversations`
- 外部代码显式关心“这次走缓存、快照还是磁盘”
- 外部代码自己决定分页、快照、单条消息刷新、锁粒度与 clone 时机

这会带来几个持续性问题：

1. 语义漂移  
   不同调用点会逐渐形成不同的“会话真相”理解。

2. 缓存泄漏  
   缓存策略不再是基础设施细节，而是扩散到业务层。

3. 锁策略分裂  
   不同路径各自拿锁、读缓存、clone 数据，难以统一优化。

4. 刷新语义混乱  
   例如“tool review 只该刷新一条消息”，却走了“补后续消息”的链路。

5. 后续优化困难  
   想做分页优化、异步刷盘、会话级锁、只读快照缓存时，需要同时修改很多外层调用点。

本计划的目标，是把这些问题收敛到一个统一的 `ConversationService` 中，类似当前提示词服务对 prompt 真相源的收口方式。

---

## 二、目标

### 1. 总目标

建立唯一的会话服务边界：

- 所有会话读
- 所有会话写
- 所有会话更新
- 所有会话分页
- 所有会话快照
- 所有会话缓存
- 所有会话持久化

都统一收口到 `ConversationService`。

### 2. 设计目标

外部调用方不再关心：

- 会话是从缓存、快照还是硬盘读取
- 是否需要自己拿会话锁
- 是否需要自己更新缓存
- 是否需要自己决定 clone 时机
- 是否需要自己判断是“全量刷新”“分页读取”还是“单条消息刷新”

外部只表达业务意图，例如：

- 取当前前台轻量快照
- 取最近窗口消息
- 取某条消息
- 取某条消息之前的 N 条
- 追加消息
- 更新某条消息
- 标记已读
- 写入 tool review 结果

### 3. 持久化目标

默认写路径改为：

- 同步更新内存真相
- 异步持久化刷盘
- 外部默认不等待落盘完成

也就是：

- 内存真相提交成功后即可返回
- 刷盘由服务内部异步 worker 完成

只有少数显式强一致场景，才允许走“等待 durable write 完成”的同步模式。

---

## 三、核心设计原则

### 1. 内存真相优先

`ConversationService` 内部维护当前 authoritative in-memory truth。

外部默认读到的是：

- 已经提交到内存的最新会话状态

而不是：

- “磁盘上最后一次成功写入的旧状态”

### 2. 同步内存提交，异步持久化

默认写流程应为：

1. 进入 `ConversationService`
2. 在服务内部完成内存状态修改
3. 更新相关会话快照 / 索引 / 分页缓存
4. 标记 dirty revision
5. 投递异步持久化任务
6. 立即返回

注意：

- “立即返回”不是指什么都不做
- 返回前必须保证内存真相已经完成提交

### 3. 读写缓存都在服务内部

外部不允许直接操作：

- `cached_app_data`
- `cached_conversations`
- 未来的分页缓存 / 快照缓存 / dirty map

这些都应成为 `ConversationService` 的内部实现细节。

### 4. 语义化 API 替代存储型 API

服务对外暴露的是业务语义接口，不是存储细节接口。

例如应优先是：

- `get_foreground_light_snapshot`
- `get_messages_before`
- `get_message_by_id`
- `patch_message`
- `append_messages`
- `write_tool_review`

而不是：

- `read_app_data_cached`
- `read_conversation_cached`
- `clone_messages_after_page`

### 5. 强弱一致性分层

默认写路径使用弱 durable 保证：

- 内存已成功
- 刷盘稍后完成

少数路径支持强 durable 模式：

- 导出
- 关闭前 flush
- 数据迁移
- 需要“必须落盘成功才算完成”的明确业务

## 四、拟定架构

## 4.1 ConversationService 职责

建议新增统一会话服务层，负责：

- 会话读取
- 会话更新
- 会话分页
- 前台轻量快照
- 单条消息读取
- 最近窗口读取
- 缓存维护
- dirty revision 管理
- 异步刷盘调度

### 4.2 服务内部建议持有的状态

- in-memory conversations truth
- 会话索引（按 `conversationId`）
- 最近窗口缓存
- 前台轻量快照缓存
- 分页缓存（可选）
- 单条消息索引（`messageId -> conversationId / position`，可选）
- dirty revision / persisted revision
- pending persist job 状态

### 4.3 持久化 worker

服务内部维护单独的 persistence worker，负责：

- 合并短时间内多次写入
- 按 revision 顺序刷盘
- 去重旧任务
- 失败重试
- 记录失败日志
- 在必要时暴露 pending / failed 状态给调试链路

建议明确以下配置参数，并在实现中提供默认值与边界校验：

- `mergeWindowMs`
  默认值：`200ms`
  推荐范围：`100ms - 500ms`
  作用：把短时间内同一会话或同一批 revision 的写入合并成一次落盘；值越小，持久化延迟越低、磁盘写放大越高；值越大，吞吐更好，但异常退出时的可丢失窗口也更大。
- `maxBatchSize`
  默认值：`500`
  推荐范围：`100 - 1000`
  作用：单次 persistence worker 处理的最大 job 数；值越小，单批时延更稳定；值越大，吞吐更高，但单次 flush 时间和失败回滚成本更高。
- `retryPolicy.maxRetries`
  默认值：`5`
  推荐范围：`0 - 10`
  作用：持久化失败后的最大重试次数；达到上限后必须打 ERROR 日志并暴露 failed 状态，供手动 flush / wait 和诊断链路读取。
- `retryPolicy.backoffStrategy`
  默认值：`exponentialBackoffWithJitter`
  推荐取值：`fixed` / `linear` / `exponentialBackoffWithJitter`
  默认建议：初始退避 `100ms`，上限 `5s`，并附加 `10% - 20%` jitter，避免连续失败时多个 worker 重试同拍。
- `workerCount`
  默认值：`1`
  推荐范围：`1 - CPU 核数`，默认仍建议 `1`
  作用：控制 persistence worker 并行度；默认单 worker 最安全，能天然保证 revision 顺序；只有在明确按会话级分片后，才允许提升到多 worker。

安全与性能权衡约束：

- 未做会话级分片前，`workerCount` 必须保持 `1`
- `mergeWindowMs` 不能为了吞吐无限拉大，否则会放大“内存真相已提交但磁盘仍未落盘”的时间窗
- `maxBatchSize` 与 `mergeWindowMs` 应联动观测：如果 batch 经常顶满，应优先分析写入热点，而不是单纯继续放大 batch
- `retryPolicy` 必须保证失败可见，不能静默吞掉重试耗尽后的最终失败

### 4.4 推荐读路径

默认读路径：

1. command / runtime 调用 `ConversationService`
2. 服务直接读内存真相或派生快照
3. 只在冷启动或服务尚未 ready 时，内部回退初始化缓存

外层不再自己决定是否读盘。

### 4.5 推荐写路径

默认写路径：

1. command / runtime 调用 `ConversationService`
2. 服务内部拿统一写锁
3. 更新内存真相
4. 更新派生缓存
5. 标记 dirty
6. 投递异步持久化
7. 返回更新结果

## 五、对外 API 草案

以下是建议收口后的服务接口方向，名称可后续微调：

### 5.1 只读接口

- `get_foreground_light_snapshot(conversation_id)`
- `get_recent_messages(conversation_id, limit)`
- `get_messages_before(conversation_id, before_message_id, limit)`
- `get_messages_after(conversation_id, after_message_id, limit)`
- `get_message_by_id(conversation_id, message_id)`
- `get_conversation_overview(conversation_id)`
- `list_unarchived_conversation_overviews()`

### 5.2 写接口

先定义统一的 `write_mode`：

- `async`（默认）
  语义：调用返回时只保证“内存真相已提交 + 已成功投递 persistence worker”；不保证磁盘已落盘。
  顺序：同一 `conversationId` 内仍需按 revision 保序；跨会话顺序不保证。
  错误：若内存提交失败或投递 worker 失败，立即返回错误；若后续异步落盘失败，由 worker 重试并记录失败日志。
  调用方是否重试：仅在收到同步返回错误时重试；若已成功返回，不应自行再次重放写入，避免重复写。
- `sync`
  语义：调用返回时保证内存真相已提交，且本次写入已完成 durable write 或明确失败。
  顺序：必须等待本次 revision 持久化完成。
  错误：任何落盘失败直接返回错误，不进入“调用方已成功”的语义。
  调用方是否重试：可在明确幂等的写场景下按调用方策略重试。
- `write_through`
  语义：同步更新内存，并立刻直写对应 shard / 索引，再更新缓存状态；适用于极少数要求最强落盘保证的链路。
  顺序：严格按当前调用顺序执行。
  错误：写盘失败立即返回，且必须保证不会留下“前台看见成功但磁盘未更新”的假成功状态。
  调用方是否重试：只建议用于可证明幂等的操作，避免重复副作用。

各写接口默认都使用 `write_mode = async`，除非调用方明确声明：

- `append_messages(conversation_id, messages, write_mode = async)`
- `patch_message(conversation_id, message_id, patch, write_mode = async)`
- `replace_message(conversation_id, message, write_mode = async)`
- `mark_conversation_read(conversation_id, write_mode = async)`
- `upsert_tool_review(call_id, review, write_mode = async)`
- `upsert_tool_review_report(batch_key, report, write_mode = async)`
- `rename_conversation(conversation_id, title, write_mode = async)`
- `toggle_conversation_pin(conversation_id, value, write_mode = async)`

### 5.3 特殊接口

- `flush_pending_persistence()`
- `wait_until_persisted(revision)`
- `get_persistence_status()`

## 六、迁移边界

### 6.1 第一批必须迁移的读路径

优先迁移本轮已经暴露问题的热点路径：

- 前台轻量快照
- `before / after` 分页
- 单条消息读取
- tool review 批次读取
- tool review 单条 detail 读取

原因：

- 这些路径最热
- 已经出现缓存策略泄漏
- 已经开始出现“不同业务自己拼读取语义”的问题

### 6.2 第一批必须迁移的写路径

- tool review 写回
- report upsert
- 标记已读
- 消息 append / patch

原因：

- 这些路径已经直接影响前台刷新与缓存一致性
- 很适合验证“同步内存提交 + 异步刷盘”模型

### 6.3 暂缓迁移的部分

第一阶段可以先不迁：

- 归档类重型链路
- 历史导出
- 非前台会话的特殊批处理
- 跨模块的大型迁移修复任务

这些可以在只读路径与常规写路径稳定后再推进。

## 七、实施方式

说明：

- 可以按阶段施工
- 但不做分阶段用户验收
- 不在“只完成一部分服务边界”时找用户做回归确认

原因：

- 会话服务是基础设施重构
- 只完成读、不完成写，或者只完成缓存收口、不完成持久化收口，都无法形成稳定可验证的最终行为
- 半成品状态下做验收，结论没有意义，反而会混淆真实问题来自“设计缺口”还是“实现缺陷”

因此本计划调整为：

- 开发阶段允许按顺序逐步落地
- 但只有在服务边界整体建立完成后，才进行一次统一回归测试
- 在统一回归完成之前，不以中间阶段结果向用户申请验收结论
## 阶段一：建立只读 ConversationService

### 目标

先把“读语义”统一。

### 范围

- 建立 `ConversationService` 只读外壳
- 服务内部托管快照 / recent / before / after / message-by-id
- command 层改为通过服务读取

## 阶段二：建立同步内存提交 + 异步持久化写模型

### 目标

把会话更新行为统一成“先内存，后落盘”。

### 范围

- 服务内部维护 dirty revision
- 增加 persistence worker
- 常规写路径迁移到 `write-back`

## 阶段三：收口高频业务写路径

### 范围

- tool review
- mark read
- append / patch message
- report upsert
- rename / pin / foreground overview patch

## 阶段四：清理旧入口与建立约束

### 范围

- 废弃旧 helper
- 禁止 command 直接读取会话缓存
- 补规则与文档

## 八、需要重点规避的风险

### 1. 内存真相与刷盘结果分叉

如果异步刷盘失败，但前台继续建立在内存真相上，就会出现“看起来成功、实际上没落盘”的情况。

应对：

- 必须记录 dirty / persisted revision
- 必须有失败日志
- 必须支持手动 flush / wait

### 2. 应用异常退出导致最近写丢失

这是 write-back 模型的天然风险。

应对：

- 合并写延迟不能过长
- 关键生命周期点要 flush
- 少数高价值链路允许强同步写

### 3. 服务边界不清导致继续绕过

如果只加一个 service，但旧 helper 不禁用，很快又会出现新路径绕过服务。

应对：

- 在阶段四明确清理旧入口
- 建立代码约束：会话读取与写入禁止直接碰缓存

### 4. 前台局部刷新与服务缓存不一致

例如 tool review 按消息刷新，如果服务内部 message index 不稳定，会导致局部刷新失效。

应对：

- 服务内部建立稳定的 `conversationId + messageId` 查询语义
- 局部刷新只通过服务提供的数据完成

### 5. 并发写冲突与锁顺序失控

如果服务内部只说“统一写锁”，但没有明确锁粒度、读写分离和多锁顺序，后面很容易重新引入吞吐瓶颈和死锁。

应对：

- 锁粒度优先使用会话级锁（conversation-level lock），按 `conversationId` 建立写互斥；只有涉及会话集合结构变更时，才允许短时间拿全局锁。
- 读写分离：优先使用读写锁或基于 revision 的乐观并发控制。只读查询应尽量走共享读锁；写操作在提交前做 revision / CAS 检查，避免读完全被串行化。
- 多把锁必须按固定顺序获取：
  1. 全局会话集合锁（如果确实需要）
  2. 会话级锁（按 `conversationId` 字典序）
  3. 消息级定位辅助结构（如 `conversationId + messageId` 索引）
  4. flush / wait 协调状态
- 禁止持锁执行长时间阻塞操作：磁盘 I/O、网络请求、媒体物化、外部工具调用、`wait_until_persisted(...)` 都必须在锁外完成。
- 对所有潜在竞争点增加超时与重试：拿锁超时必须打 WARN / ERROR 日志，并在可恢复场景下走短退避重试。

常见场景示例：

- 消息 append 与 `mark_conversation_read`
  使用同一把会话级 mutex，并在会话对象上维护 revision；`mark_conversation_read` 只更新游标字段，如果提交时 revision 已变化，则基于最新会话重新计算末尾消息后重试一次。
- tool review 与 append
  tool review 通过 `conversationId + messageId` 精确定位目标消息，提交前检查消息 revision 或消息存在性；若 append 导致会话 revision 前移，则重新读取该消息并重放 patch，必要时退回“整条消息刷新 + 再提交”。
- 手动 `flush_pending_persistence()` / `wait_until_persisted(revision)`
  只能观察或等待 worker 状态，不能反向持有会话级写锁后再等待 worker，避免形成“写锁等待 flush，flush 等待写锁释放”的死锁环。

## 九、建议的第一批落地范围

这一章仅定义内部开发推进顺序，不是独立回归点，也不是用户验收点。

为了控制风险，建议开发落地时先做：

1. 建立只读 `ConversationService`
2. 收口：
   - `light snapshot`
   - `recent window`
   - `before / after`
   - `message by id`
3. 把 tool review 读路径全部切到服务
4. 把前台切会话和局部刷新读路径切到服务

如果这一轮只做到这里，仍然只能算“内部里程碑”，不能进入统一回归。

原因：

- 只读收口已经能明显降低当前架构混乱度
- 写模型切换到异步持久化属于第二阶段，风险更高
- 先把读边界锁住，再推进写路径，成功率更高

与第七章、第十一章的关系明确如下：

- 第九章描述的是“先做什么、后做什么”的内部施工顺序
- 第七章的统一回归前提仍然不变：只有在读写边界整体建立完成，且“同步内存提交 + 异步持久化”真实落地后，才进行统一回归
- 因此，“只读 ConversationService”“读语义统一”“局部刷新改走服务”等事项，都只是统一回归的前置条件，不构成单独验收点

## 十、最终预期

完成后，系统应表现为：

- 外部不再关心缓存 / 快照 / 磁盘
- 会话读写都只通过 `ConversationService`
- 默认写入在“内存真相提交后立即返回”
- 持久化由服务内部异步完成
- 前台刷新语义更清晰：
  - 流式错位：向后补
  - tool review：按消息刷新
  - 切会话：轻量快照
  - 向上翻历史：before 分页

从架构上看，这会让会话层逐步接近“提示词服务”的收口方式：

- 外部只表达业务意图
- 服务内部负责真相、缓存、派生视图与持久化策略

---

## 十一、最终统一回归要求

只有在以下事项全部完成后，才进入统一回归测试：

1. 会话读路径已全部收口到 `ConversationService`
2. 会话写路径已全部收口到 `ConversationService`
3. 默认写入已切换为“同步内存提交 + 异步持久化”
4. 外部业务层不再直接操作会话缓存 / 快照 / 磁盘读取细节
5. 旧 helper 与绕过入口已完成清理或封禁

完成后再进行一次完整回归，并向用户提交统一回归清单。

### 建议回归清单

#### A. 会话切换与前台显示

1. 切换会话时是否始终从最新消息开始显示
2. 切换到超长会话时首屏是否仍然流畅
3. 切换会话后异步补齐是否不会串到其他会话
4. 切换后 Todo、runtimeState、tool review 面板是否与当前会话一致

#### B. 消息读取与分页

1. 前台轻量快照是否稳定返回最近窗口消息
2. 向上滚动分页是否每次准确补指定条数
3. 向上补历史后视口是否稳定，不跳动、不串会话
4. 单条消息按 `messageId` 刷新时是否只更新目标消息

#### C. 流式与中断恢复

1. 流式进行中切换会话再切回，草稿与流式状态是否能恢复
2. 流式被 stop / fail / external completed 后，`reloadForegroundConversationMessages()` 是否能正确补后续消息
3. 不同会话之间的外部 round completed / failed 事件是否不会串台

#### D. Tool Review

1. 单条 review 后是否只刷新被 review 的消息
2. 批量 review 后是否只刷新被修改的那些消息
3. submit batch 后是否正确插入或更新 report message
4. tool review 列表、detail、消息正文三者是否一致

#### E. 会话写入与异步持久化

1. 追加消息后前台是否立刻看到最新内存真相
2. 标记已读、重命名、置顶、tool review 写回后前台是否立即反映
3. 持久化 worker 失败时是否有清晰日志
4. 应用正常关闭前 flush 是否工作正常
5. 异常退出后最近写入是否符合预期容忍范围

#### F. 一致性与绕过检查

1. 新旧 command 是否都已经不再直接操作 `cached_app_data` / `cached_conversations`
2. 是否仍存在绕过 `ConversationService` 的直接会话读写入口
3. 缓存命中、分页、快照、单条消息读取是否都由服务内部统一决定

#### G. 性能与日志

1. 前台轻量快照、tool review 批次读取、分页读取的耗时日志是否回到合理区间
2. 锁持有时间是否显著下降
3. 日志是否能清楚区分内存提交成功、异步持久化成功、异步持久化失败

#### H. 异常与边界场景

1. 持久化 worker 连续失败 N 次
   预期行为：按 `retryPolicy.maxRetries` 与 `backoffStrategy` 重试，超过上限后进入 failed / degraded 状态，不再静默重试。
   性能阈值：单次重试调度开销不应持续阻塞前台写入，前台同步提交额外耗时应保持在 `50ms` 内。
   日志/告警：必须有连续失败次数、最后错误、当前 pending revision、是否已降级为 `sync` / `write_through` 的清晰日志。
   恢复验证：人工触发 `flush_pending_persistence()` 后能看到状态恢复或明确失败原因。
2. dirty revision 积压时强制关闭
   预期行为：关闭流程必须先尝试 flush；若超时仍有 dirty revision，必须记录剩余 revision 范围和会话数。
   性能阈值：关闭阶段等待 flush 的默认超时应可配置，建议默认 `3s - 10s`。
   日志/告警：必须打印剩余 dirty revision、pending job 数、是否进入“带未落盘数据退出”。
   恢复验证：重启后能够识别上次未完成 flush 的状态，并验证数据一致性是否符合允许范围。
3. 异步持久化期间发生进程崩溃
   预期行为：重启后按 persisted revision / dirty revision 恢复，必要时重放未完成 job 或承认最近 merge window 内写入丢失。
   性能阈值：恢复过程不应让首屏启动额外增加超过 `2s`。
   日志/告警：必须记录崩溃前最后 persisted revision、检测到的未完成写入范围、是否触发重放。
   恢复验证：校验会话消息、已读、tool review 等状态是否与设计的一致性语义匹配。
4. 内存不足时的降级策略
   预期行为：服务应停止扩张高成本缓存，必要时退化为更小的 recent window / message index，严重时转入 `sync` 写模式保护一致性。
   性能阈值：降级后首屏关键读路径仍应可用，不允许直接因 OOM 风险导致整条读链路不可用。
   日志/告警：必须记录触发条件、已关闭的缓存项、是否切换写模式。
   恢复验证：内存恢复后能够重新建立必要缓存，且不影响后续落盘正确性。
5. 空会话
   预期行为：读写接口对空会话返回稳定结果，不应 panic，不应误判“消息不存在”导致整会话报错。
   性能阈值：空会话快照 / 分页应稳定在 `20ms` 内。
   日志/告警：不需要 ERROR，最多 DEBUG 说明命中空会话。
   恢复验证：后续 append 第一条消息后，会话能正常进入常规路径。
6. 超大会话（如 `10000+` 消息）
   预期行为：分页、最近窗口、按消息定位仍应正确，不允许整段全量 clone 或整段全量渲染回退。
   性能阈值：recent / before / after 单次读取应保持在可接受区间，建议目标 `100ms` 内；若超过阈值，必须有性能日志。
   日志/告警：记录消息总数、分页条数、耗时、是否命中快路径。
   恢复验证：反复翻页后仍不串会话、不丢消息、不跳视口。
7. 单条消息超大（如 `100KB`）
   预期行为：按消息读取、patch、tool review 定位仍能工作，不因大消息导致索引错误或异常截断。
   性能阈值：单条消息读取与单条消息刷新目标 `50ms` 内；超过时打印具体 messageId 和 payload 大小。
   日志/告警：必须包含消息大小、路径、是否触发降级渲染或延迟物化。
   恢复验证：重启后该消息内容与 provider meta 不丢失。
8. 高并发场景（如 `100 QPS` 写入同一会话）
   预期行为：会话级锁 / revision 冲突策略正确生效，不出现乱序、重复写、死锁。
   性能阈值：同一会话 100 QPS 下不允许出现持续增长的锁等待堆积；P95 同步提交耗时应有明确观测目标并记录。
   日志/告警：必须能看到冲突重试次数、锁等待时间、最终 revision 落点。
   恢复验证：压测结束后校验消息顺序、已读游标、tool review 状态与最终 persisted revision 一致。

---

## 十二、是否建议执行

建议执行。

这不是一次“性能微调”，而是会话层基础设施收口。

如果继续保持现在的分散读取、分散缓存、分散锁逻辑，后面每加一个刷新场景、分页场景、会话补丁场景，都会继续复制一套新的读取语义，维护成本只会越来越高。

这次重构的真正价值，不只是更快，而是让后续所有会话相关优化终于有一个统一落点。
