# 20260423_压缩即闭口重开调度计划

## 一、背景

当前聊天发送链路在检测到需要上下文压缩时，会在**同一次调度**内部执行：

1. 检测是否需要压缩
2. 写入压缩消息
3. 压缩完成后继续在原调度里准备并发送后续模型请求

这套做法在实现上虽然可行，但已经暴露出明显问题：

1. 真相层已经天然分段  
   当前会话真实消息序列已经表现为：
   - 一段原始消息组
   - 一条压缩消息
   - 一段压缩后的新消息组

2. 展示层也天然分段  
   前端消息分组展示同样会自然呈现为：
   - 消息组
   - 压缩消息
   - 消息组

3. 唯独抽象层仍试图把压缩前后视作同一次调度  
   这会导致：
   - 调度边界语义不一致
   - 压缩后 continuation 语义混乱
   - 前后端对“当前这一轮是否仍在同一次调度中”的理解不一致
   - 相关缓存、用量、日志、消息分组长期出现边角 bug

因此，本次改造的目标不是继续修补“压缩后原地继续”的细节，而是**改变调度边界定义**：

**一旦发生压缩，当前调度立即闭口结束；压缩完成后，开启一次新的调度。**

---

## 二、目标

### 1. 明确压缩是调度边界事件

压缩不再被视为“同一次调度里的中间步骤”，而改为：

- 当前调度结束点
- 新调度开始点之间的边界事件

### 2. 让三层语义一致

改造后要求以下三层边界保持一致：

1. 真相层  
   - 消息组 A
   - 压缩消息
   - 消息组 B

2. 展示层  
   - 消息组 A
   - 压缩消息
   - 消息组 B

3. 抽象层 / 调度层  
   - 调度 A
   - 压缩消息
   - 调度 B

### 3. 提升稳定性，优先放弃“原地续跑”

这次改造优先级是：

- 稳定
- 边界清晰
- 便于测试

而不是追求“压缩后仍在同一次调度里无缝继续”的复杂行为。

---

## 三、核心策略

## 3.1 统一规则

以后只要在调度过程中命中上下文压缩：

1. 当前调度立刻结束
2. 写入压缩消息
3. 压缩完成后，不在原调度中继续构造和发送后续请求
4. 由系统启动一个新的调度，继续处理原始待执行意图

换句话说：

**压缩后不是 continuation，而是 restart。**

---

## 3.2 首次发送场景

### 当前错误语义

现在首次发送在 `COMPACTION-AUTO` 后，会：

- 重新 `prepare_request_context(false)`
- 继续使用原链路往下发请求

这会让：

- 压缩前后的消息仍落在同一调度里
- 中途插入的压缩消息难以定义其归属

### 目标语义

改造后：

1. 用户发送进入调度 A
2. 调度 A 检测到需要压缩
3. 调度 A 立即闭口
4. 写入压缩消息
5. 启动调度 B
6. 调度 B 基于压缩后的真相层重新准备请求并发送

---

## 3.3 工具续调场景

### 当前错误语义

现在 tool loop 在 `COMPACTION-BEFORE-TOOL-CONTINUE` 后，会：

- 在原 tool loop 内重新构造 continuation prompt
- 继续原续调链

### 目标语义

改造后：

1. 工具续调进入调度 A
2. 调度 A 检测到需要压缩
3. 调度 A 立即闭口
4. 写入压缩消息
5. 启动调度 B
6. 调度 B 以“继续处理当前会话未完成任务”的身份重新开始

也就是说：

- 不再在原 tool loop 中压缩后继续跑
- 而是把“继续处理”转成一次新的调度

---

## 四、边界定义

## 4.1 哪些内容属于旧调度

以下内容都属于压缩前旧调度：

- 旧请求的用户输入 / 工具输入
- 旧调度内已完成的工具调用
- 压缩触发前的运行态事件

这些内容在压缩后应体现在：

- 真相层历史
- 压缩消息摘要

但不应再通过“旧调度 continuation”继续传播。

## 4.2 哪些内容属于新调度

压缩后新调度只应持有：

- 压缩后的会话真相
- 继续执行所需的最小恢复意图

不应继续持有：

- 旧调度里那套“未完成但仍挂着的请求准备状态”
- 旧 continuation prompt
- 旧调度内部缓存的临时请求体

---

## 五、恢复意图模型

由于压缩后要新开调度，因此必须显式记录“接下来要继续做什么”。

本次计划要求引入一个最小恢复意图模型，至少覆盖：

### 1. 首次发送恢复

- conversation_id
- agent_id
- api_config_id
- 当前用户消息已写入这一事实
- 本次恢复属于“继续完成刚才这次发送”

### 2. 工具续调恢复

- conversation_id
- agent_id
- api_config_id
- 当前轮工具已完成这一事实
- 本次恢复属于“继续处理压缩前尚未完成的 tool loop”

恢复意图必须做到：

- 足够小
- 可序列化/可日志化
- 不直接携带庞大 prompt/request 本体

---

## 六、实施范围

## 6.1 本次要改

1. 发送前压缩命中后，不再在原调度里继续发请求
2. 工具续调前压缩命中后，不再在原 tool loop 里继续发请求
3. 引入最小恢复意图
4. 压缩完成后改为新开调度
5. 日志明确标记：
   - 当前调度闭口
   - 压缩完成
   - 新调度开始

## 6.2 本次不改

1. 不重写整个聊天发送框架
2. 不重写完整 tool loop 状态机
3. 不把所有请求入口合并成一个超大统一函数
4. 不改会话真相层的消息结构
5. 不重做前端消息分组系统

---

## 七、实现步骤

## 阶段 1：定义恢复意图与闭口规则

目标：

- 把“压缩即结束当前调度”规则钉死
- 明确恢复意图数据结构

产出：

- 恢复意图类型
- 日志语义
- 调度闭口点

## 阶段 2：首次发送链路切换为“压缩后重开”

目标：

- `COMPACTION-AUTO` 不再原地 continuation
- 压缩成功后新开一次发送调度

验证重点：

- 压缩消息之后的模型输出属于新的调度
- 前端消息组不再被同一次调度跨压缩消息切开

## 阶段 3：工具续调链路切换为“压缩后重开”

目标：

- `COMPACTION-BEFORE-TOOL-CONTINUE` 不再原地 continuation
- 压缩成功后新开一次 continuation 调度

验证重点：

- tool loop 不再跨压缩消息继续
- 压缩消息之后的新工具/新回复属于新调度

## 阶段 4：日志与回归测试补齐

目标：

- 让日志能明确看出：
  - 旧调度结束
  - 压缩成功
  - 新调度开始
- 覆盖首次发送与工具续调两条链的回归测试

---

## 八、测试要求

### 1. 首次发送压缩测试

场景：

- 首次发送命中压缩

要求：

- 当前调度结束
- 写入压缩消息
- 新调度继续完成用户这次发送
- 不允许“压缩消息后仍算同一次调度”

### 2. 工具续调压缩测试

场景：

- tool loop 中途命中压缩

要求：

- 当前 tool 调度结束
- 写入压缩消息
- 新调度继续处理未完成任务

### 3. 前端消息组测试

要求：

- 压缩消息成为调度边界
- 压缩前后两段消息属于不同调度组

### 4. 稳定性测试

要求：

- 不出现压缩后“看似继续、实际无输出”
- 不出现压缩后继续错误绑定旧调度状态
- 不出现同一次调度跨压缩边界继续写消息

---

## 九、风险与注意事项

### 风险 1：恢复意图不完整

如果恢复意图携带信息不足，压缩后新调度可能无法正确恢复原任务。

应对：

- 恢复意图只保留最小但足够的执行信息
- 先覆盖首发与 tool continue 两类核心场景

### 风险 2：重复写用户消息

首次发送链改成重开调度后，必须确保：

- 用户消息只写一次
- 新调度只是继续处理已写入输入

### 风险 3：tool loop 旧运行态残留

工具续调链切换时，需要确保旧 tool loop 的临时运行态不会继续污染新调度。

---

## 十、结论

本次改造的核心不是“继续修补压缩后 continuation 的细节”，而是承认一个更符合现状的事实：

**压缩消息本身就是调度边界。**

因此正确做法是：

- 压缩命中时，当前调度闭口结束
- 压缩完成后，新开调度继续执行

这样才能让：

- 真相层
- 展示层
- 抽象层 / 调度层

三者重新保持一致。
