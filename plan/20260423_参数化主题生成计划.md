# 20260423_参数化主题生成计划

## 1. 背景与目标

当前项目主题系统基于 daisyUI 预定义主题，通过 `data-theme` 在固定主题名之间切换。这个模式适合“挑现成主题”，不适合“用少量参数合成一套自定义主题”。

本次目标不是复刻 daisyUI 官网主题生成器的高随机策略，也不是引入大量细碎配置项，而是新增一套**参数化主题生成能力**：

- 用户只调少量高层参数。
- 系统自动推导整套主题 token。
- 生成结果优先保证稳定、耐看、可读，而不是追求极端随机性。
- 与现有 daisyUI 主题体系兼容，保留预设主题切换能力。

一句话概括：从“主题名切换”升级为“少量参数驱动的主题合成”。

---

## 2. 现状

### 2.1 当前主题切换方式

前端现状已经具备以下基础：

- `src/features/shell/composables/use-app-theme.ts`
  - 用 `APP_THEMES` 白名单限制可选主题。
  - 通过 `document.documentElement.setAttribute("data-theme", theme)` 切换主题。
- `src/style.css`
  - 使用 daisyUI `@plugin "daisyui"` 与 `@plugin "daisyui/theme"` 定义主题。
  - 主题本质是 CSS 变量集合。
- `src/features/config/views/config-tabs/AppearanceTab.vue`
  - 外观页目前只负责展示主题预览与选择。
- `src/features/config/components/ThemePreviewGrid.vue`
  - 预览卡片按主题名枚举展示。

### 2.2 当前限制

当前实现的限制也很明确：

- 主题模型是“固定主题名”，不是“主题参数 + 动态 token”。
- 主题只能从 `APP_THEMES` 里选，不能在运行时生成新主题。
- 主题预览依赖固定枚举，不支持“用户当前正在编辑的一套动态主题”。
- 主题状态只保存主题名，不保存参数或动态生成结果。

因此，如果要支持用户用少量参数生成主题，现有结构需要从“静态枚举主题”扩展为“预设主题 + 动态主题”并存模型。

## 3. 设计原则

### 3.1 不追求官方随机，追求可用稳定

官方主题生成器的随机性很强，视觉结果波动大，生成好看主题的概率不稳定。本次不以“随机”为核心目标，而以“低配置成本下生成高可用主题”为目标。

### 3.2 最少可配置项

用户不直接编辑十几种颜色 token，也不手动调整所有 daisyUI 变量。只暴露少量高层参数，让系统自动派生其余变量。

### 3.3 统一约束优先于自由漂移

以下约束在第一阶段应视为硬规则：

- `radius-box`、`radius-field`、`radius-selector` 必须统一。
- `Depth Effect` 与 `Noise Effect` 不参与随机派生，作为手动开关。
- UI 尺寸不走连续自由值，而是预设档位。
- 语义色保持稳定角色，不让主题生成把语义提示色做乱。

### 3.4 与现有 daisyUI 体系兼容

最终输出仍然是一套 CSS 变量。即：

- 可以继续让组件消费 daisyUI token。
- 不要求组件层改 class 命名。
- 动态主题只是新增 token 来源，不重写 UI 组件体系。

### 3.5 颜色标准采用“底线 + 生成 + 计算”三层组合

第一阶段不应把“颜色标准”理解为只有一条规则，而应明确采用三层组合：

1. **底线标准：WCAG 2.2**
   - 用于定义文本、图标、边框、状态元素的最小可读性门槛。
   - 这是第一阶段必须满足的硬验收条件。

2. **生成思路：Material 3 / HCT 方法论**
   - 借鉴其“少量输入 -> 角色色派生 -> 成对可读”的思路。
   - 不直接照搬其整套角色表，也不强行引入 Material token 体系。

3. **计算空间：OKLCH**
   - 作为内部颜色生成与校正的主要计算空间。
   - 目标是比 HSL / HSV 更贴近人眼感知，降低生成结果失真的概率。

这意味着：

- 是否“合格”，以 WCAG 2.2 为准。
- 如何“派生角色色”，参考 Material 的系统化思路。
- 如何“算颜色”，优先使用 OKLCH。

## 4. 主题模型

### 4.1 用户侧最小控制项

第一阶段建议把用户可调项控制在以下范围：

1. `mode`
   - `light` / `dark`
   - 决定整体明暗基线与 base 梯度方向。

2. `themeHue`
   - 主题主色相。
   - 这是整套主题的主控制参数。

3. `contrast`
   - 控制主次色差异与整体视觉张力。
   - 用于避免“所有颜色挤在一起”或“对比过于刺眼”。

4. `tone`
   - 表示颜色浅淡 / 浓烈程度。
   - 主要影响饱和度与部分亮度分布。

5. `radius`
   - 统一控制 box / field / selector 圆角。

6. `uiSizePreset`
   - 只允许预设档位，例如：`compact` / `default` / `comfortable`。
   - 用于控制字段、选择器等基础尺寸。

7. `depthEnabled`
   - 手动开关。

8. `noiseEnabled`
   - 手动开关。

说明：

- 第一阶段不建议给用户开放 `primary`、`secondary`、`accent`、`neutral` 的逐槽位独立编辑。
- 若后续确实需要精调，应作为高级模式，而不是第一阶段默认入口。

### 4.2 内部主题状态模型

当前项目主题状态只表达“预设主题名”。第一阶段应扩展为“预设主题”和“动态生成主题”并存。

建议模型：

```ts
type ThemeMode = "light" | "dark";

type GeneratedThemeControls = {
  mode: ThemeMode;
  themeHue: number;
  contrast: number;
  tone: number;
  radius: number;
  uiSizePreset: "compact" | "default" | "comfortable";
  depthEnabled: boolean;
  noiseEnabled: boolean;
};

type GeneratedThemeTokens = {
  base100: string;
  base200: string;
  base300: string;
  baseContent: string;
  primary: string;
  primaryContent: string;
  secondary: string;
  secondaryContent: string;
  accent: string;
  accentContent: string;
  neutral: string;
  neutralContent: string;
  info: string;
  infoContent: string;
  success: string;
  successContent: string;
  warning: string;
  warningContent: string;
  error: string;
  errorContent: string;
  radiusBox: string;
  radiusField: string;
  radiusSelector: string;
  sizeField: string;
  sizeSelector: string;
  border: string;
  depth: string;
  noise: string;
  colorScheme: ThemeMode;
};

type AppThemeState =
  | { kind: "preset"; name: string }
  | { kind: "generated"; controls: GeneratedThemeControls; tokens: GeneratedThemeTokens };
```

重点：

- `controls` 是用户真正编辑的来源。
- `tokens` 是由 `controls` 派生出的结果缓存。
- 第一阶段不要求用户维护 token 真源，token 只作为派生结果存在。

## 5. 颜色生成规则

### 5.1 目标不是完全随机，而是参数驱动派生

颜色策略应从“随机抽一堆 hue”转成“围绕主题主色，按规则派生”。

第一阶段建议：

- 用户侧仍可用“色相、浓淡、对比度”这类直觉参数。
- 内部颜色计算尽量转入 OKLCH 空间处理。
- 最终再序列化为 daisyUI 所需 CSS 变量。

不建议继续以 HSV 作为正式实现层的核心计算空间。原因不是 HSV 不能做，而是它在亮度与感知均匀性上不稳定，更容易导致“参数变化不顺手、结果看起来发虚或过冲”。

### 5.2 base 组规则

`base-100`、`base-200`、`base-300` 用于页面背景、层级背景与浅分隔。

硬规则：

- `light` 模式下：`base-100 -> base-200 -> base-300` 逐步变深。
- `dark` 模式下：`base-100 -> base-200 -> base-300` 逐步变亮。
- `base-content` 不开放给用户调色，自动根据明暗选择深字或浅字。

第一阶段建议：

- base 组保持低饱和，优先稳定可读。
- 可以允许轻微随主色偏移，但不应让背景显著染色到影响正文阅读。

### 5.3 primary / secondary

这是主题个性的核心来源。

规则建议：

- `primary` 直接围绕 `themeHue` 生成。
- `secondary` 从 `themeHue` 偏移得到。
- 偏移量受 `contrast` 控制。
- `tone` 决定两者的饱和度强弱与显色浓淡。

目标：

- `primary` 与 `secondary` 要有明确区分。
- 不能过近，避免视觉糊成一团。
- 不能过炸，避免页面像彩灯。

### 5.4 accent / neutral

用户期望这里也有规矩，而不是完全自由乱飞。第一阶段建议：

- `accent` 使用固定色相策略，保持点缀角色稳定。
- `neutral` 使用固定色相策略，承担中性补充角色。
- 二者可随 `mode`、`tone`、`contrast` 调整饱和度与亮度，但不建议成为独立自由主色。

备注：

- 这里的“固定色相”是设计目标，不是对 daisyUI 官方随机策略的复刻。
- 目标是稳定风格，而不是模仿官方的高随机行为。

### 5.5 语义色

`info`、`success`、`warning`、`error` 建议视为稳定语义色：

- `info`：蓝区
- `success`：绿区
- `warning`：黄橙区
- `error`：红区

规则：

- 第一阶段不随 `themeHue` 漂移。
- 仅根据 `mode` 与 `tone` 做少量明度 / 饱和度适配。
- 目标是保持语义辨识，不让主题生成破坏状态色的一致性。

### 5.6 content 色

文字内容色不开放独立编辑，自动从背景或色块反推：

- 浅底优先深字。
- 深底优先浅字。

第一阶段可以先使用稳定的黑 / 白二值策略，但是否取黑或白，不能只凭肉眼直觉，最终应通过可读性校验确认。

补充约束：

- 普通正文与背景至少满足 `4.5:1`。
- 大号文字可按 `3:1` 验收。
- 图标、边框、非文本关键 UI 元素与相邻背景至少满足 `3:1`。

这部分属于硬门槛，不应退化为“看起来差不多”。

### 5.7 对比度校正

`contrast` 不应只是“主次色 hue 角度差”，还需要参与最后的整体校正。

第一阶段建议至少校正以下关系：

- `primary` 与 `secondary` 不得过近。
- 主色与 base 背景需要有足够可见度。
- 各色块内容字与背景块必须可读。
- 整体避免出现极端冲突的高刺激组合。

这里的策略应是“受限派生 + 最终校正”，而不是“先随机，再祈祷好看”。

### 5.8 APCA 的定位

APCA 具有更强的感知导向，尤其对深色主题和低对比界面更敏感。但在第一阶段，APCA 不建议作为硬门槛，只建议作为辅助参考指标。

采用原则：

- 第一阶段正式验收仍以 WCAG 2.2 为准。
- 如实现成本允许，可在开发期同时记录 APCA 结果，用于辅助调参。
- 在 `dark` 模式调校中，若出现“WCAG 过了但肉眼仍发虚”的情况，应优先参考 APCA 结果进一步修正 token。

这样能避免一开始就把主题系统拖入过重的可访问性算法复杂度，同时保留后续演进空间。

## 6. 形态与尺寸规则

### 6.1 Radius 统一

这是明确设计要求，必须作为硬规则：

- `radius-box`
- `radius-field`
- `radius-selector`

三者始终使用同一值。

第一阶段不支持分别编辑，不做三条独立滑杆。

### 6.2 Depth / Noise

这两项不进入自动派生体系：

- `depthEnabled` 为开时，应用统一预设值。
- `noiseEnabled` 为开时，应用统一预设值。
- 关闭时统一置零。

即：

- 它们是手动选项。
- 不是随机项。
- 也不是细粒度自定义调参项。

### 6.3 UI Size

尺寸只走预设，不走自由滑杆。

建议三档：

- `compact`
- `default`
- `comfortable`

每档预设打包控制：

- `size-field`
- `size-selector`
- `border`

如后续确认需要，也可附带联动基础间距，但第一阶段不建议把 spacing 系统一起扩进去。

## 7. 应用方式

### 7.1 不改组件 class，只改 token 来源

当前项目已经依赖 daisyUI CSS 变量，第一阶段不应侵入组件层样式写法。

因此主题生成能力应只做一件事：

- 根据 `GeneratedThemeControls` 生成 `GeneratedThemeTokens`
- 再把 token 应用到当前文档根节点

### 7.2 推荐应用方式

第一阶段推荐两种实现方式中的一种：

#### 方案 A：直接写根节点 CSS 变量

在运行时执行：

```ts
document.documentElement.style.setProperty("--color-primary", value);
```

优点：

- 实现最直接。
- 不依赖额外主题名。
- 很适合边调参数边预览。

#### 方案 B：动态注入一个 generated theme 样式块

例如统一挂一个：

- `data-theme="generated"`

然后动态注入：

```css
[data-theme="generated"] {
  --color-primary: ...;
  ...
}
```

优点：

- 与当前 `data-theme` 语义更一致。
- 更方便后续导出 CSS 或保存成命名主题。

第一阶段更推荐方案 B，因为它更贴近现有主题切换模型，也更利于“预设主题 / 动态主题”并存。

### 7.3 技术选型建议

经过专项调研，第一阶段建议采用以下组合：

- **颜色计算库：`culori`**
  - 用于 OKLCH 解析、转换、序列化与颜色处理。
  - 更适合作为“自定义主题生成器”的底层积木。

- **不建议直接以 `@material/material-color-utilities` 作为正式主路线**
  - 该库方法成熟，但其输出结构围绕 Material 角色体系组织。
  - 当前项目采用 daisyUI token 体系，若直接接入会引入额外的角色映射层。
  - 更适合作为参考实现或实验对照，而不是第一阶段核心依赖。

- **对比度校验：项目内封装 WCAG 检查**
  - 不额外引入重型主题框架。
  - 只实现当前主题生成所需的对比度约束与校正逻辑。

这一选型的核心原则是：

- 底层工具尽量轻。
- 规则由项目自己掌握。
- 不为了“有现成算法”把主题架构绑到别人的角色体系上。

## 8. 设置页交互建议

第一阶段不应做成复杂编辑器，建议保持低门槛：

- 模式：浅色 / 深色
- 主题色色相：色相条或色环
- 对比度：滑杆
- 颜色浓淡：滑杆
- 圆角：滑杆
- UI 尺寸：三档切换
- Depth Effect：开关
- Noise Effect：开关

页面行为建议：

- 用户修改参数后即时预览。
- 用户确认后保存参数，而不是只保存最终 token。
- 若回到该面板，应能从已保存参数恢复当前主题状态。

### 8.1 与预设主题并存

设置页需要支持两条路径并存：

1. 继续选择现有预设主题。
2. 切换到“自定义生成主题”模式。

不要让新能力直接覆盖旧能力，避免把“快速换主题”入口做重。

## 9. 分阶段实施建议

### 阶段 1：主题合成核心打通

目标：先让“动态生成主题”跑通。

包含：

- 扩展主题状态模型，支持 `preset / generated`。
- 引入 OKLCH 颜色计算能力。
- 新增参数模型与 token 派生函数。
- 新增 WCAG 对比度校验与必要的生成后校正。
- 支持动态应用一套 generated theme。
- 设置页新增最小控制项。
- 本地持久化保存生成参数。

不包含：

- 多个自定义主题槽位管理。
- 自定义主题导入导出。
- 用户逐槽位编辑所有 token。

### 阶段 2：增强体验

在阶段 1 稳定后再考虑：

- 引入 APCA 辅助调参与更细的 dark mode 校正。
- 支持把生成主题保存为具名主题。
- 支持导出 CSS。
- 支持对少量高级 token 做精调。
- 支持更多预设风格模板。

### 9.1 实现前需先定标的规则表

在正式编码前，必须先把以下规则表收实，避免实现时边写边猜：

1. `themeHue -> primary` 的映射方式。
2. `secondary` 相对 `primary` 的 hue 偏移范围，以及 `contrast` 如何影响该偏移。
3. `tone` 如何映射到 chroma / lightness。
4. `accent`、`neutral` 的固定 hue 策略。
5. `light / dark` 下 `base-100/200/300` 的 lightness 区间。
6. 三档 `uiSizePreset` 的精确 token 值。
7. `radius` 映射到统一圆角值的规则。
8. depth / noise 开关对应的 token 值。

这份规则表应视为实现前的必要前置产物，而不是开发过程中的临时想法。

## 10. 验收标准

### 10.1 功能验收

- 用户可在设置页切换到“自定义生成主题”。
- 调整最小控制项后，界面主题即时变化。
- 刷新应用后，已保存参数能正确恢复。
- 仍可无回归地使用现有预设主题。

### 10.2 视觉验收

- `light` / `dark` 两种模式下，base 层级都稳定可读。
- `primary` 与 `secondary` 有明确区分。
- 语义色保持稳定辨识。
- 圆角统一，不出现 box / field / selector 风格割裂。
- 三档尺寸切换后，控件尺寸变化明确且协调。

### 10.3 工程验收

- 动态主题不会破坏现有 daisyUI 组件样式。
- 不引入“必须重启才能生效”的链路。
- 主题状态存储结构能清晰表达 `preset / generated`。
- 参数是主题真源，token 只是派生结果，不形成双真源。
- 颜色生成链路采用统一计算空间，不混用多套互相打架的核心算法。

### 10.4 可访问性验收

- 普通文本与背景满足 `4.5:1`。
- 大号文本满足 `3:1`。
- 关键图标、边框、状态元素与相邻背景满足 `3:1`。
- `light` 与 `dark` 两种模式下都不允许出现“标准通过但主体文字明显发虚”的高频组合；如出现，应记录并纳入后续 APCA 辅助校正范围。

## 11. 风险与边界

### 11.1 不宜第一阶段解决的问题

以下内容先不纳入第一阶段：

- 全量复刻 daisyUI 官网随机算法。
- 追求复杂颜色理论或过度精细的综合色彩系统。
- 逐组件级别的 spacing / typography 大规模参数化。
- 允许用户自由编辑十几个颜色槽位。

### 11.2 当前仍需后续实现时细化的点

本计划已经明确骨架，但以下实现细节应在开发前补成具体规则：

- `themeHue -> secondary hue` 的偏移公式。
- `contrast`、`tone` 各自映射到 OKLCH 参数的精确方式。
- `accent`、`neutral` 的固定色相选型。
- 三档 `uiSizePreset` 的具体 token 数值。
- depth / noise 的开关对应数值。
- 内容色黑 / 白切换的具体判定规则。
- 哪些 token 需要纳入 WCAG 校验，哪些只做视觉建议。

这些属于实现参数设计，不影响当前计划骨架成立。

---

## 12. 最终结论

本次主题能力改造的正确方向，不是“继续堆更多预设主题”，也不是“模仿官方高随机生成器”，而是：

- 保留现有预设主题切换。
- 新增一套参数化主题生成链路。
- 让用户只调少量高层参数。
- 系统自动派生稳定、可读、可持续维护的一整套 daisyUI token。

这条路改动集中、收益明确，也最符合当前项目现有主题架构。
