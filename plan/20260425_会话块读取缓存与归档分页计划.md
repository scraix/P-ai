# 20260425_会话块读取缓存与归档分页计划

## 背景

当前消息仓库已经完成 `Conversation -> ConversationBlock -> Message` 的目录化落盘，但上层运行时和归档 UI 仍大量沿用“整会话消息数组”的读取模型：

- `conversation_service/reads.rs` 仍然直接读取/克隆 `conversation.messages`
- `archive.rs` 仍然一次性读取整个归档会话消息
- `use-archives-view.ts` / `ArchivesView.vue` 仍然按“整会话消息列表”展示
- `runtime_cache.rs` 仍缓存完整 `Conversation`
- `state_read_app_data_cached()` 仍会把所有会话重新组装进完整 `AppData`

这导致两个核心问题：

1. 会话消息虽然已经按 block 落盘，但热路径读取仍然经常回退成整会话读取
2. 归档 UI 仍然会因为超长历史一次性加载而卡顿

本计划目标是把“按 block 读取”真正接到运行时读取链路，并引入严格受控的 block 热缓存。

## 用户目标

### 1. 会话读取与缓存

- 会话消息必须按 `ConversationBlock` 读取
- 不能再默认整会话读取 `messages`
- 缓存单位改为 block，而不是整会话
- 只缓存白名单会话：
  - 活跃联系人会话
  - 前台未归档会话
- 每个白名单会话只缓存最后 2 个 block

### 2. 归档 UI

- 归档页改为按 block 分页
- 默认只读取当前页 block
- 翻页时再读取目标 block
- 不允许进入归档页时整会话拉满

## 现状调查

### A. 已有可复用能力

`src-tauri/src/features/chat/message_store/store.rs` 已经具备以下能力：

- 按消息索引随机读取单条消息
- 按消息 ID 读前后分页
- 读取最近消息页
- 读取当前压缩段 / 上一个压缩段
- `messages.idx.json` 已带 `blockId + offset + byteLen`

这说明底层“按 block / 按消息偏移读取”的基础已经存在，不需要重做存储层。

### B. 当前主要阻塞点

#### 1. 读取层仍依赖完整 `Conversation`

关键文件：

- `src-tauri/src/features/chat/conversation_service/reads.rs`
- `src-tauri/src/features/chat/conversation_service/archive.rs`
- `src-tauri/src/features/chat/conversation_service/remote_im.rs`

典型问题：

- `read_unarchived_messages()`：直接 `conversation.messages.clone()`
- `read_recent_unarchived_messages()`：对完整 `messages` 切尾页
- `read_messages_before()` / `read_messages_after()`：仍对完整 `messages` 做切片
- `read_archive_messages()`：直接返回整条归档消息
- `read_remote_im_contact_conversation_messages()`：直接返回整条联系人消息

#### 2. 缓存仍然是完整会话

关键文件：

- `src-tauri/src/features/core/domain/runtime_cache.rs`
- `src-tauri/src/features/chat/conversation_service/core.rs`

典型问题：

- `state_read_conversation_cached()` 缓存的是完整 `Conversation`
- `cached_app_data` 仍持有完整 `AppData`
- `sync_cached_app_data_conversation()` 会把完整 `Conversation` 塞回 `cached_app_data`

#### 3. 归档前端仍按整会话模型工作

关键文件：

- `src/features/chat/composables/use-archives-view.ts`
- `src/features/archive/views/ArchivesView.vue`

典型问题：

- `selectArchive()` 先取 summary，再取整条 `get_archive_messages`
- 归档展示区直接遍历 `archiveMessages`
- 当前没有 block 页码、block 标题、前后翻页状态

### C. 目前不打算在本阶段处理的部分

以下问题已确认存在，但不在本阶段处理：

- `state_read_app_data_cached()` 仍会组装完整 `AppData`
- 发送准备链路仍然可能因为整份 `AppData` clone 偏慢
- delegate 线程消息仍未并入 block 原生存储

本阶段先把“单会话消息读取路径”和“归档 UI”切到 block 模型，不一次性重构整个 `AppData` 体系。

## 设计决策

## 1. 新增 block 元信息读取层

在 `message_store` 内补充面向上层的 block 元信息接口：

- 列出某会话的 block 列表
- 返回每个 block 的：
  - `block_id`
  - `message_count`
  - `first_message_id`
  - `last_message_id`
  - `first_created_at`
  - `last_created_at`
  - 是否当前 block
- 按 `block_id` 读取该 block 的消息

目的：

- 让上层真正基于 block 做分页，而不是继续拿整消息列表后二次切割

## 2. 引入 block 热缓存白名单

在 runtime 层新增 block 级缓存，而不是复用完整会话缓存。

缓存键：

- `conversation_id + block_id`

缓存准入白名单：

- 前台未归档会话
- 活跃联系人会话

缓存数量限制：

- 每个白名单会话最多保留最后 2 个 block

不进入常驻缓存的场景：

- 归档会话
- 非前台未归档会话
- 非活跃联系人会话

## 3. 未归档/联系人读取语义

### 未归档前台会话

- 首次进入：默认读取最后 2 个 block
- “加载更早历史”：以 block 为粒度向前加载
- `hasMoreHistory` 改为依据 block 位置判断

### 联系人会话

- 默认读取最后 2 个 block
- 联系人消息页也走同一套 block 读取逻辑
- 只对白名单联系人做 block 热缓存

## 4. 归档 UI 分页语义

归档页改为：

- 打开归档时只请求：
  - summary
  - block 列表
  - 当前页 block 消息
- 默认落在最新 block
- 点击上一页/下一页时切换 block
- 页面消息只显示当前 block

前端状态新增：

- `archiveBlocks`
- `selectedArchiveBlockId`
- `archiveBlockPageMessages`
- `archiveHasPrevBlock`
- `archiveHasNextBlock`

## 5. 兼容策略

如果某会话还未迁移到目录型 block store：

- 未归档读取先走旧逻辑
- 归档读取先走旧逻辑
- 但新接口返回的元信息中要明确标识 `legacy_fallback`

这样可以保证本次改动不强依赖所有历史数据已经完成迁移。

## 分阶段执行

### 阶段 1：补 message_store block 读取接口

目标：

- 新增 block 列表与按 block 读取能力
- 保持旧 `read_all_messages / read_recent_messages / read_messages_before` 能继续工作

输出：

- `list_ready_message_store_blocks(...)`
- `read_ready_message_store_block_messages(...)`
- 必要的 block 元信息结构体

### 阶段 2：接入 conversation_service 读取层

目标：

- 未归档读取不再默认整会话 clone
- 联系人读取不再默认整会话 clone
- 历史补页改为 block 粒度

输出：

- 新 block 读取 DTO
- 新读取命令
- 旧命令在必要场景退化为兼容层

### 阶段 3：接入 block 白名单缓存

目标：

- 在 runtime cache 中增加 block 缓存
- 只缓存白名单会话最后 2 个 block

输出：

- block cache 数据结构
- 缓存淘汰/覆盖规则
- 前台/联系人白名单判断

### 阶段 4：归档 UI 改为按 block 分页

目标：

- 归档页只读当前页 block
- 增加上一块/下一块翻页

输出：

- 归档 composable 状态更新
- 归档页按钮/页码/当前块展示

### 阶段 5：最小验证

只做与本次改动直接相关的验证：

- Rust `cargo check`
- block 读取相关单测
- 归档 composable / 前端 typecheck

不做无关全量测试，不跑 `cargo fmt`。

## 风险与注意事项

### 1. 旧接口仍可能被外部误用

如果保留 `get_archive_messages` / `get_unarchived_conversation_messages` 原语义不动，前端或其他命令仍可能继续整会话读取。

处理：

- 新接口落地后，优先把前端入口切过去
- 旧接口保留兼容，但不再作为主路径

### 2. block 边界和 UI 预期不一致

联系人会话 block 可能按 04:00 日界线拆分；普通会话按压缩边界拆分。归档 UI 必须接受“每页消息数量不固定”。

处理：

- UI 文案按“块/分页”表达，不再暗示固定条数页

### 3. materialize 仍可能放大当前页成本

即使只读一个 block，图片/音频附件 materialize 仍可能造成单页成本偏高。

处理：

- 本阶段先接受“只放大当前 block”
- 后续再按需把 materialize 继续懒化

## 完成标准

满足以下条件即可认为本阶段完成：

- 未归档前台会话默认只读最后 2 个 block
- 活跃联系人默认只读最后 2 个 block
- 白名单 block 缓存生效，缓存单位不再是整会话
- 归档 UI 默认只读当前 block，支持按 block 翻页
- 归档页不再调用整会话消息读取接口作为主路径
