# 20260425_会话块读取缓存与归档分页计划

## 定性

这份计划书已经不再是“准备怎么做”，而是 **对最终正确实现的归档说明**。

需要明确一点：**最初那版计划是有明显设计错误的**。后续实现过程中，我们连续纠偏了多次，最后落地出来的这一版，才是正确且稳定的方案。
因此，本文档不是“实现偏离计划”，而是：

- 旧计划的关键判断有误
- 现在的实现完成了对旧计划的纠正
- 后续如果继续演进，应以当前实现为准，不能退回旧思路

## 旧计划的根本问题

旧计划最大的问题，不是细节不完善，而是抽象层次弄反了。

### 1. 把“按块缓存”误写成“默认读多个块”

旧计划把“缓存最近 2 个 block”错误理解成了“高频路径默认读取最近 2 个 block”。
这是错误的。

正确语义是：

- **读取** 按需求条数精确进行
- **缓存** 只按 block 分桶和淘汰

也就是说：

- 要 1 条，就只读 1 条
- 要 4 条，就只读 4 条
- 缓存里只保存已经读过的消息
- “最后 2 个 block”只是缓存保留策略，不是读取策略

### 2. 把 block 当成了消费层概念

旧计划隐含了一种错误方向：让上层越来越感知 block。
这是不对的。

正确模型是：

- 存储层知道 `ConversationBlock`
- 服务层按消息条数、分页方向、目标消息游标来返回结果
- 前端和上层业务不需要知道自己这次拿的是哪一个 block

block 是存储与缓存实现细节，不应该泄漏成上层默认语义。

### 3. 把“局部自愈”写成了“可能的全局重建”

旧计划没有把索引修复边界定死，容易滑向“顺手重建整份列表”。
这会把本来很小、很稳的修复动作，变成昂贵又高风险的全局重算。

现在已经明确：

- 只修当前会话对应的单条索引项
- 不重建整份 `chat_index`
- 不依赖旧的整包聚合缓存

### 4. 没有把“按条读取”写成硬规则

旧计划虽然引入了 `ConversationBlock`，但并没有把 jsonl + idx 的真正价值讲清楚。
如果还是“先整块读，再在内存里裁几条”，那这套存储设计只做了一半。

当前正确实现已经定死：

- `messages.idx.json` 负责随机定位
- `blocks/*.jsonl` 负责承载正文
- 小量消息读取必须走精确读取
- 不能再退回“整块读完再裁”

## 当前正确实现

### 1. 存储模型

消息仓库已经稳定为三级结构：

- `Conversation`
- `ConversationBlock`
- `Message`

目录型会话 shard 形态为：

```text
chat/conversations/<conversation_id>/
  meta.json
  manifest.json
  messages.idx.json
  blocks/
    000000.jsonl
    000001.jsonl
    ...
```

其中：

- `messages.idx.json` 只保留最小随机读取索引
  - `messageId`
  - `blockId`
  - `offset`
  - `byteLen`
- `blocks/*.jsonl` 承载真实消息正文

这套结构已经不是实验性的，它就是当前正确实现。

### 2. 读取规则

当前读取规则已经明确，而且应该视为硬规则：

- 要 1 条消息，就只物理读取 1 条
- 要 4 条消息，就只物理读取 4 条
- 要 `before / after` 分页，就只读取命中的那些消息
- 真正跨块时，才会读到别的 block

所以当前系统的真实语义是：

**按条读取，按块缓存**

而不是：

**按块读取，再裁结果**

### 3. 缓存规则

当前块缓存的真实语义是：

- 缓存键按 `conversation_id + block` 归属管理
- 每个 block 缓存桶内部只保存“已经读过的消息”
- 不缓存整条会话
- 不缓存整块全部消息

白名单保留规则：

- 前台未归档会话
- 活跃联系人会话

数量规则：

- 每个白名单会话最多保留最后 2 个 block 的缓存桶

需要反复强调：

- **缓存 2 个 block** 不等于 **每次读取 2 个 block**
- **block 是缓存分桶单位，不是默认读取单位**

### 4. 前台会话读取语义

当前真实行为已经收敛为：

#### 首次轻量展示

- 只围绕最新 block 工作
- 只返回最后 4 条消息

#### 最近消息读取

- 只看最新 block
- 按请求条数裁剪返回

#### `before / after`

- 命中 message store 索引后，按消息精确读取
- 只有真正跨过块边界时，才会读到别的 block

#### 发送请求

- 发送前 prompt 只使用最新 block 的上下文
- 持久化仍保留完整会话语义，不会因为 prompt 裁剪而丢历史

这里已经和旧计划彻底分道扬镳：
不是“默认多块上下文”，而是“默认单块、按需跨块”。

### 5. 归档窗口语义

当前归档窗口已经统一成真正的块分页：

- `archive`：按 block 分页
- `current`：按 block 分页
- `remoteIm`：按 block 分页

顶部块切换条复用同一套交互：

- 上一块
- 下一块
- 当前块摘要
- 当前块时间范围

其中：

- 归档摘要仍只属于 `archive` 视图
- 摘要在消息内容区域内，不参与块切换条
- 块切换条固定在滚动区域上方，不会被划走

这也是对旧计划的重要修正：
不是只有归档消息能按块翻页，而是三条主链路全部统一。

### 6. 单条索引自愈

当前已经补上“**单条索引项自愈**”，这是这轮实现里非常关键的一条正确决策。

实现边界如下：

#### 允许自愈

- 会话 shard 可读，但 `chat_index` 缺少该条
- 会话 shard 可读，但该条索引项字段不一致
  - `updated_at`
  - `status`
  - `summary`
  - `archived_at`

#### 不做伪修复

- 会话目录本体不可读
- message store / JSONL 校验失败
- 消息文件损坏
- 任何导致“单会话真相无法可靠读取”的情况

#### 修复方式

- 每个 `conversation_id` 一把进程内修复锁
- 拿锁后做二次检查
- 只对这一条索引项做 upsert
- 不重建整份列表
- 不依赖 `cached_app_data`

#### 日志

会输出：

- `conversation_id`
- 触发原因
- 修复原因
- 修复前后关键字段
- 成功 / 跳过 / 失败
- 耗时

这条边界必须保持。
后续不能把它重新做回“顺手重建整份索引”。

## 已完成项

以下内容已经完成，不再属于“待实现计划”：

### A. 读取热路径块化

- 前台轻量快照
- 最近消息读取
- `messages_before / messages_after`
- 联系人最近消息读取
- 发送前 prompt 裁剪

### B. 精确消息读取

- 最近 N 条消息读取
- 根据 `messageId` 的前后读取
- 基于索引的单条消息读取

### C. 块缓存收口

- 按块分桶
- 只缓存已读取消息
- 白名单会话最后 2 个 block

### D. 归档窗口块分页

- 归档消息
- 当前未归档消息
- 联系人消息

### E. 单条索引自愈

- 挂在统一单会话读取入口
- 只修单条 item
- 不做整份列表重建

## 明确不做的事

以下内容当前没有做，而且不应该被误写成“这轮缺失”：

### 1. 不重做整套 `AppData` 架构

这轮已经把大量热路径从整份 `AppData` 上摘掉，但并没有彻底删除兼容层聚合结构。
这是有意为之，不是漏做。

### 2. 不引入全局文件事务系统

没有把“会话本体 + 列表 + runtime”重做成统一文件事务提交系统。
当前优先级是：

- 读取尽量走分片
- 单条索引异常时局部自愈

### 3. 不对委托线程消息做这轮分页改造

本轮归档窗口的块分页只覆盖：

- 归档消息
- 当前未归档消息
- 联系人消息

委托线程消息仍保持现有内存态读取语义。

### 4. 不做全局列表重建

即使发现索引问题，也只修当前会话项。
不会顺手扫描并重建整份 `chat_index`。

## 后续可选方向

如果后面继续推进，建议按下面顺序考虑：

### 1. 补强索引自愈触发面

当前自愈挂在单会话读取入口。
后续可以继续检查是否有仍绕过该入口的低频链路。

### 2. 为块缓存增加显式内存上限

目前已经有：

- 会话白名单
- 最后 2 个 block

但还没有额外写死：

- 单 block 最大字节数
- 单 block 最大缓存消息数

后续如果继续收紧内存边界，应在当前正确实现上补，不要回退到旧计划的整块缓存思路。

## 结论

这份计划书的最终结论很简单：

- 旧计划并不优秀，核心抽象就是错的
- 现在的实现不是“凑合对齐”，而是一次成功的纠偏
- 当前这一版，才是本项目在“会话块读取、块缓存、归档分页、索引自愈”上的正确方案

后续所有演进，都应以当前实现为准，而不是回头参考那版已经被证明有问题的旧计划。
