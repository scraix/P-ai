# 20260426_引导消息续调度检查点修正计划

## 背景

现有引导投送实现把核心检查点放在主助理轮次收口之后，由外层队列调度发现会话里存在 `guided` 队列消息后再处理。

这个位置只能覆盖“当前轮次完整结束后”的接管，不能覆盖工具循环中的关键边界：LLM 每次 loop 结束、工具已经执行完成、系统准备把工具结果继续喂回模型之前。

引导消息不是 assistant 消息，也不是 tool result，不能直接插入当前 tool loop 的消息尾部。正确做法是：当前 loop 闭合，保存已经完成的工具历史，再把引导消息写入正式历史，并重开下一次调度。

## 正确语义

引导消息的检查点必须位于：

- LLM 一次 loop 已经结束；
- 本轮工具调用已经执行完成；
- 工具历史已经完整形成；
- 系统即将决定是继续当前 tool loop，还是结束当前调度。

如果此时发现当前会话存在引导消息：

1. 不继续把工具结果喂回当前 LLM loop。
2. 当前调度闭合并落盘已完成的 assistant/tool 历史。
3. 外层收口后处理 guided 队列消息，把它们插入正式历史。
4. 插入后重新激活主助理，形成新的调度。

## 非目标

- 不把引导消息改成普通用户消息直接拼进当前 provider 请求。
- 不在主助理激活前提前消费 guided 队列。
- 不在工具还没执行完时打断当前工具。
- 不改变普通 normal 队列消息的处理顺序。

## 修复点

### 1. 工具循环增加引导检查

在流式与非流式 tool loop 中，工具执行完成并写入工具历史后，检查当前会话是否存在 guided 队列消息。

命中后返回一个“因引导消息闭合当前调度”的模型结果，让外层按正常 assistant 消息持久化流程收口。

### 2. 当前调度必须正常闭合

返回结果必须携带当前已形成的工具历史，确保 assistant tool_calls 与 tool result 不丢失。

如果当前 loop 没有生成可见文本，也仍然要允许落盘这条带工具历史的 assistant 消息，因为它是协议闭合的一部分。

### 3. guided 队列仍由外层收口后处理

保留现有外层收口后的 guided 队列处理：当前调度完成后，处理 guided 队列，把引导消息写入正式历史并重新激活助理。

这不是删除原检查，而是把“是否应该提前闭合当前 tool loop”的检查移动到正确时间点；外层收口后的 guided 处理仍负责真正插入引导消息与重开调度。

### 4. 防回归验证

新增最小单元检查：带工具历史且命中 guided 闭合标记的模型结果，必须被视为有效结果，不能被当成空回复重试或切模型。

## 风险

- 若只返回闭合信号但不允许空文本工具历史落盘，会导致工具调用历史丢失。
- 若在激活前消费 guided 队列，会回到旧问题：引导发生在错误时间点。
- 若在工具执行前打断，会破坏“当前 loop 自然结束后再引导”的语义。

## 完成标准

- guided 检查发生在工具执行完成后、下一次模型请求前。
- 命中 guided 时当前 tool loop 不继续请求模型。
- 当前 assistant/tool 历史正常落盘。
- guided 消息随后由外层队列处理插入历史并重开调度。
- 最小相关 Rust 测试通过。
