# 20260427_主部门联系人绑定与续调度继承计划

## 背景

当前远程联系人链路里，轮末自动外发已经部分依据本轮 `remote_im_activation_sources` 决定目标联系人；但 `contact_reply`、`contact_send_files`、`contact_no_reply` 的工具暴露条件与目标解析仍然绑定在“联系人会话”上。

这会导致主部门被联系人选为应答部门时出现语义裂缝：

- 本轮实际上由联系人消息触发；
- 轮末最终结果可以自动发往联系人；
- 但 LLM 中途却拿不到联系人工具，或工具执行时无法按本轮绑定联系人发送。

本次改动要把“联系人会话能力”收束为“本轮绑定联系人的运行时能力”，并保证压缩后续调度、follow-up、同轮续跑继承该绑定联系人。

## 目标

让主部门在被联系人选为应答部门时，也能按“本轮绑定联系人”获得 `contact_reply` / `contact_send_files` / `contact_no_reply`，并让轮末自动外发、压缩续调度、follow-up 续跑共用同一绑定联系人。

## 已确认规则

1. 调度准备阶段先绑定本轮联系人，事实源使用已有的 `remote_im_activation_sources`。
2. 只要本轮已绑定联系人，就注入 `contact_reply`、`contact_send_files`、`contact_no_reply`。
3. 是否注入这 3 个工具，不看部门是否为主部门，也不看当前会话是否为联系人会话。
4. 本轮未绑定联系人，则不注入这 3 个工具，也不会往联系人渠道发消息。
5. 联系人工具调用与轮末最终自动外发，都发往同一个本轮绑定联系人。
6. 压缩后的后续调度、follow-up、同轮续跑都必须继承该绑定联系人。
7. `contact_no_reply` 仍可抑制轮末自动外发。

## 实现范围

### 1. 固化“本轮绑定联系人”运行时上下文

- 复用调度准备阶段已收集的 `remote_im_activation_sources`。
- 明确单联系人绑定规则：
  - 本轮存在唯一有效联系人来源时，生成绑定联系人上下文；
  - 无联系人来源时，绑定为空；
  - 多联系人来源沿用现有保守语义，不自动外发，也不伪造单目标绑定。
- 将绑定结果放入轮次运行时上下文，供：
  - 工具装配；
  - 联系人工具执行；
  - 轮末自动外发；
  - 压缩后续调度 / follow-up / 同轮续跑继承。

### 2. 改造联系人工具注入条件

- 修改 `tool_assembly.rs`。
- 不再以“联系人会话”作为 `contact_reply` / `contact_send_files` / `contact_no_reply` 的暴露条件。
- 改为只看“本轮是否已绑定联系人”。
- 无绑定联系人时，不暴露这 3 个工具。

### 3. 改造联系人工具目标解析

- 修改 `builtin_remote_im.rs`。
- `contact_reply` / `contact_send_files` 的发送目标不再依赖“当前 session 对应联系人会话”反查。
- 优先从本轮绑定联系人上下文解析发送渠道与联系人。
- 若当前轮次无绑定联系人，则拒绝调用并返回清晰错误。

### 4. 统一轮末自动外发与续调度继承语义

- 统一联系人目标解析口径，工具侧与轮末自动外发共用同一绑定联系人事实源。
- 检查并补齐以下路径对绑定联系人的继承：
  - 压缩后续调度；
  - `remote_im_followup`；
  - 同轮 restart / continue 等续跑路径。
- 保持 `contact_no_reply` 对轮末自动外发的抑制语义不变。

### 5. 更新提示词与运行时说明

- 把“联系人会话中可用”改成“本轮已绑定联系人时可用”。
- 明确这 3 个工具作用于“当前轮次绑定联系人”，不是当前窗口用户。
- 补强文件发送提示：
  - 联系人发送文件时，不要把本地路径直接写进正文；
  - 应把本地绝对路径或工作区路径放进 `contact_send_files.file_paths`；
  - 如需配文，使用工具文本参数或单独调用 `contact_reply`。

### 6. 最小必要测试与验证

- 补覆盖主部门 + 联系人绑定场景的工具注入测试。
- 补覆盖无绑定联系人时工具不可用的测试。
- 补覆盖绑定联系人在压缩 / 续跑后仍保留的测试。
- 执行最小相关验证：
  - `cargo test` 针对新增/受影响测试；
  - `cargo check`；
  - 若提示词或前端类型受影响，再补最小必要 `pnpm typecheck`。

## 验收标准

1. 主部门被联系人激活时，LLM 可见 `contact_reply` / `contact_send_files` / `contact_no_reply`。
2. 这 3 个工具执行时，目标联系人来自本轮绑定上下文，而不是联系人会话反查。
3. 本轮无绑定联系人时，这 3 个工具不可用，轮末也不走联系人渠道。
4. 压缩后续调度、follow-up、同轮续跑不丢失绑定联系人。
5. `contact_no_reply` 仍可阻止轮末自动外发。
