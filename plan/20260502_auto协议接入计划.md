# auto 协议接入计划

## 目标

分两步接入 genai 协议抽象。

第一步先不新增前端能力参数列表，也不实现“模型名 -> 协议”的前端查询接口；先把当前所有文本协议的请求构造收敛到 genai 统一抽象：`ChatRequest`、`ChatOptions`、`ServiceTarget`。

第二步再新增文本模型协议 `auto`，让用户可以选择“自动协议”。现有 `openai`、`openai_responses`、`deepseek/kimi`、`gemini`、`anthropic`、`codex` 等手动协议全部保留。

`auto` 只负责让 genai 根据模型名自动解析 `AdapterKind`；供应商配置里的 `base_url`、`api_key`、模型名、能力开关、温度、最大输出、工具开关仍照常使用。

## 语义边界

- 第一阶段只统一后端 genai 抽象构造，不新增前端 auto 协议入口。
- `auto` 是文本聊天协议，不影响 STT/TTS/Embedding/Rerank。
- `auto` 不等于“自动发现 URL”，仍使用当前供应商的 `base_url`。
- `auto` 不替代 Codex OAuth 登录；Codex 专属 `codex` 协议继续保留。
- 手动协议仍然按用户指定的 `AdapterKind` 路由，不被自动推断覆盖。
- 除了 `AdapterKind` 来源不同，手动协议和 `auto` 不应有不同的消息字段裁剪策略。
- P-ai 尽可能把内部语义完整转换成 genai 抽象，字段支持由 genai adapter 自己决定。

## genai 自动匹配规则

使用 genai 默认模型名前缀解析：

- OpenAI：`gpt-*` 大多数、`o1-*`、`o3-*`、`o4-*`、`chatgpt-*`、`codex-*`
- OpenAI Responses：`gpt-5-*`、`gpt-*` 且包含 `codex` 或 `pro`
- Anthropic：`claude-*`
- Gemini：`gemini-*`
- xAI：`grok-*`
- DeepSeek：`deepseek-*`
- Zai：`glm-*`
- Cohere：`command-*`、`embed-*`
- Mimo：`mimo-*`
- Fireworks：模型名包含 `fireworks`
- Ollama：其他名称 fallback 到 Ollama

## 实现步骤

### 第一步：统一 genai 抽象构造

1. 统一 `ChatRequest` 构造
   - 新增或收敛为单一 `build_genai_chat_request`。
   - 所有文本协议都使用同一套历史消息、最新用户消息、附件、工具调用、工具结果、思维链构造。
   - 不再按 OpenAI Responses / OpenAI Chat / Gemini / Anthropic 分别裁剪消息字段。
   - 任何情况下，只要历史中存在 `reasoning_content`，都通过 `ChatMessage::with_reasoning_content` / `ContentPart::ReasoningContent` 传给 genai。
   - 缺失思维链时不为特定协议编造空串；由 genai adapter 或供应商返回决定是否接受。

2. 统一 `ChatOptions` 构造
   - 新增或收敛为单一 `build_provider_genai_chat_options`。
   - 统一设置 capture、温度、最大输出、reasoning_effort、extra_headers。
   - 不按协议预先关闭或裁剪通用选项。

3. 统一 `ServiceTarget` 构造
   - 新增或收敛为单一 `build_provider_genai_service_target`。
   - 手动协议只负责指定 `ModelIden.adapter_kind`。
   - `auto` 后续使用 `ModelSpec::Name` 或自动解析结果；不影响第一步。
   - `base_url`、`api_key`、extra headers 仍由 P-ai 供应商配置注入。

4. 工具循环同步复用统一构造
   - 工具定义、工具调用历史、工具结果都尽可能转换为 genai `Tool` / `ToolCall` / `ToolResponse`。
   - 不再由 P-ai 按目标协议决定哪些 genai 字段能不能传。
   - 仅保留无法删除的运行外围：Codex OAuth 刷新、供应商串行门、流式失败兜底、system->user fallback、百炼多模态 URL 化。

5. 验证
   - 更新请求构造相关单测。
   - 跑最小相关 Rust 测试。
   - 跑 `cargo check`。

### 第二步：新增 auto 协议与前端能力参数

1. 后端协议枚举
   - 在 `RequestFormat` 增加 `Auto`，序列化为 `auto`。
   - `from_str`、`as_str`、`is_chat_text` 纳入 `auto`。

2. 后端模型协议解析接口
   - 新增接口：输入模型名，返回 genai 解析出的 `AdapterKind` / 协议名。
   - 只反馈解析结果，不维护字段能力矩阵。

3. 前端配置
   - `ApiRequestFormat` 加入 `auto`。
   - 文本协议下拉新增 `Auto（按模型名自动匹配）`。
   - 根据 genai 已支持的参数在前端展示、持久化，并统一使用。

4. 周边入口
   - 会话可选模型、部门模型过滤、模型刷新、视觉调用、调试日志格式兼容 `auto`。

## 风险

- genai 对无法识别模型名会 fallback 到 Ollama。第二步实现 `auto` 时需要在 UI 文案和日志里明确最终解析结果，避免用户填错模型名后请求打歪。
- `auto` 可能让 `gpt-5-*` 自动走 OpenAI Responses，与用户原本选 `openai` 的手动行为不同；这是该协议的预期，但手动协议保留用于覆盖。
- `base_url` 与 adapter 类型可能不匹配，例如 Gemini 模型配 OpenAI compatible URL。此时应让请求失败并显示 provider 错误，不做猜测式修正。
- genai 当前公开 API 没看到静态 capability 查询接口；P-ai 不维护字段能力矩阵，字段支持由 adapter 构造或运行时错误反馈。
