# 快速设置与启动窗口调整计划

## 背景

当前应用启动后由 Rust `main.rs` 固定显示 `main` 配置窗口；配置页欢迎页只负责展示配置完成度。需要改为：

- 已具备可用 LLM 时，启动直接打开对话窗口。
- 缺少必要 LLM 时，启动打开快速设置窗口。
- 快速设置是独立窗口，承载一个不可跳过的基础流程；基础流程完成后自动打开对话窗。
- 欢迎页任何时候都能打开快速设置。

## 当前链路事实

- 固定窗口定义在 `src-tauri/tauri.conf.json`：`main`、`chat`、`archives`。
- 启动显示窗口在 `src-tauri/src/main.rs`，当前固定 `show_window(&app_handle, "main")`。
- 托盘入口在 `src-tauri/src/features/system/windowing.rs`，当前有配置、对话、归档。
- 前端窗口按 label 判定模式：`src/features/shell/composables/use-window-shell.ts`。
- 配置页主组件为 `src/features/config/views/ConfigView.vue`。
- 供应商编辑逻辑集中在：
  - `src/features/config/composables/use-config-core.ts`
  - `src/features/config/composables/use-config-editors.ts`
  - `src/features/config/views/config-tabs/ApiTab.vue`
- 保存配置命令为 `save_config`，会同步注册唤出热键。
- 记忆重排/嵌入绑定命令已有：
  - `save_memory_rerank_binding`
  - `save_memory_embedding_binding`
- 现有 `save_config` 会调用 `ensure_default_shell_workspace_in_config`，当前只保留一个 system 工作区；因此“shell 工作目录”这一步第一版应对接现有 system 工作区语义，避免伪造主/副目录能力。

## 判定规则

新增共享判定函数：

```ts
hasUsableTextLlm(config): boolean
```

满足以下条件才算有可用 LLM：

- 存在 `enableText === true` 的 `apiConfig`
- 对应 `assistantDepartmentApiConfigId` 非空并指向该文本模型，或可从内置主部门绑定到文本模型
- 供应商 API Key 非空；但 `codex` 的 `read_local` / `managed_oauth` 可不要求 API Key
- 模型名、base_url 非空

该判定只用于启动路由与快速设置入口提示；真正请求仍由后端校验。

## 实施步骤

### 1. 新增快速设置窗口

- 在 `tauri.conf.json` 新增窗口：
  - label：`quick-setup`
  - url：`quick-setup.html`
  - 标题：`PAI - 快速设置`
  - 建议尺寸：`760 x 820`
  - resizable：`true`
  - visible：`false`
  - decorations：`false`
- 新增入口文件：
  - `src/main-quick-setup.ts`
  - `src/apps/quick-setup/QuickSetupApp.vue`
- 复用 `UnifiedWindowApp` 的配置加载/保存能力时，给 `UnifiedWindowApp` 增加 `fixedViewMode="quickSetup"` 或新增轻量 QuickSetup 根组件。
- 为了避免把聊天和归档运行态全部挂进快速设置窗口，第一版优先做独立 QuickSetup 根组件：只加载配置、人格、聊天设置所需命令，不初始化聊天流式监听。

### 2. 启动窗口路由

在 Rust 启动阶段读取配置并判定：

- 若有可用 LLM：显示 `chat`
- 否则：显示 `quick-setup`

补充命令：

- `show_quick_setup_window`
- `complete_quick_setup_and_open_chat`：保存完成标记后打开 chat，隐藏 quick-setup

托盘菜单新增“快速设置”。

单实例激活规则改为：

- 若 chat 已可用或已显示，聚焦 chat
- 若缺少可用 LLM，聚焦 quick-setup
- 不再默认回到 main 配置窗

### 3. 快速设置基础流程

新增组件：`src/features/config/quick-setup/QuickSetupWizard.vue`。

基础流程不可跳过，步骤如下：

0. 语音与外观
   - 语言、主题、UI 字体
   - 录音相关开关只放最少项：录音热键可在第 5 步统一设置，这里只做语音功能说明与是否启用后台唤醒默认值

1. LLM 供应商
   - 新建或编辑第一个文本供应商
   - 默认推荐 DeepSeek：
     - provider name：DeepSeek
     - requestFormat：`deepseek` 或 `auto`（需要最终按现有协议习惯选定；倾向 `auto`，base_url 用 DeepSeek）
     - base_url：`https://api.deepseek.com/v1`
     - model：`deepseek-chat`
   - 必填：API Key、模型名、base_url
   - 保存后自动绑定到：
     - `selectedApiConfigId`
     - `assistantDepartmentApiConfigId`
     - 内置主部门 `apiConfigId/apiConfigIds`

2. 对话风格
   - 选择 `responseStyleId`
   - 可复用现有 `response-styles.json`

3. 用户和主部门命名与头像
   - 用户名：更新内置用户人格名称 / `userAlias`
   - 主助理名：更新当前主部门人格名称
   - 主部门名：更新内置主部门名称
   - 头像：支持使用现有 `save_agent_avatar` / `clear_agent_avatar`

4. Shell 工作目录
   - 第一版按现有全局 system 工作区处理：设置 `shellWorkspaces[0]`
   - 字段：名称、路径、访问级别固定走现有 system/full_access 归一化
   - 若后续要支持真正主/副工作目录，需要先改后端 `ensure_default_shell_workspace_in_config`，本轮不顺带重构

5. 唤出热键和录音键
   - 复用 `HotkeyTab` 的捕获逻辑，抽出 `HotkeyCaptureInput` 小组件
   - 设置 `config.hotkey`
   - 设置 `config.recordHotkey`
   - 保存配置后后端自动重注册唤出热键

6. 完成或继续高级选项
   - “完成设置”：保存配置，调用 `complete_quick_setup_and_open_chat`
   - “继续高级选项”：进入高级流程
   - 基础步骤 0-5 不允许跳过；但可以在已有值合格时直接下一步

### 4. 高级选项流程

高级流程可跳过单项。

1. 重排模型
   - 推荐 SiliconFlow：`https://api.siliconflow.cn/v1`
   - requestFormat：`openai_rerank`
   - 设置后调用 `save_memory_rerank_binding`
   - 默认模型候选建议：`BAAI/bge-reranker-v2-m3`

2. 嵌入模型
   - 推荐 SiliconFlow：`https://api.siliconflow.cn/v1`
   - requestFormat：`openai_embedding`
   - 设置后调用 `save_memory_embedding_binding`
   - 默认模型候选建议：`BAAI/bge-m3`

3. 语音转写模型
   - 推荐 SiliconFlow：`https://api.siliconflow.cn/v1`
   - requestFormat：`openai_stt`
   - 设置后绑定到 `config.sttApiConfigId`
   - 用户提示：不需要语音可以跳过
   - 注意：用户原话说“设置语音转写模型后自动设置为记忆嵌入供应商”，这与现有架构不一致，证据不足且语义冲突。应按 STT 供应商绑定到 `sttApiConfigId`，不绑定到记忆嵌入。

### 5. 欢迎页入口

在 `WelcomeTab.vue` 顶部增加按钮：

- “打开快速设置”
- 调用 `show_quick_setup_window`

配置窗口内也可在欢迎页打开 quick-setup；不强制隐藏配置窗口。

### 6. 状态持久化

新增前端/配置标记：

- `quickSetupCompleted` 建议写入 `AppConfig`，后端默认 false
- 但启动是否进入 quick-setup 不只看该标记，必须以“是否有可用 LLM”为硬条件
- 若 LLM 后续被删空，仍打开 quick-setup

若为减少后端配置 schema 变更，也可第一版不加完成标记，只基于 LLM 可用性判断。

### 7. 验证

最小必要验证：

- `pnpm build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- 新增/调整 Rust 单测：启动目标判定函数
- 前端最小单测或类型检查：快速设置可用性判定函数

## 风险与边界

- 这是启动路由 + 新窗口 + 配置写入流程，不是小 UI，必须先按计划推进。
- `save_config` 当前拒绝空 `api_configs`，快速设置第一步必须保证草稿生成合法 provider/model 后再保存。
- Shell 工作区后端目前会压成 system 工作区；本轮不承诺真正多工作区配置。
- “语音转写模型自动设置为记忆嵌入供应商”与现有模型类型冲突，应更正为绑定 STT。若必须按原话实现，会破坏记忆嵌入协议校验。
