# 20260506_VSCode工作区引用块接入计划.md

## 1. 目标

把 VS Code 当前上下文接入 P-ai，但不做会话与 IDE 的 1v1 绑定，也不做拖放连线。

首版改为：

1. P-ai 后端接收所有在线 VS Code 窗口上报的 IDE 上下文。
2. 按当前会话已配置的工作目录过滤可用上下文。
3. 在聊天输入面板上方显示一个或多个“文件行号引用块”。
4. 每个引用块有 `+ / -` 控制，用户决定本轮是否附加。
5. 发送消息时，把被附加的引用块作为 `extraTextBlocks` 一并发给 LLM。
6. 若当前会话包含多个工作目录，则可同时显示多个工作目录对应的引用块。

---

## 2. 本次范围

### 2.1 包含

1. 新增一条本地 IDE 上下文接入链路，统一接收 VS Code 插件上报。
2. 后端缓存最近一次来自各 VS Code 窗口的上下文。
3. 按会话工作目录过滤：只展示工作目录命中的 IDE 文件上下文。
4. 在输入面板上方显示引用块列表。
5. 引用块支持用户逐个开启/关闭附加。
6. 发送时把已开启的引用块转成 `extraTextBlocks`。
7. 若存在多个工作目录命中多个文件，则显示多个引用块。

### 2.2 不包含

1. 不做拖放绑定。
2. 不做会话与 IDE 窗口的 1v1 关系。
3. 不做复杂拓扑连线动画。
4. 不做整文件全文自动附加。
5. 不做 IDE 主动把内容直接写成聊天消息。
6. 不做 JetBrains 等其他 IDE 接入。

---

## 3. 需求语义

### 3.1 “始终绑定所有 VS Code”

这里的含义不是把所有 VS Code 内容无条件喂给当前会话，而是：

1. 后端统一接收所有 VS Code 窗口上报。
2. 当前会话只从这些上报中筛选“工作目录相关”的候选引用块。
3. 最终是否附加到本轮消息，仍由用户在输入面板手动决定。

### 3.2 “工作目录过滤”

过滤依据使用当前会话的 `chatWorkspaceChoices / currentChatWorkspaces`。

命中规则首版定义为：

1. IDE 上下文里的 `filePath` 标准化后，落在某个工作目录根路径之下，则视为命中。
2. 同一工作目录下若有多个 VS Code 窗口上报同一文件，优先保留最近一次更新时间最新的一条。
3. 若当前会话没有工作目录，则不展示 IDE 引用块。

### 3.3 “文件行号引用块”

引用块本质是发送前的上下文附件候选，不污染输入框正文。

首版展示信息建议至少包含：

1. 工作目录名
2. 文件相对路径
3. 行号范围
4. 选中摘要或当前选中文本预览
5. `+ / -` 按钮
6. 区分“已附加固定块”和“新到候选块”

### 3.4 已附加块固定语义

用户一旦点击 `+` 把某个 IDE 引用块附加到当前输入轮次，这个块就应固定保留，不再随着 IDE 后续选区变化被原地改写。

具体规则：

1. `+` 的瞬间，对当前 IDE 引用块做一次快照，写入当前轮次已附加列表。
2. 之后即使 IDE 继续切标签、改选区、改行号，这个已附加块内容也不变。
3. 用户只能通过 `-` 主动移除该已附加块。
4. IDE 后续新变化应显示为新的候选标签，而不是覆盖已附加块。

## 4. 协议设计

### 4.1 传输方式

首版采用单一 localhost WebSocket 服务。

原因：

1. 多个 VS Code 窗口可复用一个入口。
2. 后端可持续收到更新，不必轮询。
3. 后端收到上下文更新后通过 Tauri 事件通知前端，前端即时刷新候选引用块。
4. 前端保留低频轮询作为兜底，避免事件丢失或窗口重新挂载后状态不同步。
5. 后续若需要心跳、断线重连、双向协商也容易扩展。

### 4.2 后端监听地址

首版固定单端口，例如：

`ws://127.0.0.1:43127/ide-context`

说明：端口只是传输入口，不承载“窗口绑定”语义。

### 4.3 VS Code 上报消息

建议首版消息结构：

```json
{
  "type": "context",
  "source": "vscode",
  "instanceId": "vscode-window-uuid",
  "timestamp": "2026-05-06T21:01:00+08:00",
  "workspaceFolders": ["D:/Code/avatar_py"],
  "filePath": "D:/Code/avatar_py/main.py",
  "languageId": "python",
  "selection": {
    "startLine": 10,
    "startCharacter": 0,
    "endLine": 18,
    "endCharacter": 4,
    "text": "def upload_avatar(...):"
  },
  "visibleRanges": [
    { "startLine": 1, "endLine": 80 }
  ]
}
```

### 4.4 P-ai 内部缓存结构

后端只缓存最近状态，不直接入聊天消息。

```ts
type IdeContextSnapshot = {
  source: 'vscode'
  instanceId: string
  timestamp: string
  workspaceFolders: string[]
  filePath?: string
  languageId?: string
  selection?: {
    startLine: number
    startCharacter: number
    endLine: number
    endCharacter: number
    text: string
  }
  visibleRanges?: Array<{
    startLine: number
    endLine: number
  }>
}
```

## 5. 前端交互设计

### 5.1 展示位置

放在聊天输入面板上方，复用当前“附件 badge / 指令 badge”附近的轻量展示方式。

### 5.2 引用块状态

每个候选引用块包含两层状态：

1. `visible`：由工作目录过滤得出，是否应该显示。
2. `attached`：用户本轮是否选择附加到消息。

默认建议：

1. 命中后显示。
2. 默认不附加，由用户点 `+` 后附加。
3. 已附加状态显示 `-`，可移除。
4. 已附加块仍保留在所属工作目录分组内，并在该组中排在候选块前面，不再单独维护独立固定栏。
5. 新 IDE 变化以新的候选标签出现，供用户继续决定是否附加。
6. 前端监听后端 `ide-context-updated` 事件即时刷新候选引用块，低频轮询仅作为兜底。

### 5.3 多工作目录

如果当前会话配置了多个工作目录，则按工作目录分组显示多个引用块；同一工作目录下可有多个文件候选，但首版应控制数量，避免刷屏。

建议首版限制：

1. 每个工作目录最多显示最近 3 个候选标签。
2. 全局最多显示 6 个候选标签。
3. 若当前会话只有 1 个工作目录，则省略分组标题，直接展示引用标签。
4. 若当前会话有多个工作目录，则显示工作目录分组标题。

这样先满足“多个工作目录就显示多个引用块”，同时避免面板失控。

## 6. 发送语义

发送时不新增协议字段，继续复用现有 `extraTextBlocks`。

这里区分两类状态：

1. 候选标签：来自最新 IDE 状态，尚未附加，只用于展示与选择。
2. 已附加固定块：用户点击 `+` 后冻结的快照，真正参与本轮发送。

每个已附加固定块转换为一个文本块，例如：

```xml
<ide_context source="vscode">
工作目录: avatar_py
文件: main.py
行号: 10-18
语言: python
选中内容:
def upload_avatar(...):
</ide_context>
```

若本轮用户同时选择了多个引用块，则按显示顺序依次追加到 `extraTextBlocks`。

## 7. 实现拆分

### 7.1 Rust 后端

改动目标：

1. 在 `AppState` 中新增 IDE 上下文缓存。
2. 启动时拉起本地 WebSocket server。
3. 接收并解析 VS Code 上报的上下文。
4. 上下文写入缓存后发送 `ide-context-updated` Tauri 事件，触发前端即时刷新。
5. VS Code 客户端断开时，清理该 `clientId` 对应的未固定候选缓存，避免残留引用块。
6. 提供 Tauri command 给前端读取候选引用块。

建议新增：

1. `list_ide_context_references(session)`：按会话工作目录返回候选引用块。
2. 如有必要，再补 `get_ide_bridge_status()` 用于调试，但不作为首版必须项。

### 7.2 前端状态

改动目标：

1. 在 `UnifiedWindowApp.vue` 中持有当前候选引用块与本轮已附加引用块。
2. 会话切换、工作目录变化时刷新候选引用块。
3. 发送消息时，把已附加引用块转成 `extraTextBlocks` 传给现有发送链路。

### 7.3 输入面板 UI

主要落点：

1. `src/features/chat/components/ChatComposerPanel.vue`
2. `src/features/chat/views/ChatView.vue`

改动内容：

1. 在输入框上方新增 IDE 引用块展示区。
2. 每个块显示文件、行号、摘要。
3. `+ / -` 控制附加状态。

### 7.4 发送链路

主要落点：

1. `src/features/chat/composables/use-chat-flow.ts`
2. `src/UnifiedWindowApp.vue`

现有链路本来就支持 `extraTextBlocks`，因此本次只需要把 IDE 引用块转换后拼进去，不新增发送协议字段。

## 8. 最小可用版本口径

首版只做以下闭环：

1. 后端能收 VS Code context。
2. 前端能按工作目录看到候选引用块。
3. 用户能点 `+ / -` 选择本轮是否附加。
4. 发送时确实进入 `extraTextBlocks`。

首版不追求：

1. 历史版本管理
2. 引用块持久化到会话草稿
3. 精细 diff 感知
4. 行内代码高亮预览

## 9. 验证计划

最小相关验证：

1. 前端 `pnpm typecheck`
2. Rust `cargo check`
3. 手工验证一次发送链路：
   - 模拟一条 VS Code context
   - 前端出现候选引用块
   - 点 `+`

## 10. 风险与注意项

1. WebSocket 服务端口冲突需要优雅报错，不能阻塞主程序启动。
2. 路径比较必须统一大小写与分隔符规范，避免 Windows 下误判。
3. 不要默认发送整文件内容，只发用户明确附加的引用块。
4. 工作目录为空时应静默隐藏，不制造无意义 UI。
5. 若 VS Code 未连接，不应影响正常聊天。
