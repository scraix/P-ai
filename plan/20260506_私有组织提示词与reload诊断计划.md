# 20260506 私有组织提示词与 reload 诊断计划

## 背景

当前助理私域里的“私有人格 / 私有部门”已经不再是最初的扁平结构，但工作区内提供给 LLM 参考的 `private-organization-guide` 仍然停留在旧描述，导致 LLM 生成的 JSON 容易沿用过时心智。

同时，`reload` 已经能分别扫描：
- MCP 工作区
- Skill 工作区
- 私有人格
- 私有部门

底层也已经收集了不少 `WorkspaceLoadError`，但这些错误还没有被整理成适合 LLM 和用户理解的统一诊断结果，也没有形成“是否需要我继续修复”的后续引导。

## 现状结论

### 1. 私有组织提示词确实陈旧

当前文件：
- `src-tauri/resources/preset-skills/private-organization-guide/SKILL.md`

当前指南的主要问题：
- 私有人格示例仍以 `prompt` 为核心，没有明确推荐现在前端与主配置语义更一致的 `systemPrompt`
- 私有部门只描述了 `id/name/summary/guide/agentIds/apiConfigId`
- 没有解释当前部门的关键语义，例如：
  - 主模型与候选模型的关系
  - 部门层级/下级部门
  - 权限控制
  - 哪些字段是作者应填写，哪些字段由系统自动补齐

### 2. 私有组织加载器目前仍是“老 schema + 自动补默认值”

当前加载文件：
- `src-tauri/src/features/skill/private_org.rs`

现状：
- 私有人格文件当前支持：
  - `id`
  - `name`
  - `prompt` / `systemPrompt`（别名）
  - `tools`
  - `avatarPath`
- 私有部门文件当前支持：
  - `id`
  - `name`
  - `summary`
  - `guide`
  - `apiConfigId`
  - `agentIds`

系统自动补的字段：
- 人格：`source/scope/privateMemoryEnabled/...`
- 部门：`apiConfigIds`、`childDepartmentIds`、`permissionControl`、`source`、`scope` 等默认值

这意味着：现在不是“运行时完全不支持新结构”，而是“LLM 写入格式仍然偏旧，且私有部门 authoring schema 太窄”。

### 3. reload 底层已有错误采集，但没有形成统一诊断输出

当前链路：
- `src-tauri/src/features/skill/workspace.rs`
- `src-tauri/src/features/skill/commands.rs`
- `src-tauri/src/features/chat/model_runtime/tools_and_builtin/builtin_runtime_context.rs`

现状：
- `refresh_workspace_mcp_and_skills()` 已返回：
  - `mcp_failed`
  - `skills_failed`
  - `private_agents_failed`
  - `private_departments_failed`
- `reload` 工具会直接把这一坨结果序列化给 LLM

缺点：
- 缺少统一的“诊断摘要”
- 缺少按类别的人类可读说明
- 缺少“哪个对象坏了、坏在哪、建议怎么修”
- 缺少明确的后续动作提示，例如“是否需要我现在修复这些问题”

### 4. 前端设置页尚未接入统一 reload 结果

现状：
- MCP 页刷新走 `mcp_list_servers`
- Skill 页刷新走 `mcp_list_skills`
- `mcp_refresh_mcp_and_skills` 虽已存在，但前端目前基本未消费其完整结果

结论：
- 这次需求更适合先以“reload 工具 / 后端统一结果”为主
- 设置页可先不做大联动，只在必要位置补最小展示

## 实现目标

本次目标分两块：

### A. 修正私有组织提示词与 authoring schema

让 LLM 清楚：
- 私有人格应该怎么写
- 私有部门应该怎么写
- 哪些字段允许写
- 哪些字段不要写，由系统自动生成
- 新结构下推荐的最小正确示例是什么

### B. 给 reload 增加统一诊断与修复引导

让 `reload` 在发现问题时能明确告诉 LLM / 用户：
- 是哪一类对象出错
- 具体是哪个文件 / 哪个对象
- 错误原因是什么
- 是否建议继续修复

并让 LLM 很自然地在 reload 之后追问用户：
- “我发现这些 MCP / skill / 私有人格 / 私有部门存在格式问题，是否要我现在修复？”

## 拟议实现范围

### 1. 更新私有组织指南

修改：
- `src-tauri/resources/preset-skills/private-organization-guide/SKILL.md`

方向：
- 人格示例优先使用 `systemPrompt`
- 部门示例升级为“当前推荐写法”
- 明确说明：
  - `source/scope/createdAt/updatedAt/orderIndex/isBuiltIn...` 不需要写
  - 私有部门如果暂不支持某些新字段，要明确写“当前不支持”
  - 如果本次同步扩展了解析器，就把新增支持字段一并写进指南

### 2. 评估并最小扩展私有部门解析 schema

修改目标文件：
- `src-tauri/src/features/skill/private_org.rs`

原则：
- 只扩展对当前确实有价值、且不会引入大范围行为不确定性的字段
- 不把纯运行时字段暴露给作者

优先考虑：
- `systemPrompt` 作为人格推荐字段名继续保留兼容
- 私有部门补充对 `apiConfigIds`、`childDepartmentIds`、`permissionControl` 的支持

如果某项字段扩展风险偏大，则保守处理：
- 先更新提示词明确“当前私有部门不支持该字段”
- 但不强行在这一轮把解析器做大

## 3. 新增统一 reload 诊断结构

修改目标：
- `src-tauri/src/features/skill/types.rs`
- `src-tauri/src/features/skill/workspace.rs`
- `src-tauri/src/features/chat/model_runtime/tools_and_builtin/builtin_runtime_context.rs`
- 必要时补前端类型 `src/types/app.ts`

新增一层比 `WorkspaceLoadError` 更适合 LLM 消费的结构，例如：
- 分类：`mcp` / `skill` / `private_agent` / `private_department`
- 文件路径
- 解析出的对象名或对象 id（能拿到就带上）
- 错误说明
- 建议动作

并在 reload 结果中增加：
- 诊断摘要
- 是否存在错误
- 是否建议修复
- 建议给用户的追问文案

## 4. reload 输出改成“可直接驱动下一轮对话”

目标：
- 不是只返回原始数组
- 而是让 LLM 看到后能立刻明白应该怎么回复用户

期望效果：
- 无错误时：告诉 LLM reload 成功，哪些内容已加载
- 有错误时：告诉 LLM问题分类、定位、原因，并附一段推荐追问

示例方向：
- “检测到 2 个私有部门格式错误、1 个 skill 目录错误。建议先询问用户是否要我立即修复。”

### 5. 是否改前端

本轮默认策略：
- 不大改设置页
- 只在需要时补最小类型适配

如果后续你希望：
- 配置页也显示统一诊断摘要
- 点击“刷新”直接弹修复建议

再单独做第二轮。

## 风险与取舍

### 风险 1：私有部门 schema 一次扩太大

风险：
- 一旦把主配置里的全部部门字段都开放给私有 JSON，校验与兼容成本会明显上升

取舍：
- 优先做“最小必要字段”
- 不开放纯运行时字段

### 风险 2：reload 返回结构变复杂

风险：
- 可能影响已有调用方

取舍：
- 保留现有 `mcpFailed / skillsFailed / privateAgentsFailed / privateDepartmentsFailed`
- 在其上新增诊断字段，而不是替换旧字段

### 风险 3：主动询问“是否修复”不能变成 UI 强制弹窗

本次理解：
- 你的意思更像是“让 LLM 在 reload 结果基础上主动询问用户”
- 不是前端强制弹一个修复确认框

因此本轮优先做：
- 工具结果增强
- LLM 可读的 repair hint
