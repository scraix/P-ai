# 归档压缩 JSON 结构回正计划

## 1. 背景
- 当前 SummaryContext 链路同时服务于“正式归档”和“上下文压缩”。
- 现有 JSON 结构把记忆新增与记忆合并拆成 `newMemories + mergeGroups` 两条并行支线，执行层也分两次落库，已经出现明显结构漂移。
- 提示词、解析器、执行器目前并不是围绕同一套语义设计：
  - 提示词示例鼓励拆分表达。
  - 解析器接受多种 alias 与旧形状。
  - 执行器再把新增与合并分开处理。
- 这类“提示词一套、解析一套、执行再一套”的状态，后续只会持续积累脏兼容。

## 2. 目标
- 统一 SummaryContext 的唯一合法 JSON 结构。
- 让“单条 action 自带完整语义”成为唯一执行单位，不再把新增/合并拆成两套字段。
- 同步收口提示词、解析器、执行器，确保三层完全一致。
- 明确区分：
  - `summary` 负责交接摘要或归档结论。
  - `openLoops` 负责仍需继续推进的事项。
  - `memoryActions` 负责会话记忆新增/合并。
  - `profileActions` 负责用户画像链接/新建。

## 3. 非目标
- 不改动记忆存储表结构。
- 不新增新的工具协议。
- 不改动归档投放消息卡片格式。
- 不为旧 JSON 形状保留长期兼容解析。

## 4. 唯一合法 JSON 结构

```json
{
  "title": "短标题",
  "summary": "归档结论或压缩交接摘要正文",
  "openLoops": [
    "下一步要做什么",
    "仍未解决但确实影响后续的事项"
  ],
  "usefulMemoryIds": ["12", "19"],
  "memoryActions": [
    {
      "action": "create",
      "memory": {
        "memoryType": "knowledge",
        "judgment": "用户偏好简洁直接的任务展示",
        "reasoning": "本轮多次明确要求不要让人填写 cron，不要做多层字段展示",
        "tags": ["任务系统", "前端偏好", "极简"]
      }
    },
    {
      "action": "update",
      "sourceMemoryIds": ["31"],
      "memory": {
        "memoryType": "knowledge",
        "judgment": "用户目前常驻杭州",
        "reasoning": "本轮明确说明已从深圳搬到杭州",
        "tags": ["杭州", "居住地"]
      }
    },
    {
      "action": "merge",
      "sourceMemoryIds": ["12", "19"],
      "memory": {
        "memoryType": "knowledge",
        "judgment": "用户希望任务系统对人类极简，对 LLM 保持底层原始能力",
        "reasoning": "本轮同时要求前端简化为周/天/小时编辑，同时保留 LLM 直接写 run_at 与 cron_expression",
        "tags": ["任务系统", "LLM工具", "产品取向"]
      }
    }
  ],
  "profileActions": [
    {
      "action": "link_existing",
      "memoryId": "12"
    },
    {
      "action": "create",
      "memory": {
        "memoryType": "knowledge",
        "judgment": "用户对提示词与 JSON 结构一致性要求很高",
        "reasoning": "本轮明确反对兼容垃圾与字段语义重复",
        "tags": ["提示词", "JSON设计", "代码规范"]
      }
    }
  ]
}
```

## 5. 字段语义

### 5.1 顶层字段
- `title`
  - 当前会话短标题。
- `summary`
  - 正文摘要。
  - 在压缩场景下，负责交接当前进展、已做决定、关键约束和重要参考。
  - 在归档场景下，负责给出本轮最终结论与产出。
- `openLoops`
  - 仍需继续推进的事项列表。
  - 压缩场景推荐填写，归档场景默认 `[]`，只有确实影响后续时才填写。
- `usefulMemoryIds`
  - 本轮确实被使用、应当正向反馈的记忆 ID。
- `memoryActions`
  - 会话记忆动作列表。
  - 每条 action 自带完整目标记忆内容，不再拆成外置 merge group。
- `profileActions`
  - 用户画像动作列表。

### 5.2 `memoryActions`
- `action = "create"`
  - 表示新增一条会话记忆。
- `action = "update"`
  - 表示一条旧记忆已经与当前事实冲突，需要用当前 `memory` 替换它。
  - `sourceMemoryIds` 至少 1 条。
  - 执行路径与 `merge` 相同，本质上也是“删掉旧记忆，再写入一条新记忆”。
- `action = "merge"`
  - 表示用当前 `memory` 作为最终结果，替换/合并 `sourceMemoryIds` 指向的旧记忆。
  - `sourceMemoryIds` 允许 1 条或多条：
    - 1 条表示对旧记忆做修正/刷新。
    - 多条表示合并重复或高度相近的旧记忆。

### 5.3 `profileActions`
- `action = "link_existing"`
  - 引用一条已有用户画像记忆。
- `action = "create"`
  - 新建一条用户画像记忆。
- 用户画像新建仅允许 `knowledge / skill / event`。

## 6. 执行规则
- 提示词、解析器、执行器同次提交切换到新结构。
- 不再接受 `newMemories`、`mergeGroups`、`profileMemories` 等旧动作字段。
- 执行顺序：
  1. 先处理 `usefulMemoryIds` 的归档反馈。
  2. 再顺序执行 `memoryActions`。
  3. 最后执行 `profileActions`。
- `memoryActions.action = "merge"` 时：
  - 先写入 `memory`。
  - 再删除 `sourceMemoryIds` 中被替换的旧记忆。
  - 最终以当前 action 携带的 `memory` 为准，不再依赖任何旧记忆模板。

## 7. 提示词规则
- 提示词里只展示这一种 JSON 结构。
- 提示词里的字段说明不得与旧字段并存。
- `summary` 与 `openLoops` 必须职责分离，避免同一信息在两个字段重复表述。
- 不允许再写“兼容旧格式也可以”之类的模糊规则。

## 8. 验收标准
- 提示词示例、Rust 解析结构、Rust 执行流程完全一致。
- 模型输出单条 `memoryActions.merge` 时，系统以该条 action 内的 `memory` 作为最终记忆结果。
- 模型输出 `profileActions.link_existing` / `profileActions.create` 时，执行结果正确。
- 旧结构 JSON 不再被当作成功输入 silently 吞掉。
- 压缩场景下 `openLoops` 能进入最终交接正文，不会在链路中丢失。
