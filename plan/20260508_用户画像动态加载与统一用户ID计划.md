# 用户画像动态加载与统一用户 ID 计划

## 1. 背景与问题
- 当前用户画像链路仍然偏“旧式快照”：
  - 会话创建时通过 `build_user_profile_snapshot_block(...)` 预先生成整块画像快照。
  - 会话结构里持久化 `conversation.user_profile_snapshot`。
  - 用户画像主来源仍依赖 `profile_memory_link` 链接表，而不是画像记忆本身的语义标签。
- 这导致几个问题：
  - 画像不是按“当前用户”动态加载，而是按人格可见范围取一批全局画像。
  - 记忆层、会话层、提示词层都在碰“用户画像”，职责混杂。
  - 当前体系更像“全局画像板”，不像“当前发言人的长期画像”。
- 另一个核心问题是身份口径：
  - 记忆层不应该自己理解 OneBot / 微信 / 飞书 / 钉钉的各种平台 ID 字段。
  - “当前这个人是谁”应该由注入层先统一好，再把 canonical `user_id` 提供给记忆层。
- 用户确认的新口径：
  - 本地用户固定 `user_id = "0"`。
  - 用户画像只针对“人”，不针对群、会话、联系人路由对象。
  - 记忆层不关心平台原始字段，只关心注入层已经给好的 `user_id`。

## 2. 目标
- 建立统一的 canonical `user_id` 口径，作为用户画像的唯一主键。
- 用户画像改为“按特殊 tags 标注的普通记忆”，不再依赖 `profile_memory_link` 作为主入口。
- 提示词注入阶段按当前 `user_id` 选择对应画像，并通过会话级快照注入，而不是读取全局静态画像板。
- 本地用户固定使用 `"0"`，并在提示词角色标签与画像检索链路中统一使用。

## 3. 非目标
- 不做跨平台“同一现实人物自动合并画像”。
- 不在本次改动里重做全部长期记忆检索排序。
- 不保留长期双轨体系：`profile_memory_link` 与新标签画像体系并存。
- 不修改远程 IM 会话路由模型；群、联系人、会话仍按现有聊天链路处理。

## 4. 第一部分：如何生成用户画像

### 4.1 身份归一化职责
- 注入层负责解析并提供当前轮次 canonical `user_id`。
- 记忆层只消费 canonical `user_id`，不再理解平台字段语义。
- 推荐规则：
  - 本地聊天：`user_id = "0"`
  - 远程 IM：沿用当前注入链路已经确认的“发言人 ID”
  - 群消息：画像仍归属“发言人”，群仅影响消息路由，不参与画像主键
  - `user_id` 一律按原样字符串处理，不假定它是纯数字；它可能是 `0`、QQ 号，也可能是微信这类复杂字符 ID
  - 画像检索与标签匹配时，按 `user_id` 原值精确匹配，不做数值转换、不截断、不正规化改写

### 4.2 用户画像记忆结构
- 用户画像不再是一类“外挂链接”，而是一类带特殊 tags 的普通长期记忆。
- 推荐标签结构：
  - `profile`
  - `user_id:0` / `user_id:<canonical_id>`
  - `profile_attr:alias|fact|skill|relationship|project|preference`
- 可选标签：
  - `user_name:<display_name>`
- 标签生成硬规则：
  - 只有 `user_id:*` 允许出现用户真实 ID 字符串。
  - 除 `user_id:*` 之外，其他标签里禁止出现数字、QQ 号、手机号、群号、平台账号、数据库 ID、消息 ID、订单号等任何编号型内容。
  - 标签只保留可泛化的语义词，不把原始标识符塞进标签。
- 只有同时带有：
  - `profile`
  - `user_id:*`
  - `profile_attr:*`
  的记忆，才被视为用户画像记忆。

### 4.3 写入方式
- 后续所有“用户画像新增 / 修正 / 合并”最终都写成普通长期记忆，并自动补齐画像标签。
- 画像更新靠记忆本体完成，不再额外维护一张“画像链接表”。
- 需要提供单独的“画像记忆快速拉取”方法：
  - 直接按 `profile + user_id:* + profile_attr:*` 做 SQL 级过滤
  - 不走“先全量拉全部记忆，再在 Rust 里筛”的旧路径
  - 读取目标是“某个 user_id 的画像记忆集合”，不是“全库所有画像”

### 4.4 生成时机
- 用户画像不是“每次构建提示词时现场生成”。
- 用户画像的生成/刷新应当绑定到少数明确事件：
  - 新会话创建时
  - 当前会话识别出新的 `user_id` 时
  - 画像记忆刚被新增、更新、合并后
  - 手动整理上下文 / 归档后需要重建时
- 生成结果写入“当前会话对当前用户的画像快照”，供后续提示词直接读取。

## 5. 第二部分：如何注入上下文

### 5.1 注入原则
- 注入层只做两件事：
  - 决定当前轮次 `user_id`
  - 读取该 `user_id` 对应的会话级画像快照并注入
- 注入层不负责现场推断平台字段，也不负责临时全库扫描所有画像。

### 5.2 注入形式
- 会话里保留“当前用户画像快照”，但它不再是启动时生成的一份全局静态画像板。
- 它应该变成：
  - 与当前 `user_id` 绑定
  - 可按事件刷新
  - 提示词构建时直接读取
- 也就是说：
  - `who` 是动态决定的
  - `what profile` 是按 `user_id` 切换的
  - `prompt build` 只消费当前快照，不现场重算

### 5.3 临时注入块
- 群聊等场景下，允许把“当前发言人画像”“当前轮身份说明”“本轮临时上下文”作为临时提示块注入。
- 这类内容：
  - 只参与当前轮提示词构建
  - 不写入会话历史消息
  - 不作为长期会话记录持久化
- 优先使用现有的 prompt-only 注入位，而不是伪造一条会话消息写进历史。
- 适用场景：
  - 群聊里按当前发言人切换画像
  - 临时补充当前轮身份说明
  - 注入只对本轮有效的平台侧上下文

### 5.4 注入内容
- 提示词里注入的是“当前用户的长期稳定画像”。
- 不注入其他用户的画像。
- 群消息场景下：
  - 会话仍可挂在群
  - 但本轮提示词只注入当前发言人的画像
- 本地会话场景下：
  - 普通聊天轮次不注入本地用户画像
  - 只允许在摘要相关链路中注入 `user_id = "0"` 的本地用户画像
- 远程会话场景下：
  - 普通聊天轮次可按边界规则临时注入当前发言人画像
  - 摘要链路不注入任何画像

### 5.5 旧注入路径退场
- 这一步不再依赖：
  - `profile_memory_link` 作为主读取入口
  - `build_user_profile_snapshot_block(...)` 直接从全局画像板取一批内容
- 会话上的 `conversation.user_profile_snapshot` 若保留，也只应作为“当前 user_id 的快照容器”，不再表示全局静态画像。

## 6. 分层职责

### 6.1 注入层
- 负责生成当前轮次 `user_id`
- 负责把 `user_id` 作为当前用户身份上下文提供给提示词与记忆层
- 负责区分“会话联系人 ID”和“发言人 user_id”

### 6.2 记忆层
- 负责识别哪些记忆属于用户画像
- 负责按 `user_id` 生成/刷新画像快照
- 不关心 OneBot / 微信 / 飞书 / 钉钉原始字段名

### 6.3 会话层
- 保存“当前会话当前用户”的画像快照
- 不再把 `user_profile_snapshot` 当成创建时预烘焙的一份全局画像板
- 必须能随着 `user_id` 切换而刷新

## 7. 迁移策略

### 7.1 数据迁移原则
- 不做长期兼容双读。
- 对旧数据采用一次性迁移：
  - 将现有 `profile_memory_link` 指向的画像记忆补齐新标签
  - 迁移完成后，主读取路径切到标签识别

### 7.2 旧结构退场
- 逐步停用：
  - `profile_memory_link`
  - `build_user_profile_snapshot_block(...)`
  - `conversation.user_profile_snapshot` 作为主注入来源

## 9. 分步实施

### 9.1 第一步：统一 `user_id` 注入
- 抽出统一函数，返回当前轮次 canonical `user_id`
- 本地固定返回 `"0"`
- 远程 IM 场景返回“发言人 ID”，而非群/联系人路由 ID

### 9.2 第二步：建立画像标签规范
- 定义 `profile / user_id:* / profile_attr:*` 标签口径
- 增加画像记忆识别函数与测试
- 新增画像专用快速查询函数与测试

### 9.3 第三步：建立画像生成与刷新链路
- 新增按 `user_id` 生成用户画像快照的函数
- 该函数底层必须调用画像专用快速查询，而不是全量记忆列表
- 将刷新时机绑定到少数明确事件，而不是每次构建提示词
- 提示词构建链路改为读取当前快照

### 9.4 第四步：迁移旧画像数据
- 一次性给旧画像记忆补齐新标签
- 切换主读取路径

### 9.5 第五步：删除旧快照主路径
- 移除或降级 `user_profile_snapshot`
- 清理不再需要的旧读取与持久化代码

## 10. 风险与注意事项
- 当前聊天链路里“联系人 ID”和“发言人 ID”不是同一个概念，不能混用。
- 本地 `"0"` 必须统一按字符串处理，不能在部分路径里被当作空值或 falsy。
- 任何平台 `user_id` 都必须按不透明字符串处理，不能假定是整数、不能转 number、不能按数字范围或格式做特殊分支。
- 群消息必须明确：会话可以挂在群，画像却必须挂在人。
- 如果注入层没有把 canonical `user_id` 明确传下去，记忆层就会重新退回猜平台字段，这必须避免。

## 11. 关键边界要求
- 秘书链路不允许看到任何记忆：
  - 不注入长期记忆
  - 不注入用户画像
  - 不注入 recall memory block
  - 只允许看到秘书判断当前轮是否应答所需的最小上下文
- 常规聊天链路中，单轮最多注入 3 个画像块。
- 默认只注入当前发言人画像。
- 本地会话普通聊天中，画像注入数量固定为 0。
- 本地会话摘要链路中，只允许注入 `user_id = "0"` 的画像。
- 远程会话摘要链路中，画像注入数量固定为 0。
- 是否扩展到第 2、第 3 个画像，只看“当前出队的用户消息批次”里实际涉及了几个人。
- 画像优先级建议固定为：
  - 当前发言人
  - 当前出队消息里直接出现、且需要区分身份的其他人
  - 当前本地用户 `user_id = "0"` 或当前会话绑定的直接对话对象
- 如果拿不到画像，就直接不注入，不做兜底伪造，不把别人的画像顶上来。
- 画像注入必须是临时块，不写回消息历史，不污染归档，不作为真实聊天内容保存。
- 同一轮内如果切换了当前发言人，必须重新按新的 `user_id` 取画像，不能复用上一人的快照。
- 不允许按群 ID、联系人路由 ID、会话 ID 读取用户画像，只允许按 canonical `user_id` 读取。
- 单个画像块必须保持短小，避免把提示词重新变成“全量人物设定板”。

## 12. 验收标准
- 本地会话提示词中，当前用户身份统一为 `user_id = "0"`。
- 本地普通聊天提示词中，不注入 `user_id = "0"` 的本地画像。
- 本地摘要链路中，只注入 `user_id = "0"` 的本地画像。
- 远程会话摘要链路中，不注入任何用户画像。
- 远程 IM 群消息中，用户画像按发言人而不是按群加载。
- 用户画像读取主路径不再依赖 `profile_memory_link`。
- 新增一条带 `profile + user_id:* + profile_attr:*` 标签的记忆后，下轮提示词可动态看到它。
- 不同用户的画像不会在同一会话中错误串用。
