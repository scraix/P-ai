# 统一会话人格绑定解析计划

## 背景

当前会话运行链路里，部门与人格绑定存在多套分散解析逻辑：

- 普通发送链路通过 `resolve_conversation_bound_agent` 从 `Conversation.agent_id / department_id` 推导人格。
- 远程联系人链路通过 `RemoteImContact.bound_department_id` 与 `resolve_department_agent_pair` 推导处理部门和人格。
- 前端会话标题、联系人会话列表会基于联系人摘要与当前配置拼展示信息。
- 旧联系人会话可能存在 `Conversation.department_id / agent_id` 为空，但联系人配置里已有处理部门的漂移状态。

这会造成“展示层显示远程客服 / fairy，但发送链路读取会话记录时缺少绑定”的不一致。后续需要把“从会话获得执行部门与人格”的语义统一到一个后端入口。

## 目标

1. 提供统一的会话绑定解析方法，返回当前会话实际使用的部门与人格。
2. 明确区分“只读解析”和“可写修复”，避免列表读取、预览读取等轻量路径因为个别异常联系人而整体失败。
3. 让远程联系人会话遵守一一对应关系：
   `RemoteImContact.bound_department_id -> Conversation.department_id -> Conversation.agent_id`。
4. 保持现有发送链路错误提示清晰，异常时继续指出具体 `conversation_id / department_id / agent_id`。
5. 为后续前端标题与后端真实绑定对齐留出统一数据源。

## 非目标

- 不在本计划内重做前端会话列表结构。
- 不改变部门配置页的保存语义。
- 不改变联系人消息调度、闭嘴/张嘴、入场策略。
- 不做全仓大规模模块拆分，只收口绑定解析相关函数。

## 设计

### 1. 新增统一返回类型

在后端核心 domain 或 conversation 附近新增结构：

```rust
struct ConversationAgentBinding {
    department_id: String,
    department_name: String,
    agent_id: String,
    agent_name: String,
}
```

命名建议使用 `binding` 而不是 `agent`，因为返回值包含部门与人格，且过程包含解析与校验。

### 2. 新增只读解析函数

建议入口：

```rust
fn resolve_conversation_agent_binding(
    conversation: &Conversation,
    config: &AppConfig,
    agents: &[AgentProfile],
    remote_contacts: &[RemoteImContact],
) -> Result<ConversationAgentBinding, String>
```

解析优先级：

1. 若 `conversation.department_id` 有效：
   - 若 `conversation.agent_id` 有效且属于该部门，使用它。
   - 若 `conversation.agent_id` 为空，从部门 `agent_ids[0]` 推导。
   - 若两者不匹配，返回明确错误。
2. 若 `conversation.department_id` 为空但 `conversation.agent_id` 有效：
   - 尝试通过 `department_for_agent_id` 反查部门。
3. 若是远程联系人会话：
   - 通过 `conversation.root_conversation_id` 或 `bound_conversation_id` 找到联系人。
   - 使用联系人 `bound_department_id` 推导部门与人格。
4. 仍无法解析时返回错误：
   - 普通会话：缺少有效人格绑定或部门绑定。
   - 联系人会话：联系人处理部门缺失或不可用。

### 3. 新增可写修复函数

建议入口：

```rust
fn ensure_conversation_agent_binding(
    state: &AppState,
    conversation: &mut Conversation,
    config: &AppConfig,
    agents: &[AgentProfile],
    remote_contacts: &[RemoteImContact],
) -> Result<ConversationAgentBinding, String>
```

行为：

- 先调用只读解析。
- 若解析结果与 `conversation.department_id / agent_id` 不一致，则写回会话。
- 仅由明确需要修复数据的路径调用，例如：
  - 远程联系人会话创建。
  - 远程联系人会话复用/重绑。
  - 联系人处理部门保存。
  - 发送前对指定会话做最终兜底修复。

### 4. 容错边界

不同调用场景必须区别处理错误：

- 发送消息：硬失败，返回给前端。因为继续发送会产生错误人格或错误部门执行。
- 设置页保存处理部门：硬失败。因为用户正在配置，应立即指出部门/人格不可用。
- 联系人会话列表：软失败。单个联系人无法解析绑定时，仍应显示已有会话，只记录 warn 日志。
- 历史消息读取/预览：软失败。读取历史不应依赖当前部门模型可用。

### 5. 替换现有分散逻辑

逐步替换：

1. `resolve_conversation_bound_agent` 改为调用新的只读解析函数，并只返回 `AgentProfile` 或改造调用方直接使用 `ConversationAgentBinding`。
2. `resolve_department_agent_pair` 保留为部门配置级 helper，但不再由会话发送链路直接拼装绑定语义。
3. 远程 IM 当前的 `sync_remote_im_contact_conversation_binding` 改为调用可写修复函数。
4. 远程联系人会话列表只调用“软修复包装”，禁止把单个联系人异常扩散为整个列表失败。

## 实施步骤

1. 新增 `ConversationAgentBinding` 和只读解析函数，覆盖普通会话、部门推导、远程联系人会话三类路径。
2. 增加针对解析函数的 Rust 单测：
   - 普通会话已有部门和人格。
   - 人格为空时从部门首人格推导。
   - 联系人会话从 `bound_department_id` 推导。
   - 部门/人格不匹配时报错。
   - 联系人绑定异常时错误信息包含 `conversation_id` 与 `contact_id`。
3. 新增可写修复函数，只在需要修复的路径落盘。
4. 替换远程联系人会话创建、复用、处理部门保存处的手写同步逻辑。
5. 替换发送链路中的 `resolve_conversation_bound_agent` 使用点。
6. 保留联系人列表软失败策略，并增加回归测试，确保异常联系人不会导致全部联系人会话消失。

## 风险与注意事项

- 不能让联系人会话列表因为单个联系人绑定异常整体失败。
- 可写修复必须避免在纯读取历史时制造无关持久化 churn。
- 远程联系人会话的 `root_conversation_id` 仍应保持联系人 key，用于旧绑定回收和复用。
- 日志使用中文，包含任务名、`conversation_id`、`contact_id`、`department_id`、`agent_id` 等排障字段。
- 迁移期保留旧函数外壳，减少一次性改动面。

## 待确认

2. 远程联系人会话是否允许在联系人处理部门异常时继续保留空 `department_id / agent_id`，但禁止发送时继续执行。
3. 前端标题是否在后续阶段改为优先使用后端返回的真实绑定信息。
