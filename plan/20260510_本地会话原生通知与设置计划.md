# 20260510_本地会话原生通知与设置计划

## 背景

这项需求实际经历了三个连续阶段：

1. 调查 Windows 原生系统通知是否可行
2. 把本地会话调度完成/失败通知接到后端调度收口点
3. 增加通知设置页，让用户决定是否通知、是否声音

原先分别拆成了三份计划。现在合并为这一份，作为后续唯一的权威版本。

## 结论摘要

### 1. 技术路线

- 系统通知统一使用 `tauri-plugin-notification`
- 通知发送由 Rust 后端负责，不走前端转发
- 当前项目按“跨平台接口 + Windows 优先体验”落地

### 2. 当前落地边界

- 本地普通会话调度成功后可弹系统通知
- 本地普通会话调度失败后可弹系统通知
- 远程 IM、委托线程不进入这条通知链路
- 对应聊天窗口如果当前可见且聚焦，则不弹通知
- 设置页新增“通知”页签，提供两个开关：
  - 是否通知
  - 是否声音

### 3. 默认值

- `是否通知`：默认开启
- `是否声音`：默认关闭

这样可以保留完成提醒，但避免新用户一上来就被声音打扰。

## 可行性与平台判断

### 1. Tauri 插件能力

`tauri-plugin-notification` 本身就是统一的原生通知入口，不是 Windows 专用。

但不同平台在“声音”这件事上的支持与一致性不完全相同，所以当前实现采用：

- 通知：继续使用统一插件入口
- 声音：Windows 下按系统默认音处理
- 其他平台：不额外承诺一致提示音行为

### 2. 当前项目策略

仓库本身是 Windows 优先桌面应用，因此本次实现首先保证：

- Windows 上通知体验稳定
- Windows 上可按配置决定是否带系统默认音

## 目标范围

本计划覆盖以下目标：

1. 原生系统通知能力接入项目
2. 本地普通会话完成/失败通知接入后端调度链路
3. 设置页新增通知配置入口
4. 通知配置持久化到 `app_config.toml`
5. 后端发送通知前读取配置并决定是否静音/跳过

## 实现方案

### 1. 统一通知发送入口

在后端封装统一通知 helper，负责：

- 权限检查
- 标题/正文清洗
- Windows 下按配置决定是否附带系统默认音

### 2. 调度收口挂点

通知逻辑挂在后端现有轮次收口点：

- `emit_round_completed_event(...)`
- `emit_round_failed_event(...)`

这样可以保证：

- 由后端精确知道“什么时候真正完成”
- 不依赖前端页面是否打开
- 不需要前端监听完成后再反向发通知

### 3. 会话过滤规则

只对本地普通会话发送通知，排除：

- 委托会话
- 远程 IM 联系人会话
- 其他非普通本地聊天会话

### 4. 前台抑制规则

如果对应聊天窗口当前：

- 可见
- 且聚焦

则跳过通知，避免用户正在看着对话时还重复被提醒。

### 5. 通知文案规则

成功通知：

- 标题：人格名
- 正文：本轮回复文本摘录；为空则回退固定提示

失败通知：

- 标题：`人格名 调度失败`
- 正文：错误文本摘录；为空则回退固定提示

目标是让弹窗更像“人格在说话”，而不是系统型提示。

### 6. 设置页与配置项

前后端 `AppConfig` 增加两个字段：

- `messageNotificationEnabled`
- `messageNotificationSoundEnabled`

设置页新增一个独立的“通知”页签，只保留两个最小开关：

- 是否通知
- 是否声音

交互规则：

- 关闭“是否通知”时，声音开关置灰
- 声音开关的值保留，不强制清空

### 7. 默认行为

新配置与旧配置缺字段时，统一按以下默认值补齐：

- `messageNotificationEnabled = true`
- `messageNotificationSoundEnabled = false`

## 当前不做

本计划暂不覆盖以下能力：

1. 通知点击后跳转到指定会话
2. 远程 IM 会话完成通知
3. 委托线程完成通知
4. 更细粒度的按场景、按会话、按人格通知开关
5. 跨平台一致的声音体验适配
6. Demo 通知与用户设置强绑定

## 影响范围

- Rust 通知 helper
- Rust 调度完成/失败收口逻辑
- Rust `AppConfig` 默认值与反序列化
- 前端 `AppConfig` 类型与默认值
- 前端配置加载/保存映射

- 多语言文案

## 风险与控制

### 风险 1：旧配置缺字段

处理：

前后端都补默认值，Rust 侧继续作为最终兜底。

### 风险 2：只改了前端开关，后端仍继续发通知

处理：

以后端配置判定为准，前端只负责编辑配置。

### 风险 3：通知过于打扰

处理：

- 默认开启通知但关闭声音
- 当前聊天窗聚焦时跳过通知

### 风险 4：Demo 测试按钮被误当成正式用户行为

处理：

Demo 继续作为开发测试入口，不纳入正式通知设置的强约束范围。

## 验证方案

最小相关验证：

1. `pnpm typecheck`
2. `cargo check -p easy-call-ai`
3. 手工验证：
   - 开启通知 + 关闭声音：后台本地会话完成/失败时弹通知但无声音
   - 开启通知 + 开启声音：后台本地会话完成/失败时弹通知且有系统默认音
   - 关闭通知：后台本地会话完成/失败时不弹通知
   - 当前聊天窗聚焦：不弹通知
   - 重启后设置值保持不变
