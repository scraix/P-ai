# 文本工具自动编码读写计划

## 背景

PAI 当前文本工具链主要按 UTF-8 读取与写回。遇到 GBK、Shift_JIS、Big5 等旧编码文本时，`read`、文件阅读窗口、`apply_patch update` 会出现读取失败或无法编辑的问题。

参考 `open-claude-code` 的做法，它在编辑/写回路径中会保留读到的编码，但检测范围主要是 UTF-8 与 UTF-16LE。PAI 需要更完整地支持 Windows 常见旧编码，同时保持工具协议简单。

## 用户结论

- 不允许 LLM 指定编码。
- 工具 schema 不增加 `encoding` 参数。
- 用户不需要选择编码。
- 新文件默认 UTF-8。
- 现有文件由 IO 层自动检测编码，读出为 UTF-8 字符串，写回时使用同一编码。
- 写回不允许静默替换不可表示字符，失败时给出明确错误。

## 实现范围

1. 新增系统工具内部文本编码模块。
   - BOM 优先识别 UTF-8 / UTF-16LE / UTF-16BE。
   - 严格 UTF-8 优先。
   - 失败后使用 `chardetng` 猜测，并用 `encoding_rs` 严格解码。
   - 保留 BOM 信息，写回时恢复 BOM。
   - 使用无替换编码 API，遇到目标编码不可表示字符时失败。

2. 改造 `read` 文本读取。
   - 从 `read_to_string` 改为读 bytes 后内部解码。
   - 返回内容仍是普通字符串，不暴露编码参数给 LLM。

3. 改造文件阅读窗口读取。
   - 同样走内部 codec，避免非 UTF-8 文件预览失败。

4. 改造 `apply_patch`。
   - `add` 新文件继续写 UTF-8。
   - `update` 读取现有文件时检测编码，应用文本替换后按原编码写回。
   - 撤回/安全校验读取当前文件时同样支持非 UTF-8。
   - 备份仍保存原始 bytes，恢复保持原始字节。

## 验证

- 增加最小相关 Rust 测试：
  - GBK 文本读取能得到正确中文。
  - GBK 文件 `apply_patch update` 后仍是 GBK 字节，不被转成 UTF-8。
  - 新增文件仍按 UTF-8 写入。
  - 目标编码不可表示字符时写回失败，不做替换。
- 运行本次影响范围内的 Rust 测试。

## 不做

- 不改应用内部 JSON/TOML 配置和数据文件读写。
- 不在工具参数、提示词或 UI 中加入编码选择。
- 不把编码选择权交给 LLM。
