# 远程 IM 渠道私有状态分片计划

## 背景

当前远程 IM 渠道统一通过 `RemoteImChannelConfig.credentials` 保存连接信息和部分运行态。这个字段最终进入 `app_config.toml`，导致用户配置和轮询游标混在同一个总配置文件里。

在单渠道或低频操作下问题不明显；当多个微信渠道同时轮询时，`sync_buf` 这类高频变化的运行游标会反复触发总配置写入。即使把密钥拆到一个统一 secrets 文件，如果多个渠道仍写同一个文件，也只是把共享写入点从 `app_config.toml` 换到另一个文件，不能根治多渠道互相影响。

## 目标

- 总配置只描述“有哪些渠道”和“用户可编辑的静态设置”。
- 配置层只允许由用户设置、导入迁移等配置动作写入；后台运行期间不得把轮询结果、登录态刷新、临时 token 等运行结果写回配置。
- 运行期登录态、轮询游标不再保存在 `app_config.toml`。
- 用户写死的静态接入参数即使是密钥，也属于用户配置；只由用户保存配置时写入，不由后台运行任务改写。
- 每个远程 IM 渠道拥有独立的后端私有状态文件，多个渠道互不抢同一个持久化文件。
- 前端不直接编辑密钥文件，只通过后端命令保存静态配置、登录、退出登录、查询状态。
- 旧版本配置可平滑迁移，避免用户重新配置。
- 保存失败、迁移失败、文件锁定等情况必须返回准确错误，不允许静默吞掉。

## 推荐架构

### 1. 配置层：`app_config.toml`

只保存用户可见、低频变更、可导入导出的静态配置。

配置层的写入边界：

- 用户在设置页点击保存。
- 用户导入配置或迁移旧配置。
- 版本升级迁移需要清理旧字段。

配置层的禁止事项：

- 轮询成功不得写配置。
- 登录态刷新不得写配置。
- 后台获取或刷新的 token、session、sync cursor 不得写配置。
- 用户手工填写且运行期不会刷新/改写的接入参数仍属于配置，例如钉钉 `clientSecret`、飞书 `appSecret`。
- 后台运行任务不得通过“顺手更新 channel.credentials”的方式持久化运行结果。

远程 IM 渠道建议保留：

```text
id
name
platform
enabled
activateAssistant
receiveFiles
streamingSend
showToolCalls
allowSendFiles
publicConfig
```

其中 `publicConfig` 用来替代现在过宽泛的 `credentials`，只放用户写死的静态接入参数。例如：

- OneBot：`wsPort`、监听地址、连接模式、用户固定填写的 `accessToken`。
- 钉钉：`clientId`、`clientSecret`、`robotCode`。
- 飞书：`appId`、`appSecret`、事件接入方式、默认接收 ID 类型。
- 个人微信：`baseUrl`、`cdnBaseUrl`、`botType`、`longPollTimeoutMs`、`apiTimeoutMs`。

### 2. 渠道私有状态层：每渠道一个文件

新增后端私有目录，按平台和渠道拆分：

```text
data/
  remote-im/
    channels/
      weixin_oc/
        {channel_id}.json
      dingtalk/
        {channel_id}.json
      feishu/
        {channel_id}.json
      onebot_v11/
        {channel_id}.json
```

文件名唯一方案：

- 文件名不使用原始 `channel_id`，统一使用固定 hash 或 UUID 映射。
- 推荐第一版直接使用 `sha256(platform + "\n" + channel_id)` 的 hex 作为文件名。
- `channel_id` 只作为文件内容字段，不参与路径拼接。
- 这样路径穿越问题在设计层面消失，不依赖过滤补丁兜底。

该文件保存后端运行期产生、刷新或高频变化的字段。

个人微信：

```json
{
  "schemaVersion": 1,
  "channelId": "...",
  "platform": "weixin_oc",
  "token": "...",
  "accountId": "...",
  "userId": "...",
  "syncBuf": "...",
  "updatedAt": "..."
}
```

钉钉、飞书、OneBot 当前未发现需要本地持久化的运行期登录态；它们的用户固定接入参数继续保存在配置层。

### 3. 运行内存层

只保存进程内临时对象，不承诺重启恢复。

例如：

- 微信扫码登录中的 `sessionKey`、二维码内容。
- 微信 `contextToken` 缓存。
- WebSocket 连接句柄。
- Stream 任务句柄和 stop sender。
- UI 状态查询用的连接状态、最近错误、内存日志环形缓冲。

这些内容不写入硬盘，除非有明确的恢复需求。

## 后端接口设计

新增统一的渠道私有状态服务，建议放在：

```text
src-tauri/src/features/remote_im/channel_store.rs
```

核心能力：

```text
remote_im_channel_state_path(platform, channel_id) -> PathBuf
read_channel_private_state<T>(state, platform, channel_id) -> Result<T, String>
write_channel_private_state<T>(state, platform, channel_id, value) -> Result<(), String>
patch_channel_private_state<T>(state, platform, channel_id, patch_fn) -> Result<T, String>
delete_channel_private_state(state, platform, channel_id) -> Result<(), String>
```

写入要求：

- 每个渠道文件独立写入，互不共用全局写锁。
- 读取不加全局锁，也不因其他渠道写入被阻塞。
- 写入使用“写临时文件 + flush + rename”实现原子替换。
- `rename` 是同一文件系统内原子操作，读路径只读正式文件，不读取临时文件。
- 按渠道的 mutex 只用于同一渠道并发写入排队，不影响其他渠道。
- 明确区分“写入中排他”和“读取中排他”：第一版不需要读写锁，读操作直接读当前正式文件。
- 错误信息包含平台、渠道 ID、路径、操作名、原始错误。
- 不输出密钥明文日志。
- `channel_store.rs` 内置日志脱敏工具，禁止调用方传入原始 secret 后直接打印。

## 前端接口设计

远程 IM 设置页不再暴露通用 `credentials JSON` 作为长期入口。建议改成：

- 静态配置表单：保存到 `app_config.toml`。
- 登录态表单：仅对存在登录态的平台调用专门命令，例如个人微信扫码登录。
- 状态展示：只显示运行态状态，例如“已登录账号”“需要登录”。

建议新增或调整命令：

```text
weixin_oc_start_login(channel_id)
weixin_oc_poll_login(channel_id, session_key)
weixin_oc_logout(channel_id)
```

前端保存渠道配置时，可以继续保存用户固定填写的静态接入参数；不得保存个人微信运行期登录态和轮询游标。

## 各平台迁移策略

### 个人微信

迁移字段：

```text
token
accountId
userId
syncBuf
```

保留在配置层的字段：

```text
baseUrl
cdnBaseUrl
botType
qrPollInterval
longPollTimeoutMs
apiTimeoutMs
```

轮询路径改造：

- 轮询前读取该渠道私有状态文件，拿 `token/syncBuf`。
- 轮询成功后只更新该渠道文件的 `syncBuf`。
- 登录成功写该渠道文件的 `token/accountId/userId`。
- 退出登录删除或清空该渠道文件。

### 钉钉

不迁移。当前 `clientId/clientSecret/robotCode` 是用户写死的接入参数，运行期间不会被后台刷新写回，继续保存在配置层。

### 飞书

不迁移。当前 `appId/appSecret` 是用户写死的接入参数，运行期间不会被后台刷新写回，继续保存在配置层。飞书请求时换取的 token 不落盘。

### OneBot V11

不迁移。当前未发现本地持久化登录态，`wsPort/accessToken` 如由用户写死，继续保存在配置层。

## 兼容与迁移

首次加载配置或保存配置时执行轻量迁移。

迁移必须避免“私有状态已落地，但旧配置敏感字段清理失败后下次重复迁移”的隐患。

推荐迁移顺序：

1. 扫描 `remote_im_channels`。
2. 对每个渠道识别旧 `credentials` 中的运行态字段。
3. 先读取对应渠道私有状态文件，合并旧配置字段和已存在私有字段。
4. 只有双方确认可写后，才写入对应渠道私有状态临时文件并原子替换。
5. 所有渠道私有状态写入成功后，再写清理敏感字段后的 `app_config.toml`。
6. 如果清理配置失败，下一次迁移必须幂等：已有私有状态不得被空值覆盖，旧配置字段可再次合并。
7. 迁移失败返回明确错误；主程序可继续启动，但对应渠道状态必须展示迁移失败原因，不允许静默吞掉。

合并规则：

- 非空旧配置字段可以补齐私有状态缺失字段。
- 私有状态已有非空字段时，不被旧配置空值覆盖。
- `syncBuf` 以私有状态优先，避免旧配置回滚轮询游标。
- `updatedAt` 用于辅助排障，不作为覆盖唯一依据。

## 实施步骤

### 阶段一：存储层与迁移

- 新增 `channel_store.rs`。
- 定义各平台私有状态结构。
- 实现安全路径、读写、删除、patch。
- 实现旧个人微信 `credentials` 运行态字段到渠道私有文件的迁移。
- 保存配置时清理个人微信运行态字段。

### 阶段二：个人微信接入

- `WeixinOcCredentials` 拆成静态配置和私有状态。
- 登录成功写私有状态文件。
- 轮询读写该渠道私有状态文件。
- 发送消息、媒体上传、typing 读取私有 token。
- 退出登录清理私有状态文件。

### 阶段三：其他平台接入

- 暂不改造钉钉、飞书、OneBot 的静态配置读取路径。
- 如果后续新增“后台刷新并持久化”的平台登录态，再接入渠道私有状态文件。

### 阶段四：前端设置页收口

- 阶段一完成后，个人微信迁移会解锁后端存储验证；前端不必等所有平台后端接入完成才统一改造。
- 前端改造拆成两步：
- 第一步：保存配置时自动过滤个人微信运行态字段，避免继续把登录态/轮询游标写回配置层。
- 第二步：每个平台提供独立配置表单，替换通用 `credentials JSON`。
- 密钥状态脱敏展示。
- 保存配置失败、保存密钥失败分别准确提示。

### 阶段五：验证与归档

- 增加存储层单测：路径安全、单渠道读写、迁移、删除。
- 增加微信迁移/轮询游标写入单测。
- 增加保存配置清理敏感字段测试。
- 最小运行 `cargo check -p easy-call-ai`。
- 用户确认功能后更新计划归档报告并移动到 `plan/done/`。

## 风险与裁决点

- 是否保留 `credentials` 字段名作为兼容壳，还是引入 `publicConfig`：推荐后者，但改动面更大。
- 私有状态文件是否需要加密：当前计划先做分片和后端隔离，不做加密；如后续要提高安全等级，可再接入 Windows Credential Manager 或本机密钥加密。
- 旧配置迁移失败时是否阻止启动：推荐不阻止主程序启动，但远程 IM 对应渠道应禁用并给出明确错误。
- 当前仅迁移个人微信运行态；其他平台的用户写死配置继续保留。

## 推荐结论

推荐采用“总配置 + 每渠道私有状态文件 + 进程内临时态”的三层架构。

不要做统一 secrets 文件，因为它仍然会让多个渠道竞争同一个持久化文件。每个渠道一个文件，才能让多个微信轮询时只更新自己的游标，真正降低互相影响。
