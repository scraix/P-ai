# 20260526_meme系统简化与内存索引计划

## 一、目标

把现有 meme 贴纸系统改成更直接的文件系统规则：

1. 不再维护“可用 meme 列表”的持久索引文件。
2. 每次需要展示可用 meme、解析 `:meme名:`、或执行 `meme` 工具时，直接扫描 `.meme/` 生成内存索引。
3. 可用 meme 直接列给 LLM，看见什么名字就只能用什么名字。
4. meme 名由 `.meme/` 根目录下的文件名，以及 `.meme/` 第一层文件夹名共同组成。
5. `meme` 工具只负责把图片收入 `.meme/`，参数简化为“表情名 + 图片路径”。
6. 保留图哈希库，用于检测重复或近似重复图片，避免同一张图被反复录入。

## 二、不做什么

1. 不新增 `meme.list` 工具。
2. 不新增数据库、可用列表 JSON 索引、SQLite、向量索引。
3. 不再把 LLM 引导到“分类 + 贴纸名”的双层概念。
4. 不让工具为了查询库存而被调用；库存列表直接注入提示词和工具说明。
5. 不把图哈希库当成 meme 列表索引使用；它只回答“这张图是否已经收过或近似收过”。

## 三、新目录规则

贴纸根目录仍然是 Self Directory 下的 `.meme/`。

### 1. 单个贴纸

如果某个 meme 只有一个素材，直接放在根目录：

```text
.meme/
  坏笑.webp
  无语.webp
```

此时可用 meme 是：

```text
坏笑、无语
```

LLM 输出：

```text
:坏笑:
```

系统解析时使用：

```text
.meme/坏笑.webp
```

### 2. 多个同名贴纸

如果同一个 meme 名有多个素材，使用第一层文件夹承载多个变体：

```text
.meme/
  坏笑/
    坏笑.webp
    坏笑(2).webp
    坏笑(3).webp
```

此时可用 meme 仍然只有一个：

```text
坏笑
```

LLM 输出：

```text
:坏笑:
```

系统解析时从 `.meme/坏笑/` 中挑一个变体。

### 3. 根目录文件和同名文件夹同时存在

实现后应避免这个状态。若扫描时发现：

```text
.meme/
  坏笑.webp
  坏笑/
    坏笑(2).webp
```

以内存索引的规则处理为：

1. `坏笑/` 文件夹优先。
2. 根目录 `坏笑.webp` 视为需要整理的遗留单图。
3. 下一次入库同名 `坏笑` 时，把根目录文件并入 `坏笑/`。

## 四、入库规则

工具入库时统一归一化为 WebP，文件扩展名固定 `.webp`。

### 例子 A：第一次添加“坏笑”

LLM 调用：

```json
{
  "emotion": "坏笑",
  "path": "E:/tmp/source.png"
}
```

如果 `.meme/坏笑.webp` 和 `.meme/坏笑/` 都不存在，则生成：

```text
.meme/
  坏笑.webp
```

### 例子 B：第二次添加“坏笑”

LLM 再次调用：

```json
{
  "emotion": "坏笑",
  "path": "E:/tmp/another.png"
}
```

如果当前是：

```text
.meme/
  坏笑.webp
```

则改造成：

```text
.meme/
  坏笑/
    坏笑.webp
    坏笑(2).webp
```

流程是：

1. 创建 `.meme/坏笑/`。
2. 把原来的 `.meme/坏笑.webp` 移动到 `.meme/坏笑/坏笑.webp`。
3. 把新图片写入 `.meme/坏笑/坏笑(2).webp`。

### 例子 C：第三次添加“坏笑”

如果当前是：

```text
.meme/
  坏笑/
    坏笑.webp
    坏笑(2).webp
```

第三次入库生成：

```text
.meme/
  坏笑/
    坏笑.webp
    坏笑(2).webp
    坏笑(3).webp
```

编号规则使用英文半角括号，从 `(2)` 开始递增，选择当前目录中不存在的最小编号。

## 五、使用规则

LLM 不需要知道具体图片路径，只需要使用 meme 名：

```text
收到，我先去看一下 :坏笑:
```

后端解析消息时：

1. 扫描 `.meme/` 生成内存索引。
2. 找到 token `:坏笑:`。
3. 如果存在 `.meme/坏笑/`，从该文件夹的图片里选择一个。
4. 如果不存在文件夹，但存在 `.meme/坏笑.webp`，使用根目录文件。
5. 如果两者都不存在，按普通文本保留 `:坏笑:`，不渲染为贴纸。

变体选择沿用当前“按消息 id、meme 名、token 序号稳定选择”的思路，保证同一条消息重复渲染时不会随机跳变。

## 六、内存索引规则

每次扫描 `.meme/` 时生成类似结构，只存在内存中：

```rust
{
  "坏笑": [
    ".meme/坏笑/坏笑.webp",
    ".meme/坏笑/坏笑(2).webp"
  ],
  "无语": [
    ".meme/无语.webp"
  ]
}
```

扫描来源：

1. 根目录图片文件：文件 stem 作为 meme 名。
2. 第一层文件夹：文件夹名作为 meme 名，文件夹内图片作为变体。

忽略项：

1. `.meme/image_dhash_index.json` 等图哈希库文件不作为 meme 名来源。
2. 非图片文件。
3. 第二层及更深层目录。
4. 非法 meme 名。

排序规则：

1. meme 名按字典序输出，保证提示词稳定。
2. 同一 meme 下的变体按文件名自然顺序或字典序稳定排序。

## 七、工具简化

工具名仍然是：

```text
meme
```

新参数：

```json
{
  "emotion": "坏笑",
  "path": "E:/tmp/source.png"
}
```

字段含义：

1. `emotion`：meme 名，也是后续聊天里使用的 token 名，即 `:坏笑:`。
2. `path`：要收入贴纸库的本地图片路径。

工具 schema 只接受 `emotion` 和 `path`，不再暴露旧的“分类 + 贴纸名”双层参数。

工具说明改成类似：

```text
把当前看到的图片收入 Self Directory 的 `.meme` 贴纸库。当前可用 meme：坏笑、无语、震惊。聊天时如需发送贴纸，直接输出 `:meme名:`，例如 `:坏笑:`。不要调用本工具查询列表；只有需要新增贴纸库存时才调用。参数：emotion 为贴纸名，path 为图片路径。
```

## 八、提示词注入

当前可用 meme 需要直接列出来，例如：

```text
当前可用 meme：坏笑、无语、震惊。
如果你要发送贴纸，请直接在回答正文中写 `:meme名:`，例如 `:坏笑:`。
不要编造不存在的 meme 名，也不要调用工具查询列表。
只有当你需要把当前看到的图片收进贴纸库时，才调用 `meme` 工具。
```

如果没有可用 meme：

```text
当前没有可用 meme。
如果上下文里有适合做贴纸的图片路径，可以调用 `meme` 工具收入库存。
```

这里不再出现“分类名”或 happy/sad/angry 这类固定分类建议，避免模型继续按旧结构生成。

## 九、图哈希去重

图哈希库需要保留。它和“可用 meme 列表”是两套概念：

1. 可用 meme 列表：每次扫描 `.meme/` 根目录文件和第一层文件夹，生成内存索引，不落盘。
2. 图哈希库：记录已收入图片的感知哈希，用于新图片入库前检测重复或近似重复，可以继续落盘。

入库流程中，归一化图片后计算 dHash：

1. 如果命中已有相同或近似图片，工具返回“已存在”，不再新增文件。
2. 如果未命中，按新目录规则写入 `.meme/`。
3. 写入成功后更新图哈希库，让后续入库能识别这张图。

图哈希库里的相对路径需要跟随新目录规则：

```text
.meme/坏笑.webp
.meme/坏笑/坏笑.webp
.meme/坏笑/坏笑(2).webp
```

当第二次同名入库触发“根目录单图移动到文件夹”时，也要同步更新图哈希库中原图的相对路径：

```text
旧：.meme/坏笑.webp
新：.meme/坏笑/坏笑.webp
```

如果图哈希库损坏或缺失，可以通过扫描现有图片重建，但重建结果仍只用于去重，不用于告诉 LLM 有哪些 meme。

图哈希库中的孤儿路径不能因为文件不存在就自动删除。用户手动删除某个表情文件，通常表示这个表情不适合继续使用；保留该图片的 hash，才能在下次同图入库时继续拦截，避免被 LLM 又收回来。

## 十、图哈希库维护

`image_dhash_index.json` 继续作为重复检测数据源。实现时需要确认：

1. 它不参与可用 meme 列表。
2. 它可以在入库前用于重复检测。
3. 它可以在新文件写入或旧文件移动后更新路径。
4. 它损坏时可以按空库处理并通过扫描现有图片重建。
5. 它不能在同步时删除孤儿路径；孤儿 hash 是“已见过但当前不可用”的记录。

### 已持久化消息

历史消息里的 `provider_meta.memeSegments` 已经固化了具体图片和 base64 数据，应继续按旧数据渲染，不重新解析成新 meme。

## 十一、实现步骤

1. 重写 `.meme/` 扫描函数，产出以 meme 名为 key 的内存资产表。
2. 调整 `meme_prompt_rule_block` 和工具 description，直接列出可用 meme 名。
3. 简化 `MemeToolArgs`，只保留 `emotion + path`。
4. 修改入库路径：
   - 首次同名入库写 `.meme/<emotion>.webp`
   - 第二次同名入库改造成 `.meme/<emotion>/<emotion>.webp` 和 `.meme/<emotion>/<emotion>(2).webp`
   - 后续同名入库写入下一个编号
5. 保留 dHash 去重链路，但把职责限定为“入库重复检测”：
   - 入库前计算新图 dHash。
   - 命中重复时不新增文件。
   - 写入新文件后更新图哈希库。
   - 移动根目录旧文件到文件夹时同步更新相对路径。
   - 同步图哈希库时不删除孤儿路径。
6. 确保可用 meme 列表完全来自目录扫描，不读取 dHash 库。
7. 保持消息解析和 `provider_meta.memeSegments` 存储结构基本不变；历史字段名暂不做存储迁移，但写入值统一使用 meme 名。
8. 补充最小必要测试：
   - 根目录单图能被列出和解析。
   - 文件夹多变体优先于根目录同名文件。
   - 第一次、第二次、第三次同名入库路径正确。
   - 重复图片再次入库时被 dHash 拦截，不新增变体。
   - 根目录单图移动到文件夹后，dHash 库相对路径同步更新。
   - 删除图片文件后，孤儿 dHash 仍保留，并能拦截同图再次入库。
   - 工具参数只接受 `emotion + path`。
   - 旧 `provider_meta.memeSegments` 仍可回读。

## 十二、验收标准

1. `.meme/坏笑.webp` 存在时，提示词列出 `坏笑`，LLM 输出 `:坏笑:` 能渲染贴纸。
2. 再次收入 `坏笑` 后，目录变为 `.meme/坏笑/坏笑.webp` 与 `.meme/坏笑/坏笑(2).webp`。
3. `.meme/坏笑/` 存在时，使用 `:坏笑:` 必须从该文件夹内选择变体。
4. 相同图片或近似重复图片再次收入时，不生成新的 `.webp` 变体。
5. `.meme/image_dhash_index.json` 可以继续生成和更新，但只用于重复检测，不用于列出可用 meme。
6. 删除某个表情文件后，图哈希库保留对应 hash，同图再次入库仍会被拒绝。
7. 工具 schema 只接受 `emotion` 和 `path`。
8. 提示词中不再使用“分类名”描述 meme 用法。
9. 历史消息中的 meme 图片仍能显示。
