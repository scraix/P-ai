# 20260529_压缩打断续跑统一计划

## 背景

当前上下文压缩已经被设计成调度边界事件：命中压缩后当前调度闭口，写入压缩消息，再重开调度继续执行。

但现有实现漏掉了一个关键语义：压缩触发时，当前调度里已经完成的工作必须像用户手动打断一样先落盘。

现在手动打断和压缩打断实际走了两套逻辑：

- 手动打断：保存半成品 assistant 文本、reasoning、已完成工具历史，然后结束当前轮。
- 压缩打断：压缩接管后设置 `suppress_assistant_message=true`，跳过 assistant 消息持久化，然后压缩并续跑。

结果是：压缩发生在工具循环中间时，压缩前已经完成的工具调用、阶段性 reasoning、半成品 assistant 内容不会进入正式聊天记录。后续 prompt 又会从最后一条 `context_compaction` 开始截断历史，导致压缩前已经做过的工作从模型视角和用户聊天记录中消失。

这是实现 bug，不是预期行为。

## 目标

本次修复目标是统一“打断续跑”语义：

1. 压缩打断必须等价于一次系统触发的 stop。
2. 在任何压缩接管当前调度前，必须先落盘当前调度已经形成的 assistant/tool 历史。
3. 落盘完成后，再发起上下文压缩。
4. 压缩完成后，重新发起下一轮调度继续执行。
5. 压缩前已经完成的工作必须同时存在于：
   - 正式聊天记录；
   - 压缩摘要输入；
   - 压缩后的后续 prompt 可恢复语义中。

一句话规则：

**压缩打断 = 当前轮检查点落盘 -> 写入压缩消息 -> 重开调度续跑。**

后续实现已根据工具落盘方案调整主路径：

**工具完成 = 立即追加 tool_call/tool_result 到最后一条本助理消息；调度完成 = 更新最后一条本助理消息；压缩 checkpoint = 只作为压缩接管前兜底。**

这样可以避免把所有安全性都压在“压缩前一次性 checkpoint”上：只要工具结果已经成功返回，它就会尽快进入正式历史；压缩、停止、续跑都基于已经逐步落盘的会话事实恢复。

## 正确语义

### 1. 手动打断

手动打断保持现有语义：

1. 用户点击停止。
2. 中止正在运行的 chat/tool future。
3. 读取前端 partial 文本和后端 `inflight_completed_tool_history`。
4. 构建并持久化一条 assistant 消息。
5. 当前调度结束。

### 2. 引导打断

引导消息的语义是当前工具执行完成后切断当前调度：

1. 工具执行完成。
2. 已完成工具历史已经形成。
3. 当前调度闭合，并按正常 assistant 消息持久化。
4. 插入引导消息。
5. 重开调度。

### 3. 压缩打断

压缩打断应与引导打断一致，只是切断后插入的是压缩消息：

1. 工具执行完成，或发送前检测到需要压缩。
2. 当前调度先闭合。
3. 已经形成的 assistant/tool 历史必须持久化。
4. 使用持久化后的会话真相发起压缩。
5. 写入 `context_compaction` 消息。
6. 压缩完成后重开调度续跑。

不允许压缩路径跳过第 3 步。

## 关键不变量

### 1. 历史事实不丢失

已经完成的工作一旦对用户可见或已进入工具历史，就必须成为正式历史事实。

包括：

- 已输出的 assistant 文本；
- 已输出的 reasoning；
- 已完成并拿到结果的工具调用；
- 工具调用失败但已经形成稳定结果的失败事件。

不包括：

- 正在执行、尚未返回的工具；
- 尚未形成完整事件对的 provider tool call；
- 尚未到达后端的前端临时展示态。

### 2. 压缩输入必须来自落盘后的会话

压缩摘要不能基于压缩前旧 `source` 快照直接执行。

压缩前如果当前调度已经产生了新的 assistant/tool 历史，必须先把这些历史写入会话，再重新读取会话作为压缩输入。

否则会出现：

- 触发压缩的 token 统计包含了工具历史；
- 实际压缩摘要却不包含工具历史；
- 压缩后 prompt 从压缩消息开始，导致该工具历史彻底断档。

### 3. 压缩消息不是 assistant 工作记录的替代品

`context_compaction` 是历史压缩检查点，不是“本轮 assistant 已做工作”的替代消息。

因此不能用压缩摘要吞掉本轮未落盘 assistant/tool 历史。

### 4. 重开调度只能基于正式历史

压缩完成后的续跑不应继续携带旧调度里的临时 prompt、临时 tool loop 状态或未落盘缓存。

重开调度只能从以下来源恢复：

- 会话正式消息；
- 压缩消息；
- 可序列化的最小恢复意图。

## 实施方案

### 阶段 1：工具结果完成即落盘

主路径不再等到 stop/压缩时一次性保存工具历史，而是在每个工具成功拿到稳定结果后立刻异步持久化一对事件：

- assistant `tool_call`；
- tool `tool_result`。

追加规则保持简单：

1. 从会话尾部向上看最后一条消息。
2. 如果最后一条是当前 assistant，就把工具事件追加到该消息。
3. 如果最后一条不是当前 assistant，就新建一条 assistant 消息。
4. 以 `tool_call_id` 做去重，避免异步落盘和最终提交重复写入同一工具对。

要求：

- 落盘不阻塞工具循环主路径。
- 工具完成时同步更新 usage 相关 provider_meta，避免压缩统计继续使用过期 token 信息。
- `organize_context` 自身不作为业务工具历史保留。

### 阶段 2：最终 assistant 提交更新尾部消息

调度正常完成时，不再无条件插入新的 assistant 消息。

目标流程：

1. 如果最后一条消息是当前 assistant，保留原 `id/created_at`，用最终 assistant 内容更新它。
2. 如果最后一条不是当前 assistant，再插入新 assistant 消息。
3. 最终消息中的 tool history 与已即时落盘的 tool history 通过 `tool_call_id` 去重。

### 阶段 3：修改工具续调前压缩路径

当前 `maybe_apply_auto_compaction_before_tool_continue_genai` 会在工具历史已形成后直接运行 `run_context_compaction_pipeline`，然后返回 `CHAT_DISPATCH_RESTART_AFTER_COMPACTION`。

目标流程调整为：

1. 判断需要压缩。
2. 先确保当前调度中尚未即时落盘的 partial 文本、reasoning、工具历史通过 checkpoint 兜底写入。
3. 清理或同步 `inflight_completed_tool_history`，避免重复落盘。
4. 重新读取持久化后的会话。
5. 对新会话运行 `run_context_compaction_pipeline`。
6. 返回 `CHAT_DISPATCH_RESTART_AFTER_COMPACTION`。

禁止行为：

- 禁止在未落盘 `tool_history_events` 的情况下压缩。
- 禁止 `DeferredToolLoopOutcome::OrganizeContext` 继续用 `suppress_assistant_message=true` 吃掉已有工具历史。
- 禁止用压缩前 `source` 快照作为摘要输入。

### 阶段 4：修改 `organize_context` 工具路径

当前 `organize_context` 工具会 spawn 后台压缩并立即返回 `ok/applied/scheduled`，随后外层把这次结果视作需要 suppress assistant message 的闭口信号。

目标语义：

1. `organize_context` 工具本身仍可作为“请求压缩接管”的工具。
2. 工具返回后，外层不能静默丢弃本轮 assistant/tool 历史。
3. 外层应把 `organize_context` 之前已经完成的工具历史落盘。
4. `organize_context` 这次工具调用本身可从持久化历史中剔除，避免让模型误以为它是普通业务工具结果。
5. 压缩必须基于落盘后的会话运行。

推荐调整：

- `DeferredToolLoopOutcome::OrganizeContext` 不再直接设置 `suppress_assistant_message=true` 后返回成功结果。
- 改成返回明确的“压缩闭合请求”，由外层执行：
  - checkpoint 落盘；
  - 压缩；
  - 重开调度。

### 阶段 5：修改发送前自动压缩路径

发送前 `COMPACTION-AUTO` 命中时，通常用户消息已经持久化，还没有 assistant/tool 历史。

该路径仍需遵守统一语义：

1. 若当前调度尚未产生 assistant/tool 历史，可直接压缩。
2. 若未来该路径可能携带 partial assistant/tool 历史，必须先 checkpoint 落盘。
3. 压缩输入必须重新读取当前会话，而不是继续使用旧 snapshot。
4. 压缩完成后重开调度，不能在原 prepared context 里继续。

### 阶段 6：压缩状态前端可见

压缩是内部工具，不应作为普通业务工具卡片展示，但用户必须看到明确状态反馈。

要求：

1. 后端自动压缩 `tool_status` 必须带当前 `requestId/activationId`。
2. 后端自动压缩 `tool_status` 必须带稳定虚拟 `toolCallId`，让前端能识别 running/done/failed 是同一件事。
3. 前端横幅不能只依赖单一翻译文案，应识别 `organize_context/archive` 或“压缩/整理上下文”类状态。

### 阶段 7：续跑恢复意图

压缩结束后续跑应使用最小恢复意图，不携带庞大 prompt。

最小字段：

- conversation_id；
- agent_id；
- department_id；
- api_config_id；
- dispatch_reason：`context_compaction_followup`；
- origin_request_id；
- interrupt_reason：`context_compaction`；
- checkpoint_message_id（如果压缩前有落盘 assistant 检查点）；
- compaction_message_id。

恢复时只表达：

```text
刚才的调度因上下文整理被系统截断。请基于当前正式聊天记录和压缩摘要继续完成未闭环事项。
```

不得把旧 tool loop transient prompt 直接搬到新调度。

## 日志规范

新增或调整日志应能串起一条完整链路：

```text
[上下文整理] 开始，任务=interrupt_checkpoint，conversation_id=...，reason=context_compaction，tool_event_count=...
[上下文整理] 完成，任务=interrupt_checkpoint，conversation_id=...，assistant_message_id=...，persisted=...
[ARCHIVE-PIPELINE] 开始: task=context_compaction, ...
[ARCHIVE-PIPELINE] 上下文整理消息写入校验通过: ...
[上下文整理] 完成后续激活已入队 conversation_id=... event_id=...
```

日志必须包含：

- conversation_id；
- request_id / dispatch_id（如果有）；
- checkpoint_message_id；
- compaction_message_id；
- tool_event_count；
- trace_id；
- reason。

## 验证计划

### Rust 单元测试

至少覆盖：

1. 压缩打断前存在 completed tool history 时，必须先生成 assistant checkpoint 消息。
2. checkpoint 消息必须包含压缩前已完成工具历史。
3. 压缩摘要输入必须包含 checkpoint 消息，而不是旧 source 快照。
4. `organize_context` 触发后不能因 `suppress_assistant_message=true` 丢弃已有工具历史。
5. 空文本但有工具历史时仍落盘 assistant 消息。
6. 无文本、无 reasoning、无工具历史时不落空消息。
7. 压缩后重开调度使用 `context_compaction_followup`，不沿用旧 tool loop transient 状态。

### 最小集成验证

手动或 smoke 场景：

1. 让模型连续执行至少一个工具。
2. 在工具完成后、下一次模型续调前触发上下文整理。
3. 确认聊天记录中存在一条压缩前 assistant checkpoint 消息。
4. 确认该消息展示已完成工具历史。
5. 确认随后出现压缩消息。
6. 确认压缩结束后自动续跑。
7. 确认续跑不会重复执行已完成工具，除非模型基于新上下文主动判断需要重做。

### 日志验证

按同一 conversation_id 搜索：

```text
interrupt_checkpoint
task=context_compaction
上下文整理消息写入校验通过
context_compaction_followup
```

顺序必须是：

```text
checkpoint 落盘 -> 压缩开始 -> 压缩消息写入 -> 续跑入队/激活
```

## 非目标

- 不重做完整队列系统。
- 不改变普通用户手动 stop 的用户接口。
- 不把压缩消息改成 assistant 消息。
- 不持久化正在执行但尚未返回的工具。
- 不把旧 prepared prompt 或 provider request 原样塞进恢复意图。
- 不在本次引入新的前端状态管理库。

## 风险

### 1. 重复落盘

风险：压缩 checkpoint 和正常完成链路都尝试落同一批工具历史。

应对：

- checkpoint 成功后当前调度必须明确闭口；
- 清理对应 `inflight_completed_tool_history`；
- 后续返回 `CHAT_DISPATCH_RESTART_AFTER_COMPACTION`，不再走正常 assistant commit。

### 2. 压缩摘要仍读旧快照

风险：checkpoint 已落盘，但压缩仍使用落盘前 `source`。

应对：

- checkpoint 后强制重新读取 conversation；
- `run_context_compaction_pipeline` 调用方不得传旧 source。

### 3. 空消息判定误删工具历史

风险：assistant 文本为空时被当成空回复跳过，导致工具历史仍丢失。

应对：

- 模型结果有效性判断必须把非空 `tool_history_events` 视为有效；
- checkpoint 落盘允许“无文本但有工具历史”。

### 4. organize_context 自身污染历史

风险：把 `organize_context` 工具调用本身落进业务工具历史，导致模型续跑时看到无意义工具结果。

应对：

- checkpoint 使用 `tool_history_without_organize_context`；
- 只保留 `organize_context` 之前已经完成的业务工具历史。

## 完成标准

- 压缩路径和手动 stop 在“已完成工作必须落盘”这一点上语义一致。
- 工具续调前压缩不会丢失已完成工具历史。
- `organize_context` 触发压缩不会吞掉本轮半成品 assistant 工作。
- 压缩摘要基于 checkpoint 后的正式会话。
- 压缩后通过新调度续跑，而不是旧 tool loop 原地续调。
- 最小相关 Rust 测试通过。

## 实施顺序

1. 抽取或新增系统打断 checkpoint 落盘函数。
2. 让手动 stop 内部复用该能力，确保旧行为不退化。
3. 改造工具续调前自动压缩：先 checkpoint，再压缩，再返回重开。
4. 改造 `organize_context` 工具闭合：不再 suppress 丢历史，改为 checkpoint + 压缩 + 重开。
5. 确认发送前自动压缩读取最新会话快照。
6. 补齐单元测试。
7. 做最小手动联调和日志验证。

## 待确认

1. 压缩前 checkpoint assistant 消息是否需要 provider_meta 标记 `interruptReason=context_compaction`，便于 UI 与日志识别。
2. `organize_context` 这次工具调用本身是否完全从持久化工具历史剔除；本计划建议剔除。
3. 压缩结束后的恢复提示是否需要单独系统块，还是沿用现有 `context_compaction_followup` runtime context。
