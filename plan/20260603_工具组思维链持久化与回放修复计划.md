# 工具组思维链持久化与回放修复计划

日期：2026-06-03

## 背景

并发工具调用时，同一轮模型返回的是一个 assistant 工具调用组：一段 `reasoning_content`，对应同一条 assistant 消息里的多个 `tool_calls`。

当前实现中，工具组在流式事件与持久化阶段被拆成了多条单工具 assistant 事件，导致同一段思维链被复制到每一个工具调用/工具结果附近。这个问题会影响：

- 聊天窗口展示：思维链重复显示，像是覆盖在每个工具结果上。
- 会话持久化：落盘结构变成多条 assistant tool-call 事件，每条都带同一段 reasoning。
- 历史回放：抽象请求体会错误地回放多条 assistant，每条重复带同一段 reasoning。
- 上下文整理/压缩：checkpoint 与压缩读取到的工具历史结构已经被污染。

## 核心原则

1. 同一轮模型返回的多个工具调用必须作为一个工具组处理。
2. 一个工具组只能有一条 assistant tool-call event。
3. 一个工具组的 `reasoning_content` 只允许出现在这一条 assistant tool-call event 上。
4. 每个工具结果是独立 tool event，通过 `tool_call_id` 对应工具组里的某个 `tool_call`。
5. tool result event 永远不携带 `reasoning_content`。
6. 抽象请求体回放时，也必须保持“一条 assistant + 多条 tool”的结构。
7. 流式 UI 事件和持久化事件必须分离：流式 `assistant_tool_event` 不携带 `reasoning_content`，思维链只通过 `activity_reasoning_delta` 展示；持久化事件保留 `reasoning_content`。
8. 持久化/展示可以保留 partial 工具组；请求体回放必须裁剪成闭合子组，不能把缺少 tool result 的 `tool_call` 发给模型。
9. 同一工具组里，tool result 永远排在尾部 user sidecar 前面；user sidecar 永远在本组 tool result 后面。

```text

  role: "assistant",

  reasoning_content: Some("本轮工具组 reasoning，只回放一次"),
  tool_calls: Some([call_a, call_b]),
  tool_call_id: None,

PreparedHistoryMessage {
  role: "tool",
  text: "工具 A 结果",
  reasoning_content: None,
  tool_calls: None,
  tool_call_id: Some("call_a"),

PreparedHistoryMessage {
  role: "tool",
  text: "工具 B 结果",
  reasoning_content: None,
  tool_calls: None,

```

错误，禁止回退：

```text
assistant(reasoning, [call_a])
tool(call_a)
assistant(same reasoning, [call_b])
tool(call_b)
```

在当前数据模型里，前三类工具协议事件作为 `toolCall` 工具历史保存；最后一条普通 assistant 的 `content` 和 `reasoning_content` 保存到 assistant message 的 `parts`。抽象请求体回放时再恢复成上面的顺序。

正确的持久化消息结构：

```json
{
  "parts": [
    {
      "type": "text",
      "text": "三个并发 shell 跑完，结果是：\n\nNode.js v24.2.0\npnpm v10.29.2\nRustc 1.94.0",
      "reasoningContent": "根据三个工具结果汇总最终回答。"
    }
  ],
  "toolCall": [
    {
      "role": "assistant",
      "content": "三个并发 shell 跑完后我再汇总。",
      "reasoning_content": "先同时跑 node/pnpm/rustc。",
      "tool_calls": [
        {
          "id": "call_node",
          "call_id": "call_node",
          "type": "function",
          "function": {
            "name": "exec",
            "arguments": "{\"command\":\"node -v\"}"
          }
        },
        {
          "id": "call_pnpm",
          "call_id": "call_pnpm",
          "type": "function",
          "function": {
            "name": "exec",
            "arguments": "{\"command\":\"pnpm -v\"}"
          }
        },
        {
          "id": "call_rustc",
          "call_id": "call_rustc",
          "type": "function",
          "function": {
            "name": "exec",
            "arguments": "{\"command\":\"rustc -V\"}"
          }
        }
      ]
    },
    {
      "role": "tool",
      "tool_call_id": "call_node",
      "content": "Node.js v24.2.0"
    },
    {
      "role": "tool",
      "tool_call_id": "call_pnpm",
      "content": "pnpm v10.29.2"
    },
    {
      "role": "tool",
      "tool_call_id": "call_rustc",
      "content": "rustc 1.94.0"
    }
  ]
}
```

必须钉死：

- `parts.text` 只等于最后一条普通 assistant 的最终回答，不是工具调用正文和最终正文的拼接。
- `parts.reasoningContent` 只等于最后一条普通 assistant 的思维链。
- `toolCall` 字段只允许保存真正工具调用链：带 `tool_calls` 的 assistant 和对应 `tool` result。
- 没有 `tool_calls` 的普通 assistant 不准写入 `toolCall`。

错误，禁止回退：

```text
最终 assistant 正文 =
  assistant(tool_calls).content
  + "\n\n"
  + assistant(final).content
```

也就是禁止把工具调用轮的 `content` 拼进最终正文。工具调用轮的 `content` 属于那条 assistant tool-call event；最终正文只来自最后一条无 `tool_calls` 的 assistant。

同一轮模型响应如果供应商/SDK 返回多个连续且完全相同的 text block，后端合并文本块时只能保留一份：

```text
["三并发，都在1秒内返回：\n\n| 工具 | 版本 | ...",
 "三并发，都在1秒内返回：\n\n| 工具 | 版本 | ..."]
=>
"三并发，都在1秒内返回：\n\n| 工具 | 版本 | ..."
```

不同文本块仍要保留：

```text
["第一段", "第二段"] => "第一段\n第二段"
```

### 前端展示结构

正确的活动投影顺序：

```text
reasoning: 本轮工具组 reasoning，只显示一次
tool: read call_a result=工具 A 结果
tool: read call_b result=工具 B 结果
```

流式事件正确分工：

```text
activity_reasoning_delta: 本轮工具组 reasoning，只显示一次
assistant_tool_event: assistant(tool_calls=[call_a, call_b])，不带 reasoning_content
assistant_tool_result: tool(call_a)
assistant_tool_result: tool(call_b)
```

错误，禁止回退：

```text
reasoning: 本轮工具组 reasoning
tool: read call_a result=工具 A 结果
reasoning: 本轮工具组 reasoning
tool: read call_b result=工具 B 结果
```

### Sidecar User 顺序

截图等工具可能把图片转发为后续 user sidecar。无论调用顺序如何，写入 `messages` / `tool_history_events` 时都必须保持：

```text
assistant([call_a, call_b])
tool(call_a)
tool(call_b)
user("[desktop screenshot forwarded as user image]")
```

禁止形成：

```text
assistant([call_a, call_b])
tool(call_a)
user("[desktop screenshot forwarded as user image]")
tool(call_b)
```

实现上，写 tool result 时插入到尾部连续 user sidecar 前；写 user sidecar 时仍追加到末尾。

## 现状侦查结论

### tool_loop.rs

模型请求续轮阶段已经有正确的工具组概念：

- 同一轮 `turn_tool_calls` 会组成一条 assistant message。
- 这条 assistant message 携带本轮 `turn_reasoning`。

问题发生在后续执行工具时：

- 代码对 `turn_tool_calls` 做逐个循环。
- 每个工具都会调用 `assistant_tool_event_value(..., &turn_reasoning)`。
- 每个工具都会生成一条新的 assistant tool event。
- 因此同一段 `turn_reasoning` 被复制到多个 assistant tool event。

### message_store/store.rs

当前持久化入口仍是单工具 pair 语义：

- `append_tool_call_result_pair_to_conversation`
- `validate_tool_call_result_pair`
- `tool_calls.len() != 1` 时直接报错

这和工具组语义冲突。持久化层必须改为支持：

- 第一次看到某个工具组时追加 assistant group event。
- 每个工具结果按 `tool_call_id` 追加到同一条 assistant message 的 `tool_call` 数组中。
- 已存在的 tool result 不重复追加。
- 不为同一工具组新增第二条 assistant event。

### message_semantics.rs

`build_prepared_history_messages_from_tool_history` 已经支持一条 assistant event 携带多个 `tool_calls`。只要持久化结构正确，抽象请求体转换应保持正确结构。

需要补测试钉住：

- assistant event 多个 `tool_calls` 转换为单条 `PreparedHistoryMessage`。
- tool result 转换为多条 tool `PreparedHistoryMessage`。
- `reasoning_content` 只在 assistant 工具组消息上。

### 前端 chat-message-semantics.ts

`projectChatActivityForDisplay` 按 assistant event 输出一条 reasoning，再输出该 event 下的所有 tool calls。只要后端持久化是工具组结构，前端展示应自然变成“一条 reasoning + 多个工具”。

需要补测试钉住：

- 单个 assistant event 多 tool_calls 时，只生成一条 reasoning item。
- 两个 tool item 分别绑定自己的 tool result。

## 修改范围

### 后端工具循环

目标：同一轮 `turn_tool_calls` 只生成一条 assistant tool-call group event。

预期调整：

1. 拆分工具组事件构造函数：
   - `assistant_tool_group_history_event_value(turn_text, turn_tool_calls, turn_reasoning)`：用于持久化/回放，带 `reasoning_content`。
   - `assistant_tool_group_stream_event_value(turn_text, turn_tool_calls)`：用于流式 UI，不带 `reasoning_content`。
2. 进入工具执行前，发送一次 stream `assistant_tool_event`，内容为整组 `tool_calls`，不携带 reasoning。
3. `tool_history_events` 中只 push 一次 history assistant group event，携带 reasoning。
4. 每个工具执行完成后，只 push 对应的 tool result event。
5. 持久化时传递 group event + 当前 tool result event，不能为每个工具构造单工具 assistant event。
6. 写入 tool result 时插入到尾部 user sidecar 前，保证同一工具组 tool result 连续位于 user sidecar 之前。

### 后端持久化

目标：从单工具 pair 语义改为工具组 result 追加语义。

预期调整：

1. 重命名或新增入口，避免继续暗示 pair 语义，例如：
   - `append_tool_group_result`
   - `append_tool_call_result_to_group`
2. 校验规则改为：
   - assistant event 必须是 `role=assistant`。
   - assistant event 必须有非空 `tool_calls`。
   - tool result 必须是 `role=tool`。
   - `tool_result.tool_call_id` 必须存在于 assistant event 的 `tool_calls` 中。
3. 追加规则改为：
   - 如果目标 assistant message 还没有该工具组 assistant event，则先追加一次 group event。
   - 如果已有同一工具组 assistant event，不再追加 assistant event。
   - 如果已有相同 `tool_call_id` 的 tool result，不重复追加。
   - 否则追加当前 tool result。

工具组去重建议用 assistant event 中的 `tool_calls` id 集合判断。同一组内任一 call id 已存在时，应复用已有 group event，而不是追加新的 assistant event。

### 抽象请求体转换

目标：保持工具组。

预期调整：

1. 确认 `build_prepared_history_messages_from_tool_history` 对多 `tool_calls` 的 assistant event 只输出一条 assistant `PreparedHistoryMessage`。
2. 补充测试覆盖多工具组样本。
3. 禁止在转换过程中按单个工具拆 assistant 消息。
4. 对 partial group 只保留已有 matching tool result 的 `tool_call`；如果一个 result 都没有，则丢弃该 assistant tool-call event。

### 前端投影

目标：展示层只显示一次 reasoning。

预期调整：

1. 以前端现有 `projectChatActivityForDisplay` 为准，不主动拆 reasoning。
2. 补充测试覆盖工具组样本。
3. 如发现 result 匹配只找相邻第一条 tool result，需要确认多 tool result 连续跟随时能按 `tool_call_id` 正确匹配。

## 最小测试清单

不跑全量测试。只新增和运行以下窄测试。

### Rust：工具组事件构造

样本：

- `turn_reasoning = "先同时读取两个文件"`
- `turn_tool_calls = [call_a, call_b]`

断言：

- 只生成一条 assistant group event。
- `reasoning_content == "先同时读取两个文件"`。
- `tool_calls.len() == 2`。
- 不存在第二条 assistant event 携带相同 reasoning。
- stream event 不携带 `reasoning_content`，history event 保留 `reasoning_content`。

### Rust：持久化工具组追加

样本：

1. 初始 assistant message 为空。
2. group event 带 `call_a` 和 `call_b`。
3. 追加 `tool_result(call_a)`。
4. 再追加 `tool_result(call_b)`。

断言：

- 同一条 assistant message 下 `tool_call` 长度为 3。
- 第 0 条是 assistant group event，且 `tool_calls.len() == 2`。
- 第 1 条是 `tool_call_id=call_a`。
- 第 2 条是 `tool_call_id=call_b`。
- `reasoning_content` 只出现 1 次。

### Rust：抽象请求体转换

样本使用上面的持久化结构。

断言：

- 输出 3 条 `PreparedHistoryMessage`。
- 第 1 条 role 为 assistant，`tool_calls.len() == 2`，`reasoning_content` 为原始值。
- 第 2/3 条 role 为 tool，`reasoning_content == None`。
- partial group `assistant([call_a, call_b]) + tool(call_a)` 回放为 `assistant([call_a]) + tool(call_a)`。

### Rust：Sidecar 顺序

样本：

```text
assistant([call_a, call_b])
tool(call_a)

```

追加 `tool(call_b)` 后断言：

```text
assistant([call_a, call_b])
tool(call_a)
tool(call_b)
user(sidecar)
```

### TypeScript：前端展示投影

样本使用上面的持久化结构。

断言：

- `projectChatActivityForDisplay(...).items` 顺序为：
  - reasoning
  - tool call_a
  - tool call_b
- reasoning item 只有 1 条。
- 两个 tool item 的 resultText 分别对应 A/B。

## 验收标准

1. 并发工具调用后，聊天窗口只显示一次本轮思维链。
2. 会话落盘中，一个工具组只有一条 assistant event。
3. 会话落盘中，多个 tool result 跟随同一条 assistant group event。
4. 抽象请求体回放保持一条 assistant + 多条 tool。
5. 日志 Response 可以展示本轮 reasoning，但不能把同一段 reasoning 复制到每个工具项。
6. 所有新增测试样本必须作为防回归测试长期保留。
7. partial 工具组不会以不成对结构进入请求体。
8. screenshot/user sidecar 不会插入同一工具组的 tool results 中间。

## 非目标

- 不改变模型供应商协议适配策略。
- 不改变工具并发执行策略。
- 不改变前端工具卡视觉样式。
- 不引入全量测试要求。
- 不截断或清洗 reasoning 文本；本计划只修正归属关系。
