# 20260310 部门同步委派归档

## 1. 原始目标
- 把“多个 AI 自由聊天”收口为“主助理统筹、部门承办”的组织型协作系统。
- 明确主助理、部门、下属部门之间的职责边界。
- 让委托链具备可见性约束、禁止回退、逐级返回与同步转办语义。
- 为后续委托工具、部门配置、提示词装配与前端展示提供统一业务口径。

## 2. 最终落地结果
- 已新增完整的部门配置体系：
  - 部门名称
  - 部门概述
  - 部门办事指南
  - 驱动模型
  - 委任人格
- 已落地两种委派工具：
  - `delegate`：主助理异步委托，立即返回“委托已送达 / 委托无法送达”
  - `handoff`：委托链内部同步转办，等待子结果返回
- 已新增 delegate 持久化存储模块，使用 SQLite 记录委托记录与状态。
- 已完成部门提示词注入、调用栈限制、部门可见性限制与前端消息展示改造。

## 3. 最终业务语义
### 3.1 主助理
- 主助理直接对用户负责。
- 主助理只关心“是否需要委托”“委托是否送达”“被委托人后续如何汇报”。
- 主助理不需要理解底层执行线程、调用栈和存储细节。

### 3.2 部门
- 当前部门始终先看到自己的：
  - 部门名称
  - 部门概述
  - 部门办事指南
- 然后再看到“当前可用部门”及其概述。
- 下属部门的委任人和模型信息不会暴露给上级。

### 3.3 委托链
- 单份委托内部维持顺序执行。
- `delegate` 负责投递，不负责等待完整结果。
- `handoff` 负责在委托链内部下钻，并等待子结果返回。
- 结果严格逐级返回，不允许越级直达。

## 4. 可见性与防循环规则
- 当前部门不可见。
- 调用链中已出现过的部门不可见。
- 上游祖先和发起部门不可回退。
- 通过 `call_stack` 实现动态不可见，而不是只靠静态组织结构判断。
- 同一子任务失败后，不鼓励在条件未变化时无脑重复委托。
