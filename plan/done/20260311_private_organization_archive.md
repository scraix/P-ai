# 20260311 私有部门与私有人格加载归档

## 1. 原始目标
- 不新增高权限“部门/人格管理工具”。
- 改为让 LLM 在私有工作区内通过 JSON 声明私有人格与私有部门。
- 由 `reload` 统一扫描、校验、加载并注入运行时。
- 私有组织资产统一归属于助理私域，而不是某个具体人格。

## 2. 最终落地结果
- 已新增私有组织目录加载能力，私有对象来源于工作区目录，而不是主配置文件。
- 已支持私有人格 JSON 加载、校验、冲突检测与运行时注入。
- 已支持私有部门 JSON 加载、校验、默认模型选择、人格引用校验与运行时注入。
- 已在运行时对象上区分来源与作用域：
  - `source = private_workspace`
  - `scope = assistant_private`
- 已将 `private-organization-guide` 加入预设 Skill。
- 已通过 `reload` 返回私有人格/私有部门的加载成功与失败结果。

## 3. 最终业务语义

### 3.1 私有对象归属
- 私有人格、私有部门都属于“助理私域组织资产”。
- 它们不绑定某个具体人格，也不因切换主助理人格而变更所有权。
- 用户仍然可以直接查看、修改、删除这些文件。

### 3.2 生效方式
- LLM 或用户修改私有目录中的 JSON 文件。
- 通过 `reload` 触发扫描、校验与加载。
- 合法对象进入运行时。
- 非法对象不会生效，并返回明确错误。

### 3.3 与主配置的边界
- 私有对象不会覆盖主配置中的同 ID 对象。
- 主配置中的人格、部门、内建对象仍然保持最高优先级。
- 主配置文件本体不会被私有对象反向写回。

## 4. 实现落地要点

### 4.1 后端加载器
- 已新增私有组织加载模块：
  - `src-tauri/src/features/skill/private_org.rs`
- 已支持：
  - 扫描目录
  - 解析 JSON
  - 错误收集
  - 冲突检查
  - 引用校验
  - 运行时合并

### 4.2 目录与来源
- 已按私有组织目录进行加载。
- 私有对象来源不从 JSON 自声明读取，而是由加载流程自动赋值。
- 已自动标记：
  - `source = private_workspace`
  - `scope = assistant_private`

### 4.3 Skill
- 已加入预设 Skill：
  - `private-organization-guide`
- Skill 负责指导 LLM 如何编写 JSON 和何时调用 `reload`。
- Skill 不直接修改主配置，也不绕过系统校验。

### 4.4 前端展示
- 运行时前端模型已支持 `source / scope`。
- 私有人格与私有部门已可在界面上区分来源。
- 私有对象与主配置对象已经能够共存展示。

## 5. 已落地规则
- 私有人格不能使用保留 ID。
- 私有人格不能覆盖主配置同 ID 人格。
- 私有部门不能使用保留 ID。
- 私有部门不能覆盖主配置同 ID 部门。
- 私有部门引用的人格必须存在。
- 私有部门引用的模型必须是可用于文本对话的模型。
- 若未显式指定可用模型，系统会按当前可用文本模型规则解析默认值。

## 6. 与原计划的差异
- 原计划中的“前端来源可视化不是第一阶段必须项”已经部分落地。
- 原计划中对 owner/source 伪造的担忧，已经通过“目录归属决定来源、加载器自动赋值”收口。
- 原计划中的“私有对象是否允许前端直接编辑”等扩展问题，仍可后续另开计划继续推进。

## 7. 验证依据
- 代码中已存在私有对象来源与作用域字段的正式定义与默认值。
- 代码中已存在私有组织扫描、解析、校验、合并逻辑。
- 代码中已存在 `private-organization-guide` 预设 Skill 注册。
- 当前运行时与前端类型中都已承载私有对象来源信息。

## 8. 归档结论
- “私有部门与私有人格加载方案”已经从设计阶段进入已实现状态。
- 原 `20260311_私有部门与私有人格加载方案计划.md` 不再作为进行中的计划保留。
- 后续若继续扩展“私有对象前端直编、一键转正、私有组织高级治理”等能力，应另开新计划推进。
