# 20260311 群聊消息队列与主助理串行调度归档

## 1. 归档结论

本计划已完成主体实现，并在后续联调中根据真实业务继续收口。  
当前主会话已经切换为“队列入口 + 历史生效 + 主助理独占前台流式”的稳定模型。

本次归档确认以下设计已经落地：

- 所有主会话新消息统一先入队
- 主助理是主会话唯一允许流式输出的发言方
- 调度器基于统一状态机串行消费队列
- 入队、主助理结束、压缩结束都会触发出队检测
- 出队时先写正式历史，再决定是否激活主助理
- `history_flushed` 已携带正式落库消息，前端优先消费已确认批次
- 流式期间前端不再等待整段历史重载，而是直接显示已确认批次与当前流式预览
- 助理完成时不再整段刷新历史，而是优先补入本轮刚持久化的助理消息
- 撤回不再依赖整段重载，而是以后端删除成功为准，前端直接基于当前消息列表截断

## 2. 最终实现口径

### 2.1 主会话消息流

- 队列是入口层
- 历史是唯一生效层
- 主助理永远只有一个前台流式轮次
- 非主助理来源只能以完整消息块进入历史，不允许前台流式插话

### 2.2 前端展示口径

- 队列中的消息不直接进入主消息面板
- 只有收到 `history_flushed` 后，这批消息才被视为正式进入当前窗口
- 当前轮次流式期间，前端直接使用后端推送的已确认批次，不阻塞等待全量历史回源
- 当前轮次完成后，只补入本轮正式助理消息；仅在拿不到正式消息时才回退到整段刷新

### 2.3 撤回口径

- 用户点击某条用户消息“撤回”后，删除该条及其后的所有历史消息
- 被撤回的用户消息内容回填输入框
- 前端直接基于已持有消息列表收缩显示，不再依赖撤回后的全量重读
- 撤回后消息窗口重定位为：
  - 从主助理倒数第二条消息之后开始显示
  - 如果主助理消息不足两条，则显示全部剩余历史

## 3. 与原计划相比的实现修正

- 原计划中的“前端只展示结果，不负责竞争调度”保持不变
- 但在具体实现上，前端承担了“消费已确认批次、维持当前前台轮次显示”的职责
- 这不是前端参与调度决策，而是前端负责更快地呈现已经被后端确认的结果
- 因此业务层仍然是“后端统一调度”，显示层则采用“后端确认后前端即时呈现”的方案

## 4. 后续事项

以下属于后续优化，不再阻塞本计划归档：

- 群友/人格状态栏的交互完善
- 谁在思考时的高亮联动
- 点击人格强制发言等高级控制
- 更细的流式退场时机与视觉细节优化

## 5. 原计划正文

# 20260311 群聊消息队列与主助理串行调度计划

## 1. 背景

当前主会话已经不再是传统的一问一答，而是一个可能同时出现多种发言来源的群聊场景：

- 用户连续发言
- 任务触发写入消息
- 部门/工具人完成后回报
- 系统事件触发助理回复
- 压缩上下文期间插入系统行为

问题不在“能不能把消息写进记录”，而在“谁有资格在什么时候驱动主助理继续说话”。

如果没有统一调度，现象会非常混乱：

- 主助理正在流式输出时，新的激活事件又启动一轮回复
- 工具人回报、任务触发、用户发言互相插队
- 压缩中写入新消息，前后顺序不稳定
- 多个激活来源在极短时间内重复触发，造成重复回复或上下文错乱

因此必须把“消息记录”和“主助理执行”拆成两层：

- 记录层允许多人并发写入意图
- 执行层严格串行，只允许主助理独占流式

## 2. 核心目标

本轮目标不是做复杂智能编排，而是先建立一个绝对稳定的主会话调度内核：

1. 所有进入主会话的新消息统一先入队
2. 只有主助理可以流式输出
3. 任意并发来源都不能绕过队列直接激活主助理
4. 出队执行必须串行
5. 压缩期间禁止出队
6. 主助理回复结束后自动继续检查队列
7. 即使用户乱发、多任务同时触发、多工具人同时回报，也不会把主会话打乱

## 3. 设计原则

### 3.1 记录与执行分离

- 入队只代表“有新事件需要写入/处理”。
- 出队才代表“把事件真正写入会话，并决定是否激活主助理”。
- 不允许任何来源在入队时直接启动主助理。

### 3.2 主助理独占流式

- 主助理是主会话中唯一允许流式传输的发言方。
- 其他来源（任务、工具人、系统消息）只允许以完整消息块进入会话，不允许流式插嘴。

### 3.3 非法并发不报错，统一排队

- 并发来源不是异常，而是常态。
- 不试图阻止多来源同时进入，而是让它们统一排队并等待调度。

### 3.4 前端只展示结果，不负责竞争调度

- 是否能说、是否能压缩、是否能出队，由统一调度器判断。
- 前端不自行推断“现在可以直接触发下一轮回复”。

## 4. 统一状态机

主会话执行状态统一收敛为以下几种：

- `idle`
  - 当前没有主助理流式输出
  - 当前没有压缩流程
  - 允许执行出队

- `assistant_streaming`
  - 主助理正在流式输出
  - 禁止出队

- `organizing_context`
  - 正在执行压缩/整理上下文
  - 禁止出队

不再推荐多个分散布尔值在不同模块各自判断。  
统一由主调度器持有主状态，出队判断只认状态机。

## 5. 队列模型

建议新增主会话待处理事件队列，例如：

```ts
type ChatPendingEvent = {
  id: string;
  conversationId: string;
  createdAt: string;
  source: "user" | "task" | "delegate" | "system";
  messages: ChatMessage[];
  activateAssistant: boolean;
};
```

字段约定：

- `messages`
  - 本次要写入主会话的完整消息集合
- `activateAssistant`
  - 写入完成后是否需要触发主助理
- `source`
  - 仅用于诊断、日志、后续统计

## 6. 入队规则

以下来源统一入队：

### 6.1 用户发言

- 用户消息先入队
- `activateAssistant = true`

原因：

- 用户发言是主助理最直接的激活信号
- 但若此时主助理正在说话，不能立即插队，只能等待

### 6.2 任务触发

- 任务生成的消息先入队
- `activateAssistant = true`

原因：

- 任务触发本质上等价于一条“系统交付给主助理的新工作”
- 但也必须服从统一串行调度

### 6.3 委托/工具人完成回报

分两类：

- 需要通知主助理：
  - 入队
  - `activateAssistant = true`

- 不需要通知主助理：
  - 入队
  - `activateAssistant = false`

注意：

- 即使不通知主助理，也必须写入聊天记录，让用户看得到
- 只是它不触发主助理继续发言

### 6.4 系统内部事件

例如：

- 某些系统生成的消息卡片
- 某些由逻辑层插入的状态性消息

统一原则：

- 是否触发助理由事件自己显式携带
- 不允许靠“消息 role”自动推断

## 7. 出队触发时机

每次发生以下事件时，都要触发一次“出队检测”：

### 7.1 新事件入队后

- 目的是尽量快地消费队列
- 如果当前不可执行，就保持在队列中等待

### 7.2 主助理发言结束后

- 主助理一旦结束，主状态可能从 `assistant_streaming` 回到 `idle`
- 此时必须立刻再次检测队列

### 7.3 压缩完成后

- 压缩结束后，主状态可能从 `organizing_context` 回到 `idle`
- 此时必须再次检测队列

## 8. 出队检测规则

出队检测逻辑必须非常简单，避免复杂条件扩散。

### 8.1 不出队条件

如果满足任意一项，则直接返回：

- 当前状态为 `assistant_streaming`
- 当前状态为 `organizing_context`
- 当前没有待处理队列项

### 8.2 可以出队条件

仅当：

- 当前状态为 `idle`
- 队列非空

才允许真正执行出队。

## 9. 单次出队执行流程

单次出队建议如下：

1. 读取当前可消费的队列批次
2. 将该批次中的所有消息一次性写入主会话记录
3. 计算本批次是否需要激活主助理
4. 如果需要，则启动主助理一轮回复
5. 如果不需要，则结束本轮，等待下一次触发

### 9.1 为什么必须“先写记录，再激活助理”

因为主助理必须基于已经落入会话的最新消息进行思考。

如果边写边激活，会出现：

- 回复拿到的上下文不完整
- 会话顺序与回复时看到的顺序不一致

因此顺序必须固定：

- 先落消息
- 后激活助理

## 10. 批量出队策略

建议一次出队处理“当前时刻可见的整批事件”，而不是一条条触发一轮 LLM。

例如：

- 用户发了两条
- 一个任务同时到达
- 一个工具人刚回报

如果这些事件都还没被消费，那么一次出队应允许把它们先全部写入记录，再统一判断是否激活主助理。

这样能避免：

- 多次重复激活
- 多轮空转
- 回复基于半截上下文

## 11. 激活助理规则

建议采用“批次合并后的布尔规则”：

- 只要本批次中任意一个事件 `activateAssistant = true`
- 则本批次写入完成后激活主助理

否则：

- 只写记录，不启动主助理

## 12. 压缩与队列的关系

压缩不再是一个“随时插入的特殊流程”，而是主状态的一部分。

### 12.1 压缩开始时

- 主状态切换为 `organizing_context`
- 禁止出队

### 12.2 压缩期间

- 用户发言、任务触发、工具人回报都可以继续入队
- 但不能消费

### 12.3 压缩结束时

- 主状态切回 `idle`
- 立刻触发一次出队检测

## 13. 只记录不激活的典型场景

以下场景适合 `activateAssistant = false`：

- 某条委托结果只要求“加入聊天记录给用户看”
- 某个后台完成消息不需要主助理再总结
- 某些系统性说明信息只需要记录展示

这样可以避免主助理被无意义地唤醒。

## 14. 需要激活的典型场景

以下场景适合 `activateAssistant = true`：

- 用户发言
- 任务触发
- 被要求通知主助理的委托完成消息
- 某些明确要让主助理继续推进的系统事件

## 15. 日志与观测

调度器必须具备可观测性，建议至少记录以下日志：

- 入队成功
  - source
  - activateAssistant
  - message count
  - queue length

- 出队检测被阻塞
  - 当前状态
  - queue length

- 批次出队执行
  - batch size
  - message count
  - whether activate assistant

- 主助理发言完成后再次调度

- 压缩完成后再次调度

这些日志后续对排查“为什么这次没回复”“为什么这次延迟很高”会非常关键。

## 16. 风险与边界

### 16.1 长时间流式导致队列堆积

这是可接受的正常现象。  
主助理说话期间禁止出队，本来就是设计目标，不应视为错误。

### 16.2 大批量事件同时进入

必须允许批量写入记录，而不是逐条触发 LLM。  
否则性能和稳定性都会恶化。

### 16.3 队列事件重复入队

建议每个事件具备唯一 ID，必要时后续可以加入幂等保护。  
本轮先不强制实现复杂去重，但日志必须能追踪事件 ID。

### 16.4 压缩结束与主助理结束同时触发检测

检测逻辑必须幂等。  
即便同一时刻被调用两次，只要状态机和出队锁正确，也不应重复消费同一批队列。

## 17. 建议的实现边界

本轮建议先只做主会话调度，不扩展到所有子会话：

- 主会话实现统一队列与出队器
- 主助理实现独占流式
- 压缩流程接入同一状态机

暂不在本轮扩大到：

- 委托子会话内部的复杂并发调度
- 多个子会话之间的优先级竞争
- 更复杂的批处理合并策略

先把主会话“永不混乱”做稳定，再往外扩。

## 18. 预期收益

完成后，主会话应具备以下特性：

- 不会出现多人同时流式发言
- 不会因为用户乱发、任务同时触发而乱序
- 压缩不会和新消息竞争执行
- 消息写入与助理激活关系清晰
- 主助理始终基于完整的新记录回复

## 19. 本轮结论

这不是一个“再加几处 if 判断”的小修补，而是需要建立统一主调度器的结构性改造。

本方案的最终口径是：

- 所有新消息统一入队
- 主助理独占流式
- `idle` 之外不允许出队
- 出队时先写记录，再决定是否激活主助理
- 入队后、主助理结束后、压缩结束后都必须再次检测出队

如果后续进入实现阶段，应基于本计划继续细化：

- 队列数据结构
- 状态机持有位置
- 出队锁
- 事件来源适配层
- 调试日志与可观测性
