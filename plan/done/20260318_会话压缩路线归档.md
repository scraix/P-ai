# 会话压缩路线设计（替代自动归档）

## 目标

把“自动归档”从“会话搬迁（archive）”改为“会话内压缩（compaction）”。

核心目标：

1. 不切会话，不移动到 archived 列表。
2. 在当前会话插入一条可追溯的“压缩消息”。
3. 下一轮推理时，忽略压缩点之前的历史。
4. 组装提示词时，使用：
- 所有压缩消息（按时间顺序）
- 最后可用提示词（latest usable user prompt）

---

## 术语定义

- 压缩消息（Compression Message）：由系统生成并写入当前会话的特殊消息，承载“到某一时刻为止的上下文摘要”。
- 最后可用提示词（Latest Usable Prompt）：当前会话中，压缩点之后最近的有效 user 输入；如果不存在，则使用压缩消息的聚合文本作为本轮 user 输入。
- 压缩边界（Compression Boundary）：最后一条压缩消息所在位置。该位置之前的普通历史消息在后续推理中默认忽略。

---

## 设计原则

1. 压缩是“上下文治理”能力，不是“会话生命周期管理”。
2. 压缩结果必须可见、可审计、可追踪（作为消息落在会话中）。
3. 推理输入稳定可解释：每次都能明确“为什么只看这些消息”。
4. 不引入隐式状态机魔法：优先通过消息本身表达压缩状态。

---

## 用户可见行为

1. 触发自动压缩后，聊天记录新增一条消息（建议 role=`user`，type=`compression`）。
2. 不创建新会话，不切换 conversationId。
3. 后续回复基于压缩消息和压缩后的新输入继续。
4. UI 可显示标签：`[上下文压缩]`。

---

## Prompt 组装规则（关键）

### 规则总述

在构建下一轮模型输入时：

1. 先收集当前会话中**全部压缩消息**（按时间升序）。
2. 识别最后一条压缩消息的位置 `lastCompressionIndex`。
3. 普通历史消息仅保留 `lastCompressionIndex` 之后的部分。
4. latest user 输入采用“最后可用提示词”：
- 优先：压缩边界之后最近一条 user 消息。
- 兜底：将“全部压缩消息聚合文本”作为 latest user。

### 组装结构

- `history_messages`：
- 全部压缩消息（强制保留）
- 压缩边界之后的 assistant/tool/user 普通消息（去掉 latest user 本条）

- `latest_user_text`：
- 压缩边界之后最近一条 user 正文；若无，则使用“压缩消息聚合文本”。

### 为什么要“全部压缩消息 + 最后可用提示词”

- 全部压缩消息：保留多次压缩的累积上下文。
- 最后可用提示词：确保当前轮的即时目标明确，不被纯摘要吞没。

---

## 示例

### 原始会话

1. user：需求A
2. assistant：方案A1
3. user：改成A2
4. assistant：确认A2
5. user：补充约束X
6. assistant：已记录

触发压缩，插入：

7. user(type=compression)：
`[上下文压缩#1] 已确认：A2；约束：X；待办：实现模块M。`

继续对话：

8. user：先写模块M的数据结构

### 下一轮组 Prompt

- 保留：7（compression），8（latest user）
- 忽略：1~6
- latest_user_text = 第8条

若第8条不存在（例如仅压缩后立即触发一轮）

- latest_user_text = 聚合后的 compression 文本

---

## 数据模型建议

在 `ChatMessage` 增加可选字段：

- `message_type: Option<String>`，可取：`compression`
- `compression_meta: Option<...>`，建议包含：
- `compression_id`
- `source_range_start`
- `source_range_end`
- `trigger_reason`
- `model`
- `created_by`（system/assistant）

最小落地可先只加：

- `message_type = "compression"`
- 正文中包含结构化标题和摘要块

---

## 触发策略建议

1. 触发条件沿用现有 context 使用率阈值（如 82%）。
2. 触发后流程：
- 生成压缩文本
- 写入当前会话一条 compression 消息
- 发 `history_flushed` 让前端刷新
- 继续队列处理

3. 不再调用 archive_conversation_now。

---

## 与现有“归档”能力关系

- 保留手动归档（用户明确要归档/整理为历史）
- 自动触发场景改为压缩，不做归档迁移

即：

- 手动：Archive（会话迁移）
- 自动：Compaction（会话内压缩）

---

## 边界与冲突

1. 多次压缩：
- 保留全部 compression 消息；
- 计算边界时仅看最后一条 compression。

2. 用户回溯重写历史：
- 若回溯点早于最后 compression，可选择：
- 直接清空 compression 消息后重建；或
- 标记 compression 失效（v2 可做）。

3. 工具调用历史：
- 压缩边界之前的工具细节默认不再进入推理历史。
- 必要信息应在 compression 文本中摘要化。

4. 无 user 新输入时触发：
- latest_user_text 使用 compression 聚合文本，避免空请求。

---

## 验收标准

1. 自动压缩后不新增新会话、不切换 conversationId。
2. 会话中可见一条 compression 消息。
3. 下一轮推理不再包含压缩边界之前普通消息。
4. 下一轮 latest_user_text 符合“最后可用提示词”规则。
5. 前端能收到刷新事件并显示压缩消息。
6. 队列在压缩后可继续处理，不出现卡住。

---

## 实施步骤（建议）

1. 新增 compression 消息写入函数（会话内）。
2. 在自动触发路径把 `run_archive_pipeline` 替换为 `run_compaction_pipeline`。
3. 修改 prompt 装配：按“全部 compression + 最后可用提示词”裁剪历史。
4. 补充事件：`history_flushed`（必要）与可选 `compaction_applied`（增强可观测性）。
5. 增加单测：
- 单次压缩
- 多次压缩
- 压缩后无新 user 输入
- 压缩后队列续跑

---

## 结论

你提出的路线是明确且可实现的：

- 自动压缩不走归档迁移；
- 压缩结果作为消息落在当前会话；
- 下一轮严格按“全部压缩消息 + 最后可用提示词”组装推理输入。

这能显著降低状态复杂度，并提升可解释性与可调试性。
