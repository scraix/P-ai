## 与原计划相比的实际落地结果

### 1. 抽象层存在，但范围比原设计更收敛

- 已落地：渠道发送与平台适配层存在统一抽象，当前主要体现在 `RemoteImSdk` 及适配器路由上。
- 未单独抽象：联系人 CRUD 没有再做一层独立 service interface，而是直接由当前远程 IM 状态与命令层维护。
- 结论：系统没有完全按原抽象设计文档逐项实现，而是保留“适配器抽象 + 联系人直接管理”的更轻实现。

### 2. 联系人模型比早期“reply_mode”方案更细

- 早期计划偏向 `none / always / on_message` 的单一回复策略。
- 最终实现拆成：
  - `allow_receive`
  - `allow_send`
  - 激活模式（如关键字激活）
  - 相关冷却/触发控制
- 结论：最终产品更偏“收信控制 + 发信控制 + 激活控制”三段式，而不是只做一个 reply mode。

### 3. UI 方案按实际页面收敛，不再完全追随初稿

- 已完成：设置页存在远程 IM 联系人与渠道配置页面。
- 已完成：联系人与渠道都可以在当前配置页维护。
- 有差异：最终界面细节与 2026-03-14 的 UI 草案并不完全一致，但联系人优先、可直接管理的目标已达成。

### 4. 接口技术指南的价值已被代码实现替代

- 飞书、钉钉、NapCat 的主要接入方式已经固化到适配器代码中。
- 旧指南里很多“可选 API / 设计预案 / AstrBot 参考”已不再需要继续单独维护。
- 后续若再补平台细节，应直接补到实现说明或新的专项归档，而不是继续保留旧的总指南。

## 媒体能力归档

### 1. 出站发送

- OneBot：
  - 图片按图片消息段发送。
  - 其他文件按普通文件消息段发送。
- 飞书：
  - 图片先上传获取 `image_key` 后发送。
  - 文件先上传获取 `file_key` 后发送。
- 钉钉：
  - 图片先上传媒体后按 `sampleImageMsg` 发送。
  - 文件先上传媒体后按 `sampleFile` 发送。

### 2. 入站接收

- OneBot：
  - 解析数组消息中的 `image` / `file` 段。
  - 媒体解析成功后写入真实图片或附件，而不是仅保留 `[图片]` / `[文件]` 占位文本。
  - `file_id` 解析增加多动作兼容降级，改善相对路径与本地引用场景。
- 飞书：
  - 入站图片/文件走平台下载链路，再写入统一附件队列。
- 钉钉：
  - 入站图片/文件通过 `downloadCode` 下载后进入统一附件队列。
