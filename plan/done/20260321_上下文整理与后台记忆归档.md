# 20260321 上下文整理与后台记忆归档

## 结论

本轮目标已经落地：主会话自动整理、`organize_context` 工具整理、归档前预整理、后台记忆整理、归档提示词改造、归档查看与导出补齐，现已统一收口为“上下文整理 / 记忆整理 / 归档”三条独立能力链路。

原进行中文档 `plan/20260321_自动压缩后台记忆生成计划.md` 不再作为进行中计划保留，改由本文作为归档说明。

## 最终落地范围

### 1. 上下文整理

- 已完成：自动触发路径继续保留在主会话中执行，不再迁移会话。
- 已完成：整理结果以消息形式写入当前会话，正文统一为 `[上下文整理]` 与“整理摘要”。
- 已完成：写入后执行二次落盘校验，确认消息真实存在后才发送 `history_flushed` 事件。
- 已完成：提示词组装继续保留“历史消息在前，整理提示在最后”的顺序，满足缓存命中要求。
- 已完成：对历史 `[上下文压缩]` / “压缩摘要” 文本保留兼容识别，避免旧会话失效。

### 2. 记忆整理

- 已完成：任何一次上下文整理成功后，都会异步触发后台记忆整理任务。
- 已完成：归档前预整理也会单独触发一份后台记忆整理任务。
- 已完成：后台任务以“本会话全部上下文整理信息 + `memory_recall_table` 去重后的本会话记忆全集”为输入。
- 已完成：记忆整理输出只消费 `usefulMemoryIds`、`newMemories`、`mergeGroups`，不再要求额外 `summary` 字段。
- 已完成：支持新增记忆、记忆反馈、重复记忆合并。

### 3. 归档

- 已完成：归档主流程调整为“先做一次上下文整理，再归档”。
- 已完成：归档摘要输入会纳入本会话全部上下文整理信息，而不是只看最后一次整理。
- 已完成：归档提示词改为回顾已发生事实，禁止描述“接下来准备做什么”。
- 已完成：归档查看页支持直接看到归档消息中的上下文整理内容。
- 已完成：归档导出 JSON 只保留 `source_conversation.summary` 作为摘要来源，移除顶层重复 `summary`。

## 与原计划相比的实际落地差异

### 1. 没有引入独立持久化任务队列

- 原计划曾设想 `summary_memory_jobs.json` 一类的持久化任务队列与恢复机制。
- 最终实现没有新增独立任务存储，而是直接在整理/归档成功后 `spawn` 后台异步任务。
- 结论：本轮重点放在“主链路不卡顿”和“失败可重试”，未扩展到应用重启后的任务恢复。

### 2. 重试策略按产品口径收敛

- 上下文整理模型：最终为固定 `3` 次尝试、每次间隔 `5s`。
- 后台记忆整理：最终为固定 `3` 次尝试、每次间隔 `30s`。
- 结论：与原计划中“30 秒 + 建议 5 次”的草案不同，已按最终确认口径收敛。

### 3. 术语完成统一

- 原文档与早期实现中大量使用“压缩/compaction”表述。
- 最终用户可见文案、注释和提示词契约已统一为“上下文整理”。
- 结论：代码内部仍保留少量 `compaction` 命名，仅作为历史结构名或兼容识别，不再作为产品口径。
