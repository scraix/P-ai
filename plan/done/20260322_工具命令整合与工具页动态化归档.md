## 最终落地范围

### 1. 统一命令工具

- 已完成：原 `wait`、`reload`、`organize_context` 三类系统级能力收口为统一内置工具 `command`。
- 已完成：`command` 接收单一参数 `command: string`，支持 `help`、`reload`、`organize_context`、`wait <ms>` 四种子命令。
- 已完成：子命令返回统一 JSON 结构，包含 `ok`、`command`、`result` / `message` 等字段。
- 已完成：`wait` 参数校验收口到 `1~120000ms`，错误输入会返回明确提示。

### 2. 配置迁移与运行时装配

- 已完成：默认工具集合移除独立 `wait` / `reload` / `organize_context`，改为默认启用 `command`。
- 已完成：配置归一化会把旧 `desktop-wait`、`wait`、`reload`、`organize_context` 自动折叠进 `command`，并继承旧配置中的启用状态。
- 已完成：运行时工具装配改为按 `command` 判定和挂载，不再单独注册旧三种工具。
- 已完成：工具状态检查页会在旧人格配置缺少 `command` 时补一个运行时展示项，避免状态页缺失。

### 3. 工具页动态化

- 已完成：后端新增 `list_tool_catalog` 命令，统一返回前端展示所需的工具定义。
- 已完成：工具页不再按工具 ID 写死描述和参数说明，改为读取后端目录中的 `description` 与 `parameters`。
- 已完成：前端参数区会自动展示必填、类型、枚举和值域等摘要信息。
- 已完成：前端类型系统新增 `FrontendToolDefinition`，和后端目录返回结构对齐。

### 4. 后端目录接线

- 已完成：`main.rs` 已注册 `list_tool_catalog`。
- 已完成：聊天运行时命令聚合入口已通过 `include!("tool_catalog.rs")` 接入新命令实现。
- 已完成：目录内容已覆盖当前主要内置工具，包括 `fetch`、`websearch`、`remember`、`recall`、`screenshot`、`command`、`exec`、`apply_patch`、`task`、`delegate`、`remote_im_send`。

## 与原计划相比的实际落地差异

### 1. 工具名称最终不是 `system_command`

- 计划草案里使用的是 `system_command`。
- 最终实现收敛为更短的 `command`，对应参数也命名为 `command`。
- 结论：归档以后应以 `command` 作为唯一产品口径，不再继续使用 `system_command` 这一草案名。

### 2. 旧工具没有继续保留为运行时入口

- 草案里建议第一阶段保留 `reload`、`organize_context` 老入口，避免历史提示词立即失效。
- 最终实现选择直接收口运行时入口，只保留配置层兼容迁移。
- 结论：兼容策略从“并存”调整为“自动迁移 + 新入口统一收口”。

### 3. 工具页调试按钮被一并收敛

- 草案原本提到“仅保留动作执行器分支”。
- 当前实现里，`screenshot` / `wait` / `exec` 的页面内调试按钮已整体移除，工具页聚焦为“开关 + 描述 + 参数 + 状态”。
- 结论：本轮优先解决双端元数据重复维护问题，没有继续保留工具页内即时调试能力。

### 4. catalog 回退策略比草案更保守

- 草案设想：若某工具缺失 catalog，前端仍回退显示 `id + 通用描述`。
- 当前实现中，工具列表以后端 catalog 为准；catalog 中未返回的工具不会出现在工具页列表中。
- 结论：现阶段采用“后端单一事实来源”策略，前端没有继续保留缺省回退展示。
