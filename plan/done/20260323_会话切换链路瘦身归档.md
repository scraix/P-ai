# 会话切换链路瘦身归档（2026-03-23）

## 背景

本轮目标是把“切换会话必须等待后端返回整包 snapshot”的重链路，收敛成“前端先切、后端异步补正式消息”的轻链路，优先解决切换时的体感卡顿与白屏。

## 已完成实现

- 前端切换会话时改为立即更新当前前台会话，并优先渲染本地缓存
- 切换动作不再同步依赖 `switch_active_conversation_snapshot`
- 会话缓存锚点改为“前端当前已知最后一条正式消息”，不会再把 draft / 冻结 draft 当作锚点
- 后端新增异步补消息入口 `request_conversation_messages_after_async`
- 后端收到补消息请求后立即返回 `accepted + requestId`，随后异步查询并通过事件推送结果
- 前端新增 `easy-call:conversation-messages-after-synced` 监听，用于接收异步补回的正式消息
- 锚点失效时不再全量重拉整会话，而是回退最近 `5` 条消息
- 会话切到后台后保留最近 `5` 条缓存消息，避免再次切回时白屏
- 主会话与前台会话语义已拆开：
  - 主会话只决定列表固定位置和默认归宿
  - 前台会话只决定当前按钮高亮与正文显示
- 会话托盘已调整为：
  - 主会话固定第一个并与其他会话分隔
  - 其他会话排在其后
  - `+` 新建按钮固定最后
- 会话按钮文案已改为工作空间名，不再实时刷新时间文本
- 顶部归档入口已改为“预判断 + 三按钮确认”：
  - `归档`：摘要 + 记忆
  - `抛弃`：直接删除当前会话
  - `取消`
- `抛弃` 已从归档忙碌态中拆出，不再触发“归档中”全屏遮罩

## 实现结果

- 切换动作本身不再阻塞在后端 snapshot / recent messages / workspace labels / 流式重绑
- 前端切换后可以先立刻显示缓存，再等待后端异步补齐正式消息
- 后端不再需要在“切换瞬间”参与前台显示决策
- 会话切换期间的白屏和卡顿链路已明显缩短
- 归档与抛弃语义已明确分离，避免“其实没归档但一直显示归档中”的误导

## 验收结论

用户已完成前端功能测试，并确认“效果非常之好”“很完美了”，本计划按当前实现归档。
