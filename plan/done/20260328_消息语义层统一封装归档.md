# 20260328 消息语义层统一封装计划

## 背景

当前项目的消息原始存储结构已经定型，本次不修改落盘 schema，不做数据迁移。

现有问题不在“某一条消息坏了”，而在于“同一份原始消息，被前后端多个位置各自解释”：

- 后端聊天调度会自行把 `ChatMessage` 解释成历史消息
- 后端归档调度会再次解释 `ChatMessage`
- 归档导出会再次解释 `tool_call`
- 前端展示会再次聚合 `toolCall`
- 前端撤回判断也会再次扫描 `toolCall`

这会导致两个直接问题：

- 同一份消息在不同链路里的语义不一致
- 一旦某个消费点把 sidecar event 当成别的结构使用，就会出现 provider 400、空消息误判、重复展示等问题

本次目标不是“继续补兼容分支”，而是引入一层统一的消息语义封装，让后续所有消费方都只调用黑盒方法，不再直接猜原始字段。

## 核心原则

### 1. 原始存储结构保持不动

- 不改 `ChatMessage` 落盘结构
- 不做历史迁移
- 不新增存储层兼容字段

### 2. 只允许一个地方解释原始字段

只有“消息语义规范化层”可以直接读取这些字段：

- `parts`
- `extraTextBlocks`
- `providerMeta`
- `toolCall`
- `mcpCall`

其他层只能消费规范化后的结果，不允许再直接遍历原始 `toolCall` / `mcpCall`。

### 3. 统一的是“语义”，不是最终 JSON

前端和后端语言不同，不可能只保留一份实现代码；但可以统一：

- 一份消息语义契约
- 两端各自实现
- 一套共享黄金样例

### 4. 采用“单次规范化 + 分叉投影”，而不是所有人各自拼装

建议的处理链路：

```text
原始消息(ChatMessage)
    ↓
消息语义规范化层
    ↓
    ├─ 前端展示投影
    └─ Prompt 原子投影
            ↓
        场景组装（日常聊天 / 委托 / 归档 / 压缩）
            ↓
        供应商适配（OpenAI / DeepSeek / Gemini / Anthropic / 其他）
```

## 分层设计

## 1. 原始消息层

输入：

- 当前持久化的 `ChatMessage`

职责：

- 只负责数据承载
- 不承担“展示语义”与“Prompt 语义”

禁止：

- 在组件、命令、供应商适配器中直接扫描原始 `toolCall`

## 2. 消息语义规范化层

这是本次新增的核心黑盒模块。

职责：

- 解析消息正文、隐藏块、推理文本、媒体附件
- 解析 `toolCall` / `mcpCall` sidecar
- 统一识别工具事件顺序、事件类型、可见文本、关联 id
- 标记异常形态，例如：
  - 悬空 assistant tool call
  - 缺失 `tool_call_id` 的 tool result
  - 非法 event 结构
  - 空文本但含结构化 sidecar 的消息
- 产出稳定、可复用的中间语义对象

边界：

- 这里只做“解释”
- 不负责某个页面怎么展示
- 不负责某个 provider 要什么字段
- 不负责归档/委托/压缩场景裁剪

建议输出语义：

- `visible_text`
- `plain_text`
- `reasoning_standard`
- `reasoning_inline`
- `images`
- `audios`
- `attachments`
- `message_flags`
  - 是否压缩消息
  - 是否远程 IM 消息
  - 是否任务触发消息
- `sidecar_events`
  - assistant tool call event
  - tool result event
  - mcp event
- `event_validity`
  - 可展示
  - 可 Prompt 回放
  - 结构异常

说明：

“可展示”与“可 Prompt 回放”不能混为一谈。某些不完整 sidecar 可能可以给前端做提示，但不能原样回放给 LLM。

## 3. 前端展示投影层

职责：

- 把规范化语义投影成前端可直接消费的数据
- 可以聚合工具调用摘要
- 可以输出撤回按钮判断所需字段
- 可以输出消息卡片、归档列表、请求体预览所需字段

特点：

- 允许聚合
- 允许展示文案友好的结构
- 但不回头读取原始 `ChatMessage.toolCall`

建议能力：

- `projectMessageForDisplay`
- `projectConversationForDisplay`
- `summarizeToolActivityForDisplay`
- `inspectUndoablePatchCalls`

## 4. Prompt 原子投影层

职责：

- 把规范化语义拆成“最小可组装的 Prompt 原子消息”
- 每条消息独立、可排序、可过滤、可再次加工

特点：

- 不做前端聚合
- assistant 文本、assistant tool call、tool result 必须拆开
- 只输出 provider-agnostic 的原子结构

建议原子类型：

- `user_text`
- `assistant_text`
- `assistant_tool_call`
- `tool_result`
- `user_media`
- `message_meta_block`

说明：

这层是“后端 API 组建提示词”的唯一入口。聊天、委托、归档、压缩都从这里拿原子消息，再做各自场景加工。

## 5. 场景组装层

职责：

- 基于 Prompt 原子投影，处理业务场景差异：
  - 日常聊天
  - 委托
  - 归档
  - 压缩
  - 其他调度场景

允许做的事：

- 裁剪上下文窗口
- 跳过压缩边界前的消息
- 决定是否补时间 / 身份 / 渠道 metadata
- 决定是否保留不完整 tool trace

禁止做的事：

- 重新解释原始 `toolCall`
- 直接拼 provider JSON

## 6. 供应商适配层

职责：

- 把场景组装后的 provider-agnostic 结构映射成具体 provider 请求

允许做的事：

- 字段名转换
- 不同 provider 的结构兼容
- `tool_calls.id/type` 等 provider 特定约束补齐与校验

禁止做的事：

- 从原始消息中补语义
- 再做一遍工具 sidecar 猜测

## 四大协议 Tool Call 兼容策略

这次重构必须把“工具调用协议差异”显式纳入设计，而不是继续散落在历史回放或 provider adapter 的局部补丁里。

当前至少需要统一考虑四类协议族：

- OpenAI Chat / DeepSeek 风格
- OpenAI Responses 风格
- Gemini 风格
- Anthropic 风格

### 1. 差异现状

这些协议在工具调用上的关键差异主要有：

- 调用关联 id 的语义不同
- 工具结果回传字段不同
- 参数是字符串还是对象的要求不同
- assistant/tool 历史消息的 block 结构不同

典型差异：

- OpenAI Chat / DeepSeek：
  - 依赖 `tool_calls[*].id`
  - 工具结果通常靠 `tool_call_id` 回指
  - 参数通常接受字符串化 JSON
- OpenAI Responses：
  - `id` 与 `call_id` 不是同一个概念
  - 结构化回放时往往同时需要两者
  - 缺少 `call_id` 时，旧历史不能安全地按结构化 tool result 回放
- Gemini：
  - 参数更适合对象形态
  - 直接沿用字符串参数容易在 provider 适配时出现结构不匹配
- Anthropic：
  - `tool_use / tool_result` 语义与 OpenAI 系不同
  - 需要在最终适配时映射成其 block 结构

### 2. 规范化层不直接站队任何协议

消息语义规范化层不应该输出“OpenAI 风格 tool_call JSON”。

它应该输出协议无关的工具轨迹语义，例如：

- `invocation_id`
- `provider_call_id`
- `tool_name`
- `arguments_value`
- `arguments_text`
- `result_text`
- `result_blocks`
- `trace_status`
- `replay_capability`

建议语义说明：

- `invocation_id`
  - 对应一次工具调用的主标识
  - 默认承接原始 sidecar 中的 `tool_calls[*].id`
- `provider_call_id`
  - provider 特有的二级关联 id
  - 对 Responses 族尤其关键
  - 允许为空
- `arguments_value`
  - 统一后的 JSON Value
  - 作为 Gemini / 结构化 provider 的优先输入
- `arguments_text`
  - 原始或可回放的字符串参数
  - 作为 Chat-style provider 的兼容输入
- `replay_capability`
  - `structured`
  - `text_only`
  - `invalid`

### 3. Structured Replay 与 Text Fallback 规则

这次要把历史经验固化成统一规则：

- 当一个工具轨迹满足目标协议要求时，允许结构化回放
- 当缺失目标协议必需字段时，必须降级为文本历史
- 不允许为了“凑结构化回放”而伪造 id

重点规则：

- Responses 族如果缺少 `provider_call_id`，则：
  - 不做结构化 tool result 回放
  - 降级为普通文本历史
- Chat / DeepSeek 族如果只有 `invocation_id`，通常仍可结构化回放
- Gemini 若参数无法转成稳定对象，则降级为字符串或文本回放
- Anthropic 若 block 结构不完整，则降级文本，不能硬拼半结构化消息

### 4. 协议差异只允许留在供应商适配层

业务层不能出现这些判断：

- “如果是 Responses 就去找 `call_id`”
- “如果是 Gemini 就把 arguments 变 object”
- “如果是 Anthropic 就换成 tool_use”

这些都必须收口到供应商适配层。

职责划分应为：

- 语义规范化层：
  - 提供完整工具轨迹语义
  - 给出 `structured/text_only/invalid` 能力标记
- Prompt 原子投影层：
  - 输出协议无关的 `assistant_tool_call` / `tool_result`
- 场景组装层：
  - 决定要不要保留某段工具轨迹
  - 不处理协议字段名
- 供应商适配层：
  - 根据目标协议映射字段
  - 根据 `replay_capability` 决定结构化回放还是文本降级

### 5. 本次重构建议吸收的历史教训

此前已经遇到过 “`tool_calls[*].id` 与 Responses `call_id` 对不上” 的问题。

这说明：

- 原始存储中的工具 sidecar 可以保留精简形态
- 但中间层必须显式区分“调用项 id”与“provider 调用链 id”
- 不能继续把二者混成一个 `tool_call_id` 概念到处传

因此本次重构建议：

- 在规范化层里显式保留 `invocation_id` 与 `provider_call_id`
- 在 Prompt 原子投影中保留这两个字段
- 在 provider adapter 中按协议做最终映射
- 对缺少必要 id 的历史数据统一走降级路径，而不是各处单独兜底

## 不纳入本次范围

- 修改持久化 schema
- 历史数据迁移
- 新增数据库或中间存储层
- 改变归档页面视觉样式
- 改变各 provider 现有协议能力边界

## 风险

### 风险 1：规范化层做得过重，重新变成大杂烩

应对：

- 规范化层只解释，不做业务裁剪
- 场景裁剪一律留在场景组装层

### 风险 2：前端展示语义与 Prompt 回放语义混淆

应对：

- 明确区分“展示投影”和“Prompt 原子投影”
- 不允许一个输出同时承担两种职责

### 风险 3：迁移消费方不彻底，旧逻辑继续偷读 `toolCall`

应对：

- 在计划实施时逐一替换现有消费点
- 代码评审重点检查是否仍有直接扫描 `toolCall` 的新旧路径并存

### 风险 4：测试只覆盖 happy path，边界数据继续漏

应对：

- 共享 fixture 必须覆盖异常结构与历史脏数据形态
- 先补黄金样例，再迁消费方

## 完成判定

满足以下条件视为开发完成：

- 原始 `ChatMessage` 存储结构保持不变
- 前端存在统一的展示投影入口，不再直接扫描原始 `toolCall`
- 后端存在统一的 Prompt 原子投影入口，聊天/归档/压缩/委托共用
- 场景差异只在“场景组装层”体现
- provider 适配层不再自行解释原始 sidecar
- 共享黄金样例覆盖关键正常/异常路径
- 前后端测试通过
- 生产验证后再决定是否归档计划
