# 全局运行上下文渐进接入归档（2026-03-30）

## 1. 背景

当前项目里与“谁发起、从哪来、要投到哪去”有关的 ID 在多条链路中反复拆装：

- 聊天发送链路会在不同层之间分别传 `session_id`、`conversation_id`、`agent_id`
- 工具调用链路会再次解析会话来源
- 任务调度链路会再次推导目标会话
- 委托链路、联系人链路各自也有一套局部语义

这导致同一个动作在不同阶段容易出现：

- 会话 ID 丢失
- 会话 ID 被回退覆盖
- agent / department / source 语义漂移
- 日志里多个字段彼此不一致

本计划目标不是一次性全仓重构，而是先新增一套统一上下文结构体，让后续改动有统一挂点，并按需渐进接入。

## 2. 命名原则

字段名要求“看名字就知道语义”，不依赖注释补全：

- `origin_*`：最初从哪里发起
- `target_*`：本次最终准备投到哪里
- `root_*`：跨线程/跨委托场景下的主链归属
- `executor_*`：这次由谁执行
- `*_reason`：为什么触发这次动作

禁止新增低信息量字段名：

- `source_kind`
- `trigger_kind`
- `session_kind`
- 语义不清的裸 `conversation_id` / `agent_id`

除非当前结构体只描述单一层含义，否则字段名必须显式写出 `origin / target / root / executor` 前缀。

## 3. 第一版结构体

新增一个全局运行上下文结构体，建议命名为：

- `RuntimeContext`

第一版字段只放最容易漂移、最值得统一的 ID 与原因信息：

1. `request_id`
2. `dispatch_id`
3. `origin_conversation_id`
4. `target_conversation_id`
5. `root_conversation_id`
6. `executor_agent_id`
7. `executor_department_id`
8. `model_config_id`
9. `event_source`
10. `dispatch_reason`

说明：

- `request_id`：一次完整请求/动作链路的追踪号
- `dispatch_id`：一次具体投递动作的追踪号
- `origin_conversation_id`：最初发起该动作的会话
- `target_conversation_id`：当前这一步最终要写入/投递的会话
- `root_conversation_id`：委托/子线程等分支场景下归属的主链会话
- `executor_agent_id`：当前真正执行该动作的人格
- `executor_department_id`：当前执行所归属的部门
- `model_config_id`：当前使用的模型配置 ID
- `event_source`：该动作来自哪类入口，例如 `user_message / task_trigger / remote_im / delegate_result`
- `dispatch_reason`：本次为什么发生投递，例如 `user_send / task_due / delegate_done / remote_im_inbound / task_due_fallback_to_main`

## 4. 实现边界

这轮只做“新增结构体 + 在关键链路开始挂载”，不做全仓强制替换。

### 4.1 这轮要做

1. 在 Rust 侧新增 `RuntimeContext`
2. 为该结构体补一个最小构造 helper
3. 先接入最容易出错的几条链路：
   - 聊天发送入口
   - `task` 工具创建任务
   - 任务调度与任务投递
   - 委托入口
4. 关键日志优先改为从 `RuntimeContext` 读取 ID
5. 新代码优先传 `RuntimeContext`

### 4.2 这轮不做

1. 不要求所有函数立刻改签名
2. 不强行替换所有旧的 `conversation_id` / `agent_id` 参数
3. 不改前端数据模型
4. 不要求 remote IM / memory / archive 一次性全部接入

## 5. 渐进接入策略

### 5.1 第一阶段

先让以下链路生成并持有 `RuntimeContext`：

- `send_chat` 入口
- `task` create
- task scheduler tick -> dispatch
- delegate dispatch

这些入口生成 context 后，向下游优先传 context，本轮涉及到的函数能接就接，不能接的保持旧参数。

### 5.2 第二阶段

以后任何涉及以下内容的代码修改，都顺手接入 `RuntimeContext`：

- 会话切换
- 任务提醒
- 联系人入站
- 委托结果回发
- 运行日志

也就是说，后续采用“碰到就接”的迁移方式，不单独发起一次全仓大迁移。

## 6. 验证

### 6.1 Rust

1. 新增单测，确认 `RuntimeContext` 在 task / delegate / send chat 入口构造正确
2. 覆盖“副会话发起任务后仍回副会话”的场景
3. 覆盖“桌面副会话丢失时才回主会话”的场景
4. `cargo check`

### 6.2 人工验证

1. 在副会话创建任务，确认 `origin_conversation_id` 与 `target_conversation_id` 正常
2. 在委托链路里检查 `root_conversation_id` 是否稳定
3. 查看日志，确认不再出现同一动作里多个 ID 互相矛盾

## 7. 风险

最大风险不是代码量，而是“新增了 context，但旧逻辑仍继续随手猜 ID”，导致上下文和现有参数并存时再次分叉。

所以这轮的硬规则是：

- 新增 `RuntimeContext` 后，凡是已经拿到 context 的代码，日志和下游调用优先使用 context
- 不允许在已经有 context 的函数里再次从裸 `session_id` 二次猜会话

## 8. 已完成实现

- Rust 侧已新增统一上下文结构体 `RuntimeContext`
- `SendChatRequest` 与 `ChatPendingEvent` 已支持携带 `RuntimeContext`
- 用户 `send_chat` 入口会生成并下传第一份运行上下文
- 聊天队列触发主助理轮次时会沿用已有上下文，不再只依赖散装 `trace_id`
- `task` 创建入口已开始生成上下文用于日志
- 任务调度投递会生成 `task_trigger` 运行上下文，并将 `requestId / dispatchId` 写入运行日志
- 委托触发与委托结果回发已开始挂载 `RuntimeContext`
- 旧链路暂未强制重构，未接入入口当前显式保持 `runtime_context: None`

## 9. 实现结果

- 运行上下文已经有了统一挂点，后续新链路不需要再重新发明一套 ID 传递方式
- 任务、委托、聊天入口这几条最容易出现 ID 漂移的链路，已经开始共享同一套上下文字段
- 本轮采用渐进接入策略，没有引入一次性全仓重构风险

## 10. 验收结论

用户已完成测试，并确认“测试完美，审查没任何异议”，本计划按当前实现归档。
