# 计划工具与聊天计划模式协议化归档（2026-04-15）

## 1. 背景

本轮目标是把“先计划、确认后执行”的习惯，从普通自然语言提醒升级成正式协议：

- 后端提供专门的 `plan` 工具
- 前端能把计划呈现为独立卡片并提供确认入口
- 计划模式成为聊天链路中的正式状态，而不是一段临时拼接文案

用户在生产环境中已完成验证，并确认当前版本可用，因此按本归档收口。

## 2. 实际落地

### 2.1 工具协议

已新增内置 `plan` 工具，签名保持最小化：

- `action`
- `context`

运行规则与最初计划一致：

- `present`：呈现执行计划，并中止当前轮继续实施
- `complete`：汇报计划完成结果

工具说明里已加入“关键疑问已消除才能 present”的约束。

### 2.2 聊天前端

已完成以下交互：

- `plan.present` 渲染为计划卡片
- 卡片展示计划正文
- 卡片底部展示“我同意，并执行计划”
- `plan.complete` 渲染为完成汇报
- 计划模式在输入区提供独立按钮
- `Shift+Tab` 可切换计划模式

确认执行计划时，还会自动关闭当前会话的计划模式。

### 2.3 计划模式状态

这部分实现最终偏离了最初“前端发送额外提示块”的想法，收口为：

- 前端只发送“切换计划模式”的信号
- 计划模式本质上是后端会话运行时状态
- 后端在组装 prompt 时统一决定是否注入计划模式要求

这样可以保证：

- 后端脱离前端也能运行
- 不需要前端拼装计划提示正文
- 计划模式开关不会再依赖输入框本地临时状态

### 2.4 注入位置

为保证模型稳定感知计划模式，计划提示最终没有注入到普通正文尾部，而是注入到“最新用户消息元信息”区域，位置在说话人/时间文本之前。

这是本轮调试后确认的正确位置。

### 2.5 压缩恢复

压缩链路按简化规则落地：

- 自后向前查找最近计划事件
- 如果先遇到 `plan_complete`，说明没有待执行计划
- 如果先遇到 `plan_present`，则把该计划正文写入压缩摘要

这样在上下文压缩后，仍能恢复最后一个未完成计划。

## 3. 本轮收口时的关键修正

这轮实现中反复重写过多次，最终确认并修正了以下关键问题：

1. `plan.present` 结果必须被外层视为有效回复，不能继续续跑。
2. 计划提示必须注入到最新用户元信息区域，而不是普通附加文本尾部。
3. `Shift+Tab` 优先级必须高于普通 `Tab`，并且只在输入区语境里生效。
4. 计划模式不能走高频写盘；最终收口为运行时会话状态。
5. 工具目录、工具状态、工具运行时三条链路都必须补齐 `plan`。
6. 对缺失工具项，最终统一采用“拿不到时默认开启”的口径，但不自动写回配置。

## 4. 与原计划的偏离

有两点与初版计划不同：

### 4.1 没有引入 `plan_id`

首版仍然没有加 `plan_id`，继续使用“向上查找最近未完成计划”的简化规则。

这是有意控制复杂度的结果，不属于漏做。

### 4.2 计划模式不是前端额外文本块

初版计划里曾经保留“发送时额外附加一段固定提示词”的表述。

最终实现没有把这个责任留给前端，而是改成：

- 前端切换运行时计划状态
- 后端统一决定注入与否

这个偏离是正确收口，不是功能缺失。

## 5. 当前结果

本轮已经实现：

- `plan` 工具协议
- 计划卡片与确认执行
- 计划完成汇报
- 计划模式按钮与 `Shift+Tab`
- 后端统一注入计划提示
- 压缩恢复未完成计划
- 设置页工具目录中展示 `plan`

并且用户已明确反馈“终于好了”，可视为生产验证通过。
