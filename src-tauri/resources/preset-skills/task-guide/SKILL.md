# Task Guide

## 核心目标
- 任务系统不是日历，而是长期推进任务的面板。
- 把你要持续推进的事情写成任务，而不是只在对话里临时提一句。
- 当前追踪任务由系统自动选择，不要试图手动指定。

## 什么时候该创建任务
- 目标会持续多轮对话。
- 需要在未来某个时间继续跟进。
- 需要周期性检查、监控、复盘或提醒。
- 任务完成前，需要持续知道自己为什么在做，以及下一步该做什么。

## 字段怎么写
- `goal`：任务最终要达成什么；它也同时充当任务标题。
- `why`：为什么要做这件事，用来防止后续推进时丢上下文、走偏方向。
- `todo`：当前下一步要做什么；尽量写得短、具体、能执行。

## 触发器怎么写
- 不写 `runAtLocal` 时，任务会立刻进入待执行状态。
- 只写 `runAtLocal` 时，任务会在这个时间点推进一次。
- 同时写 `runAtLocal`、`everyMinutes`、`endAtLocal` 时，任务会从这个时间开始周期推进，并在 `endAtLocal` 后停止。
- `everyMinutes` 必须同时配 `runAtLocal` 和 `endAtLocal`，不要只写间隔不写开始/结束时间。
- 写 `runAtLocal` 时，以界面里已经显示给你的当前本地 RFC3339 时间为准，直接照这个格式填写，不要自己换算时区。
- `runAtLocal` / `endAtLocal` 写到秒即可，并带时区偏移；不要写毫秒。
- 任务系统会自动提醒，不要在创建任务后自己空等。

## 等待规则
- 创建了任务，就把“等待到下次时间点”这件事交给任务系统。
- 不要自己回复“我会等到明天/一小时后再来”。
- 不要为了等下次触发而原地挂机、空转、持续占用当前轮次。
- 如果现在无事可做，就更新好任务状态，然后结束当前回复。
- 下次该推进时，任务系统会再次把任务推回来。

## 禁止行为
- 禁止创建 `at` 或 `every` 任务后，自己在当前对话里说“我先等一下”。
- 禁止用 `wait`、空转思考、重复自言自语来模拟定时任务。
- 禁止把“未来再做”只写在自然语言里，不落到任务系统。
- 禁止用户已经有任务面板时，再发明一套临时提醒机制。

## 正确心智
- 你负责定义任务与维护状态，不负责亲自计时。
- 任务系统负责提醒，LLM 负责在被提醒时继续推进。
- 如果任务需要明天、下一小时、每天下午继续做，就创建或更新任务，而不是承诺自己记住。
- 只有“现在立刻继续做”时，才留在当前轮次继续执行。

## 反模式对比
- 错误：创建任务后回复“好的，我明天再检查一次。”
- 正确：创建任务，说明已记录，等待系统到点提醒。
- 错误：设置了完整的 `runAtLocal + everyMinutes + endAtLocal` 周期任务后，自己调用等待工具或空等。
- 正确：更新 `trigger` 后结束当前轮次。
- 错误：长期监控任务每做一轮就直接完成。
- 正确：更新 `todo`，必要时补充 `why`，继续等待下一次触发。

## 长期任务的正确维护方式
- 不要每推进一轮就直接完成任务。
- 长期任务应该更新 `todo`，必要时补充 `why`，然后继续等待下一次触发。
- 只有真正结束时，才调用完成任务。
- 如果确实无法继续，使用失败完成，并把原因写进 `completionConclusion`。

## 推荐工作流
1. 发现一个需要持续推进的目标。
2. 立刻创建任务，写清 `goal`、`why`、`todo`、`trigger`。
3. 每推进一次，就更新 `todo`。
4. 如果方向或背景发生变化，再补充更新 `why`。
5. 需要调整下一次节奏时，更新 `trigger`。
6. 真正结束时，再完成任务。

## 质量要求
- 不要写空泛 `goal`，例如“继续看看”“处理一下”。
- `why` 要能解释“为什么这件事值得继续推进”，而不只是重复 `goal`。
- `todo` 必须体现当前下一步，不要写成最终总结。
- 任务完成时，`completionConclusion` 要说明为什么完成，成功或失败都一样。

## 典型例子
- 跟踪一个项目一周内的发布动态。
- 每天固定检查一个数据源是否有新变化。
- 分阶段完成一篇报告、调研或代码迁移。
- 对一个长期观察对象做持续记录。
