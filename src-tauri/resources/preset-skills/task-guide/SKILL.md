# Task Guide

## 核心目标
- 任务系统不是日历，而是长期推进任务的面板。
- 把你要持续推进的事情写成任务，而不是只在对话里临时提一句。
- 当前追踪任务由系统自动选择，不要试图手动指定。

## 什么时候该创建任务
- 目标会持续多轮对话。
- 需要在未来某个时间继续跟进。
- 需要周期性检查、监控、复盘或提醒。
- 任务完成前，需要持续积累阶段状态与笔记。

## 字段怎么写
- `title`：一句话说清任务是什么。
- `cause`：为什么要做这件事，任务的起因是什么。
- `goal`：最终想达成什么结果。
- `flow`：大致推进路径，分几步做。
- `todos`：当前具体待做项，尽量短、能执行。
- `statusSummary`：任务当前进行到哪里，不是最终结论。
- `stageKey`：当前阶段的短标签，例如 `collecting`、`watching`、`drafting`。
- `appendNote`：每完成一小步、发现一个阻塞、拿到一个新证据，就追加一条笔记。

## 触发器怎么写
- 不写 `runAt` 时，任务会立刻进入待执行状态。
- 只写 `runAt` 时，任务会在这个时间点推进一次。
- 同时写 `runAt`、`everyMinutes`、`endAt` 时，任务会从这个时间开始周期推进，并在 `endAt` 后停止。
- `everyMinutes` 必须同时配 `runAt` 和 `endAt`，不要只写间隔不写开始/结束时间。
- 写 `runAt` 时，以界面里已经显示给你的当前本地 RFC3339 时间为准，直接照这个格式填写，不要自己换算时区。
- `runAt` / `endAt` 写到秒即可，并带时区偏移；不要写毫秒。
- 任务系统会自动提醒，不要在创建任务后自己空等。

## 等待规则
- 创建了任务，就把“等待到下次时间点”这件事交给任务系统。
- 不要自己回复“我会等到明天/一小时后再来”。
- 不要为了等下次触发而原地挂机、空转、持续占用当前轮次。
- 如果现在无事可做，就更新好任务状态，然后结束当前回复。
- 下次该推进时，任务系统会再次把任务推回来。

## 禁止行为
- 禁止创建 `at` 或 `every` 任务后，自己在当前对话里说“我先等一下”。
- 禁止用 `wait`、空转思考、重复自言自语来模拟定时任务。
- 禁止把“未来再做”只写在自然语言里，不落到任务系统。
- 禁止用户已经有任务面板时，再发明一套临时提醒机制。

## 正确心智
- 你负责定义任务与维护状态，不负责亲自计时。
- 任务系统负责提醒，LLM 负责在被提醒时继续推进。
- 如果任务需要明天、下一小时、每天下午继续做，就创建或更新任务，而不是承诺自己记住。
- 只有“现在立刻继续做”时，才留在当前轮次继续执行。

## 反模式对比
- 错误：创建任务后回复“好的，我明天再检查一次。”
- 正确：创建任务，说明已记录，等待系统到点提醒。
- 错误：设置了 `runAt + everyMinutes` 任务后，自己调用等待工具或空等。
- 正确：更新 `trigger` 后结束当前轮次。
- 错误：长期监控任务每做一轮就直接完成。
- 正确：更新 `statusSummary`、`stageKey`、`appendNote`，继续等待下一次触发。

## 长期任务的正确维护方式
- 不要每推进一轮就直接完成任务。
- 长期任务应该更新 `statusSummary`、`stageKey`、`appendNote`，然后继续等待下一次触发。
- 只有真正结束时，才调用完成任务。
- 如果确实无法继续，使用失败完成，并把原因写进 `completionConclusion`。

## 推荐工作流
1. 发现一个需要持续推进的目标。
2. 立刻创建任务，写清 `title`、`goal`、`trigger`。
3. 每推进一次，就更新 `statusSummary` 和 `appendNote`。
4. 阶段变化时，更新 `stageKey`。
5. 需要调整下一次节奏时，更新 `trigger`。
6. 真正结束时，再完成任务。

## 质量要求
- 不要写空泛标题，例如“继续看看”“处理一下”。
- `statusSummary` 必须体现当前状态，不要写成最终总结。
- `appendNote` 优先记录新变化、结论、阻塞和下一步线索。
- 任务完成时，`completionConclusion` 要说明为什么完成，成功或失败都一样。

## 典型例子
- 跟踪一个项目一周内的发布动态。
- 每天固定检查一个数据源是否有新变化。
- 分阶段完成一篇报告、调研或代码迁移。
- 对一个长期观察对象做持续记录。
